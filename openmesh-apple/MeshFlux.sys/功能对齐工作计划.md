## 技术架构对齐方案

### 核心架构对比

#### 1. **Extension 架构对比**

**MeshFluxMac (应用级)**:
```swift
主应用 → VPNManager → NETunnelProviderManager
            ↓
       ExtensionProfile (加载/管理)
            ↓
    vpn_extension_macos.appex (应用级)
```

**MeshFlux.sys (系统级)**:
```swift
主应用 → SystemExtensionManager → NETunnelProviderManager
                                        ↓
                                  vpn_extension_macx.appex (系统级)
                                  (bundle: com.meshnetprotocol.OpenMesh.macsys.vpn-extension)
```

**关键差异**:
- 系统级 extension 需要通过 `OSSystemExtensionRequest` 安装
- 系统级 extension 运行在独立进程（root 权限）
- 数据共享必须通过 App Group（`group.com.meshnetprotocol.OpenMesh.macsys`）
- Extension bundle identifier 不同

#### 2. **数据共享机制**

**App Group 配置**:
```xml
<!-- MeshFlux.sys -->
<key>com.apple.security.application-groups</key>
<array>
    <string>group.com.meshnetprotocol.OpenMesh.macsys</string>
</array>

<!-- MeshFluxMac -->
<key>com.apple.security.application-groups</key>
<array>
    <string>group.com.meshnetprotocol.OpenMesh</string>
</array>
```

**共享目录结构**:
```
App Group Root (group.com.meshnetprotocol.OpenMesh.macsys)/
├── MeshFlux/
│   ├── singbox_config.json        # sing-box 配置
│   ├── routing_rules.json         # 路由规则（如使用）
│   ├── command.sock               # UNIX socket（通信）
│   └── ...                        # 其他共享文件
├── Library/
│   └── Caches/
│       ├── Working/               # 工作目录
│       └── stderr.log             # Go 错误日志
└── ...
```

#### 3. **Go 代码库集成**

**依赖确认**:
```bash
# MeshFlux.sys 需要确保使用 go-cli-lib
cd go-cli-lib
make apple  # 构建 XCFramework
```

**框架位置**:
```
openmesh-apple/lib/OpenMeshGo.xcframework  # 由 go-cli-lib 构建
```

**Xcode 项目配置**:
- 确保 MeshFlux.sys 和 vpn_extension_macx 都链接了 `OpenMeshGo.framework`
- Framework 路径应指向 `../lib/` 目录

## 关键技术挑战与解决方案

### Challenge 1: 系统级 vs 应用级 Extension 差异

**问题**: MeshFluxMac 使用应用级 extension，而 MeshFlux.sys 使用系统级 extension

**解决方案**:
```swift
// 在 ExtensionProfile.load() 中指定正确的 bundle identifier
let targetBundleID = "com.meshnetprotocol.OpenMesh.macsys.vpn-extension"

// 或者通过 localizedDescription 匹配
let managers = try await NETunnelProviderManager.loadAllFromPreferences()
let manager = managers.first { 
    $0.localizedDescription == "MeshFlux X" 
}
```

### Challenge 2: App Group 数据共享

**问题**: 系统级 extension 无法直接访问主应用的沙盒目录

**解决方案**:
```swift
// 1. 统一使用 App Group 作为共享根目录
static let sharedDirectory: URL = {
    guard let url = FileManager.default.containerURL(
        forSecurityApplicationGroupIdentifier: "group.com.meshnetprotocol.OpenMesh.macsys"
    ) else {
        fatalError("Missing App Group container")
    }
    return url
}()

// 2. 确保权限正确（系统级 extension 以 root 运行）
try? FileManager.default.setAttributes(
    [.posixPermissions: 0o777],
    ofItemAtPath: sharedDirectory.path
)
```

### Challenge 3: 不影响其他 Target

**问题**: 如何确保修改不破坏 MeshFluxMac 等其他 Target

**解决方案**:
```bash
# 1. 在修改前备份 project.pbxproj
cp ../MeshFlux.xcodeproj/project.pbxproj ../MeshFlux.xcodeproj/project.pbxproj.backup

# 2. 使用 git 追踪变更
git add ../MeshFlux.sys/
git commit -m "WIP: MeshFlux.sys refactor"

# 3. 每次修改后编译其他 Target
xcodebuild -scheme MeshFluxMac clean build
xcodebuild -scheme vpn_extension_macos clean build

# 4. 如果发现影响了其他 Target，立即回滚
git checkout ../MeshFlux.xcodeproj/project.pbxproj
```

## 风险与缓解措施

### Risk 1: 系统级 extension 兼容性问题

**风险**: macOS 版本差异导致 extension 行为不一致

**缓解措施**:
- 在多个 macOS 版本上测试（13.x, 14.x, 15.x）
- 添加版本检测和降级方案
- 记录详细的错误日志

### Risk 2: App Group 权限问题

**风险**: 沙盒限制导致无法访问共享目录

**缓解措施**:
- 启动时自动修复权限（chmod 777）
- 在 Info.plist 中添加 `NSSystemExtensionUsageDescription`
- 测试时检查 entitlements 配置

### Risk 3: Go 代码库版本不匹配

**风险**: OpenMeshGo.framework 版本过旧或缺少功能

**缓解措施**:
- 定期从 go-cli-lib 重新构建 framework
- 在 CI/CD 中自动化构建流程
- 添加版本检查机制

### Risk 4: 影响其他 Target

**风险**: 修改意外破坏 MeshFluxMac 等其他 Target

**缓解措施**:
```bash
# 每次提交前执行
./scripts/test_all_targets.sh

# 如果发现破坏，立即回滚
git reset --hard HEAD~1
```
