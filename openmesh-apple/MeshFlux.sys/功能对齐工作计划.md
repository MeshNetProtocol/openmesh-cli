# MeshFlux.sys 功能对齐工作计划

## 项目概述

**目标**: 将 MeshFlux.sys（系统级 extension）改造成与 MeshFluxMac（应用级 NetworkExtension）具有相同 UI 和能力的应用，同时保持系统级 extension 的架构优势。

**参考项目**:
1. **MeshFlux.sys** - 当前系统级 extension 实现（需要改造）
2. **MeshFluxMac** - 应用级 NetworkExtension 实现（UI/能力对标）
3. **SFM.System** (sing-box) - 系统级 extension 参考实现
4. **go-cli-lib** - Go 代码库依赖

## 技术架构对齐方案

### 核心架构调整

#### 1. **Extension 架构对比**

**MeshFluxMac (应用级)**:
```swift
主应用 → VPNManager → NETunnelProviderManager
            ↓
       ExtensionProfile (加载/管理)
            ↓
    vpn_extension_macos.appex (应用级)
```

**MeshFlux.sys (系统级)**:
```swift
主应用 → SystemExtensionManager → NETunnelProviderManager
                                        ↓
                                  vpn_extension_macx.appex (系统级)
                                  (bundle: com.meshnetprotocol.OpenMesh.macsys.vpn-extension)
```

**关键差异**:
- 系统级 extension 需要通过 `OSSystemExtensionRequest` 安装
- 系统级 extension 运行在独立进程（root 权限）
- 数据共享必须通过 App Group（`group.com.meshnetprotocol.OpenMesh.macsys`）
- Extension bundle identifier 不同

#### 2. **数据共享机制**

**App Group 配置**:
```xml
<!-- MeshFlux.sys -->
<key>com.apple.security.application-groups</key>
<array>
    <string>group.com.meshnetprotocol.OpenMesh.macsys</string>
</array>

<!-- MeshFluxMac -->
<key>com.apple.security.application-groups</key>
<array>
    <string>group.com.meshnetprotocol.OpenMesh</string>
</array>
```

**共享目录结构**:
```
App Group Root (group.com.meshnetprotocol.OpenMesh.macsys)/
├── MeshFlux/
│   ├── singbox_config.json        # sing-box 配置
│   ├── routing_rules.json         # 路由规则（如使用）
│   ├── command.sock               # UNIX socket（通信）
│   └── ...                        # 其他共享文件
├── Library/
│   └── Caches/
│       ├── Working/               # 工作目录
│       └── stderr.log             # Go 错误日志
└── ...
```

#### 3. **Go 代码库集成**

**依赖确认**:
```bash
# MeshFlux.sys 需要确保使用 go-cli-lib
cd go-cli-lib
make apple  # 构建 XCFramework
```

**框架位置**:
```
openmesh-apple/lib/OpenMeshGo.xcframework  # 由 go-cli-lib 构建
```

**Xcode 项目配置**:
- 确保 MeshFlux.sys 和 vpn_extension_macx 都链接了 `OpenMeshGo.framework`
- Framework 路径应指向 `../lib/` 目录

## 关键技术挑战与解决方案

### Challenge 1: 系统级 vs 应用级 Extension 差异

**问题**: MeshFluxMac 使用应用级 extension，而 MeshFlux.sys 使用系统级 extension，两者在以下方面存在差异：

1. **安装方式**:
   - 应用级：直接保存至 Preferences
   - 系统级：需要 `OSSystemExtensionRequest` + 用户授权

2. **运行权限**:
   - 应用级：受限于 App Sandbox
   - 系统级：root 权限，但需要通过 App Group 共享数据

3. **Bundle Identifier**:
   - 应用级：`com.meshnetprotocol.OpenMesh.vpn-extension`
   - 系统级：`com.meshnetprotocol.OpenMesh.macsys.vpn-extension`

**解决方案**:
```swift
// 在 ExtensionProfile.load() 中指定正确的 bundle identifier
let targetBundleID = "com.meshnetprotocol.OpenMesh.macsys.vpn-extension"

// 或者通过 localizedDescription 匹配
let managers = try await NETunnelProviderManager.loadAllFromPreferences()
let manager = managers.first { 
    $0.localizedDescription == "MeshFlux X" 
}
```

### Challenge 2: App Group 数据共享

**问题**: 系统级 extension 无法直接访问主应用的沙盒目录

**解决方案**:
```swift
// 1. 统一使用 App Group 作为共享根目录
static let sharedDirectory: URL = {
    guard let url = FileManager.default.containerURL(
        forSecurityApplicationGroupIdentifier: "group.com.meshnetprotocol.OpenMesh.macsys"
    ) else {
        fatalError("Missing App Group container")
    }
    return url
}()

// 2. 所有配置文件都放在共享目录
let configURL = sharedDirectory
    .appendingPathComponent("MeshFlux")
    .appendingPathComponent("singbox_config.json")

// 3. 确保权限正确（系统级 extension 以 root 运行）
try? FileManager.default.setAttributes(
    [.posixPermissions: 0o777],
    ofItemAtPath: sharedDirectory.path
)
```

### Challenge 3: Go 代码库集成

**问题**: 确保 MeshFlux.sys 使用正确的 go-cli-lib 版本

**解决方案**:
```bash
# 1. 从 go-cli-lib 构建 XCFramework
cd /Users/hyperorchid/MeshNetProtocol/openmesh-cli/go-cli-lib
make apple

# 2. 验证框架位置
ls -la ../openmesh-apple/lib/OpenMeshGo.xcframework

# 3. 在 Xcode 项目中链接框架
# - 添加 Framework Search Paths
# - Link Binary With Libraries: OpenMeshGo.framework
```

**Xcode 项目配置** (project.pbxproj):
```
// 确保 MeshFlux.sys 和 vpn_extension_macx 都引用同一个 OpenMeshGo.framework
// 路径应为相对路径：$(SRCROOT)/../lib/
```

### Challenge 4: Extension 生命周期管理

**问题**: 系统级 extension 的生命周期与主应用不同步

**解决方案**:
```swift
// 1. 主应用通过 heartbeat 文件与 extension 通信
let heartbeatURL = sharedDirectory
    .appendingPathComponent("heartbeat")

// 2. Extension 定期检查 heartbeat
DispatchSourceTimer.makeTimerSource(queue: heartbeatQueue)
    .schedule(deadline: .now(), repeating: 10.0)
    .setEventHandler { [weak self] in
        self?.checkHeartbeat()
    }

// 3. 若连续多次未检测到 heartbeat，认为主应用已退出
if heartbeatMissCount >= 3 {
    stopVPN()  // 主动关闭 VPN
}
```

## 风险与缓解措施

### Risk 1: 系统级 extension 兼容性问题

**风险**: macOS 版本差异导致 extension 行为不一致

**缓解措施**:
- 在多个 macOS 版本上测试（13.x, 14.x, 15.x）
- 添加版本检测和降级方案
- 记录详细的错误日志

### Risk 2: App Group 权限问题

**风险**: 沙盒限制导致无法访问共享目录

**缓解措施**:
- 启动时自动修复权限（chmod 777）
- 在 Info.plist 中添加 `NSSystemExtensionUsageDescription`
- 测试时检查 entitlements 配置

### Risk 3: Go 代码库版本不匹配

**风险**: OpenMeshGo.framework 版本过旧或缺少功能

**缓解措施**:
- 定期从 go-cli-lib 重新构建 framework
- 在 CI/CD 中自动化构建流程
- 添加版本检查机制

### Risk 4: Extension 被系统终止

**风险**: 系统资源紧张时 extension 可能被杀死

**缓解措施**:
- 实现自动重连机制
- 监控 extension 健康状态
- 添加心跳检测和恢复逻辑
