# OpenMesh 平台冷启动与社区接力执行总方案（V2 边界重定义）

## 0. 文档目的

本文档用于统一产品、UE、客户端、链上与社区运营策略，解决以下核心问题：

1. 在“商用提供者尚未形成规模”时，新用户下载 App 后如何快速形成可用闭环。  
2. 当官方停止提供临时受限访问后，如何保证新用户仍能稳定拿到可用的私人全量配置。  
3. 商用部分如何在不依赖 Cloudflare 中间层的前提下，直接基于区块链信息完成发现与使用。  

本文档是执行方案，不包含具体代码实现细节。

---

## 1. 新边界（与旧方案相比的明确变化）

## 1.1 商用提供者边界（强约束）

1. 商用信息源改为“App 直接读取链上数据”，不再以 Cloudflare 作为商用发现核心路径。  
2. 商用查询、过滤、排序等交互在本地完成。  
3. 智能合约存储商用供应商核心索引信息；配置详情通过链上字段 `url/metadataURI` 指向实际内容。  
4. 文档中关于商用部分只描述：  
   - App 内保留哪些界面位置和元素  
   - 这些界面具备哪些核心功能  
   不描述 Cloudflare 承载商用查询的方案。  

## 1.2 冷启动核心目标重定义

目标从“平台自身长期提供可达能力”调整为“官方临时引导 + 社区接力长期可用”：

1. 早期：官方提供受限、临时的 VPN 引导能力（仅用于帮助用户到达社区和配置入口）。  
2. 过渡期：官方可作为“志愿者”提供私人全量配置（免费），用于启动用户规模。  
3. 成熟期：官方退出临时引导后，由社区（Twitter/Reddit/GitHub/QQ群/微信群/线下）维护私人配置供给。  
4. 全程保持对外口径：App 提供的是临时受限服务与配置发现能力，不承诺官方长期全流量代理。  

## 1.3 UE 总目标重定义

1. 解决冷启动可用性：新用户低学习门槛，先连通再选择。  
2. 兼顾老用户效率：老用户进入 App 后可快速直达目标服务类型（私人/商用）并快速切换。  
3. 支持进阶用户：具备快速自建私人节点并分享给朋友的路径。

## 2. 当前系统基线（结合现有仓库）

## 2.1 openmesh-apple 现状（客户端）

1. 已具备 Provider 安装闭环：下载配置、写入 Profile、hash 记录、切换生效。  
2. 已具备 Market/推荐/离线导入能力。  
3. 当前体验问题：首连路径仍需用户理解“市场/导入”，不是“一键可用”。  
4. 当前商用路径仍偏向中心化 API 组织方式，未完全切换到链上直读模型。  

## 2.2 MeshNetProtocol.github.io 现状（辅助平台）

你已具备私有与商用工具网站基础能力：

1. 私有工作台（无需钱包）用于配置生成与导入指导。  
2. 商用入口（MetaMask + Base 网络）用于链上动作。  
3. 合约已存在：`ProtocolRegistry.sol`、`SupplierProfile.sol`。  
4. 网站已具备 seed/config 模板与工具化入口，可作为社区供给中枢。  

## 2.3 market-api 现状（兼容资产）

`market-api` 已有只读市场能力，但在 V2 边界下：

1. 不再作为商用发现主路径。  
2. 可在迁移期作为兼容 fallback。  
3. 长期目标是商用查询从 App 直连链上与 URL 内容完成。

## 3. 产品总架构（V2）

## 3.1 三层来源模型（Source of Truth）

1. Bootstrap 临时引导源（官方临时）  
2. Community 私人配置源（社区长期）  
3. Onchain 商用索引源（链上长期）  

三层并行存在，但职责不同，不互相替代。

## 3.2 分层职责

### A. Bootstrap 临时引导层（官方）

职责：
1. 让新用户在无配置时先具备最低可达能力。  
2. 帮用户到达社区入口和配置分发点（Twitter/Reddit/GitHub/网站）。  

限制：
1. 受限、临时、可下线。  
2. 不承诺长期全量可用。  

### B. Community 私人供给层（长期主力）

职责：
1. 提供可持续的私人全量配置供给。  
2. 支持社区维护、轮换、镜像、备份。  

形态：
1. 公共社区帖、仓库、Wiki、公告板。  
2. 私密群组内的分享链接/二维码。  

### C. Onchain 商用层（商业化）

职责：
1. 提供可验证、可计费、可结算的商用服务索引。  
2. 通过合约字段指向配置详情 URL。  

原则：
1. App 直接读取链上索引。  
2. 本地完成筛选、排序、过滤与缓存。

## 4. 目标用户路径（新老用户双模式）

## 4.1 新用户（低门槛）

路径：
1. 打开 App -> 点击“快速连通（临时受限）”。  
2. 获得临时受限访问能力。  
3. 一键打开“社区配置入口”（网站/社媒）。  
4. 导入私人全量配置。  
5. 切换到私人全量配置后，退出临时模式。  

成功标准：
1. 用户无需理解区块链即可完成首次可用。  
2. 用户在 1-3 分钟内拿到可用全量配置。  

## 4.2 老用户（高效率）

路径：
1. 打开 App 默认进入“快速切换台（最近使用）”。  
2. 一步切换私人/商用/收藏服务。  
3. 不强制显示新手引导流程。  

成功标准：
1. 老用户 10 秒内完成服务切换。  
2. 无冗余新手弹窗打断。  

## 4.3 进阶用户（自建分享）

路径：
1. 进入“私人节点工作台”。  
2. 快速生成/更新配置。  
3. 分享给朋友（URL/二维码/文件）。  

成功标准：
1. 自建和分享路径闭环独立可用。  
2. 不依赖商用链上流程。

## 5. 商用模块界面与核心功能（按你的要求）

本章节只描述“App 中保留什么界面元素、具备什么功能”。

## 5.1 保留的界面位置

1. 主导航保留 `Market` 或 `Commercial` 入口。  
2. 在服务切换台中保留“商用服务”分组。  
3. 在供应商详情页保留“链上信息区块”。  

## 5.2 商用列表页元素

1. 搜索框（名称、标签、供应商 ID）。  
2. 过滤器（价格区间、可用状态、更新时间、链网络）。  
3. 排序器（价格、稳定性、最新、评分）。  
4. 列表卡片（名称、标签、价格、状态、更新时间、链上标识）。  

## 5.3 商用详情页元素

1. 供应商基础信息。  
2. 链上状态（active/suspended/expiry）。  
3. 配置详情 URL 与内容校验状态。  
4. 操作按钮：安装、购买、续费、收藏。  

## 5.4 商用模块核心功能

1. 直接从链上读取供应商索引数据。  
2. 通过链上 `metadataURI/url` 拉取配置详情。  
3. 在本地完成筛选/排序/搜索/分页缓存。  
4. 校验配置合法性后安装为本地 provider。  
5. 在需要支付或签名时触发钱包流程。

## 6. 信息源与数据流设计

## 6.1 信息源优先级

按使用场景拆分，不做单一来源强依赖：

1. 首次可达：Bootstrap 临时引导源  
2. 日常全量：Community 私人配置源  
3. 商业化选择：Onchain 商用索引源  

## 6.2 商用数据流（链上直连）

1. App 读取合约索引（Registry + Profile 状态）。  
2. 得到供应商列表和 `metadataURI/url`。  
3. App 拉取 URL 指向的配置内容。  
4. 本地校验后写入本地 provider。  
5. 本地索引更新用于快速筛选与离线回显。  

## 6.3 私人社区数据流

1. App 打开社区入口（官网、GitHub、社媒公告）。  
2. 用户获取社区维护的私人配置链接/文件。  
3. App 导入并安装。  
4. App 记录来源标签（官方临时/社区/朋友分享）供后续治理。  

---

## 7. Bootstrap 临时引导与官方退出机制

## 7.1 Bootstrap 期策略

1. 官方提供受限临时配置，仅保证到达社区和配置入口。  
2. 官方可提供志愿者私人全量配置（免费）作为早期启动工具。  
3. App 明确提示该能力为“临时引导”，并鼓励尽快迁移到社区/自建配置。  

## 7.2 官方退出前置条件

满足以下条件再逐步退出官方临时引导：

1. 社区已有稳定维护渠道（至少 2-3 个独立入口）。  
2. 社区配置更新频率和可用率达到门槛。  
3. 新用户在无官方引导时仍能完成首连闭环。  

## 7.3 官方退出执行步骤

1. 发布退出公告和迁移指南。  
2. App 内引导入口切换为社区入口优先。  
3. 受限临时配置逐步降权并下线。  
4. 监控新用户可用率，必要时短期恢复应急引导。

## 8. UE 信息架构（兼顾新老用户）

## 8.1 首页结构（建议）

首屏仅两个主动作：

1. `快速连通（临时受限）`  
2. `我有配置 / 导入配置`  

次级动作：
1. `一键自建私人节点`  
2. `浏览商用服务`  

## 8.2 老用户模式

1. 记住最近使用服务类型。  
2. 默认展示“最近使用 + 收藏 + 可用状态”。  
3. 隐藏新手引导卡片，可手动再打开。  

## 8.3 新手模式

1. 分步提示只保留 3 步：连通 -> 找配置 -> 导入。  
2. 所有文案避免区块链术语过早出现。  
3. 每步都给“失败时下一步”。

## 10. 分阶段执行计划（V2）

## Phase A：冷启动闭环（当前最高优先）

目标：
1. 新用户无需钱包快速可用。  
2. 能顺利迁移到社区私人全量配置。  

交付：
1. 首屏双主动作 + 三步引导。  
2. 官方临时受限配置接入。  
3. 社区入口聚合页（官网/社媒/GitHub）。  

验收：
1. 新用户 3 分钟内完成“下载 -> 可用”。  
2. 新用户导入社区配置成功率达到目标门槛。  

## Phase B：商用链上直连

目标：
1. 商用列表完全由链上索引驱动。  
2. 本地筛选、排序、缓存可用。  

交付：
1. 商用列表页与详情页元素完整。  
2. `metadataURI/url` 配置拉取与校验闭环。  
3. 钱包流程仅在支付/签名动作出现。  

验收：
1. 在无 Cloudflare 商用接口情况下可完成商用发现与安装。  

## Phase C：新老用户体验分层优化

目标：
1. 老用户极速切换。  
2. 新手低学习成本。  

交付：
1. 老用户快速切换台。  
2. 新手引导可折叠与可跳过。  
3. 收藏与最近使用策略。  

## Phase D：官方退出与社区自治

目标：
1. 官方停止临时引导后，新用户可用性不显著下降。  

交付：
1. 退出公告机制与应急恢复机制。  
2. 社区维护手册与镜像策略。  
3. 长期监控看板。

## 11. 运营与合规口径

对外统一描述：

1. OpenMesh App 提供临时受限连接与配置发现能力。  
2. 社区和服务提供者负责长期全量服务供给。  
3. 平台不承诺官方长期自营全流量代理。  

---

## 12. 核心风险与对策

## 12.1 社区源不稳定

风险：
1. 社区链接失效、更新不及时。  

对策：
1. 多入口冗余（官网 + GitHub + 社媒 + 群组）。  
2. 配置来源评级和健康检查提示。  

## 12.2 链上读取成本和性能

风险：
1. 直接链上读取可能慢或受 RPC 不稳定影响。  

对策：
1. 多 RPC 备选。  
2. 本地索引缓存与增量刷新。  

## 12.3 配置 URL 安全风险

风险：
1. metadata URL 内容被篡改或失效。  

对策：
1. 强制完整性校验与来源提示。  
2. 安装前风险告警与失败回退。  

## 12.4 App 审核风险

风险：
1. 若口径不清易被理解为官方长期 VPN 服务。  

对策：
1. 明确“临时受限引导”定位。  
2. 产品文案中避免“官方长期代理”表述。  

---

## 13. 成功标准（V2）

1. 冷启动：新用户首次可用成功率显著提升。  
2. 迁移：用户可从官方临时引导平滑迁移到社区私人配置。  
3. 去中心：商用发现不依赖 Cloudflare 主链路。  
4. 效率：老用户能快速找到并切换目标服务。  
5. 可持续：官方退出临时引导后，新用户可用性保持稳定。  

---

## 14. 决策摘要（最终）

1. 商用：链上直连 + 本地筛选，Cloudflare 非主路径。  
2. 冷启动：官方临时受限访问仅做引导，不做长期承载。  
3. 长期供给：社区私人配置接力，官方逐步退出。  
4. UE：新手低门槛，老用户高效率，进阶用户可自建可分享。  
5. 工具协同：`MeshNetProtocol.github.io` 作为私有/商用操作与社区入口中枢。
