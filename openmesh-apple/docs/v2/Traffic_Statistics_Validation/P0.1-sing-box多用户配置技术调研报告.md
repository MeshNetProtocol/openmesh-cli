# P0.1 - sing-box 多用户配置技术调研报告

> 任务编号：P0.1
> 创建时间：2026-03-22
> 状态：✅ 已完成

## 1. 调研目标

验证 sing-box 是否支持多用户配置，并能够按用户标识统计流量，为 OpenMesh V2 商用供应商架构提供技术可行性依据。

---

## 2. 核心发现

### 2.1 多用户支持

✅ **sing-box 原生支持多用户配置**

sing-box 通过以下协议支持多用户：
- **Shadowsocks** (推荐：AEAD 2022 ciphers)
- **Trojan**
- **VLESS**

### 2.2 流量统计能力

✅ **sing-box 提供多种流量统计方案**

1. **SSM API** (Shadowsocks Server Management API)
   - RESTful API 服务
   - 专门用于管理 Shadowsocks 多用户服务器
   - 支持用户管理和流量状态持久化

2. **V2Ray API**
   - gRPC 基础的统计服务
   - 支持按 inbound/outbound/user 统计流量
   - 提供细粒度的流量监控

3. **Clash API**
   - HTTP/WebSocket 基础的控制接口
   - 提供运行时监控和管理
   - 支持连接跟踪和流量统计

## 3. Shadowsocks 多用户配置详解

### 3.1 服务端配置示例

```json
{
  "inbounds": [
    {
      "type": "shadowsocks",
      "listen": "::",
      "listen_port": 8080,
      "network": "tcp",
      "method": "2022-blake3-aes-128-gcm",
      "password": "<server_password>",
      "users": [
        {
          "name": "user1",
          "password": "<user1_password>"
        },
        {
          "name": "user2",
          "password": "<user2_password>"
        }
      ],
      "managed": true,
      "multiplex": {
        "enabled": true
      }
    }
  ]
}
```

**关键字段说明：**
- `method`: 推荐使用 `2022-blake3-aes-128-gcm` (AEAD 2022 加密)
- `password`: 服务器主密码
- `users`: 用户列表，每个用户有独立的 `name` 和 `password`
- `managed`: **必须设置为 `true`** 才能被 SSM API 管理
- `multiplex`: 启用多路复用，提高性能和抗检测能力

### 3.2 客户端配置示例

```json
{
  "outbounds": [
    {
      "type": "shadowsocks",
      "server": "example.com",
      "server_port": 8080,
      "method": "2022-blake3-aes-128-gcm",
      "password": "<server_password>:<user_password>",
      "multiplex": {
        "enabled": true
      }
    }
  ]
}
```

**关键点：**
- 客户端密码格式：`<server_password>:<user_password>`
- 通过不同的 `user_password` 区分不同用户

## 4. SSM API 详解

### 4.1 SSM API 配置

```json
{
  "type": "ssm-api",
  "listen": "127.0.0.1",
  "listen_port": 9000,
  "servers": {
    "/": "ss-in"
  },
  "cache_path": "/var/lib/sing-box/ssm-cache.json",
  "tls": {}
}
```

**关键功能：**
- `servers`: 映射 HTTP 端点到 Shadowsocks inbound tag
- `cache_path`: 流量和用户状态持久化路径（服务器停止时自动保存，启动时恢复）

### 4.2 SSM API 规范

SSM API 遵循 Shadowsocks 官方规范：
- 规范文档：[shadowsocks-specs/2023-1-shadowsocks-server-management-api-v1.md](https://github.com/Shadowsocks-NET/shadowsocks-specs/blob/main/2023-1-shadowsocks-server-management-api-v1.md)
- 提供标准的 RESTful API 用于：
  - 用户管理（添加/删除/更新用户）
  - 流量统计查询
  - 服务器状态监控

## 5. 流量统计实现机制

### 5.1 代码层面的流量跟踪

根据 sing-box 源码分析：

**文件：`protocol/shadowsocks/inbound_multi.go`**

```go
type MultiInbound struct {
    // ...
    service  shadowsocks.MultiService[int]
    users    []option.ShadowsocksUser
    tracker  adapter.SSMTracker  // 流量跟踪器
}

func (h *MultiInbound) SetTracker(tracker adapter.SSMTracker) {
    h.tracker = tracker
}
```

**文件：`adapter/ssm.go`**

```go
type SSMTracker interface {
    TrackConnection(conn net.Conn, metadata InboundContext) net.Conn
    TrackPacketConnection(conn N.PacketConn, metadata InboundContext) N.PacketConn
}
```

**核心机制：**
1. `MultiInbound` 支持多用户服务
2. `SSMTracker` 接口负责跟踪每个连接的流量
3. 通过 `InboundContext` 中的用户标识区分不同用户
4. 每个连接和数据包连接都会被包装以统计流量

### 5.2 V2Ray API 统计

**文件：`experimental/v2rayapi/stats.go`**

```go
// 按 inbound 统计
readCounter = append(readCounter, s.loadOrCreateCounter("inbound>>>"+inbound+">>>traffic>>>uplink"))
writeCounter = append(writeCounter, s.loadOrCreateCounter("inbound>>>"+inbound+">>>traffic>>>downlink"))

// 按 user 统计
readCounter = append(readCounter, s.loadOrCreateCounter("user>>>"+user+">>>traffic>>>uplink"))
writeCounter = append(writeCounter, s.loadOrCreateCounter("user>>>"+user+">>>traffic>>>downlink"))
```

**统计维度：**
- `inbound>>>tag>>>traffic>>>uplink`: 入站上传流量
- `inbound>>>tag>>>traffic>>>downlink`: 入站下载流量
- `user>>>name>>>traffic>>>uplink`: 用户上传流量
- `user>>>name>>>traffic>>>downlink`: 用户下载流量

## 6. 推荐方案

### 6.1 数据平面协议选择

**推荐：Shadowsocks 2022 + SSM API**

**理由：**
1. ✅ 原生多用户支持
2. ✅ 标准化的管理 API (SSM API)
3. ✅ 流量统计和状态持久化
4. ✅ 密码学安全（AEAD 2022）
5. ✅ 抗被动检测能力
6. ✅ 与当前客户端路径最接近

**备选方案：**
- **Trojan**: 需要 TLS，适合伪装成 HTTPS 流量
- **VLESS**: V2Ray 生态，功能丰富但配置复杂
- **Hysteria2**: 基于 QUIC，高性能但需要 UDP 支持

### 6.2 流量统计方案选择

**推荐：SSM API + V2Ray API 组合**

| 方案 | 用途 | 优势 |
|------|------|------|
| **SSM API** | 用户管理、基础流量统计 | 简单、RESTful、状态持久化 |
| **V2Ray API** | 细粒度流量监控 | 实时、多维度、gRPC 高性能 |

**实施建议：**
- Phase 0/1: 使用 SSM API 验证基础流量统计
- Phase 2: 引入 V2Ray API 实现实时监控和细粒度统计

## 7. 技术可行性结论

### 7.1 核心问题验证

| 需求 | sing-box 支持情况 | 验证结果 |
|------|------------------|---------|
| 多用户配置 | ✅ 原生支持 | **可行** |
| 按用户标识统计流量 | ✅ 支持（通过 SSM/V2Ray API） | **可行** |
| 动态用户管理 | ✅ 支持（SSM API） | **可行** |
| 流量桶续费触发 | ✅ 可通过 API 查询实现 | **可行** |
| 状态持久化 | ✅ 支持（cache_path） | **可行** |

### 7.2 风险评估

**低风险：**
- ✅ sing-box 是成熟项目，多用户功能稳定
- ✅ SSM API 遵循官方规范，有标准实现
- ✅ 社区活跃，文档完善

**需要验证的点：**
- ⚠️ 流量统计的实时性和准确性（需要 Phase 0 实测）
- ⚠️ 高并发场景下的性能表现（需要 Phase 1 压测）
- ⚠️ 状态持久化的可靠性（需要测试异常重启场景）
