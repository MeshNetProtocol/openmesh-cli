> 任务编号：P0.4
> 状态：📝 规划中

## 1. 任务目标

实现流量采集服务，将 sing-box 节点的流量统计定期上报到记账服务：
- 读取 sing-box 的流量统计数据
- 定期上报到记账服务
- 查询流量桶状态
- 根据流量桶状态决定是否阻断连接

---

## 2. 技术调研

### 2.1 sing-box 流量统计 API

根据 [sing-box 官方文档](https://sing-box.sagernet.org/configuration/experimental/v2ray-api/)，sing-box 提供两种 API：

**选项 A：V2Ray API (gRPC)**
- 协议：gRPC
- 功能：按用户、按入站、按出站统计流量
- 优势：精确的用户级流量统计
- 劣势：需要 gRPC 客户端，配置复杂

**选项 B：Clash API (RESTful HTTP)**
- 协议：HTTP REST
- 功能：连接跟踪、流量统计、代理管理
- 优势：简单易用，HTTP 接口
- 劣势：可能不支持按用户统计

**选项 C：独立采集服务（推荐用于测试）**
- 不依赖 sing-box API
- 通过其他方式获取流量数据（如日志解析、系统监控）
- 优势：实现简单，不需要修改 sing-box 配置
- 劣势：精度可能不如原生 API

### 2.2 技术选型建议

**Phase 0 测试阶段**：选项 C - 独立采集服务
- 原因：快速验证流量统计方案，无需深入 sing-box API
- 实现：模拟流量数据或使用简单的流量监控

**生产环境**：选项 A - V2Ray API
- 原因：精确的用户级流量统计
- 需要：启用 sing-box 的 V2Ray API 配置

## 3. 架构设计

### 3.1 系统架构

```
┌─────────────────────────────────────────────────────┐
│                  sing-box 节点                       │
│  ┌──────────────────────────────────────────────┐   │
│  │  Shadowsocks Inbound (多用户)                 │   │
│  │  - alice: 流量统计                            │   │
│  │  - bob: 流量统计                              │   │
│  │  - charlie: 流量统计                          │   │
│  └──────────────────────────────────────────────┘   │
│                      ↓                               │
│  ┌──────────────────────────────────────────────┐   │
│  │  V2Ray API / Clash API (可选)                 │   │
│  └──────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────────────┐
│            流量采集服务 (Traffic Collector)          │
│  ┌──────────────────────────────────────────────┐   │
│  │  1. 定期查询 sing-box 流量统计                 │   │
│  │  2. 计算流量增量（与上次查询的差值）            │   │
│  │  3. 上报到记账服务                            │   │
│  │  4. 查询流量桶状态                            │   │
│  │  5. 如果流量桶耗尽，通知 sing-box 阻断         │   │
│  └──────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────────────┐
│              记账服务 (Metering Service)             │
│  - 接收流量上报                                      │
│  - 扣减流量桶                                        │
│  - 返回流量桶状态                                    │
└─────────────────────────────────────────────────────┘
```

### 3.2 流量采集流程

```
1. 启动流量采集服务
   ↓
2. 每 10 秒执行一次采集循环
   ↓
3. 查询 sing-box 当前流量统计
   ↓
4. 计算流量增量 = 当前值 - 上次值
   ↓
5. 如果增量 > 0，上报到记账服务
   ↓
6. 记账服务返回流量桶状态
   ↓
7. 如果流量桶耗尽 (action: "block")
   ├─ 记录日志
   └─ (可选) 通知 sing-box 阻断用户连接
   ↓
8. 更新上次统计值
   ↓
9. 等待下一个周期
```

## 4. 实现方案

### 4.1 Phase 0 简化方案（推荐）

**目标**：快速验证流量统计和上报机制

**实现**：
1. 不启用 sing-box API
2. 流量采集服务模拟流量数据
3. 或者通过系统工具（如 `netstat`）监控连接流量
4. 定期上报到记账服务

**优势**：
- 实现简单，快速验证
- 不需要修改 sing-box 配置
- 专注于测试记账服务的流量管理逻辑

**劣势**：
- 流量数据不精确
- 无法真正按用户统计

### 4.2 生产方案（V2Ray API）

**配置 sing-box 启用 V2Ray API**：

```json
{
  "experimental": {
    "v2ray_api": {
      "listen": "127.0.0.1:8080",
      "stats": {
        "enabled": true,
        "inbounds": ["ss-in"],
        "outbounds": ["direct"],
        "users": ["alice", "bob", "charlie"]
      }
    }
  }
}
```

**流量采集服务**：
- 使用 gRPC 客户端连接 V2Ray API
- 查询用户流量统计
- 计算增量并上报
