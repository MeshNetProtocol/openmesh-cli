## 2. sing-box 流量统计方案

### 2.1 V2Ray API

**官方文档**：[sing-box V2Ray API](https://sing-box.sagernet.org/configuration/experimental/v2ray-api/)

**特性**：
- 基于 gRPC 协议
- 支持按 inbound、outbound、user 级别统计
- 与 V2Ray 统计服务协议兼容

**限制**：
- ⚠️ **用户级统计只支持 VMess 协议**
- VMess 使用 email 作为用户标识
- **Shadowsocks、SOCKS、HTTP 等协议不支持按用户统计**

**结论**：❌ 不适用于我们的 Shadowsocks 2022 方案

### 2.2 Clash API

**官方文档**：[sing-box Clash API](https://sing-box.sagernet.org/configuration/experimental/clash-api/)

**特性**：
- RESTful HTTP 接口
- 实时流量监控
- 连接跟踪（每个连接的详细信息）
- 兼容 Clash dashboard UI

**能力**：
- 可以查询所有活动连接
- 每个连接包含：源地址、目标地址、代理链、流量统计
- 提供全局流量统计

**限制**：
- ⚠️ **不提供按用户聚合的流量统计**
- 只能看到连接级别的数据

**结论**：⚠️ 可以使用，但需要自己实现用户级聚合

## 3. Shadowsocks 2022 多用户配置

**官方文档**：[sing-box Shadowsocks](https://sing-box.sagernet.org/configuration/inbound/shadowsocks)

**多用户配置示例**：
```json
{
  "type": "shadowsocks",
  "method": "2022-blake3-aes-128-gcm",
  "password": "server_password",
  "users": [
    {
      "name": "alice",
      "password": "user_password_1"
    },
    {
      "name": "bob",
      "password": "user_password_2"
    }
  ]
}
```

**特点**：
- 每个用户有独立的密码
- 服务端可以识别不同用户的连接
- 但 **sing-box 不提供按用户统计流量的 API**

## 4. 可行的解决方案

### 方案 A：切换到 VMess 协议 + V2Ray API

**实现**：
- 将 Phase 0 的 Shadowsocks 2022 改为 VMess
- 启用 V2Ray API
- 使用 gRPC 查询用户级流量统计

**优势**：
- ✅ 原生支持用户级流量统计
- ✅ 官方 API，稳定可靠
- ✅ 有现成的 Python gRPC 客户端库

**劣势**：
- ❌ 需要重新配置 Phase 0 环境
- ❌ VMess 性能可能不如 Shadowsocks 2022
- ❌ 需要重新测试

**工作量**：2-3 天

### 方案 B：Clash API + 自定义聚合层

**实现**：
- 使用 Clash API 获取连接级别数据
- 编写聚合服务，根据连接元数据识别用户
- 定期轮询 Clash API，累加每个用户的流量

**优势**：
- ✅ 保持 Shadowsocks 2022 配置
- ✅ HTTP API，易于集成

**劣势**：
- ❌ 需要自己实现用户识别逻辑
- ❌ 连接元数据可能不包含用户信息
- ❌ 轮询方式可能有延迟

**工作量**：3-4 天

### 方案 C：外部流量监控（eBPF/iptables）

**实现**：
- 使用 eBPF 或 iptables 在系统层面监控流量
- 根据端口或其他标识区分用户
- 独立于 sing-box 的流量统计

**优势**：
- ✅ 不依赖 sing-box API
- ✅ 可以监控任何协议
- ✅ 精确的系统级统计

**劣势**：
- ❌ 实现复杂度高
- ❌ 需要 root 权限
- ❌ 跨平台兼容性问题

**工作量**：5-7 天

### 方案 D：保持 Phase 0 的模拟流量方案

**实现**：
- 继续使用 Phase 0 的模拟流量采集服务
- 不集成真实的 sing-box 流量统计
- 专注于开发其他功能（管理页面、钱包等）

**优势**：
- ✅ 无需修改现有代码
- ✅ 可以快速推进其他功能
- ✅ 架构和 API 已验证

**劣势**：
- ❌ 不是真实的流量统计
- ❌ 无法用于生产环境

**工作量**：0 天

## 5. 推荐方案

### 推荐：方案 A（切换到 VMess + V2Ray API）

**理由**：
1. **官方支持**：V2Ray API 是 sing-box 官方提供的用户级流量统计方案
2. **稳定可靠**：经过充分测试，有完整的文档
3. **长期可维护**：不依赖 hack 或 workaround
4. **性能可接受**：VMess 性能足够好，且支持多种优化

**实施步骤**：
1. 修改 sing-box 节点配置（Shadowsocks → VMess）
2. 启用 V2Ray API
3. 实现 gRPC 客户端
4. 修改流量采集服务
5. 重新运行 Phase 0 测试

**时间估算**：2-3 天

## 7. 技术参考

### V2Ray API 配置示例

```json
{
  "experimental": {
    "v2ray_api": {
      "listen": "127.0.0.1:8080",
      "stats": {
        "enabled": true,
        "inbounds": ["vmess-in"],
        "outbounds": ["direct"],
        "users": ["alice@example.com", "bob@example.com"]
      }
    }
  }
}
```

### VMess 多用户配置示例

```json
{
  "type": "vmess",
  "tag": "vmess-in",
  "listen": "127.0.0.1",
  "listen_port": 8001,
  "users": [
    {
      "name": "alice",
      "uuid": "b831381d-6324-4d53-ad4f-8cda48b30811",
      "alterId": 0
    },
    {
      "name": "bob",
      "uuid": "a1b2c3d4-5678-90ab-cdef-1234567890ab",
      "alterId": 0
    }
  ]
}
```

### Python gRPC 客户端示例

```python
import grpc
from v2ray_api import StatsServiceStub

channel = grpc.insecure_channel('127.0.0.1:8080')
stub = StatsServiceStub(channel)

# 查询用户流量
response = stub.GetStats(pattern="user>>>alice@example.com>>>traffic>>>uplink")
print(f"Alice uplink: {response.stat.value} bytes")
```
