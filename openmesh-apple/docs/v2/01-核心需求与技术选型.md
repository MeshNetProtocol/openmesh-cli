# 核心需求与技术选型

> **文档目的**：明确 V2 的核心业务需求，对比技术方案，说明为什么选择 Hysteria2。

---

## 1. 核心业务需求

### 1.1 产品定位

**OpenMesh = VPN 行业的 OpenRouter**

让用户像切换 AI 模型一样切换 VPN 供应商，同时保持去中心化结算。

### 1.2 关键需求

| 需求 | 说明 | 优先级 |
|------|------|--------|
| **供应商快速切换** | 用户可以一键切换供应商，无需重新安装配置 | P0 |
| **多用户流量统计** | 每个用户的流量独立统计，支持多节点汇总 | P0 |
| **预付费配额管理** | 用户预付 USDC 购买流量配额，中心化扣减 | P0 |
| **去中心化支付** | 用户直接向供应商支付，平台不托管资金 | P0 |
| **链上供应商发现** | 供应商通过链上注册合约被发现 | P1 |
| **钱包自动支付** | 解锁一次后支持小额自动支付 | P1 |

### 1.3 用户场景

```
场景 1：新用户首次使用
用户 → 创建钱包 → 浏览供应商 → 选择供应商 → 购买流量 → 开始使用

场景 2：切换供应商
用户 → 查看当前配额 → 选择新供应商 → 一键切换 → 继续使用

场景 3：配额耗尽
用户使用中 → 配额不足提醒 → 追加购买 → 继续使用
```

## 2. 技术约束

### 2.1 硬约束（不可改变）

1. **不修改 sing-box 源码**
   - 原因：维护成本高，升级困难
   - 影响：只能使用 sing-box 原生支持的协议和功能

2. **虚拟网卡层不变**
   - 原因：客户端已实现 TUN 虚拟网卡
   - 影响：只需修改 sing-box 的 outbound 配置

3. **客户端已封装 sing-box Go 代码**
   - 原因：MeshFluxMac 已集成 sing-box
   - 影响：协议切换只需修改配置文件

### 2.2 软约束（可以权衡）

1. **性能要求**：单用户吞吐量 > 100 Mbps
2. **并发要求**：支持 1000+ 并发用户
3. **统计精度**：流量统计误差 < 1%
4. **响应延迟**：API 响应时间 < 100ms

## 3. 协议对比分析

### 3.1 候选协议

我们评估了三个候选协议：

1. **Shadowsocks** - 当前使用的协议
2. **VMess** - V2Ray 的主要协议
3. **Hysteria2** - 基于 QUIC 的新一代协议

### 3.2 对比表

| 维度 | Shadowsocks | VMess | Hysteria2 |
|------|-------------|-------|-----------|
| **用户级流量统计** | ❌ 不支持 | ✅ 支持（V2Ray API） | ✅ 原生支持（Traffic Stats API） |
| **性能** | 中等 | 中等 | 优秀（QUIC） |
| **配置复杂度** | 低 | 中 | 低 |
| **sing-box 支持** | ✅ 原生支持 | ✅ 原生支持 | ✅ 原生支持 |
| **需要修改源码** | ❌ 无法实现 | ❌ 不需要 | ❌ 不需要 |
| **统计 API** | 无 | V2Ray gRPC API | HTTP REST API |
| **社区成熟度** | 高 | 高 | 中 |
| **抗审查能力** | 中 | 中 | 高 |

### 3.3 详细分析

#### Shadowsocks

**优点**：
- 配置简单
- 社区成熟
- sing-box 原生支持

**致命缺陷**：
- ❌ **不支持用户级流量统计**
- Shadowsocks 协议本身不区分用户
- V2Ray API 无法为 Shadowsocks 提供用户级统计
- 即使使用 `managed: true` 多用户模式，也只能统计总流量

**结论**：无法满足核心需求，不可行。

#### VMess

**优点**：
- ✅ 支持用户级流量统计（通过 V2Ray API）
- sing-box 原生支持
- 社区成熟

**缺点**：
- 性能一般（基于 TCP）
- 配置相对复杂
- 需要配置 TLS
- V2Ray API 是 gRPC，集成相对复杂

**结论**：可行，但不是最优选择。

#### Hysteria2

**优点**：
- ✅ **原生支持用户级流量统计**（内置 Traffic Stats API）
- ✅ 性能优秀（基于 QUIC）
- ✅ 配置简单
- ✅ sing-box 原生支持
- ✅ HTTP REST API，集成简单
- ✅ 抗审查能力强

**缺点**：
- 社区相对较新（但已有生产案例）

**结论**：最优选择。

## 4. 最终决策：Hysteria2

### 4.1 决策理由

基于**第一性原理**和**奥卡姆剃刀原则**，我们选择 Hysteria2：

1. **满足核心需求**
   - 原生支持用户级流量统计
   - 不需要修改 sing-box 源码
   - 只需修改配置文件

2. **技术优势**
   - 基于 QUIC，性能优于 TCP
   - 内置 Traffic Stats API，集成简单
   - HTTP REST API，易于调用

3. **最简方案**
   - 虚拟网卡层不需要改动
   - 客户端只需修改 outbound 配置
   - 服务端配置简单直接

### 4.2 Hysteria2 核心能力

#### 用户级流量统计

Hysteria2 内置 Traffic Stats API：

```bash
# 获取所有用户流量统计
GET http://server:port/traffic

# 响应示例
{
  "users": {
    "user_0x1234abcd": {
      "tx": 1048576,      # 上传字节数
      "rx": 5242880       # 下载字节数
    },
    "user_0x5678efgh": {
      "tx": 2097152,
      "rx": 10485760
    }
  }
}
```

#### 多用户认证

```json
{
  "auth": {
    "type": "password",
    "password": "your_password"
  },
  "users": {
    "user_0x1234abcd": "password1",
    "user_0x5678efgh": "password2"
  }
}
```

#### 性能特性

- 基于 QUIC 协议
- 支持 0-RTT 连接
- 内置拥塞控制（Brutal CC）
- 支持多路复用

## 6. 架构影响

### 6.1 客户端变化

```
变化前（Shadowsocks）：
TUN → sing-box → Shadowsocks outbound → 服务器

变化后（Hysteria2）：
TUN → sing-box → Hysteria2 outbound → 服务器
     ↑ 不变              ↑ 只改这里
```

```
无法获取用户级流量统计

变化后（Hysteria2）：
```
