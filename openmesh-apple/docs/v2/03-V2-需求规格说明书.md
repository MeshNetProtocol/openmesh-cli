# 03-V2-需求规格说明书

## 1. 文档目的

本文档定义 V2 的正式需求，覆盖：

- 官方临时受限访问
- 商用供应商
- 私人节点 / DIY
- 钱包与自动支付

---

## 2. 核心术语

### 2.1 流豆账户

用户自己的链上钱包。  
它持有 Base 主网 / 测试网资产，并负责 x402 自动签名。

### 2.2 商用供应商

通过链上注册被发现、直接向用户收取 USDC、可快速切换的流量供应商。

### 2.3 官方临时受限访问

由官方维护的第一个供应商，用于冷启动和兜底。  
它不是中心化平台特权能力，而是第一个官方 provider。

### 2.4 流量桶

商用供应商向用户提供的可消耗额度。  
例如：

- 免费 40MB
- 续费 40MB

### 2.5 支付收据

供应商在确认 x402 支付后签发的凭条，用于证明用户已支付某个流量桶。

### 2.6 用量凭条

供应商对某段时间或某个桶的实际流量使用情况出具的签名凭条。  
后续 token 激励必须以此为依据。

---

## 3. 产品原则

1. OpenMesh 主体是供应商切换，不是安装向导。
2. 商用供应商目录来自链上。
3. 官方临时受限访问和私人节点可无钱包使用。
4. 商用场景的热路径必须支持自动小额支付。
5. 平台不做中心化账本。
6. sing-box 只承担数据平面职责。

---

## 4. 场景需求

## 4.1 官方临时受限访问

要求：

- 新用户在无任何配置时可立即连接
- 可以继续使用当前简单单节点服务端方案
- 不强制要求先创建钱包
- 可选日限额

## 4.2 私人节点 / DIY

要求：

- 用户可导入本地配置
- 不要求钱包或支付
- 与商用场景并列存在

## 4.3 商用供应商

要求：

- 用户已创建或恢复钱包，并持有一定 USDC
- 用户可浏览链上供应商目录
- 用户可快速切换供应商
- 供应商可先提供一个免费桶
- 桶剩余量低于阈值时，供应商发起 `402`
- 客户端在自动支付限制内自动签名支付
- 供应商验证成功后放行下一个桶

---

## 5. 商用供应商正式需求

### F-COM-01 链上发现

客户端必须支持从 Base 主网 / 测试网固定 RPC 读取供应商注册合约。

### F-COM-02 metadata 模型

每个供应商 metadata 至少应包含：

- `providerId`
- `displayName`
- `description`
- `configServiceURL`
- `meteringServiceURL`
- `supportedProtocols`
- `pricingModel`
- `owner`
- `serviceSigners`

### F-COM-03 快速切换

用户切换供应商时，不得要求先删除旧配置再重新走传统安装流程。  
客户端必须支持在本地 `tun` 不变的前提下快速切换远端供应商配置。

### F-COM-04 免费桶

供应商应支持先提供一个免费桶，例如 40MB。  
免费桶大小可配置，但文档默认参考值为 40MB。

### F-COM-05 预续费阈值

供应商不得等到流量桶完全耗尽后才发起续费请求。  
当剩余流量低于阈值时，应发起下一桶的 `402` 续费。

建议：

- 阈值可按固定 MB 或百分比设置
- 默认建议为“剩余 5MB”或“剩余 20%”

### F-COM-06 x402 自动支付

当供应商返回 `402` 时，客户端应在以下条件全部满足时自动完成支付：

- 钱包已解锁
- 单次支付金额不超过用户设置上限
- 当日累计支付不超过用户设置上限

### F-COM-07 热路径放行

供应商不得把“链上余额轮询到账”作为热路径放行条件。  
热路径应以 x402 校验通过并完成支付确认作为准入依据。

### F-COM-08 收据与凭条

供应商在支付成功后必须签发支付收据。  
供应商在会话结束或桶结算时必须签发用量凭条。

### F-COM-09 凭条恢复

客户端和供应商都必须保存最近有效凭条。  
任一方丢失时，应允许向另一方恢复。

### F-COM-10 后续 token 依据

商用场景中真实、可验证的最新用量凭条必须作为后续 token 激励的依据。

---

## 6. 钱包需求

### F-WALLET-01 用户自持

流豆账户必须是用户自己持有助记词的钱包，不得被产品叙事写成平台托管账号。

### F-WALLET-02 安全落盘

助记词和私钥不得明文落盘。  
允许以加密 blob 形式存储，但必须由设备绑定能力保护。

### F-WALLET-03 signer session

用户在成功解锁钱包后，系统应进入 signer session。  
在 signer session 存活期间，允许后续小额支付自动签名。

### F-WALLET-04 signer session 退出条件

以下任一条件触发时，必须销毁内存中的私钥明文：

- 用户手动锁定
- signer session 超时
- App 退出
- 达到每日自动支付上限

### F-WALLET-05 自动支付限制

设置页必须至少提供：

- 单次最大自动支付额度
- 每日最大自动支付额度
- signer session 持续时长

---

## 7. sing-box 数据平面需求

### F-DP-01 用户标识

商用供应商必须能把用户的钱包地址或地址派生值下发到数据平面，作为可统计、可停供的用户标识。

### F-DP-02 会话统计

数据平面必须支持至少以下粒度之一：

- 地址级统计
- 地址 + session 级统计

### F-DP-03 停供能力

当用户未续费或超出授权范围时，供应商必须能够停止该地址或该 session 的继续供给。

---

## 8. 验收结论

只有当以下条件同时满足，商用场景才算成立：

1. 用户可从链上发现供应商
2. 用户可快速切换供应商
3. 用户可在 signer session 内自动完成小额支付
4. 供应商能在阈值到达前触发下一桶续费
5. 供应商能签发并恢复凭条
6. 平台不依赖中心化账本
