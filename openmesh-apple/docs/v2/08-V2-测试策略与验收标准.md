# 08-V2-测试策略与验收标准

## 1. 文档目的

本文档用于定义 MeshFlux macOS 客户端 V2 升级的测试策略、测试边界、测试分层、Mock 规则、关键测试范围和最终验收标准。

本文档重点回答以下问题：

- V2 应如何测试
- 哪些模块必须写自动化测试
- 哪些能力只做手工验收
- Mock 如何设计
- 哪些测试失败必须阻断合并
- 最终达到什么门槛才算可以进入发布或评审阶段

本文档不负责描述具体实现代码，也不替代详细技术设计与实施计划文档。  
其中模块目录、协议拆分和功能边界以前序文档为准。

---

## 2. 测试目标

V2 测试体系必须同时覆盖以下目标：

### 2.1 正确性
确保钱包、链上供应商、x402 支付和安装接入行为符合需求与设计预期。

### 2.2 稳定性
确保 V2 新功能不会破坏现有 VPN、配置管理、安装和导航能力。

### 2.3 安全性
确保私钥、恢复短语、解锁、签名和 Keychain 使用符合安全约束，不出现明文泄露和错误持久化。

### 2.4 可回归
确保后续修改不会悄悄破坏已完成模块，尤其是加密、签名、授权和配置安装链路。

### 2.5 可持续集成
所有核心测试应能在本地和 CI 中稳定执行，不依赖真实网络、真实 Keychain 和真实外部服务。

---

## 3. 测试原则

## 3.1 单元测试优先
V2 以单元测试为主，尤其是：

- Blockchain 层
- Service 层
- Store / Coordinator 层
- 加密、签名、授权和状态机相关逻辑

## 3.2 Mock 一切外部依赖
所有以下依赖必须 Mock：

- RPC 节点
- HTTP 服务
- Keychain
- LocalAuthentication
- 时间与随机数源（如需要）
- 供应商配置下载响应

## 3.3 UI 不做 XCTest 主战场
OpenMesh 视图层以手工验收为主，不把 SwiftUI 细节作为单元测试主目标。  
自动化测试优先聚焦：

- ViewModel
- Store
- Service
- Blockchain 基础层

## 3.4 测试失败即阻断
任何核心单元测试失败，都不允许合并代码。  
不允许以“后面再补”作为理由跳过测试。

## 3.5 安全测试不可妥协
以下测试必须存在且通过：

- 错误密码拒绝解锁
- 恢复短语恢复结果一致
- 过期签名拒绝
- Keychain Mock 替代真实 Keychain
- 无真实网络请求

## 3.6 每完成一模块立即补测试
实现顺序必须与测试补充同步，而不是最后集中补测试。

---

## 4. 测试范围总览

V2 测试分为四层：

1. **基础层单元测试**
2. **服务与状态层单元测试**
3. **集成级流程测试（仍使用 Mock）**
4. **手工验收测试**

---

## 5. 测试分层设计

## 5.1 第一层：基础层单元测试

### 覆盖对象
- `ContractABI`
- `BlockchainRPCClient`
- `WalletCrypto`
- `EIP712Signer`

### 目标
验证纯算法、纯编码、纯解码和底层协议行为。

### 特点
- 不依赖 UI
- 不依赖真实 Keychain
- 不依赖真实链
- 可使用固定测试向量

---

## 5.2 第二层：服务与状态层单元测试

### 覆盖对象
- `MeshWalletStore`
- `OnChainSupplierService`
- `SupplierMetadataService`
- `X402PaymentClient`
- `ConfigAcquisitionService`
- `X402PaymentFlowCoordinator`

### 目标
验证业务流程编排、错误处理、状态机和依赖注入行为。

### 特点
- 所有外部依赖必须可 Mock
- 重点看状态变化和输出结果
- 不要求渲染真实界面

---

## 5.3 第三层：集成级流程测试（Mock 集成）

### 覆盖对象
- 钱包创建 → 解锁 → 签名
- 供应商加载 → metadata 合并
- 支付确认 → 授权生成 → 配置获取 → 安装接入

### 目标
用 Mock 依赖串起来验证主流程，不访问真实链和真实服务。

### 特点
- 仍属于自动化测试
- 但覆盖多个模块串联
- 用于发现接口契约不一致和阶段切换错误

---

## 5.4 第四层：手工验收测试

### 覆盖对象
- `OpenMeshView`
- 导航插入行为
- Wallet Setup / Unlock 弹窗
- Supplier Payment Confirm Sheet
- DIY 页面
- 与现有 App 导航、Dashboard、Profiles 的兼容性

### 目标
验证视觉、交互、可理解性和整体连通性。

### 特点
- 不追求自动化
- 必须形成手工验收清单
- 是发布前最后一道用户体验检查

---

## 7. Mock 基础设施设计

## 7.1 MockBlockchainRPCClient

### 作用
替代真实链上 RPC。

### 必备能力
- 预设 `eth_call` 返回
- 预设 `eth_getLogs` 返回
- 预设余额返回
- 可注入错误

### 目标
让所有合约读取、供应商服务和余额测试不依赖真实网络。

---

## 7.2 MockKeychainManager

### 作用
替代真实 Keychain。

### 必备能力
- 保存加密私钥
- 读取加密私钥
- 保存地址
- 读取地址
- 保存 identity
- 判断钱包是否存在
- 可注入保存失败 / 读取失败

### 目标
保证测试不写系统真实 Keychain，避免副作用污染。

---

## 7.3 MockHTTPSession

### 作用
替代真实 HTTP 请求。

### 必备能力
- 根据请求 URL 或 Request 返回预设数据
- 预设状态码
- 预设 header
- 可注入网络错误
- 统计请求次数（用于重试逻辑测试）

### 目标
覆盖：
- metadata 获取
- x402 配置拉取
- 402 重试逻辑
- 非法响应处理

---

## 7.4 MockLocalAuthentication

### 作用
替代系统生物识别能力。

### 必备能力
- 模拟生物识别成功
- 模拟生物识别失败
- 模拟设备不支持

### 目标
让钱包解锁测试不依赖设备真实 Touch ID / Face ID。

---

## 7.5 TestHelpers

### 建议内容
- 构造 HTTP 响应
- 构造 `SupplierRegistered` 事件 log
- 构造已解锁的 `MeshWalletStore`
- 构造常见 ABI 调用 data
- 提供固定测试助记词和期望地址

### 原则
- 测试向量集中管理
- 不把这些常量泄露到生产代码中

## 8. 自动化测试范围详细设计

## 8.1 `ContractABITests.swift`

### 目标
验证 ABI 编码与解码正确性。

### 至少覆盖
- `address` 编码
- `uint256` 编码
- `bool` 编码
- `bytes32` 编码
- `string` 解码
- selector 拼接正确
- 非法返回数据时抛错

### 关键断言
- 编码结果长度正确
- selector 正确
- 解码结果值正确
- 错误输入不会静默通过

---

## 8.2 `BlockchainRPCClientTests.swift`

### 目标
验证 JSON-RPC 层请求与错误映射。

### 至少覆盖
- `eth_call` 请求成功
- `eth_getLogs` 请求成功
- `balanceOf` 请求成功
- HTTP 非 200
- JSON 非法
- RPC error 返回
- 空字段返回
- 错误映射为 `BlockchainError`

### 关键断言
- 请求方法名正确
- 参数结构正确
- 返回数据能转换为预期格式

---

## 8.3 `WalletCryptoTests.swift`

### 目标
验证助记词、地址派生和加解密逻辑。

### 至少覆盖
- 生成 12 词助记词
- 已知助记词派生固定地址
- 加密后可正确解密
- 错误密码无法解密
- 地址格式正确
- 私钥加密输出不是明文

### 关键断言
- 助记词数量为 12
- 地址长度为 42 且以 `0x` 开头
- 同一助记词派生地址稳定
- 错误密码不会返回部分明文

---

## 8.4 `EIP712SignerTests.swift`

### 目标
验证 EIP-712 digest 和签名能力。

### 至少覆盖
- digest 长度正确
- 签名格式正确
- 签名可恢复地址
- 过期签名被拒绝
- 非法时间窗口被拒绝
- 不同 nonce 产生不同 digest

### 关键断言
- `recoverAddress` 与预期地址一致
- 已过期 `validBefore` 抛出 `.x402SignatureExpired`
- digest 具备确定性

---

## 8.5 `MeshWalletStoreTests.swift`

### 目标
验证钱包状态中心行为。

### 至少覆盖
- 创建钱包成功
- 恢复钱包成功
- 相同恢复短语派生地址一致
- 正确密码解锁成功
- 错误密码解锁失败
- 后台后自动锁定
- 刷新余额成功
- 无钱包状态识别正确
- Keychain 失败时错误传播正确

### 关键断言
- 创建后状态为 `.unlocked`
- 恢复后地址正确
- 错误密码后状态不应是假解锁
- `handleAppDidEnterBackground()` 后状态回到 `.locked`

---

## 8.6 `SupplierMetadataServiceTests.swift`

### 目标
验证 metadata 拉取和解析。

### 至少覆盖
- 正常 JSON 解析成功
- 缺少必要字段时报错
- `config_urls` 为空时报错
- 非法价格格式时报错
- 非法 JSON 报错
- 网络错误报错

### 关键断言
- 正常 metadata 正确映射到模型
- 错误 metadata 不会静默成功

---

## 8.7 `OnChainSupplierServiceTests.swift`

### 目标
验证链上供应商目录读取逻辑。

### 至少覆盖
- 正常读取注册事件
- 过滤非活跃供应商
- 合并 metadata 成功
- metadata 单项失败不影响整体
- RPC 失败时报错
- 缓存命中
- `forceRefresh` 失效缓存
- 空供应商列表返回空结果

### 关键断言
- 活跃条件判断正确
- 供应商数量正确
- metadata 字段合并正确
- 单个 supplier 出错不拖垮全部

---

## 8.8 `X402PaymentClientTests.swift`

### 目标
验证 x402 授权生成和网络请求逻辑。

### 至少覆盖
- 授权对象生成成功
- header 格式正确
- 402 后重试成功
- 402 多次后失败
- 过期授权被拒绝
- 错误余额或错误输入被阻断
- 网络请求带上正确 header

### 关键断言
- Authorization header 存在
- `scheme/network/from/to/value` 正确
- 请求次数符合重试预期
- 超出重试上限时报错

---

## 8.9 `ConfigAcquisitionServiceTests.swift`

### 目标
验证支付后配置获取与安装接入。

### 至少覆盖
- 获取配置成功并调用安装接入点
- 配置 URL 非法时报错
- HTTP 失败时报错
- 空数据时报错
- 安装接入抛错时正确传播
- 成功后不遗留临时脏状态

### 关键断言
- `ProviderInstallWindowManager` 的接入方法被调用
- 传入的 supplierId / displayName / configData 正确

---

## 8.10 `X402PaymentFlowCoordinatorTests.swift`

### 目标
验证完整流程协调状态机。

### 至少覆盖
- 无钱包时阻断
- 钱包锁定时阻断
- 余额不足时阻断
- 进入确认阶段
- 用户确认后进入授权阶段
- 获取配置后进入安装阶段
- 成功进入 `success`
- 任一阶段失败进入 `failed`

### 关键断言
- 阶段迁移顺序正确
- 失败时不会继续进入下一阶段
- 成功时状态闭环完整

## 10. 安全测试专项

## 10.1 敏感信息不落盘测试

### 验证点
- 助记词不进入 SharedPreferences
- 私钥不进入 SharedPreferences
- 私钥不写入普通文件
- 配置下载时不夹带额外密钥材料

---

## 10.2 日志脱敏测试

### 验证点
- 普通日志不输出私钥
- 普通日志不输出恢复短语
- 普通日志不输出完整 authorization
- 地址只展示缩略形式或安全形式

---

## 10.3 锁定与清理测试

### 验证点
- 进入后台自动锁定
- 支付结束后清理临时授权对象
- 解锁失败不保留半解锁状态
- 流程取消时清理临时上下文

---

## 11.1 链上异常

- OpenMesh 不崩溃

### 验证点

- 402 重试达到上限后失败
- 余额不足时阻断

## 11.4 安装异常
### 验证点
- 配置下载失败不写脏状态
- 安装失败不写半成品映射
- 不影响既有 provider/profile 体系

## 12. 测试执行策略

## 12.1 模块完成即测
每完成一个模块，立即补该模块的单元测试，再进入下一模块。

## 12.2 提交前必跑
每次提交前至少运行：

- 本模块测试
- 受影响的回归测试
- 核心钱包 / 支付 / 供应商测试

## 12.3 合并前全量跑
合并前必须运行：

- `MeshFluxMacTests/blockchain/` 全部测试
- 关键手工验收清单
- 既有主流程回归检查

---

## 13. 测试门禁与阻断规则

以下任一情况出现，禁止合并：

1. 单元测试有失败  
2. 有测试被故意跳过且无明确豁免  
3. blockchain 目录覆盖率低于门槛  
4. 出现真实 HTTP 请求  
5. 测试写入真实 Keychain  
6. 钱包安全关键测试缺失  
7. 支付签名关键测试缺失  
8. 现有主流程回归失败  

---

## 14. 验收标准

## 14.1 自动化测试门槛

| 项目 | 要求 |
|---|---|
| 单元测试通过率 | 100% |
| blockchain 目录代码覆盖率 | ≥ 90% |
| 测试执行总时长 | ≤ 30 秒 |
| 真实 HTTP 请求 | 0 |
| 真实 Keychain 写入 | 0 |

---

## 14.2 功能验收门槛

必须同时满足：

- OpenMesh 导航可见并可进入
- 钱包创建 / 恢复 / 解锁可用
- 供应商列表可加载
- DIY 页面可用
- 支付确认可用
- 商业供应商闭环可走通
- 安装结果可进入现有体系
- 链上异常时应用不崩溃
- V1 现有功能不被破坏

---

## 14.3 安全验收门槛

必须同时满足：

- 私钥仅存于 Keychain
- 私钥密文使用 AES-256-GCM
- KDF 使用 PBKDF2-SHA256
- 助记词仅展示一次且不落盘
- 后台自动锁定有效
- 签名只在用户明确确认支付后执行
- `validBefore` 过期会被拒绝
- nonce 不复用

---

## 16. 测试结果输出要求

每轮测试完成后，至少输出以下内容：

- 通过测试列表
- 失败测试列表
- 覆盖率结果
- 是否有真实网络调用
- 是否有真实 Keychain 副作用
- 当前阻断项
- 是否达到下一阶段准入条件

## 18. 关键结论

V2 的测试体系不能停留在“写几个 happy path 单测”的层面，而必须形成一套完整的质量门禁：

1. **基础层必须可验证**  
   ABI、RPC、加密、签名都必须有确定测试向量。

2. **业务层必须可 Mock**  
   钱包、供应商、支付、配置获取不能依赖真实链和真实服务。

3. **界面层以手工验收为主**  
   重点验证体验、文案、状态和兼容性，而不是堆 UI 单测。

4. **安全测试必须和功能测试同等重要**  
   私钥、助记词、解锁、签名和日志脱敏都属于阻断级要求。

5. **最终标准不是“基本能跑”，而是“可回归、可持续集成、可发布”**  
   只有当自动化测试、手工验收、安全门槛和兼容性回归同时满足时，V2 才能算真正完成。

这份测试策略文档，将作为 V2 开发、联调、回归和发布前验收的统一质量基线。
