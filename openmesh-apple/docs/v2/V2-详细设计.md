## 1. TrafficQuota（流量配额）

### 1.1 数据结构

```json
{
  "quotaId": "quota_abc123",
  "providerId": "provider_a",
  "userAddress": "0x1234567890abcdef",
  "totalQuota": 500000000,
  "usedUpload": 50000000,
  "usedDownload": 150000000,
  "remaining": 300000000,
  "priceRate": 100,
  "purchasedAt": "2026-03-24T10:00:00Z",
  "updatedAt": "2026-03-24T11:00:00Z",
  "expiresAt": null
}
```

### 1.2 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| quotaId | string | 配额唯一标识 |
| providerId | string | 供应商 ID |
| userAddress | string | 用户钱包地址 |
| totalQuota | int64 | 总配额（字节） |
| usedUpload | int64 | 已用上传（字节） |
| usedDownload | int64 | 已用下载（字节） |
| remaining | int64 | 剩余配额（字节） |
| priceRate | decimal | 购买时价格（1 USDC = X MB） |
| purchasedAt | timestamp | 购买时间 |
| updatedAt | timestamp | 最后更新时间 |
| expiresAt | timestamp\|null | 过期时间（null = 永久有效） |

### 1.3 数据库表设计

```sql
CREATE TABLE traffic_quotas (
    quota_id TEXT PRIMARY KEY,
    provider_id TEXT NOT NULL,
    user_address TEXT NOT NULL,
    total_quota BIGINT NOT NULL,
    used_upload BIGINT DEFAULT 0,
    used_download BIGINT DEFAULT 0,
    remaining BIGINT NOT NULL,
    price_rate DECIMAL(10,2) NOT NULL,
    purchased_at TIMESTAMP NOT NULL,
    updated_at TIMESTAMP NOT NULL,
    expires_at TIMESTAMP,
    UNIQUE(provider_id, user_address)
);

CREATE INDEX idx_user_provider ON traffic_quotas(user_address, provider_id);
CREATE INDEX idx_updated_at ON traffic_quotas(updated_at);
```

{

### 3.2 数据库表设计

```sql
CREATE TABLE purchase_records (
    record_id TEXT PRIMARY KEY,
    provider_id TEXT NOT NULL,
    user_address TEXT NOT NULL,
    usdc_amount DECIMAL(18,6) NOT NULL,
    bandwidth_purchased BIGINT NOT NULL,
    price_rate DECIMAL(10,2) NOT NULL,
    payment_signature TEXT NOT NULL,
    tx_hash TEXT,
    purchased_at TIMESTAMP NOT NULL
);

CREATE INDEX idx_user_purchases ON purchase_records(user_address, purchased_at DESC);
```

## 4. API 接口定义

### 4.1 Hysteria2 Traffic Stats API

**获取流量统计**：
```bash
GET http://node:8080/traffic
Authorization: Bearer {stats_secret}

Response:
{
  "users": {
    "user_0x1234abcd": {
      "tx": 1048576,
      "rx": 5242880
    }
  }
}
```

### 4.2 Metering Service API

**查询用户配额**：
```bash
GET /api/v1/users/{userId}/quota

Response:
{
  "quotaId": "quota_abc123",
  "totalQuota": 500000000,
  "usedUpload": 50000000,
  "usedDownload": 150000000,
  "remaining": 300000000,
  "priceRate": 100
}
```

**流量上报（内部）**：
```bash
POST /api/v1/metering/report
{
  "nodeId": "node_a",
  "userId": "user_0x1234abcd",
  "txIncrement": 1000000,
  "rxIncrement": 5000000
}

Response:
{
  "status": "ok",
  "remaining": 294000000,
  "action": "continue"
}
```

### 4.3 Payment Service API

**预付费购买流量**：
```bash
POST /api/v1/users/{userId}/purchase
{
  "usdcAmount": 5.0,
  "paymentSignature": "0xabcdef...",
  "txHash": "0x123456..."
}

Response:
{
  "quotaId": "quota_abc123",
  "bandwidthPurchased": 500000000,
  "priceRate": 100,
  "newTotalQuota": 800000000,
  "purchasedAt": "2026-03-24T10:00:00Z"
}
```

### 4.4 Config Service API

**生成配置**：
```bash
POST /api/v1/config/generate
{
  "userAddress": "0x1234abcd",
  "providerId": "provider_a"
}

Response:
{
  "config": {
    "type": "hysteria2",
    "server": "node-a.example.com",
    "server_port": 443,
    "password": "temp_password_xyz",
    "tls": {
      "enabled": true,
      "server_name": "node-a.example.com"
    }
  },
  "expiresAt": "2026-03-25T10:00:00Z"
}
```

## Phase 0: Hysteria2 技术验证（3-5天）

**目标**：验证 Hysteria2 用户级流量统计能力和多节点汇总的可行性。

**任务**：
- 安装 Hysteria2 服务端
- 配置多用户认证
- 启用 Traffic Stats API
- 配置 sing-box 客户端

**输出**：
- Hysteria2 服务端运行
- Traffic Stats API 可访问
- 客户端可连接

### P0.2 单节点验证（1天）

**任务**：
- 配置 2 个测试用户
- 验证用户级流量统计

- 对比实际流量和统计流量

**验收标准**：
- ✅ Traffic Stats API 返回用户级数据
- ✅ 流量统计准确度 > 99%
- ✅ 多用户流量互不干扰

### P0.3 多节点验证（2天）

**任务**：
- 部署 3 个 Hysteria2 节点

## Phase 2: Payment Service 开发（1周）

**目标**：实现预付费购买流量功能。

- 实现配额追加逻辑

**核心逻辑**：
```python
def purchase_bandwidth(user_address, usdc_amount, signature):
    # 1. 验证签名
    verify_x402_signature(signature, usdc_amount)

    # 2. 计算流量
    bandwidth = usdc_amount * price_rate  # 如 5 USDC * 100MB = 500MB

    # 3. 更新配额
    update_quota(user_address, bandwidth)

    # 4. 记录购买
    save_purchase_record(user_address, usdc_amount, bandwidth)
```

# 风险与应对方案

> **文档目的**：识别 V2 系统的技术风险和实施风险，提供应对方案和回滚策略。

---

## 1. 技术风险

### 1.1 Hysteria2 流量统计不准确

**风险描述**：
- Hysteria2 的 Traffic Stats API 统计不准确
- 误差超过 1%
- 多节点汇总出现重复计数

**可能性**：低
**影响**：高（核心功能）

**缓解措施**：
1. Phase 0 重点验证流量统计准确性
2. 对比实际流量和统计流量
3. 测试多种场景（单节点、多节点、并发）
4. 建立性能基准数据

**应对方案**：
- 如果误差 < 1%：可接受，继续使用
- 如果误差 1-5%：评估业务影响，考虑补偿机制
- 如果误差 > 5%：回滚到 VMess + V2Ray API 方案

---

### 1.2 Hysteria2 性能不达标

**风险描述**：
- 单用户吞吐量 < 100 Mbps
- 并发用户性能下降严重
- 延迟过高

**可能性**：低
**影响**：中

**缓解措施**：
1. Phase 0 进行性能基准测试
2. 使用 iperf3 测试吞吐量
3. 测试不同并发场景
4. 优化服务端配置

**应对方案**：
- 调整 Hysteria2 配置参数（拥塞控制、带宽限制）
- 升级服务器硬件
- 如果无法优化，考虑回滚到其他协议

---

### 1.3 Traffic Stats API 限制

**风险描述**：
- API 调用频率限制
- API 响应时间过长
- API 不稳定

**可能性**：中
**影响**：中

**缓解措施**：
1. Phase 0 测试 API 调用频率
2. 测试 API 响应时间
3. 实现错误处理和重试机制
4. 设计降级方案

**应对方案**：
- 调整拉取频率（从 10 秒调整到 30 秒）
- 实现本地缓存
- 实现降级逻辑（API 失败时使用上次数据）

---

### 1.4 多节点流量汇总复杂度

**风险描述**：
- 流量汇总逻辑复杂
- 节点切换时流量统计不连续
- 节点故障影响统计

**可能性**：中
**影响**：中

**缓解措施**：
1. Phase 0.3 重点验证多节点场景
2. 设计简单的汇总逻辑
3. 实现增量计算
4. 测试节点故障场景

**应对方案**：
- 简化汇总逻辑（定期拉取 + 增量计算）
- 实现节点状态监控
- 实现容错机制（节点故障时跳过）

---

### 1.5 配额扣减并发问题

**风险描述**：
- 并发扣减导致超额使用
- 数据库锁竞争
- 性能下降

**可能性**：中
**影响**：高

**缓解措施**：
1. 使用数据库事务保证原子性
2. 实现乐观锁或悲观锁
3. 测试并发扣减场景
4. 监控数据库性能

**应对方案**：
- 使用 PostgreSQL 的行级锁
- 实现配额预留机制
- 优化数据库索引
- 如果性能不足，考虑使用 Redis

---

## 4. 备选方案

### 4.1 方案 A：VMess + V2Ray API

**适用场景**：Hysteria2 验证失败

**优点**：
- 成熟稳定
- 社区支持好
- 用户级流量统计可靠

**缺点**：
- 性能一般（基于 TCP）
- gRPC API 集成复杂
- 配置相对复杂

**切换成本**：中

---

### 4.2 方案 B：简化版（单节点统计）

**适用场景**：多节点汇总过于复杂

**优点**：
- 实现简单
- 风险低
- 快速上线

**缺点**：
- 功能受限
- 无法支持多节点
- 扩展性差

**切换成本**：低

---

### 4.3 方案 C：混合方案

**适用场景**：部分功能无法实现

**优点**：
- 灵活性高
- 可以分阶段实施
- 降低风险

**缺点**：
- 架构复杂
- 维护成本高

**切换成本**：中

## 6. 应急预案

### 6.1 生产环境故障

**场景**：生产环境 Hysteria2 节点故障

**应急步骤**：
1. 立即切换到备用节点
2. 通知用户（如果影响范围大）
3. 排查故障原因
4. 修复并恢复服务
5. 事后总结

---

### 6.2 流量统计异常

**场景**：流量统计出现大量异常数据

**应急步骤**：
1. 暂停配额扣减
2. 排查异常原因
3. 修复数据
4. 恢复正常扣减
5. 补偿受影响用户

---

### 6.3 配额超额使用

**场景**：用户配额被超额扣减

**应急步骤**：
1. 立即停止扣减
2. 排查原因（并发问题？统计错误？）
3. 修复逻辑
4. 恢复用户配额
5. 道歉并补偿
