# V2 概要设计文档

> **文档目的**：描述 V2 版本的最终产品形态、软件开发的最终目标、技术选型说明、架构图、模块功能描述、模块之间交互关系。

---

## 1. 产品概述

### 1.1 产品定位

**OpenMesh = VPN 行业的 OpenRouter**

让用户像切换 AI 模型一样切换 VPN 供应商，同时保持去中心化结算。

### 1.2 核心需求

| 需求 | 说明 | 优先级 |
|------|------|--------|
| **供应商快速切换** | 用户可以一键切换供应商，无需重新安装配置 | P0 |
| **多用户流量统计** | 每个用户的流量独立统计，支持多节点汇总 | P0 |
| **预付费配额管理** | 用户预付 USDC 购买流量配额，中心化扣减 | P0 |
| **去中心化支付** | 用户直接向供应商支付，平台不托管资金 | P0 |
| **链上供应商发现** | 供应商通过链上注册合约被发现 | P1 |
| **钱包自动支付** | 解锁一次后支持小额自动支付 | P1 |

### 1.3 用户场景

**场景 1：新用户首次使用**
```
用户 → 创建钱包 → 浏览供应商 → 选择供应商 → 购买流量 → 开始使用
```

**场景 2：切换供应商**
```
用户 → 查看当前配额 → 选择新供应商 → 一键切换 → 继续使用
```

**场景 3：配额耗尽**
```
用户使用中 → 配额不足提醒 → 追加购买 → 继续使用
```

### 1.4 最终产品目标

实现一个去中心化的 VPN 服务平台，用户可以：
- 自由选择和切换供应商
- 使用自己的钱包进行支付
- 透明的流量统计和配额管理
- 无需信任中心化平台

## 2. 技术选型

### 2.1 技术约束

#### 硬约束（不可改变）

1. **不修改 sing-box 源码**
   - 原因：维护成本高，升级困难
   - 影响：只能使用 sing-box 原生支持的协议和功能

2. **虚拟网卡层不变**
   - 原因：客户端已实现 TUN 虚拟网卡
   - 影响：只需修改 sing-box 的 outbound 配置

3. **客户端已封装 sing-box Go 代码**
   - 原因：MeshFluxMac 已集成 sing-box
   - 影响：协议切换只需修改配置文件

#### 软约束（可以权衡）

1. **性能要求**：单用户吞吐量 > 100 Mbps
2. **并发要求**：支持 1000+ 并发用户
3. **统计精度**：流量统计误差 < 1%
4. **响应延迟**：API 响应时间 < 100ms

### 2.2 协议对比分析

| 维度 | Shadowsocks | VMess | Hysteria2 |
|------|-------------|-------|-----------|
| **用户级流量统计** | ❌ 不支持 | ✅ 支持（V2Ray API） | ✅ 原生支持（Traffic Stats API） |
| **性能** | 中等 | 中等 | 优秀（QUIC） |
| **配置复杂度** | 低 | 中 | 低 |
| **sing-box 支持** | ✅ 原生支持 | ✅ 原生支持 | ✅ 原生支持 |
| **需要修改源码** | ❌ 无法实现 | ❌ 不需要 | ❌ 不需要 |
| **统计 API** | 无 | V2Ray gRPC API | HTTP REST API |
| **社区成熟度** | 高 | 高 | 中 |
| **抗审查能力** | 中 | 中 | 高 |

#### Shadowsocks - 不可行

**致命缺陷**：
- ❌ 不支持用户级流量统计
- Shadowsocks 协议本身不区分用户
- 即使使用多用户模式，也只能统计总流量

#### VMess - 可行但不是最优

**优点**：
- ✅ 支持用户级流量统计（通过 V2Ray API）
- sing-box 原生支持

**缺点**：
- 性能一般（基于 TCP）
- V2Ray API 是 gRPC，集成相对复杂

#### Hysteria2 - 最优选择 ✅

**优点**：
- ✅ 原生支持用户级流量统计（内置 Traffic Stats API）
- ✅ 性能优秀（基于 QUIC）
- ✅ 配置简单
- ✅ HTTP REST API，集成简单
- ✅ 抗审查能力强

### 2.3 最终决策：Hysteria2

基于**第一性原理**和**奥卡姆剃刀原则**，选择 Hysteria2：

1. **满足核心需求**
   - 原生支持用户级流量统计
   - 不需要修改 sing-box 源码
   - 只需修改配置文件

2. **技术优势**
   - 基于 QUIC，性能优于 TCP
   - 内置 Traffic Stats API，集成简单
   - HTTP REST API，易于调用

3. **最简方案**
   - 虚拟网卡层不需要改动
   - 客户端只需修改 outbound 配置
   - 服务端配置简单直接

### 2.4 Hysteria2 核心能力

#### 用户级流量统计

Hysteria2 内置 Traffic Stats API：

```bash
GET http://server:port/traffic
Authorization: Bearer stats_secret

Response:
{
  "users": {
    "user_0x1234abcd": {
      "tx": 1048576,      # 上传字节数
      "rx": 5242880       # 下载字节数
    }
  }
}
```

#### 多用户认证

```json
{
  "auth": {
    "type": "password",
    "password": "your_password"
  },
  "users": {
    "user_0x1234abcd": "password1",
    "user_0x5678efgh": "password2"
  }
}
```

#### 性能特性

- 基于 QUIC 协议
- 支持 0-RTT 连接
- 内置拥塞控制（Brutal CC）
- 支持多路复用

#### 模块组成

```
MeshFluxMac
├── UI Layer
│   ├── 供应商列表视图
│   ├── 钱包管理视图
│   ├── 配额监控视图
│   └── 设置视图
├── Business Logic Layer
│   ├── WalletManager（钱包管理）
│   ├── ProviderSwitcher（供应商切换）

│   └── PaymentService（支付服务）
├── Network Layer
```

#### 关键模块职责

**TUN Manager（不变）**
- 创建和管理虚拟网卡
- 路由规则配置
- 流量过滤

**sing-box Wrapper**
- 启动/停止 sing-box 进程
- 动态更新配置
- 监控连接状态

配置示例：
```json
{
  "inbounds": [
    {
      "type": "tun",
      "tag": "tun-in",
      "interface_name": "utun3",
      "inet4_address": "172.19.0.1/30"
    }
  ],
  "outbounds": [
    {
      "type": "hysteria2",
      "tag": "proxy",
      "server": "example.com",
      "server_port": 443,
      "password": "user_password",
      "tls": {
        "enabled": true,
        "server_name": "example.com"
      }
    }
  ]
}
```

**WalletManager**
- 创建/恢复钱包
- 钱包加密存储（macOS Keychain）
- Signer Session 管理
- x402 支付签名

**ProviderSwitcher**
- 从链上读取供应商列表
- 请求供应商配置
- 动态切换 sing-box outbound
- 不中断 TUN 层

**QuotaMonitor**
- 定期查询配额余额
- 配额不足提醒
- 触发购买流程

### 3.4 服务端架构

#### Hysteria2 节点

**职责**：
- 用户认证（基于 password）
- 流量转发
- 用户级流量统计（原生）
- 暴露 Traffic Stats API

**配置示例**：
```json
{
  "listen": ":443",
  "tls": {
    "cert": "/path/to/cert.pem",
    "key": "/path/to/key.pem"
  },
  "auth": {
    "type": "password",
    "password": "your_secret"
  },
  "trafficStats": {
    "listen": ":8080",
    "secret": "stats_secret"
  }
}
```

**Traffic Stats API**：
```bash
GET http://node:8080/traffic
Authorization: Bearer stats_secret

Response:
{
  "users": {
    "user_0x1234abcd": {
      "tx": 1048576,
      "rx": 5242880
    }
  }
}
```

#### Metering Service

**职责**：
- 定期拉取各节点 Traffic Stats API
- 计算流量增量
- 按用户 ID 汇总多节点流量
- 中心化扣减配额
- 配额不足时返回阻断指令

**数据流**：
```
1. 定时任务（每 10 秒）
   ↓
2. 并发拉取所有节点 Traffic Stats API
   ↓
3. 计算流量增量（当前值 - 上次值）
   ↓
4. 按用户 ID 汇总
   ↓
5. 原子性扣减数据库中的配额
   ↓
6. 返回配额状态（充足 / 不足）
```

#### Payment Service

**职责**：
- 验证 x402 支付签名
- 根据服务商价格兑换流量配额
- 记录购买历史
- 更新用户配额

#### Config Service

**职责**：
- 根据用户地址和配额状态生成 Hysteria2 配置
- 下发客户端配置
- 管理短期凭据

## 4. 模块交互关系

### 4.1 流量统计流

```
用户使用流量
    ↓
Hysteria2 节点统计（原生）
    ↓
暴露 Traffic Stats API
    ↓
Metering Service 定期拉取
    ↓
计算增量并汇总
    ↓
中心化扣减配额
    ↓
更新数据库
```

### 4.2 支付流

```
用户选择购买流量
    ↓
客户端检查 Signer Session
    ↓
生成 x402 支付签名
    ↓
调用 Payment Service
    ↓
验证签名和金额
    ↓
根据价格兑换配额
    ↓
更新用户配额
    ↓
返回购买结果
```

### 4.3 配置下发流

```
用户选择供应商
    ↓
客户端请求 Config Service
    ↓
Config Service 检查用户配额
    ↓
生成 Hysteria2 配置
    ↓
返回配置给客户端
    ↓
客户端更新 sing-box outbound
    ↓
重启 sing-box（TUN 层不变）
    ↓
连接建立
```

### 4.4 供应商切换流

```
用户点击切换供应商
    ↓
客户端请求新供应商配置
    ↓
Config Service 返回新配置
    ↓
客户端动态更新 sing-box outbound
    ↓
sing-box 重新连接
    ↓
TUN 层保持不变，用户无感知
```

## 5. 核心设计决策

### 5.1 为什么 TUN 层不需要改动？

```
TUN 层职责：
- 接收系统流量
- 根据路由规则转发

sing-box outbound 职责：
- 接收 TUN 转发的流量
- 使用特定协议（Hysteria2）发送到远程

两者解耦，修改 outbound 不影响 TUN
```

**架构变化**：
```
变化前（Shadowsocks）：
TUN → sing-box → Shadowsocks outbound → 服务器

变化后（Hysteria2）：
TUN → sing-box → Hysteria2 outbound → 服务器
     ↑ 不变              ↑ 只改这里
```

### 5.2 为什么使用中心化扣减？

**优点**：
- 简单直接，无需双向签名
- 无需凭条备份和恢复
- 原子性操作，避免超额使用
- 实时性要求低

**缺点**：
- 需要信任 Metering Service

**权衡**：
- 商用场景下，供应商运营 Metering Service
- 用户选择信任的供应商
- 平台提供开源代码，供应商可自行部署

### 5.3 为什么定期拉取而不是实时推送？

**定期拉取（推荐）**：
- 简单可靠
- 节点无需主动连接 Metering Service
- 容错性好（拉取失败可重试）

**实时推送**：
- 复杂度高
- 需要维护长连接
- 节点故障影响统计

**决策**：Phase 0 使用定期拉取，Phase 2 可选实时推送优化。

## 6. 扩展性设计

### 6.1 多供应商支持

```
用户可以同时：
- 在供应商 A 有 500MB 配额
- 在供应商 B 有 1GB 配额
- 在供应商 C 有 200MB 配额

每个供应商独立：
- 独立的 Metering Service
- 独立的配额管理
- 互不感知
```

### 6.2 多节点支持

```
供应商 A 部署：
- 节点 A1（美国）
- 节点 A2（欧洲）
- 节点 A3（亚洲）

Metering Service 汇总：
- 拉取 A1、A2、A3 的 Traffic Stats
- 按用户 ID 汇总
- 统一扣减配额
```

### 6.3 性能优化路径

**Phase 0-1**：
- 单机 Metering Service
- SQLite 数据库
- 定期拉取（10 秒）

**Phase 2+**：
- 分布式 Metering Service
- PostgreSQL + Redis
- 实时推送 + 定期对账

## 7. 安全设计

### 7.1 用户认证

- Hysteria2 使用 password 认证
- 每个用户独立密码
- 密码由 Config Service 生成
- 短期有效（24 小时）

### 7.2 流量统计安全

- Traffic Stats API 需要 secret
- 只有 Metering Service 可以访问
- 节点之间互不可见

### 7.3 支付安全

- x402 签名验证
- 防重放攻击
- 金额校验
