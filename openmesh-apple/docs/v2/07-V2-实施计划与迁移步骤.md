# 07-V2-实施计划与迁移步骤

## 1. 文档目的

本文档用于将 MeshFlux macOS 客户端 V2 的需求、总体架构和详细技术设计，转化为**可执行的实施计划与迁移步骤**。

本文档重点回答以下问题：

- V2 应按照什么顺序实施
- 每个阶段要修改哪些模块
- 每个阶段的产出物是什么
- 每个阶段完成后如何验证
- 哪些阶段之间存在依赖关系
- 出现问题时如何在阶段边界上停止、回退或延后

本文档不是编码规范，也不是测试用例清单。  
它是 V2 升级的**工程路线图**和**集成顺序控制文档**。

---

## 2. 实施原则

V2 的实施必须遵守以下原则：

### 2.1 先骨架，后能力，最后闭环
先完成导航、目录、状态骨架，再接入钱包、链上供应商、支付和安装闭环，避免一开始就把所有复杂度压到一起。

### 2.2 先低风险基础设施，后高风险业务集成
先完成区块链配置、协议、RPC、加密、Keychain、Store，再接入安装与支付闭环，降低调试复杂度。

### 2.3 每阶段都必须可独立验证
每个阶段结束后，都应具备清晰的可验证结果，而不是只“写了一半代码”。

### 2.4 不跨阶段修改高风险核心区
`VPNController.swift`、`VPNManager.swift`、现有 `TrafficMarketView.swift`、`ProfilesView.swift` 的核心逻辑不得在无明确必要的情况下被跨阶段改写。

### 2.5 保持主线可编译
任何阶段完成后，主工程都必须保持可编译、可运行、可回退。

### 2.6 尽量新增，少量改旧
优先通过新增文件实现 V2，尽量避免对现有成熟文件做大面积侵入式改造。

---

## 3. 实施范围与阶段边界

本次实施覆盖以下新增能力：

- 新增 `OpenMesh` 导航入口
- 新增 OpenMesh 主界面骨架
- 新增区块链基础设施层
- 新增钱包创建 / 恢复 / 解锁能力
- 新增链上供应商读取与展示能力
- 新增 x402 支付前置能力
- 新增商业供应商安装接入能力
- 新增 DIY 自建入口与导入接入

本次实施不覆盖以下内容：

- iOS 端区块链功能
- 供应商注册后台
- 法币购买入口
- 历史账单系统
- 多钱包 / 多账户
- 评分评论系统

---

## 4. 总体实施顺序概览

V2 建议按以下 9 个阶段推进：

1. **阶段 0：基线审计与工程准备**
2. **阶段 1：导航与 OpenMesh UI 骨架**
3. **阶段 2：区块链基础设施层**
4. **阶段 3：钱包安全层与钱包状态中心**
5. **阶段 4：链上供应商读取与展示**
6. **阶段 5：DIY 自建路径接入**
7. **阶段 6：x402 支付前置能力**
8. **阶段 7：安装接入与业务闭环**
9. **阶段 8：回归、稳定性与发布准备**

这 9 个阶段不是并行自由发挥，而是有明确依赖关系的串行主线。  
其中，**阶段 3、阶段 6、阶段 7** 是风险最高的三个阶段。

---

## 5. 阶段 0：基线审计与工程准备

## 5.1 目标

在正式开发前，确认当前 `main` 分支的真实工程基线，建立 V2 实施所需的目录、协议和最小工程准备工作。

## 5.2 主要任务

### 任务 0-1：确认现有源码落点
确认以下文件的真实存在与职责：

- `views/NavigationPage.swift`
- `views/TrafficMarketView.swift`
- `views/ProfilesView.swift`
- `views/ProviderInstallWizard.swift`
- `core/ProviderInstallWindowManager.swift`
- `core/VPNController.swift`
- `core/VPNManager.swift`
- `core/AppConfig.swift`
- `core/OfflineImportWindowManager.swift`

### 任务 0-2：创建新目录骨架
新建以下目录：

```text
MeshFluxMac/core/blockchain/
MeshFluxMac/core/blockchain/protocols/
MeshFluxMac/core/blockchain/models/
MeshFluxMac/core/blockchain/abi/
MeshFluxMac/core/blockchain/crypto/
MeshFluxMac/core/blockchain/rpc/
MeshFluxMac/core/blockchain/services/
MeshFluxMac/core/blockchain/stores/
MeshFluxMac/views/openmesh/
```

### 任务 0-3：建立协议文件
先创建空骨架或最小实现：

- `BlockchainRPCClientProtocol.swift`
- `KeychainManagerProtocol.swift`
- `URLSessionProtocol.swift`

### 任务 0-4：建立错误与配置文件
创建：

- `BlockchainConfig.swift`
- `BlockchainError.swift`

### 任务 0-5：测试目录准备
在测试 target 中预建：

```text
MeshFluxMacTests/blockchain/
MeshFluxMacTests/blockchain/Mocks/
```

## 5.3 输出物

- 新目录全部建立
- 协议骨架可编译
- `BlockchainConfig.swift` / `BlockchainError.swift` 可编译
- 工程结构已具备后续承载条件

## 5.4 验收标准

- 工程可编译
- 新目录和新文件已进入 target
- 不影响现有 App 运行
- 现有 Dashboard / Profiles / TrafficMarket 不受影响

## 5.5 风险与控制

### 风险
- 新文件未加入 target
- 新目录命名不统一
- 过早引入业务逻辑导致基线不稳定

### 控制措施
- 阶段 0 只允许做骨架与协议准备
- 不允许在本阶段接入 UI 业务逻辑
- 不允许修改 VPN 相关逻辑

## 6.2 主要任务

### 任务 1-1：修改 `NavigationPage.swift`
新增：

- `case openMesh`
- 标题：`OpenMesh`
- 图标
- `contentView` 挂载 `OpenMeshView`

### 任务 1-2：创建 OpenMesh 基础 UI 文件
新增：

- `OpenMeshView.swift`
- `OpenMeshTopBarView.swift`
- `OpenMeshSegmentControl.swift`
- `OpenMeshSupplierListView.swift`
- `OpenMeshDIYView.swift`

### 任务 1-3：实现静态分段切换
实现：
- `链上供应商`
- `DIY 自建`

先使用静态占位数据，不接真实供应商。

### 任务 1-4：接入主题
确保新增 UI 使用 `MeshFluxTheme.swift` 的既有风格。

## 6.3 当前阶段允许的状态

此阶段允许出现：

- 顶部钱包状态为静态占位
- 供应商列表为 mock 数据
- DIY 页面按钮为空实现或打印日志

此阶段不允许出现：

- 真实 RPC 调用
- 真实钱包逻辑
- 真实支付逻辑
- 安装接入逻辑

## 6.4 输出物

- 用户能在导航栏中看到 `OpenMesh`
- 点进去后能看到完整页面骨架
- 可切换两个分段
- 不破坏原有导航行为

## 6.5 验收标准

- `NavigationPage` 可编译
- `OpenMeshView` 可展示
- 导航切换正常
- VPN 连接状态逻辑不受影响
- `groups / connections / logs / profiles` 行为不变

## 6.6 风险与控制

### 风险
- 新导航插入后导致原有排序混乱
- OpenMesh 页面样式与现有产品割裂
- 把链上逻辑写进 View 里

### 控制措施
- 本阶段 UI 只做静态骨架
- 不允许从 View 内部直接访问外部网络
- 所有真实能力延后到后续阶段

## 7. 阶段 2：区块链基础设施层

## 7.1 目标

建立链上调用的最小基础设施，使后续钱包、供应商和支付模块有统一底层可用。

## 7.2 主要任务

### 任务 2-1：实现 `BlockchainRPCClient.swift`
支持最小 JSON-RPC 能力：

- `eth_call`
- `eth_getLogs`
- USDC `balanceOf`

### 任务 2-2：实现 `ContractABI.swift`
支持 V2 所需最小 ABI 编解码：

- `address`
- `uint256`
- `bool`
- `bytes32`
- `string`
- tuple 与基础数组（按需要最小实现）

### 任务 2-3：实现链配置
在 `BlockchainConfig.swift` 中配置：

- Base mainnet / sepolia
- USDC 地址
- Protocol Registry 地址
- x402 网络名

### 任务 2-4：实现基础测试
新增以下测试：

- `BlockchainRPCClientTests.swift`
- `ContractABITests.swift`

## 7.3 输出物

- 可从统一 RPC Client 发起只读调用
- 可解码基本合约返回
- 可调用 USDC `balanceOf`
- 基础测试可运行

## 7.4 验收标准

- 不依赖真实 UI 即可测试
- `BlockchainRPCClientProtocol` 可被 Mock 替换
- ABI 编解码有确定测试向量
- RPC 层错误能统一映射到 `BlockchainError`

## 7.5 风险与控制

### 风险
- 直接把业务逻辑写入 RPC 层
- ABI 编解码过度设计
- 过早依赖真实链环境

### 控制措施
- RPC 层只做基础通信与解码
- 只实现 V2 需要的最小 ABI
- 所有对外接口必须可 Mock

## 8. 阶段 3：钱包安全层与钱包状态中心

## 8.1 目标

完成钱包创建、恢复、解锁、锁定和余额读取能力，并建立 `MeshWalletStore` 作为钱包统一状态中心。

## 8.2 主要任务

### 任务 3-1：实现 `WalletCrypto.swift`
支持：

- BIP-39 12 词助记词生成
- BIP-44 地址派生
- 私钥加密 / 解密
- 地址计算

### 任务 3-2：实现 `KeychainManager.swift`
支持：

- 保存加密私钥
- 保存 salt
- 保存 `MeshWalletIdentity`
- 读取钱包身份
- 判断钱包是否存在

### 任务 3-3：实现 `MeshWalletStore.swift`
支持：

- `checkWalletState()`
- `createWallet(...)`
- `restoreWallet(...)`
- `unlock(...)`
- `unlockWithBiometrics()`
- `lock()`
- `refreshBalance()`
- `handleAppDidEnterBackground()`

### 任务 3-4：创建钱包相关 UI
新增：

- `MeshWalletSetupView.swift`
- `MeshWalletUnlockView.swift`
- `MeshWalletAddressSheet.swift`

### 任务 3-5：接入 OpenMesh 顶部状态条
让 `OpenMeshTopBarView` 从静态占位改为真实状态驱动。

### 任务 3-6：实现后台自动锁定
在 App 生命周期中或通过通知，调用 `walletStore.handleAppDidEnterBackground()`。

## 8.3 输出物

- 可创建钱包
- 可恢复钱包
- 可解锁 / 锁定
- 可显示地址与余额
- 应用退后台自动锁定

## 8.4 验收标准

- 助记词仅显示一次
- 私钥不明文存储
- 解锁成功后可刷新余额
- 后台后状态回到 locked
- 相关测试通过：
  - `MeshWalletStoreTests.swift`
  - `WalletCrypto` 测试
  - `KeychainManager` Mock 测试

## 8.5 风险与控制

### 风险
- 助记词误落盘
- 解锁失败后出现半解锁状态
- 把 Keychain 细节暴露到 UI
- 密钥在多个对象间乱传

### 控制措施
- 钱包能力统一收口到 `MeshWalletStore`
- 敏感数据仅通过 Keychain 和内存短驻留
- 不允许在 View 层保存私钥或助记词

## 9. 阶段 4：链上供应商读取与展示

## 9.1 目标

用真实链上数据替换 OpenMesh 的静态供应商占位数据，完成链上供应商读取、metadata 合并、缓存和错误态展示。

## 9.2 主要任务

### 任务 4-1：实现 `SupplierMetadataService.swift`
支持：
- 拉取 metadata JSON
- 校验 JSON 结构
- 返回 `SupplierMetadata`

### 任务 4-2：实现 `OnChainSupplierService.swift`
支持：
- `eth_getLogs` 读取注册事件
- 筛选活跃供应商
- 调只读函数读取 supplier 数据
- 合并 metadata
- 缓存结果

### 任务 4-3：实现 `OnChainSupplierViewModel`
支持状态：

- `idle`
- `loading`
- `loaded`
- `empty`
- `error`
- `refreshing`

### 任务 4-4：实现供应商卡片 UI
新增：

- `OpenMeshSupplierCardView.swift`

并把 `OpenMeshSupplierListView` 接入真实状态。

### 任务 4-5：安装状态融合
将现有已安装 provider / profile 状态与链上供应商列表进行融合展示，标记“已安装”。

## 9.3 输出物

- OpenMesh 中能看到真实链上供应商列表
- 列表支持刷新
- metadata 解析成功则展示完整卡片
- metadata 失败时仍可降级显示基础信息

## 9.4 验收标准

- 供应商列表能从真实或 Mock 链返回
- 页面有完整 loading / empty / error 状态
- 不因单个 metadata 异常而导致整个列表崩溃
- `OnChainSupplierServiceTests.swift` 可通过

## 9.5 风险与控制

### 风险
- RPC 调用慢导致 UI 卡顿
- metadata 不规范导致崩溃
- 列表刷新和缓存状态混乱

### 控制措施
- 所有链上调用放后台
- ViewModel 统一状态机
- metadata 失败单项降级，不全局失败
- 缓存 TTL 固定，避免重复读链

## 10. 阶段 5：DIY 自建路径接入

## 10.1 目标

把 OpenMesh 中的 DIY 分段从静态说明页，升级为真正可用的引导入口与导入接入点。

## 10.2 主要任务

### 任务 5-1：完善 `OpenMeshDIYView.swift`
加入：

- 大标题
- 引导说明
- 商业 / DIY 模式对比
- 主按钮：打开自建教程
- 次按钮：直接导入本地配置

### 任务 5-2：接入外部教程链接
实现按钮打开官方教程网页。

### 任务 5-3：接入现有导入能力
让次按钮调用既有：

- `OfflineImportWindowManager`
- 或现有 Import Profile 入口

### 任务 5-4：校验降级路径
在链上不可用、钱包未创建、RPC 失败时，DIY 区域仍可正常使用。

## 10.3 输出物

- DIY 页面完整可用
- 用户可打开自建教程
- 用户可直接进入本地配置导入路径

## 10.4 验收标准

- DIY 路径不依赖钱包状态
- DIY 路径不依赖链上服务
- 即使供应商列表失败，DIY 仍可用
- 不影响既有离线导入能力

## 10.5 风险与控制

### 风险
- DIY 页面被错误绑定到钱包状态
- 导入按钮误改现有导入逻辑
- 说明页和产品术语不统一

### 控制措施
- DIY 仅作为新入口，不重写导入实现
- 保持与现有 import / offline import 逻辑解耦
- 文案统一使用 V2 需求术语

## 11. 阶段 6：x402 支付前置能力

## 11.1 目标

建立商业供应商配置拉取前的支付授权能力，使“使用该供应商”不再只是按钮展示，而是具备真实支付前置。

## 11.2 主要任务

### 任务 6-1：实现 `EIP712Signer.swift`
支持：
- digest 构造
- secp256k1 签名
- 基础时间窗口校验

### 任务 6-2：实现 `X402PaymentClient.swift`
支持：
- 构造 `X402Authorization`
- 使用授权头请求配置资源
- 处理 402 / 重试逻辑

### 任务 6-3：新增支付确认 Sheet
新增：

- `SupplierPaymentConfirmSheet.swift`

### 任务 6-4：实现支付上下文模型
用于在 UI 中承载：

- 供应商名称
- 当前余额
- 预计费用
- 用户确认动作

### 任务 6-5：把“使用该供应商”按钮接入支付前置
当前阶段先做到：
- 钱包状态检查
- 余额检查
- 弹出确认框
- 确认后获取 `X402Authorization`

此阶段**还不要求**完成安装接入闭环。

## 11.3 输出物

- 可生成授权对象
- 可发起带 x402 头的请求
- 用户点“使用该供应商”时会进入支付确认
- 余额不足会阻断

## 11.4 验收标准

- `EIP712SignerTests.swift` 通过
- `X402PaymentClientTests.swift` 通过
- 支付确认 UI 可展示真实金额与余额
- 未确认前不会继续执行请求
- 钱包锁定时会先要求解锁

## 11.5 风险与控制

### 风险
- 时间窗口失效导致授权不可用
- 签名和地址不匹配
- 余额检查与 UI 展示单位不一致
- 支付逻辑直接写进 View

### 控制措施
- 支付前置逻辑统一下沉到 `X402PaymentClient` + Coordinator
- 金额统一使用精确类型
- UI 只做确认与结果展示

## 12. 阶段 7：安装接入与业务闭环

## 12.1 目标

把“支付成功后的配置获取”接入现有安装闭环，形成完整的商业供应商使用链路。

## 12.2 主要任务

### 任务 7-1：实现 `ConfigAcquisitionService.swift`
支持：
- 选择有效 `configURL`
- 发起带授权头的请求
- 获取配置原始数据
- 将结果交给安装接入点

### 任务 7-2：为 `ProviderInstallWindowManager.swift` 增加接入点
新增建议方法：

```swift
@MainActor
func installImportedProviderConfig(
    supplierID: String,
    displayName: String,
    configData: Data
) async throws
```

### 任务 7-3：实现 `X402PaymentFlowCoordinator`
串联完整流程：

- 钱包检查
- 余额检查
- 支付确认
- 授权生成
- 配置获取
- 安装接入
- 状态返回

### 任务 7-4：把供应商卡片主按钮接到 Coordinator
使用户真正能从卡片完成一次完整闭环。

### 任务 7-5：安装完成后刷新安装状态
安装成功后：

- 刷新供应商列表“已安装”状态
- 让结果能在既有配置体系中被识别

## 12.3 输出物

- 用户可以完成：
  - 选择供应商
  - 钱包检查
  - 支付确认
  - 配置获取
  - 安装接入
- 安装成功后结果进入现有体系

## 12.4 验收标准

- 真实或 Mock 配置下载后能进入既有安装终点
- 安装失败时不写脏状态
- 支付失败时不进入安装流程
- 已安装状态可正确回显
- 不破坏既有 `ProviderInstallWizard` 行为

## 12.5 风险与控制

### 风险
- 现有安装终点接入方式不兼容
- 配置数据格式与旧安装链路不一致
- 失败后 provider/profile 映射半写入
- OpenMesh 自己复制出第二套安装逻辑

### 控制措施
- 统一以 `ProviderInstallWindowManager` 为安装接入点
- 不允许在 OpenMesh 页面里直接实现安装落地
- 安装前后状态写入必须集中控制

## 17. 停止点与回退策略

## 17.1 安全停止点

以下阶段结束后，都可以作为安全停止点：

- 阶段 1：只有导航与静态 UI
- 阶段 3：钱包能力闭环，但未接支付
- 阶段 5：供应商浏览 + DIY 完成，但未接支付安装
- 阶段 6：支付前置完成，但未接安装终点

### 原则
如果某阶段风险过高，可在该阶段边界停止，不强行推进下一阶段。

---

## 17.2 回退策略

### 导航级回退
若 OpenMesh 主流程严重不稳定，可先隐藏 `.openMesh` 入口，保留代码但不开放 UI。

### 支付级回退
若 x402 支付链路不稳定，可在阶段 6 停止，不进入阶段 7 安装接入。

### 安装级回退
若现有安装终点接入存在高风险，可保留支付与浏览能力，延后商业安装闭环。

### 安全级回退
若钱包密钥保护未达到要求，必须回退到“只读浏览，不开放钱包与支付”。

---

## 18. 管理要求与执行纪律

实施过程中，必须遵守以下纪律：

1. 每个阶段提交前必须可编译  
2. 每个阶段提交后必须有明确验证结果  
3. 未完成当前阶段验收前，不得跳阶段做下一阶段核心逻辑  
4. 不允许为赶进度把复杂业务逻辑直接塞进 View  
5. 不允许为了省事把敏感数据塞进 SharedPreferences  
6. 不允许为了快速闭环而复制现有安装逻辑形成第二套实现  
7. 不允许在没有测试替身的情况下把真实链上依赖写死到核心模块

---

## 19. 关键结论

V2 的实施不能以“功能堆砌”的方式进行，而必须按**骨架 → 基础设施 → 钱包 → 供应商 → DIY → 支付 → 安装闭环 → 回归发布**的顺序推进。

这样做的核心价值在于：

- 风险逐层暴露，而不是一次性爆炸
- 每阶段都能独立验证与止损
- 现有 VPN / 配置 / 安装核心能力始终被隔离保护
- OpenMesh 新能力可以逐步生长，而不是推倒现有产品重来

这份实施计划既是开发顺序控制文档，也是 V2 项目的阶段验收基线。
