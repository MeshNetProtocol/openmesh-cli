# 07-V2-实施计划与迁移步骤

## 1. 文档目的

本文档定义 V2 的实施顺序。  
重点不是按当前代码最少修改量排序，而是按产品战略和冷启动节奏排序。

---

## 2. 实施原则

1. 官方临时受限访问和 DIY 先保持可用
2. 商用供应商架构按第一性原理重设
3. 优先建立”链上发现 + 预付费购买 + 快速切换”的主闭环
4. 采用预付费模型，中心化扣减流量配额
5. 平台提供开源记账服务代码，服务商可自行部署

---

## 3. 分阶段实施

### 阶段 A：文档与数据模型定稿

目标：

- 定义 provider metadata（包括价格信息）
- 定义 TrafficQuota（流量配额）
- 定义 PurchaseRecord（购买记录）
- 定义 signer session

### 阶段 B：客户端钱包和 signer session

目标：

- 创建 / 恢复钱包
- 设备绑定加密存储
- 解锁一次后支持预付费支付

### 阶段 C：链上供应商发现

目标：

- Base 主网 / 测试网切换
- registry 读取
- metadata 解析（包括价格信息）

### 阶段 D：商用供应商参考服务端

目标：

- Config Service
- Payment Service（验证 x402 支付，兑换流量配额）
- Metering Service（流量汇总，中心化扣减）
- sing-box Adapter（流量统计上报）

### 阶段 E：预付费购买流量

目标：

- 用户通过 x402 支付 USDC
- 服务商验证支付并兑换流量配额
- 流量配额永久有效，直到用完

### 阶段 F：OpenRouter 式快速切换

目标：

- 商用供应商间快速切换
- 不反复走传统安装向导
- 当前供应商状态正确刷新

### 阶段 G：异步对账

目标：

- 异步链上对账
- 提现手续费逻辑预留

---

## 4. 冷启动推广节奏对齐

### 第一阶段：先有人能用

- 官方临时受限访问
- DIY 自建节点

### 第二阶段：再有人愿意试商用

- 链上供应商目录
- 钱包和预付费购买

### 第三阶段：验证快速切换和复购

- 预付费购买流量
- 供应商切换
- 追加购买

### 第四阶段：再挂 token 经济

- 真实用量数据沉淀
- token 激励和平台分成扩展
