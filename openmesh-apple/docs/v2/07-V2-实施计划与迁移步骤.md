# 07-V2-实施计划与迁移步骤

## 1. 文档目的

本文档定义 V2 的实施顺序。  
重点不是按当前代码最少修改量排序，而是按产品战略和冷启动节奏排序。

---

## 2. 实施原则

1. 官方临时受限访问和 DIY 先保持可用
2. 商用供应商架构按第一性原理重设
3. 优先建立“链上发现 + 自动支付 + 快速切换”的主闭环
4. 不把中心化账本作为必做前提

---

## 3. 分阶段实施

### 阶段 A：文档与消息模型定稿

目标：

- 定义 provider metadata
- 定义 PaymentReceipt / UsageReceipt
- 定义 signer session
- 定义流量桶和预续费阈值

### 阶段 B：客户端钱包和 signer session

目标：

- 创建 / 恢复钱包
- 设备绑定加密存储
- 解锁一次后支持自动小额支付

### 阶段 C：链上供应商发现

目标：

- Base 主网 / 测试网切换
- registry 读取
- metadata 解析

### 阶段 D：商用供应商参考服务端

目标：

- Config Service
- Metering Service
- Receipt Service
- sing-box Adapter

### 阶段 E：x402 流量桶续费

目标：

- 免费桶
- 阈值触发下一桶 `402`
- 自动支付
- 支付后立即放行

### 阶段 F：OpenRouter 式快速切换

目标：

- 商用供应商间快速切换
- 不反复走传统安装向导
- 当前供应商状态正确刷新

### 阶段 G：凭条恢复与对账

目标：

- 客户端和供应商双边恢复
- 异步链上对账
- 提现手续费逻辑预留

---

## 4. 冷启动推广节奏对齐

### 第一阶段：先有人能用

- 官方临时受限访问
- DIY 自建节点

### 第二阶段：再有人愿意试商用

- 链上供应商目录
- 钱包和自动支付

### 第三阶段：验证快速切换和复购

- 流量桶续费
- 供应商切换
- 凭条恢复

### 第四阶段：再挂 token 经济

- 真实用量凭条沉淀
- token 激励和平台分成扩展
