# 04-V2-总体架构设计

## 1. 文档目的

本文档定义 V2 的系统级架构，重点覆盖商用供应商场景。

---

## 2. 总体架构

```text
┌────────────────────────────────────────────────────────────┐
│                      Discovery Plane                       │
│ Base Registry Contract | Provider Metadata | Price Info    │
└────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌────────────────────────────────────────────────────────────┐
│                       Client Plane                         │
│ OpenMesh UI | Wallet | Signer Session | Local TUN         │
│ Config Installer | Provider Switcher | Purchase Flow      │
└────────────────────────────────────────────────────────────┘
              │                    │
              ▼                    ▼
┌──────────────────────┐   ┌────────────────────────────────┐
│   Payment Flow       │   │        Data Plane              │
│ x402 Prepayment      │   │ sing-box Cluster | User/Auth   │
│ USDC → Quota         │   │ Traffic Stats | Report         │
└──────────────────────┘   └────────────────────────────────┘
              │                    │
              └────────────┬───────┘
                           ▼
┌────────────────────────────────────────────────────────────┐
│                    Provider Plane                          │
│ Payment Service | Metering Service | Reconciliation        │
│ Quota Management | Provider Contract Withdrawal           │
└────────────────────────────────────────────────────────────┘
```

---

## 3. 设计原则

1. Discovery 去中心化
2. 用户直接向供应商预付 USDC 购买流量
3. 流量配额永久有效，直到用完
4. 供应商独立定价，价格差异化竞争
5. 中心化扣减流量配额，无需双向签名和凭条备份
6. 平台抽成在供应商提现时发生
7. 钱包安全优先于自动支付便利
8. 本地 `tun` 保持不变，商用变化主要发生在远端配置和支付流程
9. 平台提供开源记账服务代码，供应商可自行部署

---

## 4. Discovery Plane

职责：

- 供应商注册
- metadata 发现
- signer 公钥发现
- 主网 / 测试网切换

链上注册合约建议包含：

- `providerId`
- `owner`
- `metadataURL`
- `configServiceURL`
- `meteringServiceURL`
- `priceRate` (如 1 USDC = 100MB)

---

## 5. Client Plane

职责：

- 钱包创建 / 恢复 / 解锁
- signer session 管理
- 读取供应商目录（包括价格信息）
- 预付费购买流量（通过 x402）
- 请求配置
- 安装配置
- 查询流量配额余额

注意：

- 客户端继续保留官方临时受限访问和私人节点路径
- 商用路径不再被当前 Shadowsocks 编辑器和节点 UI 结构限制

---

## 6. Provider Plane

每个商用供应商至少应包含以下逻辑模块：

### 6.1 Config Service

职责：

- 根据用户地址和当前会话签发数据平面配置
- 返回短期有效配置

### 6.2 Metering Service

职责：

- 接收节点上报的流量统计
- 按用户 ID 汇总流量（支持多节点并行）
- 中心化扣减流量配额
- 配额不足时返回阻断指令

### 6.3 Payment Service

职责：

- 验证 x402 支付签名
- 根据服务商价格兑换流量配额
- 记录支付历史

### 6.4 Reconciliation Watcher

职责：

- 异步对账
- 审计链上支付是否与记账服务记录一致

注意：

- 它不负责热路径放行

### 6.5 Provider Contract Withdrawal

职责：

- 管理供应商收入提现
- 在提现时向平台结算手续费

注意：

- 平台手续费发生在供应商提现阶段
- 不发生在用户支付热路径

---

## 7. Data Plane

`sing-box` 的职责被限制为：

- 用户或会话认证
- 流量转发
- 用量统计
- 动态停供

不负责：

- 最终账本
- 平台 token
- 供应商注册
- 供应商发现

---

## 8. 商用场景主流程

### 8.1 选择供应商

1. 客户端读取链上 registry
2. 拉取 provider metadata（包括价格信息，如 1 USDC = 100MB）
3. 用户点击”使用该供应商”

### 8.2 预付费购买流量

1. 客户端检查 signer session
2. 用户选择购买流量（如支付 5 USDC）
3. 客户端通过 x402 签名支付
4. 供应商记账服务验证支付并兑换成流量配额（如 5 USDC × 100MB = 500MB）
5. 用户获得流量配额，永久有效直到用完

### 8.3 使用流量

1. 客户端请求供应商配置
2. 供应商检查用户流量配额
3. 如果配额充足，供应商下发配置
4. 客户端安装配置并开始使用
5. 供应商节点统计流量并定期上报到记账服务
6. 记账服务中心化扣减流量配额

### 8.4 配额不足

1. 记账服务检测到用户配额不足
2. 返回”流量配额不足”状态
3. 供应商节点阻断用户连接
4. 客户端提示用户购买更多流量

### 8.5 追加购买

1. 用户选择继续购买流量（如再支付 3 USDC）
2. 客户端通过 x402 签名支付
3. 供应商记账服务验证支付并追加流量配额（如 3 USDC × 100MB = 300MB）
4. 用户总配额增加，可以继续使用

**关键特性**:
- ✅ 预付费模型，无需实时计费
- ✅ 中心化扣减，无需双向签名和凭条备份
- ✅ 流量永久有效，直到用完
- ✅ 支持追加购买，配额累加

---

## 9. 官方临时受限访问与商用场景的关系

官方临时受限访问不是另一套系统，而是第一个官方 provider。  
差异只在于：

- 可以继续使用简单单节点脚本
- 可无钱包使用
- 可简化流量显示和限额
- 不要求完整商用凭条和计费闭环
