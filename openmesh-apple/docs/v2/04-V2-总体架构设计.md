# 04-V2-总体架构设计

## 1. 文档目的

本文档定义 MeshFluxMac V2 的系统级架构。  
重点回答：

- 新能力应该放在哪里
- 如何与旧系统并存
- 商用供应商、私人节点、流豆账户和授权支付如何协同

---

## 2. 架构目标

### 2.1 产品目标

让 `OpenMesh` 成为统一路由入口：

- 官方受限通道像平台内置免费模型一样承担冷启动和兜底
- 商用供应商像云模型一样被浏览和切换
- 私人节点像本地模型一样被接入和启用
- 流豆账户与授权支付为商用路径提供底层能力

### 2.2 工程目标

- 保留旧系统稳定能力
- 复用现有配置与 VPN 终点
- 新能力主要以新增模块和重构 OpenMesh 实现
- 整体结构支持灰度与回滚

---

## 3. 架构前提

### 3.1 当前 OpenMesh 入口已存在

真实代码中，`OpenMesh` 已经作为菜单栏 Tab 存在，并挂接到 `OpenMeshView()`。  
因此架构设计应以“升级已有入口”为前提。

### 3.2 旧系统必须并行保留

以下能力在本轮是并行保留的：

- Dashboard
- Market
- Profiles
- Offline Import
- 现有 VPN 控制

### 3.3 现有安装链路必须复用

商用供应商最终必须接入现有配置安装与 VPN 选择体系，而不是自造独立终点。

## 5. 分层设计

## 5.1 App Shell 层

职责：

- 承载菜单栏和现有主入口
- 保持旧页面可用
- 提供 OpenMesh 的挂接点

对应真实代码：

- `OpenMeshMacApp.swift`
- 现有菜单栏 Tab

## 5.2 OpenMesh 体验层

职责：

- 展示当前路由供应商
- 展示流豆账户状态
- 展示官方受限通道状态
- 承载商用供应商与私人节点两个分段
- 承载确认 Sheet、异常提示和切换结果

对应方向：

- `views/openmesh/OpenMeshView.swift`
- `OpenMeshTopBarView.swift`
- `OpenMeshSupplierListView.swift`
- `OpenMeshDIYView.swift`

## 5.3 Official Access 层

职责：

- 管理官方受限通道配置
- 管理每日限额
- 为冷启动和兜底提供默认连接来源

关键要求：

- 冷启动阶段优先按设备匿名标识限额
- 账户创建后可叠加链上地址做风控
- 可由自建最小节点或合规 B2B 上游承载

## 5.4 Provider Catalog 层

职责：

- 读取商用供应商目录
- 合并元数据
- 输出可直接渲染的供应商卡片模型

输入：

- 链上注册表
- metadata
- 本地缓存

输出：

- 商用供应商列表
- 当前供应商候选集

## 5.5 Private Node Hub 层

职责：

- 管理私人节点入口
- 复用本地导入能力
- 输出私人节点卡片与当前启用状态

输入：

- 本地导入配置
- 用户维护的私人节点定义

输出：

- 私人节点列表
- 可启用的私人节点路由

## 5.6 Wallet / Payment 层

职责：

- 管理流豆账户状态
- 执行余额读取
- 触发授权支付
- 为商用供应商切换提供支付凭证

输入：

- Keychain 中的敏感信息
- Base 链资产与授权数据

输出：

- 流豆账户状态
- 授权支付结果

## 5.7 Integration 层

职责：

- 将商用供应商和私人节点的结果接回旧系统
- 触发现有配置安装或导入逻辑
- 回写当前路由状态

关键要求：

- 不接管 VPN 控制层
- 不复制第二套安装流程

## 6. 核心业务流

### 6.1 官方受限通道流

1. 用户首次进入 OpenMesh
2. 系统发现当前没有任何可用配置或现成路由
3. Official Access 层根据设备匿名标识判定当日剩余额度
4. 若额度可用，则为用户提供官方受限通道
5. OpenMesh 顶部将当前路由显示为官方受限通道
6. 用户可在此基础上继续访问目录、链上和商用切换流程

### 6.2 商用供应商流

1. 用户进入 OpenMesh
2. 体验层显示当前路由供应商与流豆账户
3. Provider Catalog 层加载商用供应商列表
4. 用户选择供应商并点击“使用 / 切换”
5. Wallet / Payment 层检查账户、余额、授权支付
6. Integration 层获取配置并接入现有安装链路
7. 当前路由供应商状态回写到体验层

### 6.3 私人节点流

1. 用户进入 OpenMesh
2. 切换到“私人节点”
3. 查看已有私人节点或进入导入 / 引导
4. 选择或导入节点
5. Integration 层调用现有导入 / 配置体系
6. 当前路由来源更新为私人节点

### 6.4 当前路由同步流

系统应维护一个面向 OpenMesh 的“当前路由视图模型”，用于把以下来源统一成同一类状态：

- 当前官方受限通道
- 当前商用供应商
- 当前私人节点
- 当前未知 / 遗留配置

这样 OpenMesh 顶部卡片不会因为来源不同而割裂。

## 7. 与旧系统的关系

### 7.1 与旧 Market 的关系

- 旧 Market 本轮保留
- OpenMesh 不应把旧 Market 的逻辑硬塞进去
- 新链路稳定前，旧 Market 是对照组和回退路径

### 7.2 与 Profiles / Offline Import 的关系

- 私人节点优先复用现有导入和配置能力
- Profiles 继续承担配置可见与可管理职责
- OpenMesh 只负责新体验入口，不替代所有旧管理页

### 7.3 与 VPN 控制层的关系

- `VPNController.swift` / `VPNManager.swift` 保持边界
- OpenMesh 只产出“应该接入什么配置”和“当前路由怎么展示”
- 真正的连接控制仍然交给现有系统

---

## 8. 数据归属

### 8.1 面向用户的状态

- 当前路由供应商
- 官方受限通道状态
- 流豆账户状态
- 商用供应商列表
- 私人节点列表
- 支付与接入阶段状态

### 8.2 敏感状态

- 钱包密钥材料
- 授权签名
- 支付上下文
- 私人节点敏感配置内容

### 8.3 复用旧状态

- 已安装配置
- 已选配置
- VPN 是否连接
- 旧市场安装结果

---

## 9. 灰度与回滚架构

### 9.1 回滚层次

系统应支持按以下层次关闭新能力：

1. 整体隐藏 OpenMesh 商用功能
2. 仅保留官方受限通道和私人节点
3. 仅关闭授权支付能力
4. 保留私人节点，关闭商用供应商
5. 完全回退到旧 Market + 旧 Profiles

### 9.2 原则

- 新链路必须可关闭
- 关闭后旧系统仍能正常运行
- 数据和 UI 不应因回退而互相污染

---

## 10. 架构结论

V2 的正确架构不是“用区块链模块改写整个 VPN 客户端”，而是：

- 以现有 `MeshFluxMac` 为壳
- 以 `OpenMesh` 为新体验入口
- 以官方受限通道先解决冷启动
- 以商用供应商目录、私人节点中心、流豆账户和授权支付为新增能力层
- 最终把结果安全接回现有安装与 VPN 控制体系
