# 04-V2-总体架构设计

## 1. 文档目的

本文档用于定义 MeshFlux macOS 客户端 V2 升级的总体架构设计，说明系统在引入 OpenMesh、金豆账户、链上供应商和 x402 支付能力后，应如何在现有代码基础上进行增量扩展。

本文档重点回答以下问题：

- V2 升级后的系统边界是什么
- 新旧模块之间如何协作
- 新增模块应如何分层
- 哪些现有模块必须复用，哪些新模块必须新增
- 关键业务链路应如何穿过系统架构

本文档不展开到具体类实现、接口签名和测试代码层级，这些内容在后续详细技术设计与测试策略文档中定义。

---

## 2. 架构设计目标

V2 总体架构必须同时满足以下目标：

### 2.1 增量扩展
在不破坏 V1 既有 VPN、配置管理、供应商安装和本地导入能力的前提下，增加 OpenMesh 新能力。

### 2.2 分层清晰
新架构必须将界面、业务流程、区块链调用和本地安全持久化明确分层，避免视图层直接承担网络、签名和密钥处理责任。

### 2.3 用户抽象优先
底层引入钱包、链上目录、USDC 和 x402 支付等能力，但面向用户应被抽象为“金豆账户”“供应商”“使用该供应商”等可理解表达。

### 2.4 可复用现有终点
链上供应商安装流程应尽量复用现有配置安装终点，而不是推翻当前安装向导和 VPN 接入闭环。

### 2.5 可降级
链上能力不可用时，OpenMesh 不应拖垮整个客户端。DIY、自有配置和既有 VPN 主流程必须仍可继续使用。

### 2.6 安全隔离
钱包私钥、解锁态和签名能力必须与现有普通配置状态分离，采用更高安全等级的本地管理边界。

---

## 3. 架构前提与现实约束

V2 总体架构建立在以下现实前提之上：

### 3.1 现有主导航不可被重构替换
当前主导航已经稳定存在，V2 采用“插入一个新入口”的方式扩展，而不是重写整个导航系统。

### 3.2 现有安装体系是可复用资产
当前已存在推荐供应商展示、市场管理、安装窗口和安装向导，说明“安装”不是空白能力。V2 应在安装前增加钱包与支付前置，而不是重做安装终点。

### 3.3 现有 VPN 控制层属于高风险核心区
`VPNController`、`VPNManager` 等核心控制链路必须尽量保持不变。V2 的职责是让新的供应商配置最终能进入既有控制系统，而不是替换控制系统。

### 3.4 现有本地配置体系必须保留
`Profiles`、离线导入、已有 provider/profile 映射关系都是当前系统稳定资产。V2 必须与其兼容。

### 3.5 区块链能力当前基本为空白
现有代码中尚未形成钱包、合约调用、签名和 x402 支付层，因此 V2 需要新增一组独立基础设施模块。

## 4. V2 系统边界

V2 架构涉及四类系统边界。

### 4.1 客户端内部边界
即 MeshFlux macOS 客户端自身，负责：

- OpenMesh 主界面
- 钱包状态与解锁
- 链上供应商列表展示
- 支付确认
- 配置获取前置
- 与既有安装流程的衔接

### 4.2 链上边界
即 Base 链及其相关协议合约，负责：

- 供应商目录注册信息
- 供应商状态查询
- USDC 余额读取
- 支付授权所依赖的链上语义约束

### 4.3 供应商侧边界
即供应商 metadata 与配置服务，负责：

- 提供供应商展示信息
- 提供受授权保护的配置下载入口
- 接受 x402 授权并返回配置数据

### 4.4 既有客户端能力边界
即当前已存在且需复用的本地能力，主要包括：

- 配置安装与管理
- provider/profile 状态映射
- VPN 启停与扩展控制
- 本地导入与离线安装能力

## 5. 总体架构原则

### 5.1 单向依赖原则
所有新模块必须遵守单向依赖：

Views  
→ ViewModels / Services  
→ Blockchain Layer  
→ Keychain / SharedPreferences / Existing Core Services

### 5.2 复用终点原则
链上能力只负责“把商业供应商路径带到安装终点”，不负责替代安装终点本身。

### 5.3 新能力隔离原则
OpenMesh 新增逻辑应尽量集中在新增文件和新增模块中，避免把现有成熟模块改造成高耦合复杂体。

### 5.4 敏感数据分层原则
普通业务状态与敏感钱包数据不能混存。  
地址、非敏感标识和缓存可进入普通状态层；私钥、密文、salt、解锁凭证必须进入专门安全层。

### 5.5 失败可回退原则
任何链上调用、余额查询、metadata 获取或支付失败，都必须能回退到安全状态，并向用户提供清晰反馈。

## 7. 分层架构设计

### 7.1 展示层（Presentation Layer）

#### 7.1.1 职责
展示层负责：

- OpenMesh 页面结构展示
- 状态渲染
- 用户输入采集
- 导航切换
- 调起上层业务流程

#### 7.1.2 典型模块
- `NavigationPage.swift`（新增 `.openMesh`）
- `OpenMeshView.swift`
- `OpenMeshSupplierListView.swift`
- `OpenMeshSupplierCardView.swift`
- `OpenMeshDIYView.swift`
- `MeshWalletTopBarView.swift`
- `MeshWalletSetupView.swift`
- `MeshWalletUnlockView.swift`
- `SupplierPaymentConfirmSheet.swift`

#### 7.1.3 约束
展示层不得直接：

- 调用 RPC
- 直接访问合约
- 持有私钥
- 直接进行签名
- 直接改写现有 VPN 控制逻辑

### 7.2 应用层（Application Layer）

#### 7.2.1 职责
应用层是视图与底层服务之间的编排层，负责：

- 聚合页面所需状态
- 协调多步业务流程
- 控制页面级 loading/error/success 状态
- 管理用户动作触发后的阶段流转

#### 7.2.2 典型模块
- `OpenMeshViewModel` 或等价页面协调器
- `MeshWalletStore`
- `OnChainSupplierViewModel`
- `X402PaymentFlowCoordinator`

#### 7.2.3 典型职责划分
##### `MeshWalletStore`
负责：
- 检查是否已存在钱包
- 创建钱包
- 恢复钱包
- 解锁与锁定
- 刷新余额
- 提供签名入口

##### `OnChainSupplierViewModel`
负责：
- 加载供应商列表
- 管理空状态、错误状态、刷新状态
- 转换链上模型为可展示模型

##### `X402PaymentFlowCoordinator`
负责：
- 处理“使用该供应商”主流程
- 串联钱包检查、余额检查、支付确认、授权签名、配置获取和安装接入

#### 7.2.4 约束
应用层不得直接持有 View 引用，不负责底层密码学细节实现。

### 7.3 领域服务层（Domain / Service Layer）

#### 7.3.1 职责
服务层负责封装具体业务能力，为应用层提供可组合调用的稳定接口。

#### 7.3.2 典型模块
- `OnChainSupplierService`
- `SupplierMetadataService`
- `WalletService`
- `X402PaymentClient`
- `ConfigAcquisitionService`

#### 7.3.3 服务层职责说明
##### `OnChainSupplierService`
负责：
- 读取链上供应商注册记录
- 过滤活跃供应商
- 拉取 metadata
- 生成供应商领域模型

##### `SupplierMetadataService`
负责：
- 拉取 metadata JSON
- 验证格式
- 生成 UI 所需字段

##### `WalletService`
负责：
- 助记词与地址派生
- 私钥加解密协作
- 钱包状态基础操作

##### `X402PaymentClient`
负责：
- 生成支付授权请求
- 组织 EIP-712 待签名数据
- 组装 x402 Authorization
- 校验授权时效与输入合法性

##### `ConfigAcquisitionService`
负责：
- 使用授权信息请求供应商配置
- 解析配置内容
- 把结果传递给既有安装系统

#### 7.3.4 约束
服务层不直接处理界面，不承担导航逻辑。

### 7.4 区块链基础层（Blockchain Layer）

#### 7.4.1 职责
区块链基础层负责所有底层链上只读调用、ABI 编解码、签名摘要构造与密码学辅助，不负责页面状态管理。

#### 7.4.2 典型模块
- `BlockchainConfig`
- `BlockchainRPCClient`
- `ContractABI`
- `EIP712Signer`
- `OnChainSupplier`
- `X402Authorization`

#### 7.4.3 能力范围
- Base 链 RPC 调用
- `eth_call`
- `eth_getLogs`
- USDC `balanceOf`
- 供应商目录事件或只读函数调用
- EIP-712 digest 构造
- secp256k1 签名封装

#### 7.4.4 边界限制
本层不应直接依赖 SwiftUI。  
本层不应自行决定用户交互流程。  
本层不应处理窗口管理。

### 7.5 安全持久化层（Secure Persistence Layer）

#### 7.5.1 职责
安全持久化层负责：

- 敏感钱包密钥材料存储
- 地址与公开标识存储
- 普通业务状态缓存
- 与现有 SharedPreferences 协同

#### 7.5.2 建议划分
##### Keychain
用于保存：
- 加密后的私钥
- salt
- 生物识别保护相关凭据
- 其他高敏感安全材料

##### SharedPreferences / 本地非敏感状态
用于保存：
- 钱包地址
- 最近使用的 UI 分段
- 已加载的供应商缓存
- 已安装状态映射的继续复用
- 非敏感的流程恢复标志

#### 7.5.3 边界
私钥、助记词原文不得进入 SharedPreferences。  
展示层不得直接读写 Keychain。

## 8. 与现有系统的集成架构

### 8.1 导航集成

#### 8.1.1 目标
在现有 `NavigationPage` 中新增 `.openMesh`，作为一个常驻导航入口。

#### 8.1.2 原则
- 只增加，不重排既有含义
- 不改变 `dashboard / groups / connections / logs / profiles` 的既有职责
- `openMesh` 始终可见，不依赖 VPN 已连接状态

#### 8.1.3 结果
OpenMesh 成为独立页面入口，而不是把现有 `TrafficMarketView` 硬改成链上供应商页。

### 8.2 与现有市场能力的关系

#### 8.2.1 V1 市场能力定位
当前 `TrafficMarketView` 和 `ProviderMarketManagerView` 可视为 V1 现有中心化市场体系。

#### 8.2.2 V2 设计原则
V2 不强行替换 V1 市场体系，而是新增一条链上供应商能力路径：

- V1：中心化推荐供应商与市场管理
- V2：链上供应商目录与支付前置路径

#### 8.2.3 长期演进方向
短期内双路径可并存。  
未来是否收敛为统一供应商市场，由后续版本决定，不在 V2 总体架构中强制实现。

### 8.3 与现有安装流程的关系

#### 8.3.1 安装终点复用原则
V2 商业供应商流程在完成支付授权和配置获取后，必须尽量复用现有安装向导和安装终点。

#### 8.3.2 架构定位
现有安装体系是“终端执行层”；  
V2 新增的是“前置获取层”。

#### 8.3.3 结果
V2 的商业安装流程应抽象为：

链上供应商选择  
→ 钱包检查  
→ 支付确认  
→ x402 授权  
→ 配置获取  
→ 接入既有安装流程  
→ 写入既有 provider/profile 状态映射  
→ VPN 可按现有方式使用

### 8.4 与现有 Profiles / Offline Import 的关系

#### 8.4.1 Profiles
Profiles 继续作为本地配置管理中心存在，不被 OpenMesh 替代。

#### 8.4.2 DIY
DIY 模式不是另起一套配置系统，而是为现有离线导入和自建配置能力提供更清晰的产品入口与解释层。

#### 8.4.3 结果
DIY 的主价值在架构上是“新入口 + 新引导”，不是“新安装内核”。

### 8.5 与现有 VPN 控制层的关系

#### 8.5.1 保持边界
`VPNController` / `VPNManager` 继续作为连接控制核心存在。

#### 8.5.2 OpenMesh 的职责
OpenMesh 只负责把新获得的商业配置或 DIY 配置送入既有配置体系，不直接接管连接控制。

#### 8.5.3 结果
V2 不改变“配置选择 + provider/profile 映射 + 启动 VPN”的既有总逻辑。

## 9. 核心业务流架构

### 9.1 业务流一：进入 OpenMesh 并浏览供应商

```text
用户点击 OpenMesh
  → OpenMeshView 加载
  → MeshWalletStore 检查钱包状态
  → OnChainSupplierViewModel 加载供应商列表
  → OnChainSupplierService 读取链上目录
  → SupplierMetadataService 拉取 metadata
  → 返回 UI 展示模型
  → 页面渲染供应商卡片与 DIY 入口
```

#### 架构意义
这是 V2 的默认浏览链路。  
它体现了“UI 不直接访问链”“ViewModel 编排”“Service 读取数据”的分层原则。

### 9.2 业务流二：创建或恢复金豆账户

```text
用户点击创建/恢复金豆账户
  → MeshWalletSetupView 收集输入
  → MeshWalletStore 发起创建或恢复
  → WalletService 生成或恢复私钥与地址
  → KeychainManager 保存密文与 salt
  → SharedPreferences 保存公开地址与非敏感状态
  → MeshWalletStore 更新为 unlocked / locked 状态
  → UI 刷新余额与地址展示
```

#### 架构意义
这是钱包安全边界的核心链路。  
密钥材料和 UI 状态必须通过 Store 和安全层隔离。

### 9.3 业务流三：商业供应商支付安装

```text
用户点击“使用该供应商”
  → X402PaymentFlowCoordinator 启动
  → MeshWalletStore 检查账户状态
      → 无账户：进入创建/恢复流程
      → 已锁定：进入解锁流程
      → 已解锁：继续
  → 刷新余额并校验余额是否足够
  → 展示支付确认 Sheet
  → 用户确认
  → X402PaymentClient 构造授权数据
  → MeshWalletStore 提供签名能力
  → ConfigAcquisitionService 携带授权获取配置
  → 交给现有安装流程安装
  → 更新安装状态映射
  → 返回成功或失败结果给 UI
```

#### 架构意义
这是 V2 最核心的跨层流程。  
它必须把新增能力串起来，但仍落回既有安装体系。

### 9.4 业务流四：DIY 自建路径

```text
用户切换到 DIY 分段
  → OpenMeshDIYView 渲染说明与按钮
  → 用户选择“打开教程”
      → 调起外部浏览器
  或
  → 用户选择“直接导入本地配置”
      → 调用既有 Offline Import / Import Profile 流程
  → 后续进入既有 Profiles / VPN 使用链路
```

#### 架构意义
DIY 路径本质上是“新入口指向旧能力”。  
它不依赖钱包和链上供应商模块，是重要的独立降级通道。

## 11. 状态模型架构

### 11.1 钱包状态模型
建议钱包至少具有以下状态：

- `noWallet`
- `locked`
- `unlocking`
- `unlocked`
- `loading`
- `error`

这些状态由 `MeshWalletStore` 持有并对 UI 暴露。

### 11.2 供应商列表状态模型
建议供应商列表至少具有以下状态：

- `idle`
- `loading`
- `loaded`
- `empty`
- `error`
- `refreshing`

这些状态由 `OnChainSupplierViewModel` 管理。

### 11.3 支付安装状态模型
建议支付安装流程至少具有以下阶段：

- `walletCheck`
- `balanceCheck`
- `confirming`
- `authorizing`
- `fetchingConfig`
- `installing`
- `success`
- `failed`

这些阶段不直接持久化为长期业务状态，而是作为一次性流程状态存在于应用层协调器中。

---

## 12. 数据归属与持久化架构

### 12.1 公开数据
可普通保存的数据包括：

- 钱包地址
- 供应商展示缓存
- 最近一次查看分段
- 最近一次刷新时间
- 已安装供应商标识

### 12.2 敏感数据
只能进入安全持久化层的数据包括：

- 私钥密文
- salt
- 生物识别相关保护材料
- 解锁所需的敏感派生信息

### 12.3 临时内存态
只应驻留内存的数据包括：

- 已解锁的私钥明文
- 本次支付授权原始待签名数据
- 用户刚创建时显示的恢复短语原文

---

## 13. 错误处理与降级架构

### 13.1 错误分类
V2 总体架构中的错误应至少分为四类：

- 链上读取错误
- 钱包状态错误
- 支付授权错误
- 配置获取 / 安装错误

### 13.2 错误传播原则
底层错误以明确 Error 类型向上抛出；  
应用层将其翻译为用户可理解的状态；  
展示层只负责展示，不自己拼装底层错误细节。

### 13.3 降级策略

#### 链上目录失败
- 供应商列表显示错误态
- DIY 分段仍可用
- 既有 Profiles / Dashboard 仍不受影响

#### 钱包未创建或锁定
- 不阻断浏览
- 仅在需要支付时触发创建/解锁流程

#### 支付失败
- 停在安全状态
- 不进入安装终点
- 提供重试或返回

#### 配置获取失败
- 不修改既有已安装状态
- 允许重新发起流程

## 14. 安全架构原则

### 14.1 最小暴露
只在必要时解锁，只在必要时签名，只在必要时读取敏感材料。

### 14.2 最小持久化
不保存不必要的敏感中间结果。

### 14.3 最小侵入
尽量不将钱包逻辑扩散到既有 VPN 和安装核心模块中。

### 14.4 自动回收
应用退后台后自动锁定钱包并清理内存敏感态。

## 15. 可测试性架构要求

为了让后续测试文档能够落地，总体架构必须具备以下可测试性特征：

- Service 层与 RPC 层可注入 Mock
- Keychain 层可替换为 Mock 实现
- WalletStore 不依赖真实网络
- PaymentClient 可在无真实链上环境下完成授权构造测试
- 配置获取流程可通过假响应验证，不依赖真实供应商服务
- 新增区块链目录可形成独立测试覆盖目标

## 16. 架构决策总结

### 16.1 关键决策一：OpenMesh 是新入口，不是旧页面改名
这样可以避免把现有 Market、Profiles、VPN 控制强行揉成一块。

### 16.2 关键决策二：支付是安装前置，而不是安装替代
这样可以最大化复用当前安装终点和状态映射体系。

### 16.3 关键决策三：DIY 是独立路径，不依赖商业支付链路
这样可以保证链上异常时仍有可用入口，也符合目标用户差异。

### 16.4 关键决策四：钱包能力独立成层
这样可以把高敏感逻辑与普通客户端状态隔离，降低安全和耦合风险。

### 16.5 关键决策五：区块链能力下沉到底层
这样可以让未来的供应商、支付、余额等能力共用统一基础设施，而不是在页面里各自拼装。

## 18. 结论

MeshFlux V2 的总体架构，不是把现有客户端重做成一个“钱包优先”的全新应用，而是在现有 VPN 客户端骨架上，增加一套围绕 OpenMesh 的新能力层。

其核心架构思路是：

- 导航层新增入口
- 应用层新增流程协调
- 服务层新增供应商、钱包与支付能力
- 底层新增区块链与安全持久化基础设施
- 终点复用现有安装、配置与 VPN 控制体系

这使得 V2 能在保持既有稳定性的同时，引入链上供应商、金豆账户、x402 支付与 DIY 自建能力，并为后续版本继续演进留出清晰边界。
