# 04-V2-总体架构设计

## 1. 文档目的

本文档定义 V2 的系统级架构，重点覆盖商用供应商场景。

---

## 2. 总体架构

```text
┌────────────────────────────────────────────────────────────┐
│                      Discovery Plane                       │
│ Base Registry Contract | Provider Metadata | Signer Set    │
└────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌────────────────────────────────────────────────────────────┐
│                       Client Plane                         │
│ OpenMesh UI | Wallet | Signer Session | Local TUN         │
│ Config Installer | Receipt Store | Provider Switcher      │
└────────────────────────────────────────────────────────────┘
              │                    │
              ▼                    ▼
┌──────────────────────┐   ┌────────────────────────────────┐
│   Session / Payment  │   │        Data Plane              │
│ x402 Flow | Bucket   │   │ sing-box Cluster | User/Auth   │
│ Payment Receipt      │   │ Traffic Stats | Stop/Resume    │
└──────────────────────┘   └────────────────────────────────┘
              │                    │
              └────────────┬───────┘
                           ▼
┌────────────────────────────────────────────────────────────┐
│                    Provider Plane                          │
│ Config Service | Metering Service | Receipt Service       │
│ Reconciliation Watcher | Provider Contract Withdrawal     │
└────────────────────────────────────────────────────────────┘
```

---

## 3. 设计原则

1. Discovery 去中心化
2. 热路径不依赖平台中心化账本
3. 用户直接向供应商支付
4. 平台抽成在供应商提现时发生
5. 钱包安全优先于自动支付便利
6. 本地 `tun` 保持不变，商用变化主要发生在远端配置和支付会话

---

## 4. Discovery Plane

职责：

- 供应商注册
- metadata 发现
- signer 公钥发现
- 主网 / 测试网切换

链上注册合约建议包含：

- `providerId`
- `owner`
- `metadataURL`
- `configServiceURL`
- `meteringServiceURL`
- `signerRoot` 或 signer 列表

---

## 5. Client Plane

职责：

- 钱包创建 / 恢复 / 解锁
- signer session 管理
- 读取供应商目录
- 请求配置
- 安装配置
- 自动支付
- 保存收据和凭条

注意：

- 客户端继续保留官方临时受限访问和私人节点路径
- 商用路径不再被当前 Shadowsocks 编辑器和节点 UI 结构限制

---

## 6. Provider Plane

每个商用供应商至少应包含以下逻辑模块：

### 6.1 Config Service

职责：

- 根据用户地址和当前会话签发数据平面配置
- 返回短期有效配置

### 6.2 Metering Service

职责：

- 读取数据平面用量
- 维护流量桶剩余量
- 在阈值触发时发起下一桶续费

### 6.3 Receipt Service

职责：

- 生成支付收据
- 生成用量凭条
- 提供恢复接口

### 6.4 Reconciliation Watcher

职责：

- 异步对账
- 审计链上支付是否与凭条一致

注意：

- 它不负责热路径放行

### 6.5 Provider Contract Withdrawal

职责：

- 管理供应商收入提现
- 在提现时向平台结算手续费

注意：

- 平台手续费发生在供应商提现阶段
- 不发生在用户支付热路径

---

## 7. Data Plane

`sing-box` 的职责被限制为：

- 用户或会话认证
- 流量转发
- 用量统计
- 动态停供

不负责：

- 最终账本
- 平台 token
- 供应商注册
- 供应商发现

---

## 8. 商用场景主流程

### 8.1 选择供应商

1. 客户端读取链上 registry
2. 拉取 provider metadata
3. 用户点击“使用该供应商”

### 8.2 打开会话

1. 客户端检查 signer session
2. 请求供应商配置
3. 供应商先发放免费桶或直接进入付费流程
4. 客户端安装远端配置并开始使用

### 8.3 预续费

1. Metering Service 发现桶剩余量低于阈值
2. 供应商返回 `402`
3. 客户端若满足额度限制则自动签名支付
4. 供应商校验成功后放行下一个桶
5. 供应商签发支付收据

默认建议：

- 免费桶：40MB
- 续费桶：40MB
- 预续费阈值：剩余 5MB 或剩余 20%

说明：

- 预续费成功后，当前会话应无缝延续
- 若预续费失败，则允许用户继续消耗完当前桶剩余额度，再暂停后续供给

### 8.4 结束和恢复

1. 会话结束时，供应商签发最新用量凭条
2. 客户端和供应商都保存最新凭条
3. 任一方丢失时可向另一方恢复

---

## 9. 官方临时受限访问与商用场景的关系

官方临时受限访问不是另一套系统，而是第一个官方 provider。  
差异只在于：

- 可以继续使用简单单节点脚本
- 可无钱包使用
- 可简化流量显示和限额
- 不要求完整商用凭条和计费闭环
