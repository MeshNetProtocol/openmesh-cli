# 05-V2-详细技术设计

## 1. 文档目的

本文档用于在《04-V2-总体架构设计》的基础上，进一步下沉到**目录、文件、类型、接口、状态、流程和集成点**层级，形成可直接指导实现的详细技术设计。

本文档重点回答：

- 新代码具体落在哪些目录和文件
- 每个新增文件的职责是什么
- 关键类型与方法签名如何设计
- 钱包、链上供应商、x402 支付和安装接入如何串联
- 如何与现有 `NavigationPage`、`ProviderInstallWindowManager`、`Profiles`、`VPNController` 等模块集成
- 如何保证测试可注入、实现可控、风险可收敛

本文档不重复展开需求背景与总体架构原则；相关内容以 `03-V2-需求规格说明书.md` 和 `04-V2-总体架构设计.md` 为准。

---

## 2. 设计输入与约束

本设计建立在以下前提之上：

1. 当前主 App 目标 `MeshFluxMac` 的主要源码目录为：
   - `core/`
   - `views/`

2. 当前主导航 `NavigationPage.swift` 仍只有：
   - `dashboard`
   - `groups`
   - `connections`
   - `logs`
   - `profiles`

3. V2 必须满足以下硬性约束：
   - 只能在 `dashboard` 之后新增 `.openMesh`
   - 不改写现有 VPN 核心控制逻辑
   - 不改写 `TrafficMarketView` 现有业务逻辑
   - 新功能必须满足分层架构
   - 敏感密钥材料不得明文存储
   - 测试必须支持 Mock 注入

4. V2 的关键新增能力包括：
   - OpenMesh 新入口
   - 链上供应商目录
   - 金豆账户（钱包）能力
   - x402 支付前置
   - DIY 引导
   - 安装结果接入现有安装闭环

## 3. 目录与文件落点设计

## 3.1 设计原则

为了与当前工程结构保持一致，同时避免把新模块散落到多个全新顶层目录中，V2 详细设计采用以下落点策略：

- UI 文件统一落在 `views/openmesh/`
- 区块链、钱包、支付相关逻辑统一落在 `core/blockchain/`
- 安全持久化文件放在 `core/`
- 只对极少数现有文件做**受控修改**
- 尽量通过新增文件实现功能，避免扩散式侵入

## 4.1 `views/NavigationPage.swift`

### 修改目标
在不改变现有 case 顺序语义的前提下，新增 `.openMesh`。

### 修改点
1. 新增枚举 case：
   - `case openMesh`

2. 插入位置：
   - `case dashboard`
   - `case openMesh`
   - `case groups`

3. 新增 title：
   - `"OpenMesh"`

4. 新增图标：
   - `"globe.americas.fill"`

5. `visible(vpnConnected:)`
   - `openMesh` 不单独判断
   - 通过 `default: return true` 始终可见

6. `contentView(vpnController:)`
   - 新增：
     - `case .openMesh: OpenMeshView(vpnController: vpnController)`

### 修改示例

```swift
enum NavigationPage: String, CaseIterable, Identifiable {
    case dashboard
    case openMesh
    case groups
    case connections
    case logs
    case profiles

    var title: String {
        switch self {
        case .dashboard: return "Dashboard"
        case .openMesh: return "OpenMesh"
        case .groups: return "出站组"
        case .connections: return "连接"
        case .logs: return "日志"
        case .profiles: return "配置列表"
        }
    }

    private var iconImage: String {
        switch self {
        case .dashboard: return "text.and.command.macwindow"
        case .openMesh: return "globe.americas.fill"
        case .groups: return "rectangle.3.group.fill"
        case .connections: return "list.bullet.rectangle.portrait.fill"
        case .logs: return "doc.text.fill"
        case .profiles: return "list.bullet.rectangle.fill"
        }
    }

    func visible(vpnConnected: Bool) -> Bool {
        switch self {
        case .groups, .connections: return vpnConnected
        case .logs: return AppConfig.showLogsInUI
        default: return true
        }
    }

    @ViewBuilder
    func contentView(vpnController: VPNController) -> some View {
        switch self {
        case .dashboard:
            DashboardView(vpnController: vpnController)
        case .openMesh:
            OpenMeshView(vpnController: vpnController)
        case .groups:
            GroupsView(vpnController: vpnController)
        case .connections:
            ConnectionsView(vpnController: vpnController)
        case .logs:
            LogsView(vpnController: vpnController)
        case .profiles:
            ProfilesView()
        }
    }
}
```

## 4.2 `core/ProviderInstallWindowManager.swift`

### 修改目标
为链上供应商下载得到的配置内容提供一个**受控接入点**，使 V2 不需要复制一套安装终点。

### 新增建议方法

```swift
@MainActor
func installImportedProviderConfig(
    supplierID: String,
    displayName: String,
    configData: Data
) async throws
```

### 设计说明
- 输入为已经拿到的配置文件原始数据
- 方法内部负责：
  1. 将数据写入临时文件或转换为当前安装流程可接受的形式
  2. 调用现有安装窗口 / 安装向导终点
  3. 在成功后回写现有 provider/profile 状态映射
- 该方法是 **V2 安装接入点**
- 不改变既有中心化 `MarketService` 的安装路径

### 不允许的做法
- 在 OpenMesh UI 中直接复制现有安装逻辑
- 在 X402PaymentClient 中直接写 SharedPreferences
- 在区块链层直接调窗口管理

## 8.3 `core/blockchain/abi/ContractABI.swift`

### 职责
实现 V2 所需的最小 ABI 编解码。

### 支持范围
- `address`
- `bool`
- `uint256`
- `uint64`
- `bytes32`
- `string`
- `tuple`
- 基础数组

### 选择器
必须预置以下 selector：
- `getSupplier(bytes32)`
- `isSupplierActive(bytes32)`
- `balanceOf(address)`
- `authorizationState(address,bytes32)`
- 如后续合约确定，还可补充：
  - `getSupplierHashesByOwner(address)`
  - `getSupplierCount()`

### 设计原则
- 只实现 V2 用得到的最小集合
- 避免引入大型第三方 ABI 库
- 将测试向量写入单元测试

## 10.2 `core/blockchain/services/OnChainSupplierService.swift`

### 职责
读取链上供应商目录，合并 metadata，输出可展示模型。

### 核心接口
```swift
actor OnChainSupplierService {
    private let rpc: BlockchainRPCClientProtocol
    private let httpSession: URLSessionProtocol
    private let metadataService: SupplierMetadataService
    private let cacheTTL: TimeInterval

    init(
        rpc: BlockchainRPCClientProtocol,
        httpSession: URLSessionProtocol = URLSession.shared,
        cacheTTL: TimeInterval = 300
    )

    func fetchActiveSuppliers(forceRefresh: Bool = false) async throws -> [OnChainSupplier]
    func clearCache()
}
```

### 内部流程
1. `eth_getLogs` 读取 `SupplierRegistered` 事件
2. 提取 supplier hash 列表
3. 对每个 hash：
   - 调 `isSupplierActive(hash)`
   - 调 `getSupplier(hash)`
4. 将链上结果组装为 `OnChainSupplier`
5. 拉取 metadata
6. 合并价格、名称、地区、标签、配置 URL
7. 返回列表

### 缓存策略
- 默认 TTL：300 秒
- `forceRefresh = true` 时忽略缓存
- metadata 缓存可与基础链上缓存共用一次快照

### 降级规则
- metadata 失败：保留链上基础信息
- RPC 失败：抛出 `networkUnavailable`
- 某个供应商单独失败：不阻断全部结果

## 11. 支付安装协调器设计

## 11.1 `X402PaymentFlowCoordinator`（建议放在 `core/blockchain/services/`）

### 职责
串联“使用该供应商”的完整流程。

### 核心接口
```swift
@MainActor
final class X402PaymentFlowCoordinator: ObservableObject {
    enum Phase: Equatable {
        case idle
        case walletCheck
        case balanceCheck
        case confirming
        case authorizing
        case fetchingConfig
        case installing
        case success
        case failed(String)
    }

    @Published private(set) var phase: Phase = .idle

    func startFlow(for supplier: OnChainSupplier) async throws
    func confirmAndContinue() async throws
    func cancel()
}
```

### 流程设计
#### 阶段 1：钱包检查
- 无钱包 → 抛出 `.walletNotFound`
- 已锁定 → 抛出 `.walletLocked`
- 已解锁 → 继续

#### 阶段 2：余额检查
- 读取余额
- 若不足 → `.insufficientBalance`

#### 阶段 3：展示确认
- UI 根据 coordinator 状态弹出确认 Sheet

#### 阶段 4：构造授权
- 使用 `X402PaymentClient.buildAuthorization`

#### 阶段 5：拉取配置并安装
- 使用 `ConfigAcquisitionService.fetchAndInstall`

#### 阶段 6：成功 / 失败回写
- 更新 phase
- 刷新列表中安装状态

### 设计原则
- Coordinator 不直接构建界面
- View 不自行串联多步业务逻辑
- 复杂状态统一收敛在协调器中

## 15. 错误与提示设计

- 网络不可用
- RPC 返回非法

- 配置获取失败

- 向用户呈现业务语义
- 日志中可保留更详细错误
- 失败后允许回退或重试

## 18. 本文档的关键结论

V2 详细技术设计的核心，不是“再加几个页面”，而是在当前 `MeshFluxMac` 的 `core + views` 结构上，受控地新增一条完整能力链：

- `NavigationPage` 提供新入口
- `views/openmesh/` 承担所有新 UI
- `core/blockchain/` 承担钱包、链上供应商、x402 支付与服务编排
- `KeychainManager` 承担敏感数据安全存储
- `ProviderInstallWindowManager` 提供现有安装终点的接入点
- `VPNController / VPNManager` 保持稳定，不被区块链逻辑污染

这样设计的结果是：

- 架构边界清晰
- 与现有工程贴合
- 便于测试
- 风险收敛
- 后续实现时可直接按文件落地
