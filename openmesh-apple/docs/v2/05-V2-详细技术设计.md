# 05-V2-详细技术设计

## 1. 文档目的

本文档把总体架构落到 `MeshFluxMac` 的文件级设计上。  
重点说明：

- 真实工程里改哪些文件
- 新增哪些模块
- 如何把 OpenMesh 升级成真实产品

---

## 2. 设计原则

1. **优先改造已有 OpenMesh 骨架，不重新起炉灶。**
2. **旧系统只做最小接入。**
3. **官方受限通道、商用供应商与私人节点共用同一套“当前路由”表达。**
4. **流豆账户和授权支付必须下沉到服务层。**
5. **商用链路复用旧安装终点，私人链路复用旧导入终点。**
6. **冷启动能力优先于商业闭环。**

---

## 3. 需要修改的现有文件

## 3.1 `OpenMeshMacApp.swift`

目标：

- 保持现有 `OpenMesh` Tab
- 不再把文档假设成“新增入口”
- 允许为 OpenMesh 注入真实 Store 或 Environment

建议改动：

- 维持 `MenuBarTab.openMesh`
- 在 `openMeshTabContent` 中接入真实状态容器
- 增加必要的 feature flag 或 capability 开关

## 3.2 `views/openmesh/OpenMeshView.swift`

目标：

- 从静态壳升级成真实主页面容器

职责：

- 管理分段状态
- 组合当前供应商卡片、流豆账户条、官方受限通道卡、商用供应商列表、私人节点列表
- 承接错误和加载态

## 3.3 `views/openmesh/OpenMeshTopBarView.swift`

目标：

- 真实展示当前路由供应商和流豆账户状态

必须支持：

- 当前为官方受限通道
- 当前为商用供应商
- 当前为私人节点
- 当前未选择供应商
- 账户未创建 / 已锁定 / 已解锁

## 3.4 `views/openmesh/OpenMeshSupplierListView.swift`

目标：

- 从 mock 列表升级成真实商用供应商目录

必须支持：

- 列表渲染
- 筛选 / 排序
- “当前使用中”状态
- “使用 / 切换”主动作
- 加载、空状态、错误状态

## 3.5 `views/openmesh/OpenMeshDIYView.swift`

目标：

- 升级为“私人节点”界面，而不是纯说明页

建议升级为两段结构：

- 已有私人节点列表
- 新增 / 导入入口

## 3.6 `core/ProviderInstallWindowManager.swift`

目标：

- 仅增加 OpenMesh 接入点，不改变旧市场主逻辑

建议：

- 增加一个供 OpenMesh 调用的轻量适配方法
- 让 OpenMesh 在拿到配置或 provider 结果后走现有安装终点

## 3.7 `core/OfflineImportWindowManager.swift`

目标：

- 作为私人节点导入终点继续复用

## 4. 建议新增目录

```text
MeshFluxMac/
  core/
    openmesh/
      bootstrap/
      models/
      stores/
      services/
      payments/
      wallet/
      integration/
```

说明：

- 使用 `openmesh` 目录而不是泛化为 `blockchain`，避免把产品目标缩成“区块链模块”
- 底层链上能力可以放在 `openmesh/wallet` 或 `openmesh/payments` 内，再按需要细分

## 5. 建议新增类型

## 5.1 模型层

### `OpenMeshCommercialProvider`

字段建议：

- `id`
- `displayName`
- `summary`
- `providerType`
- `pricingDisplay`
- `regions`
- `tags`
- `isCurrent`
- `isAvailable`
- `metadataVersion`

### `OpenMeshPrivateNode`

字段建议：

- `id`
- `displayName`
- `sourceType`
- `configReference`
- `lastUsedAt`
- `isCurrent`

### `CurrentRouteState`

统一表达当前路由来源：

- `.none`
- `.officialLimited(channelID, displayName, quotaDetail)`
- `.commercial(providerId, displayName, detail)`
- `.privateNode(nodeId, displayName, detail)`
- `.legacy(profileId, displayName)`

### `FlowCreditAccountState`

统一表达流豆账户状态：

- `.notCreated`
- `.locked(addressHint)`
- `.ready(balanceDisplay, addressHint)`
- `.error(message)`

## 5.2 Store / ViewModel 层

### `OpenMeshStore`

职责：

- 作为 OpenMesh 页面主状态容器
- 暴露当前路由状态、官方受限通道状态、商用供应商列表、私人节点列表、分段状态
- 协调刷新、切换、异常和局部 loading

### `OfficialAccessStore`

职责：

- 管理官方受限通道状态
- 展示剩余额度
- 在无现成路由时提供默认引导网络

### `FlowCreditWalletStore`

职责：

- 创建 / 恢复 / 解锁流豆账户
- 读取余额
- 暴露账户状态

### `CommercialProviderCatalogStore`

职责：

- 加载商用供应商目录
- 做筛选、排序和缓存

### `PrivateNodeStore`

职责：

- 管理私人节点列表
- 对接导入结果
- 同步当前启用状态

---

## 5.3 Service 层

### `OfficialAccessService`

职责：

- 获取官方受限通道配置
- 计算和查询剩余额度
- 输出受限通道可用性状态

### `OfficialQuotaService`

职责：

- 以设备匿名标识作为初始限额 key
- 在用户创建流豆账户后叠加地址维度
- 执行每日 `100MB` 配额规则

### `CommercialProviderService`

职责：

- 读取链上供应商目录
- 获取 metadata
- 输出 `OpenMeshCommercialProvider`

### `PrivateNodeService`

职责：

- 管理私人节点的本地描述信息
- 与现有配置或导入能力做映射

### `FlowCreditWalletService`

职责：

- 管理钱包底层能力
- 地址、余额、解锁、签名

### `AuthorizationPaymentService`

职责：

- 执行授权支付
- 返回配置获取所需的支付凭证或支付结果

### `OpenMeshInstallCoordinator`

职责：

- 串联账户检查、授权支付、配置获取、旧安装终点接入
- 完成商用供应商切换

### `CurrentRouteResolver`

职责：

- 从当前配置、当前 provider 关联和 OpenMesh 状态中解析“当前路由供应商”

## 7. 关键流程设计

## 7.1 商用供应商切换流程

1. 用户点击“使用该供应商”
2. `OpenMeshStore` 请求 `FlowCreditWalletStore` 检查账户状态
3. 若账户可用，弹出授权支付确认 Sheet
4. 用户确认后，`OpenMeshInstallCoordinator` 调用 `AuthorizationPaymentService`
5. 支付成功后拉取配置
6. 调用旧安装终点
7. 成功后刷新当前路由和供应商列表状态

## 7.2 私人节点启用流程

1. 用户在私人节点分段选择已有节点或导入配置
2. `PrivateNodeStore` 更新本地节点状态
3. 通过现有配置导入或配置切换逻辑接回旧系统
4. `CurrentRouteResolver` 回写当前路由状态

## 7.3 当前路由刷新流程

触发时机：

- OpenMesh 页面出现
- 官方受限通道启用成功
- 商用供应商切换成功
- 私人节点启用成功
- 配置切换通知到达

---

## 8. 特殊约束

1. 不允许在 `OpenMeshView` 里直接做链上调用或安装落地
2. 不允许让 `OpenMeshStore` 直接依赖 VPN 控制实现细节
3. 不允许让商用供应商和私人节点维护两套相互冲突的当前路由状态
4. 不允许把新链路硬塞进 `TrafficMarketView.swift`

---

## 9. 结论

详细设计的核心不是“再造一套新客户端”，而是：

- 升级已有 `OpenMesh` 骨架
- 增加官方受限通道能力以解决冷启动
- 新增 `openmesh` 业务层
- 让商用供应商和私人节点共享同一个 OpenMesh 入口
- 让所有新结果最终落回现有 `MeshFluxMac` 稳定终点
