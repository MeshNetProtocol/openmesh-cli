本文档把总体架构落到实现对象、消息和模块层级。

## 2. 关键对象

### 2.1 Provider Metadata

建议字段：

- `providerId`
- `displayName`
- `description`
- `supportedProtocols`
- `configServiceURL`
- `meteringServiceURL`
- `paymentServiceURL`
- `priceRate` (如 1 USDC = 100MB)
- `minPurchaseAmount` (最小购买金额)
- `maxPurchaseAmount` (最大购买金额)

### 2.2 TrafficQuota（流量配额）

建议字段：

- `quotaId`
- `providerId`
- `userAddress`
- `totalQuota` (总配额，字节)
- `usedUpload` (已用上传，字节)
- `usedDownload` (已用下载，字节)
- `remaining` (剩余，字节)
- `priceRate` (购买时的价格比率)
- `purchasedAt`
- `updatedAt`
- `expiresAt` (null = 永久有效)

### 2.3 PurchaseRecord（购买记录）

建议字段：

- `recordId`
- `providerId`
- `userAddress`
- `usdcAmount`
- `bandwidthPurchased` (购买的流量，字节)
- `priceRate`
- `paymentSignature`
- `txHash` (链上交易哈希，可选)
- `purchasedAt`

### 2.4 SignerSessionState

建议字段：

- `isUnlocked`
- `singlePaymentLimit`
- `dailyPaymentLimit`
- `dailySpent`
- `unlockExpiresAt`
- `lockReason`

## 3. 客户端模块

## 3.1 OpenMesh 商用目录层

职责：

- 读取链上供应商列表
- 加载 metadata
- 展示商用供应商卡片
- 触发切换

## 3.2 Wallet / Signer Session 层

职责：

- 创建 / 恢复钱包
- 解锁钱包
- 管理 signer session
- 签名 x402 预付费支付

## 3.3 Quota Monitor

职责：

- 查询流量配额余额
- 监控配额使用情况
- 配额不足时提示用户购买

## 3.4 Provider Switch Coordinator

职责：

- 请求供应商配置
- 安装供应商配置
- 管理当前商用供应商状态

---

## 4. 商用供应商参考服务端模块

## 4.1 Config Service

职责：

- 根据用户地址和流量配额状态生成接入配置
- 返回短期凭据

## 4.2 Payment Service

职责：

- 验证 x402 预付费支付签名
- 根据服务商价格兑换流量配额
- 记录购买历史

## 4.3 Metering Service

职责：

- 接收节点上报的流量统计
- 按用户 ID 汇总流量（支持多节点并行）
- 中心化扣减流量配额
- 配额不足时返回阻断指令

## 4.4 Reconciliation Watcher

职责：

- 异步对账
- 审计链上支付是否与记账服务记录一致

注意：

- 它不负责热路径放行

## 4.5 Provider Withdrawal Service

职责：

- 在供应商请求提现时执行平台手续费逻辑
- 对接供应商收款合约和平台分成逻辑

注意：

- 该服务不进入用户支付热路径

## 4.6 sing-box Adapter

职责：

- 把地址或地址派生值映射成数据平面用户
- 下发、更新、停用用户配置
- 汇总节点统计并上报到 Metering Service

## 5. 数据平面协议建议

V1 参考实现建议优先选择：

- `Shadowsocks multi-user + managed`

原因：

- 官方多用户能力明确
- 管理 API 明确
- 最接近当前现有客户端路径

但协议层必须保持开放，未来可支持：

- `Trojan`
- `Hysteria2`
- `VLESS`

前提是它们满足：

- 地址 / session 级身份
- 流量统计
- 动态停供

## 6. 预付费购买流程

### 6.1 购买流量

用户通过 x402 向服务商预付 USDC，直接兑换成流量配额。

示例：
- 用户支付 5 USDC
- 服务商价格：1 USDC = 100MB
- 用户获得 500MB 流量配额

### 6.2 流量使用

1. 用户请求配置
2. 服务商检查用户流量配额
3. 如果配额充足，下发配置
4. 用户开始使用流量
5. 节点统计流量并定期上报到 Metering Service
6. Metering Service 中心化扣减流量配额

### 6.3 配额不足

1. Metering Service 检测到用户配额不足
2. 返回"流量配额不足"状态
3. 节点阻断用户连接
4. 客户端提示用户购买更多流量

### 6.4 追加购买

用户可以随时追加购买流量，配额累加。

### 6.5 关键特性

- ✅ 预付费模型，无需实时计费
- ✅ 中心化扣减，无需双向签名
- ✅ 流量永久有效，直到用完
- ✅ 支持追加购买，配额累加
- ✅ 服务商独立定价

## 8. 关键结论

1. 官方临时受限访问继续走简单单节点方案
2. 商用供应商必须引入独立的 provider 参考服务端
3. `sing-box` 是数据平面，不是账本
4. 核心流程是”预付费购买 → 使用流量 → 中心化扣减 → 配额不足阻断”
5. 平台提供开源记账服务代码，服务商可自行部署
