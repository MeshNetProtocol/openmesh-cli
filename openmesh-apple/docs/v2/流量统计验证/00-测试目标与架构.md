## 1. 测试目标

### 1.1 核心目标

验证多节点并行架构下的流量汇总和预付费计费能力：

1. ✅ **多节点流量汇总**: 用户可在多个并行 sing-box 节点间切换，所有节点流量统计汇总到统一记账服务
2. ✅ **预付费购买流量**: 用户通过 x402 向服务商预付 USDC，直接兑换成流量配额
3. ✅ **中心化扣减**: 记账服务中心化扣减流量余额，余额耗尽时阻断连接
4. ✅ **多用户并发**: 每个 sing-box 节点支持多个账号同时访问
5. ✅ **动态用户管理**: 支持通过 API 动态添加用户和充值流量（不修改配置文件）
6. ✅ **多会话支持**: 一个用户可以同时开启多个连接（类似 AI 用户多窗口访问）
7. ✅ **服务商价格差异**: 不同服务商的带宽价格不同（如 1 USDC = 100MB vs 80MB）

### 1.2 计费模型

**预付费模型**:
```
用户 --x402 支付 USDC--> 服务商 --兑换--> 流量配额 --使用--> 中心化扣减 --耗尽--> 阻断
```

**关键特性**:
- **支付时机**: 预付费，不是实时计费
- **支付对象**: 用户直接向服务商支付 USDC
- **价格差异**: 每个服务商独立定价（如服务商 A: 1 USDC = 100MB，服务商 B: 1 USDC = 80MB）
- **流量有效期**: 永久有效，直到用完
- **扣减方式**: 记账服务中心化扣减，无需双向签名和凭条备份
- **服务商独立**: 每个服务商独立的流量账户，互不感知

## 2. 测试架构

### 2.1 整体拓扑（并行节点架构）

```
                                    ┌──────────────────────────────────┐
                                    │   统一记账服务 (Metering Service) │
                                    │   - 汇总所有节点的流量统计        │
                                    │   - 管理流量桶（充值、消耗）      │
                                    │   - 提供管理 API                  │
                                    └──────────────────────────────────┘
                                              ▲         ▲         ▲
                                              │         │         │
                                    上报流量统计│         │         │上报流量统计
                                              │         │         │
                    ┌─────────────────────────┴─────────┴─────────┴─────────────┐
                    │                                                             │
                    │                                                             │
┌─────────────┐     │     ┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐
│  用户客户端  │     │     │ sing-box 节点 A  │  │ sing-box 节点 B  │  │ sing-box 节点 C  │
│  (机器 1)   │─────┼────→│  (机器 2)        │  │  (机器 2)        │  │  (机器 2)        │
│             │可切换│     │  - 多用户入站    │  │  - 多用户入站    │  │  - 多用户入站    │
│  - curl     │     │     │  - 流量统计      │  │  - 流量统计      │  │  - 流量统计      │
│  - 浏览器   │     │     │  - Direct 出站   │  │  - Direct 出站   │  │  - Direct 出站   │
└─────────────┘     │     └────────┬─────────┘  └────────┬─────────┘  └────────┬─────────┘
                    │              │                     │                     │
                    └──────────────┼─────────────────────┼─────────────────────┘
                                   │                     │                     │
                                   ↓                     ↓                     ↓
                               ┌────────────────────────────────────────────────┐
                               │              互联网                             │
                               └────────────────────────────────────────────────┘
```

**关键特性**:
- 用户可以在节点 A、B、C 之间**自由切换**
- 同一用户可能**同时使用多个节点**（多窗口、多设备）
- 每个节点**独立统计**本节点的流量
- 所有节点的流量统计**上报到统一记账服务**
- 记账服务**汇总流量**并统一管理流量桶

### 2.2 节点角色说明

#### 用户客户端 (机器 1)
- **角色**: 流量发起端
- **工具**: curl、浏览器、或其他网络工具
- **连接方式**:
  - 直接连接到 sing-box 节点（Shadowsocks 客户端）
  - 可以切换连接不同的节点
- **用户标识**: 通过 Shadowsocks 用户名/密码传递用户 ID

#### sing-box 服务节点 (机器 2，多个实例)
- **入站**: Shadowsocks 多用户服务（启用 `managed: true`）
- **出站**: Direct 直连互联网
- **流量统计**:
  - 统计本节点所有用户的流量
  - 实时记录每个用户的上传/下载字节数
- **上报机制**:
  - 定期（如每 10 秒）向记账服务上报流量统计
  - 或通过 webhook 实时推送
- **流量控制**:
  - 查询记账服务获取用户流量桶状态
  - 如果流量桶耗尽，阻断用户连接

#### 统一记账服务 (Metering Service)
- **流量汇总**:
  - 接收所有节点上报的流量统计
  - 按用户 ID 汇总流量（累加所有节点的流量）
- **流量桶管理**:
  - 每个用户独立的流量桶
  - 支持动态充值（通过 API，每次 40MB）
  - 实时计算剩余流量
- **管理 API**:
  - 用户管理（添加/查询/删除）
  - 流量充值
  - 流量查询
  - 流量桶状态查询
- **存储**: SQLite 或 Redis

## 3. 核心难题与解决方案

### 3.1 难题 1：多节点并行服务同一账号

**问题描述**:
- 用户可能同时使用多个 sing-box 节点（多窗口、多设备）
- 用户可能在节点之间切换（从节点 A 切换到节点 B）
- 每个节点独立统计流量，如何保证不重复计费？

**解决方案**:
```
1. 每个节点独立统计本节点的流量
2. 节点定期上报流量增量到记账服务
3. 记账服务按用户 ID 汇总所有节点的流量
4. 记账服务统一管理流量桶，避免重复扣减
```

**上报机制**:
```json
// 节点 A 上报
POST /api/v1/metering/report
{
  "node_id": "node_a",
  "user_id": "user_0x1234abcd",
  "upload_bytes": 1000000,      // 自上次上报以来的增量
  "download_bytes": 5000000,
  "timestamp": "2026-03-22T14:30:00Z"
}

// 节点 B 同时上报
POST /api/v1/metering/report
{
  "node_id": "node_b",
  "user_id": "user_0x1234abcd",
  "upload_bytes": 500000,
  "download_bytes": 2000000,
  "timestamp": "2026-03-22T14:30:05Z"
}

// 记账服务汇总
// user_0x1234abcd 总流量 = 1000000 + 500000 + 5000000 + 2000000 = 8500000 字节
```

### 3.2 难题 2：流量统计汇总到记账服务

**问题描述**:
- 多个 sing-box 节点分散部署
- 如何实时或准时地将流量统计汇总到记账服务？
- 如何处理网络延迟、节点故障等异常情况？

**解决方案 A：定期上报（推荐用于测试）**
```
1. 每个节点每 10 秒上报一次流量增量
2. 记账服务接收上报并累加到用户流量桶
3. 如果上报失败，节点本地缓存，下次重试
```

**解决方案 B：实时推送（生产环境）**
```
1. 节点与记账服务建立 WebSocket 连接
2. 流量变化时实时推送
3. 记账服务实时更新流量桶
```

**解决方案 C：混合模式（推荐用于生产）**
```
1. 节点定期（如每 10 秒）批量上报流量增量
2. 当流量桶接近耗尽时（如剩余 < 5MB），实时查询记账服务
3. 记账服务返回是否允许继续使用
```

### 3.3 难题 3：流量桶统一管理

**问题描述**:
- 用户在多个节点同时使用，如何避免超额使用？
- 如何保证流量桶扣减的原子性？

**解决方案**:
```
1. 流量桶只在记账服务管理（单一数据源）
2. 节点上报流量时，记账服务原子性扣减流量桶
3. 如果流量桶不足，记账服务返回错误，节点阻断连接
```

**流量桶扣减逻辑**:
```python
# 伪代码
def report_traffic(user_id, upload_bytes, download_bytes):
    with transaction():
        bucket = get_bucket(user_id)
        total_bytes = upload_bytes + download_bytes

        if bucket.remaining < total_bytes:
            # 流量桶不足，拒绝
            return {"status": "insufficient", "remaining": bucket.remaining}

        # 扣减流量桶
        bucket.used_upload += upload_bytes
        bucket.used_download += download_bytes
        bucket.remaining -= total_bytes
        bucket.save()

        return {"status": "ok", "remaining": bucket.remaining}
```

### 3.4 难题 4：节点如何知道用户流量桶状态

**问题描述**:
- 节点需要知道用户流量桶是否耗尽
- 如何避免频繁查询记账服务？

**解决方案**:
```
1. 节点上报流量时，记账服务返回流量桶状态
2. 节点缓存流量桶状态（如剩余流量）
3. 当缓存的剩余流量 < 阈值时，实时查询记账服务
4. 如果流量桶耗尽，节点阻断用户连接
```

**上报响应示例**:
```json
POST /api/v1/metering/report
{
  "node_id": "node_a",
  "user_id": "user_0x1234abcd",
  "upload_bytes": 1000000,
  "download_bytes": 5000000
}

// 响应
{
  "status": "ok",
  "user_id": "user_0x1234abcd",
  "remaining": 34000000,  // 剩余流量
  "action": "continue"    // continue | block
}
```

## 4. 流量桶管理

### 4.1 预付费购买流量

**购买流程**:
```
1. 用户选择服务商（如服务商 A）
2. 查看服务商价格（如 1 USDC = 100MB）
3. 用户通过 x402 支付 USDC（如支付 5 USDC）
4. 服务商记账服务收到支付，兑换成流量配额（5 USDC × 100MB = 500MB）
5. 用户获得 500MB 流量配额，永久有效直到用完
```

**服务商价格差异示例**:
```
服务商 A: 1 USDC = 100MB  (高速节点)
服务商 B: 1 USDC = 80MB   (中速节点)
服务商 C: 1 USDC = 150MB  (经济节点)
```

### 4.2 流量配额数据结构

```json
{
  "user_id": "user_0x1234abcd",
  "provider_id": "provider_a",
  "total_quota": 500000000,      // 总配额（字节）500MB
  "used_upload": 50000000,       // 已用上传（字节）50MB
  "used_download": 150000000,    // 已用下载（字节）150MB
  "remaining": 300000000,        // 剩余（字节）300MB
  "price_rate": 100,             // 价格比率：1 USDC = 100MB
  "purchased_at": "2026-03-22T10:00:00Z",
  "updated_at": "2026-03-22T14:30:00Z",
  "expires_at": null             // 永久有效
}
```

**HTTP API 设计**:

```bash
  "remaining": 500000000,
  "purchased_at": "2026-03-22T15:00:00Z"

```

### 4.4 流量扣减逻辑（中心化）

```
1. 用户发起连接
2. 节点统计流量并定期上报到记账服务
3. 记账服务中心化扣减流量配额
4. 如果剩余配额 > 0，返回 "continue"
5. 如果剩余配额 <= 0，返回 "block"，节点阻断连接
```

**扣减示例**:
```python
# 伪代码
def report_traffic(user_id, upload_bytes, download_bytes):
    with transaction():
        quota = get_quota(user_id)
        total_bytes = upload_bytes + download_bytes

        if quota.remaining < total_bytes:
            # 流量配额不足，拒绝
            return {"status": "insufficient", "remaining": quota.remaining, "action": "block"}

        # 中心化扣减流量配额
        quota.used_upload += upload_bytes
        quota.used_download += download_bytes
        quota.remaining -= total_bytes
        quota.save()

        return {"status": "ok", "remaining": quota.remaining, "action": "continue"}
```

**关键特性**:
- ✅ 中心化扣减，无需双向签名
- ✅ 无需凭条备份和恢复
- ✅ 原子性操作，避免超额使用
- ✅ 简单直接，易于实现


### 7.2 避免死循环的方案

**问题**: 如果在同一台机器上运行客户端和服务端，可能形成流量死循环

**解决方案**: 使用 2 台机器物理隔离
```
机器 1 (客户端) → 网络 → 机器 2 (服务端) → 互联网
```

**备选方案**: 如果只有一台机器，使用网卡绑定
```json
// 客户端 sing-box 配置
{
  "outbounds": [
    {
      "type": "shadowsocks",
      "bind_interface": "lo0"  // 绑定本地回环
    }
  ]
}

// 服务端 sing-box 配置
{
  "outbounds": [
    {
      "type": "direct",
      "bind_interface": "en0"  // 绑定物理网卡
    }
  ]
}
```

## 10. 风险与应对

### 10.1 技术风险

| 风险 | 影响 | 应对方案 |
|------|------|---------|
| sing-box 不支持用户 ID 透传 | 无法实现多级流量归属 | 使用 V2Ray API 或自定义协议头 |
| 流量统计不准确 | 计费错误 | 对比多个统计源，校准算法 |
| 死循环导致测试失败 | 无法验证 | 使用网卡绑定或路由规则隔离 |

### 10.2 实施风险

| 风险 | 影响 | 应对方案 |
|------|------|---------|
| 配置复杂度高 | 调试困难 | 逐步增加节点，先验证两级再扩展 |
| 测试环境不稳定 | 结果不可靠 | 使用 Docker 容器隔离环境 |
