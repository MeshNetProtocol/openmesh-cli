# 02-V2-现状系统分析与差距评估

## 1. 文档目的

本文档用于基于当前 `main` 分支的真实代码结构，对 MeshFlux macOS 客户端的现状进行梳理，并评估其与 V2 升级目标之间的差距。  
本文档的作用不是提出最终实现方案，而是先回答三个问题：

1. 当前代码到底已经具备哪些能力  
2. 当前代码的主要结构和边界是什么  
3. 要实现 V2，还缺哪些关键能力与架构层

本文档是后续总体架构设计、详细技术设计和实施计划的现实基础。

## 3. 当前代码结构概览

## 3.1 App 目标目录结构

当前 `MeshFluxMac` 主体代码的结构，至少可以确认分为以下几个层面：

- `core/`：偏系统控制、窗口管理、VPN 控制、命令客户端、应用级配置
- `views/`：偏 SwiftUI 视图、页面、安装向导、主题、导航定义
- `Assets.xcassets`：资源
- `OpenMeshMacApp.swift`：应用入口
- `OpenMeshMac.entitlements`：主应用权限配置

这说明当前工程已经具备一定的“视图层 / 核心控制层”分离，但尚未形成 V2 文档要求中的完整区块链分层架构。

## 4. 当前系统已确认的行为与能力

## 4.1 当前主导航结构

`NavigationPage.swift` 当前定义的主导航为：

- `dashboard`
- `groups`
- `connections`
- `logs`
- `profiles`

现状说明：

1. 当前主导航中**没有** `openMesh`
2. `groups` 和 `connections` 仅在 VPN 已连接时显示
3. `logs` 是否显示由 `AppConfig.showLogsInUI` 控制
4. `profiles` 始终存在，且当前标题为“配置列表”

这说明 V2 文档中提到的“在 Dashboard 和 Market 之间插入 OpenMesh”的说法，在当前真实代码中需要重新解释：  
因为当前主导航里并不存在名为 `Market` 的导航 case，V1 的“市场能力”并不是以一个 `NavigationPage.market` 的形式直接存在。

## 4.2 当前日志入口策略

`AppConfig.swift` 中已经明确：

- `showLogsInUI = false`

这意味着当前商业化版本默认隐藏日志入口。  
这也说明当前产品已经存在“商业化裁剪开关”的设计习惯，V2 可以延续这种“通过配置控制非核心调试入口”的思路。

## 4.3 当前 Dashboard 的职责

`DashboardView.swift` 当前主要承担以下职责：

- 显示当前选中的配置名称
- 显示 VPN 连接状态与切换开关
- 连接时显示状态网格（如内存、连接数、流量）
- 响应 `selectedProfileDidChange`
- 响应供应商更新状态通知
- 读取当前已选 profile 与供应商更新标记

这说明当前 Dashboard 的定位非常明确：  
它是“当前配置 + 当前连接状态 + 当前运行状态”的总览页，而不是市场页，也不是设置页。

## 4.4 当前 Profiles 的职责

`ProfilesView.swift` 当前已经形成一套完整的本地配置管理能力，包括：

- 列出本地 `Profile`
- 新建配置
- 导入配置
- 编辑配置
- 删除配置
- 切换当前选中配置
- 同步 `SharedPreferences.selectedProfileID`
- 通过 `NotificationCenter` 广播配置变化

这说明：

1. 本地配置能力已经是一个成熟、可复用的能力闭环
2. V2 不需要重做“配置最终落地与选择”逻辑
3. V2 更合理的做法，是在链上供应商支付/下载完成后，接入这条已有的配置落地链路，而不是旁路重造一套新配置系统

## 4.5 当前推荐供应商与市场入口

`TrafficMarketView.swift` 当前已经承担一部分“市场入口”职责：

- 调用 `MarketService.shared.fetchMarketRecommendedCached()` 拉取推荐供应商
- 展示供应商卡片
- 支持“安装 / 重新安装 / 更新”三种动作样式
- 支持打开“供应商市场”窗口
- 支持打开“导入安装”窗口
- 维护本地已安装供应商哈希与待初始化规则集状态

这说明当前的供应商能力并不是不存在，而是已经以“推荐供应商视图 + 市场管理窗口 + 安装向导”的组合方式存在。

## 4.6 当前供应商市场管理能力

`ProviderMarketManagerView.swift` 表明，当前市场系统已经具备：

- Marketplace / Installed 双分段
- 搜索
- 区域过滤
- 排序
- 刷新
- 已安装供应商管理
- 更新判断
- 重新安装
- 卸载

这说明 V1 的市场能力其实已经相当完整，只是它的数据源仍然是中心化市场服务，而非链上供应商目录。

## 4.7 当前安装链路

`ProviderInstallWindowManager.swift` + `ProviderInstallWizard.swift` 已经形成比较完整的安装向导链路：

- 打开独立安装窗口
- 显示安装步骤与状态
- 默认调用 `MarketService.shared.installProvider(...)`
- 安装完成后刷新本地安装状态
- 维护安装过程中的步骤、进度、异常提示
- 与本地 `SharedPreferences` 中的安装状态映射联动

这里非常关键：  
当前系统已经具备“供应商配置安装”的成熟 UI 和流程控制层。  
因此，V2 更合理的做法不是推翻安装向导，而是把“链上供应商选择 + x402 支付 + 拉取配置”作为安装前置阶段，然后尽量复用现有安装终点。

## 4.8 当前 VPN 控制链路

`VPNController.swift` 与 `VPNManager.swift` 可以确认当前 VPN 控制层已经非常具体，包含：

- `ExtensionProfile` 模式优先
- legacy `VPNManager` 回退模式
- 连接/断开控制
- 扩展 reload
- provider message 通信
- urltest
- select outbound
- 连接后自动初始化 pending rule sets
- 根据当前已选 profile 和已安装 provider 判断是否允许连接

特别值得注意的是：

1. `VPNController.canStartVPN()` 会检查：
   - 当前是否存在 `selectedProfileID`
   - 当前 profile 是否映射到一个已安装 provider

2. 当前系统已经把“选择配置 + 选择供应商”与“是否允许启动 VPN”绑定起来

这意味着 V2 不能绕开现有 profile/provider 映射关系直接启动连接，否则会破坏当前运行逻辑。

## 5. 当前架构特征总结

基于已确认代码，当前系统具有以下明显特征。

## 5.1 现有系统是“VPN 客户端优先”的产品结构

当前代码的核心主轴仍然是：

- 选择配置
- 安装配置
- 启动 VPN
- 查看连接和运行状态

供应商市场是为这条主轴服务的。  
也就是说，当前产品不是“钱包应用上挂一个 VPN”，而是“VPN 客户端中嵌入供应商市场能力”。

这对 V2 很重要：  
V2 的 OpenMesh 设计不能反客为主，不能把主产品变成一个过度强调链上概念的钱包界面。

## 5.2 当前系统已存在成熟的安装闭环

当前已有：

- 推荐供应商视图
- 市场管理视图
- 安装窗口管理器
- 安装向导
- 卸载窗口
- 已安装状态映射
- update 检查

这意味着 V2 最具价值的策略是“复用安装闭环”，而不是“重做安装闭环”。

## 5.3 当前系统的数据源仍然偏中心化

从已确认代码看，当前供应商展示和安装仍围绕 `MarketService` 展开。  
推荐列表、安装过程、更新检查，都是当前市场服务体系的一部分。

这与 V2 要求的“链上原生供应商目录”之间存在根本差异。

## 5.4 当前系统已有 SharedPreferences / 本地配置状态映射

代码中已经大量使用本地状态映射，例如：

- `selectedProfileID`
- `installedProviderIDByProfile`
- `installedProviderPackageHash`
- `installedProviderPendingRuleSetTags`
- `providerUpdatesAvailable`

这说明当前应用已经依赖一套成熟的本地状态持久化机制。  
因此，V2 新增的钱包身份、解锁状态、链上供应商缓存、支付授权缓存等，必须谨慎设计与现有持久化边界的关系。

## 5.5 当前系统尚未形成独立的区块链层

尽管现有代码已经具备清晰的 `core` 与 `views` 结构，但尚未看到这些 V2 所需的层：

- Blockchain Config
- RPC Client
- ABI 编解码
- EIP-712 签名
- 钱包 Store
- Keychain 安全持久层
- 链上供应商 Service

这意味着当前项目对 V2 而言，最缺的不是界面骨架，而是区块链与支付基础设施层。

## 6. 当前代码与 V2 目标的主要差距

## 6.1 差距一：导航模型与文档描述不完全一致

V2 原始需求文档使用了“Dashboard 与 Market 之间新增 OpenMesh Tab”的说法。  
但当前真实代码里，主导航 case 只有：

- dashboard
- groups
- connections
- logs
- profiles

并没有 `market` case。

这说明后续设计必须做一个明确决策：

- 是在当前 `NavigationPage` 中新增 `openMesh`，并把它放在 `dashboard` 与 `groups` 之间
- 而不是机械地照抄“Dashboard 与 Market 之间”这句文字

也就是说，**真实代码基线应高于文档措辞**。

## 6.2 差距二：当前市场数据源不是链上目录

V2 需要：

- 从 Base 链协议合约读取供应商目录
- 再读取 metadata
- 再形成链上供应商列表

但当前系统的市场与推荐逻辑仍围绕 `MarketService`。  
这说明 V2 必须新增一条新的供应商数据读取链，而不是在原有 `TrafficMarketView` 上简单改字段。

## 6.3 差距三：当前没有钱包能力

当前已确认代码中，没有看到以下能力已经存在：

- 钱包身份模型
- 助记词生成
- HD 派生
- 私钥加密存储
- 钱包解锁
- Base 链 USDC 余额读取

这意味着钱包能力是从零新增，而不是对现有模块做轻微扩展。

## 6.4 差距四：当前没有 x402 支付前置阶段

当前安装向导默认直接执行 `MarketService.shared.installProvider(...)`。  
V2 需要在这个安装动作之前增加：

- 钱包状态检查
- 余额检查
- 支付确认
- x402 授权签名
- 带授权头拉取配置

因此，V2 的关键设计问题不是“怎么安装”，而是“如何在不破坏现有安装终点的前提下，为安装动作增加支付前置阶段”。

## 6.5 差距五：当前没有独立的 DIY 业务路径

当前系统已经有：

- 本地导入配置
- Offline Import 入口

但这还不是一个完整的 DIY 用户引导路径。  
V2 要求的是：

- 明确的 DIY 说明区
- 自建教程入口
- 与商业购买路径清晰分流

所以，DIY 不是“已有能力完全不用做”，而是“已有导入能力可复用，但缺少产品级引导层”。

## 6.6 差距六：当前没有 V2 所要求的分层架构

V2 文档要求形成：

- Views
- ViewModels / Services
- Blockchain Layer
- Keychain / Persistence

而当前已确认代码更多是：

- Views
- Core Managers / Controllers / WindowManagers
- SharedPreferences / ProfileManager / MarketService（下游依赖）

说明现有架构并非不可扩展，但还没有为区块链模块预留标准化层次。

## 6.7 差距七：当前安全模型不足以承载钱包

现有系统已经有本地配置、共享目录、App Group、VPN Extension 相关的安全与存储实践。  
但钱包系统需要的是另一套安全等级更高的能力：

- 助记词只显示一次
- 私钥加密存 Keychain
- 解锁方式管理
- 解锁态生命周期控制
- 后台自动锁定
- 签名不出设备

这些能力并不能直接从当前 Market / Profile / VPN 逻辑中自然获得。

## 7. 现有代码中可复用的部分

V2 并不是从空白开始。以下能力应优先复用。

## 7.1 可复用：主导航承载机制

虽然当前没有 `openMesh`，但 `NavigationPage.swift` 已经是成熟的主导航定义点。  
因此：

- 新增 `.openMesh`
- 增加标题、图标
- 在 `contentView` 中挂接 `OpenMeshView`

这是合理且低侵入的扩展路径。

## 7.2 可复用：主题系统

`MeshFluxTheme.swift` 已存在，说明新 UI 不应另起一套视觉体系。  
OpenMesh 的所有界面组件应沿用现有主题 token，而不是自创独立视觉语言。

## 7.3 可复用：本地配置落地与选择机制

`ProfilesView`、`ProfileManager`、`SharedPreferences.selectedProfileID` 已形成稳定闭环。  
V2 商业供应商安装成功后，仍应落回现有配置系统，而不是绕过它。

## 7.4 可复用：安装向导与安装窗口

`ProviderInstallWindowManager` 和 `ProviderInstallWizard` 已经具备成熟的安装步骤呈现与异常处理能力。  
V2 更适合把它作为安装终点复用，而不是整体替换。

## 7.5 可复用：VPN 启动约束与扩展控制

`VPNController` / `VPNManager` 已经处理好了：

- 连接状态
- 扩展/legacy 双轨兼容
- provider message
- reload
- URLTest
- outbound 选择

V2 不应重新设计 VPN 控制，而应把目标限定为“让新供应商/新配置正确接入已有控制体系”。

## 7.6 可复用：Offline Import 相关能力

DIY 路径至少可以复用已有的本地导入与离线安装能力。  
也就是说，DIY 不是从 0 到 1，而是从“底层能力已存在”提升到“用户路径与界面表达完整”。

## 8. 高风险区域与设计约束

## 8.1 高风险区域一：VPN 控制层

`VPNController.swift` 和 `VPNManager.swift` 是高风险核心区。  
任何 V2 设计都应遵循：

- 不重写现有连接管理机制
- 不改变 extension / legacy fallback 的逻辑
- 不改变 provider message 的既有协议
- 不破坏 `canStartVPN()` 对 profile/provider 映射的要求

## 8.2 高风险区域二：现有安装闭环

现有安装闭环已经与本地状态映射、provider 更新检查、已安装供应商管理紧密耦合。  
如果 V2 直接推翻 `MarketService.installProvider(...)` 所承载的终端行为，风险会很高。

更安全的做法是：

- 新增支付前置
- 新增链上拉取前置
- 但安装落地尽量保持兼容

## 8.3 高风险区域三：SharedPreferences 现有键空间

当前系统已经使用多组本地状态映射。  
如果 V2 把钱包私钥、敏感解锁态等也混入现有 SharedPreferences 体系，会带来边界混乱和安全风险。

更合理的做法是：

- 地址、非敏感状态可与现有本地状态共存
- 私钥与敏感密文必须进入独立 Keychain 管理层
- 钱包持久层与现有 provider/profile 状态层应逻辑分离

---

## 9. 本轮审计后的关键结论

## 9.1 结论一：V2 应被定义为“增量架构扩展”，而不是重构

当前代码已经有稳定的：

- 主导航
- Dashboard
- 配置管理
- 市场管理
- 安装向导
- VPN 控制

因此，V2 的最佳路径不是推翻重做，而是增量增加：

- 新导航入口
- 新钱包层
- 新链上目录层
- 新支付前置层
- 新 DIY 引导层

## 9.2 结论二：当前最缺的是基础设施层，而不是界面骨架

从 UI 骨架来看，当前项目并不空白。  
真正缺失的是：

- 区块链 RPC 能力
- 合约/事件读取能力
- 钱包密钥与签名能力
- Keychain 安全层
- x402 支付授权层

因此，V2 的技术重点不在“再做几个页面”，而在“建立一套新基础设施，并把它安全接入现有产品”。

## 9.3 结论三：当前“市场”在真实代码中是一个能力集合，不是一个导航枚举

这意味着后续文档中凡是提到“Dashboard 与 Market 之间”时，都应该以真实代码结构重新表述。  
推荐表述应改为：

- 在现有 `NavigationPage` 中，于 `dashboard` 之后插入 `openMesh`
- 不改变 `groups / connections / logs / profiles` 的既有语义

## 9.4 结论四：现有安装链路应作为 V2 的终点复用

V2 新增的链上供应商与支付逻辑，应该更像“安装前置阶段”：

- 链上读取供应商
- 钱包解锁
- x402 支付
- 授权拉取配置

而不是替代当前安装终点。  
最终仍应优先落回现有配置安装与 provider/profile 状态映射体系。
