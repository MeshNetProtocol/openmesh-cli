# 02-V2-现状系统分析与差距评估

## 1. 文档目的

本文档基于当前真实代码和当前真实服务端脚本，判断：

- 现状已经有什么
- 商用供应商场景还缺什么
- 哪些旧能力可以保留

---

## 2. 当前客户端事实

### 2.1 本地模式

当前 `MeshFluxMac` 是 Apple `packet-tunnel-provider` 客户端。  
本地默认配置模式为：

- `tun` inbound
- `shadowsocks` outbound
- `selector(tag=proxy)` 切换节点

这意味着：

- 它适合全局透明代理
- 它适合官方临时受限访问和简单自建节点
- 它不天然等于“商用多用户结算体系”

### 2.2 当前 UI 事实

当前节点编辑和切换 UI 明显偏向：

- Shadowsocks 节点
- 本地 profile
- 已安装配置

因此商用供应商场景不能被现有 UI 设计束缚。

---

## 3. 当前服务端事实

### 3.1 当前 shell 脚本的能力

当前官方脚本只生成：

- 单 sing-box 节点
- 单 Shadowsocks 入站
- 单密码
- 无 `users`
- 无 API
- 无动态管理

### 3.2 当前 shell 脚本适用场景

它已经足够支撑：

- 官方临时受限访问
- DIY 自建节点
- 极简试用入口

### 3.3 当前 shell 脚本不适合的场景

它不能直接支撑：

- 商用多用户
- 按地址流量统计
- 小额自动支付
- 流量桶续费
- 供应商签名凭条
- 多节点汇总

---

## 4. 商用场景的关键差距

### 差距 1：没有链上供应商发现

当前客户端还没有把供应商目录定义为链上注册合约来源。

### 差距 2：没有商用服务端参考实现

当前只有极简单节点脚本，没有：

- provider metadata
- config service
- metering service
- receipt service

### 差距 3：没有钱包 signer session

当前没有解决：

- 私钥如何安全落盘
- 如何避免每次小额支付都重新输入口令
- 如何限制单次和每日自动支付额度

### 差距 4：没有流量桶和预续费模型

当前还没有定义：

- 免费桶
- 续费桶
- 预续费阈值
- 停供规则

### 差距 5：没有凭条真相源

当前没有正式定义：

- 支付收据
- 用量凭条
- 双方恢复机制
- 后续 token 激励的依据

---

## 5. 设计决策

1. **官方临时受限访问继续沿用简单 sing-box 节点。**
2. **商用供应商架构另起一套，不受当前 shell 脚本限制。**
3. **客户端继续保留本地 `tun` 主路线。**
4. **商用热路径采用 x402 自动支付，不等待链上轮询。**
5. **平台不做中心化账本。**
