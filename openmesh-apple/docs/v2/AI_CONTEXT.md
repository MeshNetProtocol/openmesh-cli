## 项目概述

**项目名称**：OpenMesh V2 升级
**核心定位**：OpenMesh 要成为 VPN 行业的 OpenRouter，实现供应商快速切换和去中心化结算。

**当前状态**：Phase 0 准备阶段，技术选型已完成，准备开始 Hysteria2 技术验证。

---

## 核心问题

我们需要实现一个**多用户多服务器的流量统计和配额管理系统**，但面临以下约束：

### 硬约束（不可改变）
1. **不能修改 sing-box 源码** - 维护成本高，升级困难
2. **虚拟网卡层不能改动** - 客户端已实现 TUN 虚拟网卡
3. **客户端已封装 sing-box Go 代码** - 只能通过配置文件修改行为

### 核心需求
1. **用户级流量统计** - 每个用户的流量独立统计
2. **多节点流量汇总** - 用户可以在多个节点间切换，流量需要汇总
3. **预付费配额管理** - 用户预付 USDC 购买流量配额，中心化扣减
4. **供应商快速切换** - 用户可以一键切换供应商，无需重新配置

## 技术决策

### 为什么选择 Hysteria2？

经过对比 Shadowsocks、VMess 和 Hysteria2 三个协议：

| 协议 | 用户级流量统计 | 性能 | 集成复杂度 | 结论 |
|------|--------------|------|-----------|------|
| Shadowsocks | ❌ 不支持 | 中等 | 低 | **不可行** |
| VMess | ✅ V2Ray API | 中等 | 中（gRPC） | 可行但不是最优 |
| Hysteria2 | ✅ 原生 Traffic Stats API | 优秀（QUIC） | 低（HTTP REST） | **最优选择** |

**决策理由**：
1. Hysteria2 原生支持用户级流量统计（Traffic Stats API）
2. 基于 QUIC 协议，性能优秀
3. HTTP REST API，集成简单
4. sing-box 原生支持，只需修改配置文件
5. 虚拟网卡层完全不需要改动

## 系统架构

### 总体架构

```
┌─────────────────────────────────────────────────────────────┐
│                     Discovery Plane                         │
│          Base Registry Contract | Provider Metadata         │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                      Client Plane                           │
│  MeshFluxMac 客户端                                          │
│  - 钱包管理                                                  │
│  - 供应商切换                                                │
│  - 配额监控                                                  │
│  - TUN 虚拟网卡（不变）                                      │
│  - sing-box 核心（Inbound: TUN | Outbound: Hysteria2）      │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼ Hysteria2 协议
┌─────────────────────────────────────────────────────────────┐
│                      Data Plane                             │
│  Hysteria2 节点集群（多节点）                                │
│  - 用户认证                                                  │
│  - 流量转发                                                  │
│  - Traffic Stats API（用户级流量统计）                       │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼ Traffic Stats API
┌─────────────────────────────────────────────────────────────┐
│                    Provider Plane                           │
│  - Metering Service（拉取流量、汇总、扣减配额）              │
│  - Payment Service（x402 支付验证、配额兑换）                │
│  - Config Service（生成 Hysteria2 配置）                     │
└─────────────────────────────────────────────────────────────┘
```

### 关键数据流

**流量统计流**：
```
用户使用流量
  → Hysteria2 节点统计（原生）
  → 暴露 Traffic Stats API
  → Metering Service 定期拉取（每 10 秒）
  → 计算增量并汇总
  → 中心化扣减配额
```

**支付流**：
```
用户购买流量
  → 客户端生成 x402 支付签名
  → Payment Service 验证签名
  → 根据价格兑换配额（如 1 USDC = 100MB）
  → 更新用户配额
```

**配置下发流**：
```
用户选择供应商
  → 客户端请求 Config Service
  → 生成 Hysteria2 配置（包含临时密码）
  → 客户端更新 sing-box outbound
  → 重启 sing-box（TUN 层不变）
```

## 实施计划

### Phase 0: Hysteria2 技术验证（3-5天）
**目标**：验证 Hysteria2 用户级流量统计能力和多节点汇总的可行性

**关键验收标准**：
- ✅ Traffic Stats API 返回用户级数据
- ✅ 流量统计准确度 > 99%
- ✅ 多节点流量汇总无重复计数
- ✅ 节点切换流量连续统计

### Phase 1: Metering Service 开发（1-2周）
**目标**：实现完整的流量统计和配额管理服务

**核心功能**：
- 定期拉取各节点 Traffic Stats API
- 计算流量增量
- 按用户 ID 汇总多节点流量
- 原子性扣减配额
- 配额查询 API

### Phase 2: Payment Service 开发（1周）
**目标**：实现预付费购买流量功能

**核心功能**：
- x402 支付签名验证
- USDC → 流量配额兑换
- 购买记录存储
- 配额追加逻辑

### Phase 3: 客户端集成（2周）
**目标**：客户端支持 Hysteria2 和流量监控

**核心功能**：
- Hysteria2 配置生成
- 配额查询和显示
- 钱包集成（Keychain 加密存储）
- x402 支付签名

### Phase 4: 供应商发现（1周）
**目标**：实现链上供应商发现和快速切换

**核心功能**：
- 读取 Base 链上 Registry 合约
- 供应商列表缓存
- 快速切换逻辑

**获取流量统计**：
```bash

Authorization: Bearer {stats_secret}

Response:

    "user_0x1234abcd": {
      "tx": 1048576,      # 上传字节数
      "rx": 5242880       # 下载字节数
    }
  }
}
```

**客户端配置**（只需修改 outbound）：
```json
{
  "inbounds": [

      "type": "tun",              // ← 不变
      "tag": "tun-in",
      "interface_name": "utun3",
      "inet4_address": "172.19.0.1/30"

  ],
  "outbounds": [
    {
      "type": "hysteria2",        // ← 只改这里

      "server_port": 443,
      "password": "user_password",
```

## 关键设计决策

### 1. 为什么 TUN 层不需要改动？

```
TUN 层职责：接收系统流量 → 根据路由规则转发
sing-box outbound 职责：接收 TUN 转发的流量 → 使用特定协议发送

两者解耦，修改 outbound 不影响 TUN
```

### 2. 为什么使用中心化扣减？

**优点**：
- 简单直接，无需双向签名
- 无需凭条备份和恢复
- 原子性操作，避免超额使用
- 实时性要求低

**权衡**：
- 需要信任 Metering Service
- 商用场景下，供应商运营 Metering Service
- 用户选择信任的供应商

### 3. 为什么定期拉取而不是实时推送？

**定期拉取（推荐）**：
- 简单可靠
- 节点无需主动连接 Metering Service
- 容错性好（拉取失败可重试）

**决策**：Phase 0 使用定期拉取，Phase 2 可选实时推送优化。

## 风险与应对

### 主要技术风险

| 风险 | 可能性 | 影响 | 应对方案 |
|------|--------|------|---------|
| Hysteria2 统计不准确 | 低 | 高 | Phase 0 重点验证，误差 > 5% 则回滚到 VMess |
| 多节点汇总复杂 | 中 | 中 | 简化逻辑（定期拉取 + 增量计算） |
| 配额扣减并发问题 | 中 | 高 | 使用数据库事务 + 行级锁 |

### 回滚方案

**Phase 0 回滚**：
- 触发条件：Hysteria2 验证失败
- 回滚到：VMess + V2Ray API 方案
- 成本：低（仅验证阶段）
