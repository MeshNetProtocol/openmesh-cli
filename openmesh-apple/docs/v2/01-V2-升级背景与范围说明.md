# 01-V2-升级背景与范围说明

## 1. 文档目的

本文档用于定义 MeshFlux macOS 客户端 V2 升级的业务背景、版本目标、建设范围、排除范围以及成功标准。  
本文档不展开具体模块实现细节，也不替代详细技术设计、测试设计和 AI 执行约束文档。其作用是为后续的需求、架构、实施和验收提供统一的范围基线。

---

## 2. 升级背景

### 2.1 V1 当前能力概述

MeshFlux 当前版本（V1）是一款基于 macOS NetworkExtension 框架构建的客户端，已经具备以下核心能力：

- Dashboard：VPN 开关、连接状态与实时流量统计
- Market：从中心化服务端拉取供应商列表，下载并安装 VPN 配置，并支持供应商更新
- Profiles：管理本地 VPN 配置文件，支持本地 JSON 导入
- Connections / Groups / Logs：提供运行时路由管理与日志查看能力

V1 的供应商数据来源是中心化 API 服务，当前系统中不存在链上供应商目录、内嵌钱包或链上支付逻辑。

### 2.2 升级动机

V2 的升级目标不是替换 V1 的现有稳定能力，而是在保持既有 VPN 核心流程、配置安装能力和本地配置使用方式不被破坏的前提下，为产品增加一条新的能力路径：

- 面向普通用户，提供更简单的链上供应商浏览与购买体验
- 面向技术用户，提供 DIY 自建模式入口
- 面向未来演进，建立钱包、链上供应商和 x402 支付的基础设施
- 在产品层面对区块链复杂性进行抽象，避免将技术细节直接暴露给普通用户

因此，V2 本质上是一轮增量式能力升级，而不是一次对现有产品形态的推翻重构。

---

## 3. V2 升级目标

### 3.1 总体目标

在现有 Dashboard 与 Market 之间新增一个 OpenMesh 主入口，使客户端形成“现有 VPN 功能 + 链上供应商能力 + 内嵌钱包能力 + DIY 引导能力”的统一体验。

### 3.2 业务目标

V2 需要达成以下四项核心业务目标：

#### 目标一：引入链上原生供应商目录

客户端不再只依赖中心化 Market 数据源，还应具备从 Base 链智能合约中读取商业供应商目录的能力。链上供应商应来源于协议注册信息，而不是客户端静态内置数据。

#### 目标二：引入内嵌钱包能力

客户端需要为用户提供一个轻量 EVM 钱包能力，但在产品表达上不直接强调区块链钱包概念，而是抽象为“金豆账户”。其底层资产映射为 Base 链上的 USDC，并用于后续支付流程。

#### 目标三：建立 x402 付费安装链路

用户在购买或安装商业供应商配置前，需要能够完成一次面向供应商的 x402 支付授权。该支付流程应具备金额确认、余额检查和失败提示能力。

#### 目标四：提供 DIY 自建引导入口

对于不希望购买商业服务、而是更倾向自建节点或团队内部使用的技术型用户，客户端需要提供清晰的 DIY 入口，能够引导用户前往自建教程或直接导入本地配置，而不强制其进入商业购买路径。

---

## 4. 目标用户与价值定位

### 4.1 消费型用户

这类用户通常不了解区块链技术，更关注的是能否简单、稳定地获取可用服务。  
V2 对这类用户的价值在于：

- 降低理解链上系统的门槛
- 通过“金豆账户”完成余额与支付抽象
- 提供从账户创建、充值、选供应商到安装连接的一体化路径

### 4.2 DIY 技术用户

这类用户更关心灵活性、可控性和自建能力，不一定需要商业购买路径。  
V2 对这类用户的价值在于：

- 提供明确的 DIY 自建教程入口
- 保持本地导入配置能力
- 允许其绕过商业供应商购买流程

### 4.3 商业供应商

商业供应商不在本客户端中完成注册流程，而是通过独立的 vendor-console 网站在链上完成注册与元数据发布。  
V2 客户端的职责是读取供应商信息、展示信息并完成支付与安装，不承担供应商注册后台的职责。

---

## 5. 本次升级范围

### 5.1 产品范围

本次升级包含以下产品层面的变更：

#### 5.1.1 导航结构升级
在现有 Dashboard 与 Market 之间新增 OpenMesh 入口，形成新的主导航结构，但不删除或破坏现有 Tab 的功能。

#### 5.1.2 OpenMesh 主界面
新增一个统一承载 V2 能力的主界面，至少应包含：

- 钱包状态与余额展示区
- 链上供应商浏览区
- DIY 模式入口区

其整体视觉风格需要与现有 MeshFlux 主题保持一致。

#### 5.1.3 钱包能力
提供钱包创建、恢复、解锁、余额读取、充值引导等能力，并将其对用户表达为“金豆账户”，而不是直接暴露复杂链上术语。

#### 5.1.4 链上供应商能力
支持从链上协议读取供应商注册信息，获取供应商元数据，并在客户端中展示供应商卡片、价格和基本状态。

#### 5.1.5 x402 支付与安装
在用户购买或安装商业供应商配置前，必须展示支付确认，并在确认后完成支付授权，再拉取配置并接入现有安装流程。

#### 5.1.6 DIY 引导
提供自建教程入口，并保留本地导入配置的使用方式，使 DIY 用户在离线或链上服务不可用时仍能继续使用部分能力。

## 6. 范围边界说明

### 6.1 本次升级必须满足的边界

V2 必须遵守以下边界：

- 不影响 V1 任何现有功能
- 不要求新增中心化服务端改造
- 支持 macOS 13.0 及以上版本
- 链上服务不可用时，系统必须能优雅降级，而不是崩溃
- 离线状态下仍应允许访问 DIY 分段和现有本地配置能力

### 6.2 本次升级不改变的部分

V2 不以重写现有 VPN 核心能力为目标，因此以下能力原则上保持不变：

- 现有 VPN 连接控制逻辑
- 现有本地配置导入与安装主流程
- 现有 Dashboard / Profiles / Connections / Groups / Logs 的稳定能力
- 现有主题体系与视觉语言基础

这意味着，V2 更偏向于在现有产品骨架上增加新能力入口，而不是重新定义整个客户端。

---

## 7. 排除范围

根据现有需求定义，以下内容明确不属于 V2 范围：

- iOS 版本的区块链功能
- 供应商注册功能
- 法币换金豆能力
- 历史账单查询
- 多钱包支持
- 供应商评分与评论系统

其中，历史账单、多钱包、评分评论等能力被明确放入后续版本规划，不应在 V2 实施过程中扩展范围。

---

## 8. 非功能目标

### 8.1 性能目标

V2 应满足以下关键性能要求：

- 供应商列表首次加载时间不超过 5 秒
- 钱包解锁响应时间不超过 2 秒
- 包含 x402 支付的配置安装流程不超过 30 秒

### 8.2 安全目标

V2 应遵守以下安全基线：

- 私钥存储于 Keychain，并使用 AES-256-GCM 加密
- 签名操作不离开本地设备
- 助记词仅展示一次，不持久化
- 应用进入后台后自动锁定钱包

### 8.3 可用性与降级目标

V2 不应把链上能力变成单点阻断条件。即便链上服务暂时不可用，也必须保证：

- DIY 引导仍可访问
- 本地配置使用方式仍可继续
- 客户端给出明确错误提示而非异常退出

### 8.4 一致性目标

V2 的界面设计需要与现有 MeshFlux 风格保持统一，包括颜色体系、卡片风格和整体交互语义的一致性，避免新增模块像“另一个产品”。

---

## 9. 成功标准

当以下条件全部满足时，可认为 V2 升级达成阶段性目标：

### 9.1 产品目标达成
- 新增 OpenMesh 入口并完成主流程接入
- 用户可在客户端内查看链上供应商
- 用户可创建或使用金豆账户完成支付前置操作
- 用户可完成一次商业供应商的付费安装流程
- 用户可进入 DIY 模式并继续使用本地导入能力

### 9.2 工程边界达成
- 不破坏 V1 现有功能
- 不依赖新增服务端改造
- 在链上异常、离线等情况下具备合理降级能力

### 9.3 质量目标达成
- 满足既定性能目标
- 满足既定安全约束
- 测试通过并达到后续测试文档规定的验收门槛

---

## 10. 关键假设与依赖

V2 的实施建立在以下假设之上：

- Base 链相关合约与 USDC 支付机制可被客户端稳定访问
- 商业供应商通过独立系统完成注册并发布可读取元数据
- 现有本地配置安装主流程可复用为商业配置安装的最终落点
- 客户端具备引入钱包、签名与 RPC 能力所需的依赖基础

如上述假设发生变化，后续总体架构设计和实施计划需要同步调整。

## 12. 结论

MeshFlux V2 的本质，不是将现有客户端改造成一个全新的区块链产品，而是在保持 V1 稳定能力的基础上，增加一套围绕 OpenMesh 的新入口和新能力体系。  
这次升级的核心特征是：

- 增量升级，不是推翻重构
- 能力扩展，不是破坏替换
- 面向用户抽象复杂性，不是直接暴露底层技术
- 兼顾商业供应商与 DIY 用户两类路径，不是只服务单一人群

这一定义，将作为后续需求、架构、设计、实施和验收工作的统一前提。
