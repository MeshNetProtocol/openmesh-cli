# 01-V2-升级背景与范围说明

## 1. 文档目的

本文档用于定义 MeshFluxMac V2 升级的背景、产品目标、范围边界、排除范围和成功标准。  
它回答两个问题：

1. 为什么当前系统要升级
2. 本轮升级到底做到哪里为止

---

## 2. 当前背景

### 2.1 当前系统已具备的能力

`MeshFluxMac` 已经具备以下稳定能力：

- VPN 启停与运行状态展示
- 本地配置管理
- 市场供应商浏览、安装、更新、卸载
- 本地导入配置
- 连接、分组、日志等运行态能力
- 菜单栏主入口

同时，真实代码里已经出现了一个 `OpenMesh` 菜单栏 Tab 和一套静态 UI 骨架，但它目前还没有承载完整业务能力。

### 2.2 当前系统的局限

当前产品的核心叙事仍然是“VPN 客户端 + 配置安装工具”，而不是“供应商路由平台”。  
主要局限包括：

- 商用供应商更像安装列表，而不是可比较、可切换的服务目录
- 私人节点还不是独立的“本地模型式”入口
- 钱包、结算和按需支付能力不存在
- 用户无法通过统一账户和统一交互完成商用供应商授权接入
- OpenMesh 虽然有 UI 入口，但还只是静态占位

---

## 3. 新的产品设想

### 3.1 核心定位

**MeshFlux V2 要做的是 VPN 行业的 OpenRouter。**

新的产品理解如下：

- 商用流量供应商 = 大模型供应商
- 私人流量供应商 = 本地模型 / 自托管模型
- OpenMesh = 模型选择器式的统一入口
- 当前路由供应商 = 当前正在使用的模型
- 流豆账户 = 用户的统一余额和授权账户
- 授权支付 = 类似按次 / 按量的推理结算

### 3.2 设计原则

1. **界面应首先表达“当前在用谁”。**
2. **用户的核心动作是“切换供应商”，不是“安装”。**
3. **私人节点必须像本地模型一样独立接入。**
4. **底层虽然使用 Base / x402 / USDC / 智能合约，但主界面尽量不直接暴露这些术语。**
5. **旧系统先保留，V2 通过新增和重构 OpenMesh 的方式渐进上线。**

### 3.3 用户侧命名

本轮升级统一采用以下命名：

- `流豆`：用户侧余额与结算单位，英文可写作 `Flow Credits`
- `流豆账户`：钱包的用户侧表达
- `商用供应商`：链上注册、需要授权支付后接入的供应商
- `私人节点`：用户自己的节点、团队节点或导入配置
- `授权支付`：x402 支付授权的用户侧表达

---

## 4. 本轮升级目标

### 4.1 产品目标

V2 需要把当前系统升级成一个“供应商聚合与切换平台”，而不是继续堆叠安装功能。

用户在 OpenMesh 中应该能够：

1. 看见当前路由供应商
2. 浏览和比较商用供应商
3. 像切换模型一样切换供应商
4. 使用流豆账户完成授权支付
5. 把私人节点像本地模型一样接入

### 4.2 工程目标

工程层面必须满足：

1. 只改造 `openmesh-apple/MeshFluxMac`
2. 当前 Dashboard / Market / Profiles / VPN 能力保持可用
3. V2 主要通过新增 OpenMesh 业务层和新增界面完成
4. 旧界面不在本轮删除
5. 所有高风险旧模块只允许最小接入，不允许大改

---

## 5. 目标用户

### 5.1 普通用户

特点：

- 更关心“能不能快速用起来”
- 不希望理解区块链和协议细节
- 更喜欢类似 AI 模型选择器的统一体验

价值：

- 快速浏览商用供应商
- 直观看到价格、地区、标签和当前使用状态
- 用流豆账户完成授权支付
- 不需要理解底层 Base、USDC、x402

### 5.2 技术用户

特点：

- 有自建节点、团队节点或导入配置的需求
- 追求可配置、可控
- 不一定愿意走商用支付路径

价值：

- 可以把私人节点当作“本地模型”接入
- 私人节点不依赖流豆账户
- 可以直接导入本地配置，或维护自己的私有供应商

### 5.3 商用供应商

本轮里，商用供应商仍然不在客户端内注册。  
客户端只负责：

- 读取供应商目录
- 展示供应商能力
- 触发授权支付
- 拉取配置并接入现有安装链路

---

## 6. 本轮范围

### 6.1 明确在范围内

本轮升级包括：

- 重新定义 OpenMesh 的产品定位和交互
- 保留现有旧界面，新增并重构 OpenMesh 新体验
- 商用供应商目录读取与展示
- 私人节点入口和接入路径
- 流豆账户能力
- 基于 Base 的授权支付能力
- 商用供应商授权后的自动配置接入
- 渐进式灰度和回退策略

### 6.2 明确不在范围内

本轮不包括：

- 删除旧的 Market / Profiles / Dashboard
- 改造 iOS 版本
- 供应商后台和合约部署文档
- 完整财务系统、发票系统、历史账单中心
- 多钱包并行管理
- 评论、评分、推荐算法

### 6.3 必须保持不变的旧能力

以下能力不是本轮改造目标：

- 现有 VPN 控制逻辑
- 现有安装闭环
- 现有本地导入能力
- 现有 Profiles、Connections、Groups、Logs 的职责
- 现有主题和菜单栏主入口

---

## 7. 核心边界

### 7.1 产品边界

- OpenMesh 不得被重新定义为“安装中心”
- 商用供应商路径的主语义是“使用 / 切换”
- 私人节点路径必须独立于流豆账户
- 主界面不以 USDC、Base、x402 作为主要文案

### 7.2 技术边界

- 不重写 `VPNController.swift` / `VPNManager.swift`
- 不直接改坏 `TrafficMarketView.swift` 的旧逻辑
- 不在 UI 层直接做链上调用、密钥操作和安装落地
- 不在本轮删除旧页面和旧能力

---

## 8. 成功标准

### 8.1 产品成功

满足以下条件可认为产品方向成立：

1. 用户进入 OpenMesh 后能立刻识别当前路由供应商
2. 用户能像选模型一样比较和切换商用供应商
3. 用户能创建、恢复、解锁流豆账户
4. 用户能完成一次授权支付并成功接入商用供应商
5. 用户能独立使用私人节点路径

### 8.2 工程成功

满足以下条件可认为工程策略成立：

1. 旧界面和旧功能未被破坏
2. 新能力主要通过新增模块和新增界面实现
3. 商用供应商链路可按功能开关独立关闭
4. 私人节点链路即使链上异常仍可用

### 8.3 发布成功

满足以下条件才允许进入稳定测试：

1. OpenMesh 主流程可演示
2. 旧系统回归通过
3. 支付、安装、导入均有失败处理
4. 风险与回滚开关明确

---

## 9. 关键假设

本轮方案建立在以下假设之上：

- Base 链和相关合约可被客户端稳定访问
- 商用供应商存在可读取的目录和 metadata
- x402 授权支付可以为配置获取提供有效凭证
- 现有安装链路可以作为商用供应商接入终点复用
- 现有导入和配置体系足以承接私人节点

如果这些假设变化，架构和实施计划需要同步调整。

---

## 10. 结论

本轮升级不是“给 VPN 客户端加一点区块链功能”。  
它本质上是在现有 `MeshFluxMac` 上，建立一个新的产品层：

**OpenMesh 作为统一路由入口，让商用供应商像云模型一样被调用，让私人节点像本地模型一样被接入。**
