# 01-V2-升级背景与范围说明

## 1. 文档目的

本文档回答三个问题：

1. 为什么当前系统必须升级
2. 本轮升级要达到什么产品形态
3. 哪些能力在本轮内，哪些能力明确延后

---

## 2. 当前背景

### 2.1 当前客户端已有的基础

`MeshFluxMac` 已经具备：

- Apple `packet-tunnel-provider` 客户端骨架
- 本地配置管理
- 配置导入和安装
- VPN 启停和状态查看
- 简单的 OpenMesh 入口

### 2.2 当前服务端已有的基础

目前官方脚本已经能快速部署一个极简 sing-box 节点：

- 单节点
- 单 Shadowsocks 入站
- 单密码
- 无多用户
- 无账单
- 无动态配置

这条路径对以下场景已经足够：

- 官方临时受限访问
- DIY 自建节点
- 演示和快速试用

### 2.3 当前系统的根本局限

当前体系还不能支撑商用供应商场景，因为缺少：

- 链上供应商发现
- 供应商身份与 signer 体系
- 小额自动支付
- 多用户 / 多会话数据平面
- 按地址统计流量
- 供应商签名凭条
- OpenRouter 式快速切换

---

## 3. 新的产品设想

### 3.1 核心定位

**OpenMesh 要成为 VPN 行业的 OpenRouter，同时保持供应商发现和结算的去中心化。**

这意味着：

- 用户可以像切换模型一样切换 VPN 供应商
- 供应商目录来自链上注册，而不是中心化后台
- 用户直接向供应商支付 USDC
- 平台抽成在供应商提现时发生
- 后续 token 经济建立在真实流量消费凭条之上

### 3.2 基本叙事

1. 官方临时受限访问是第一个官方供应商
2. 商用供应商通过链上注册被发现
3. 私人节点和 DIY 仍然独立存在
4. 流豆账户是用户的钱包，不是平台账户
5. x402 预付费购买流量是商用场景核心

---

## 4. 本轮目标

### 4.1 产品目标

用户在商用场景下应该能够：

1. 创建或恢复自己的钱包
2. 在 Base 主网 / 测试网之间切换
3. 从链上发现供应商
4. 查看供应商 metadata 和定价
5. 选择供应商并快速切换
6. 在不反复输入口令的前提下自动支付小额费用
7. 获取双方签名的支付和用量凭条

### 4.2 技术目标

本轮必须明确设计以下体系：

1. 供应商注册合约与 metadata 模型
2. 商用供应商参考服务端
3. 钱包解锁和自动支付 signer session
4. 预付费购买流量模型
5. 用量凭条模型和恢复机制
6. sing-box 在商用场景中的职责边界

---

## 5. 本轮范围

### 在范围内

- `MeshFluxMac` 客户端升级
- 官方临时受限访问继续保留
- 商用供应商架构重设
- 链上供应商发现
- x402 自动支付交互
- 钱包 signer session 设计
- 凭条与恢复设计
- 供应商参考实现设计

### 不在范围内

- token 铸造和分发细则
- 推荐算法和声誉系统
- 多链支持
- 完整运营后台
- 最终的供应商经济模型

---

## 6. 成功标准

本轮文档和设计成功，至少满足以下条件：

1. 官方临时受限访问、自建节点、商用供应商、私人节点四种场景边界清楚
2. 商用供应商路径不依赖平台中心化账本
3. 用户钱包安全方案兼顾易用和安全
4. 商用场景支持自动支付和快速切换
5. sing-box 只承担数据平面职责，不承担中心化账本职责
