# 06-V2-数据模型与安全设计

## 1. 文档目的

本文档用于定义 MeshFlux macOS 客户端 V2 升级中的**核心数据模型、持久化边界、安全约束、密码学处理方式和敏感信息保护规则**。

本文档重点回答以下问题：

- V2 中有哪些核心数据模型
- 这些数据模型分别存放在哪里
- 哪些数据属于公开数据，哪些属于敏感数据
- 钱包创建、恢复、解锁、签名、支付授权的安全边界是什么
- Keychain、SharedPreferences、内存态分别承担什么职责
- 日志、缓存、错误和测试中如何避免泄露敏感信息

本文档不定义 UI 布局，不替代总体架构和详细技术设计文档。  
其中目录落点、类型拆分和接口组织以 `05-V2-详细技术设计.md` 为准。

---

## 2. 设计目标

V2 的数据模型与安全设计必须满足以下目标：

### 2.1 安全性
私钥、恢复短语、解锁态和签名材料必须受到严格保护，任何时候不得以明文形式落入普通持久化层。

### 2.2 最小暴露
只在必要时生成、读取和使用敏感数据；不在不必要的模块中传播敏感信息。

### 2.3 分层清晰
公开数据、业务数据、缓存数据、敏感数据和临时内存态必须分层管理，避免边界混乱。

### 2.4 可恢复
用户在设备重启、应用重启后，仍可通过密码或恢复短语重新进入可用状态。

### 2.5 可测试
所有加密、解密、RPC、Keychain 和 HTTP 依赖必须可替换为 Mock，以支持稳定测试。

### 2.6 与现有系统兼容
V2 的新增数据模型不能污染现有 provider/profile/VPN 的状态体系，也不能让现有 SharedPreferences 承担不该承担的敏感职责。

---

## 3. 数据分级模型

V2 所有数据按敏感程度分为四级。

## 3.1 一级：公开业务数据

这类数据可在 UI 中展示，可进入普通缓存层，不需要加密存储。

包括：

- 钱包公开地址
- 链上供应商公开标识
- 供应商名称、地区、标签、价格、网站
- 供应商 metadata URI
- 配置下载 URL（若本身不是密钥或一次性令牌）
- 已安装状态标识
- 最近刷新时间
- 当前选中的 OpenMesh 分段

这类数据可进入：
- SharedPreferences
- 内存缓存
- 供应商列表缓存
- 非敏感日志

---

## 3.2 二级：业务受限数据

这类数据本身不是密钥，但会影响业务一致性，不应随意篡改或暴露过多细节。

包括：

- x402 授权对象
- 支付请求金额
- 授权有效时间
- nonce
- 某次安装流程的中间状态
- provider/profile 映射接入结果
- 某次配置下载响应内容

这类数据原则上：
- 可短期存在内存中
- 必要时可保存非敏感摘要或标识
- 不应长期明文持久化完整原文
- 失败时应及时清除

---

## 3.3 三级：敏感持久化数据

这类数据不能进入普通偏好存储，必须进入安全持久化层。

包括：

- 加密后的私钥密文
- 私钥加密使用的 salt
- 钱包身份对象（含地址、创建时间、解锁方式）
- 生物识别相关保护配置
- 用于校验解锁的一次性安全材料

这类数据必须进入：
- Keychain
- 或等价系统安全存储

---

## 3.4 四级：极高敏感临时数据

这类数据原则上只允许驻留内存，在使用完成后必须尽快擦除。

包括：

- 私钥明文
- 助记词原文
- seed
- 解密后的 secp256k1 原始密钥字节
- 待签名 digest 的完整原始中间材料
- 用户刚创建时显示的恢复短语

这类数据不得：
- 写入 SharedPreferences
- 写入普通日志
- 写入调试输出
- 写入 crash 上报正文
- 写入临时文本文件
- 写入自动恢复状态

---

## 4. 核心数据模型设计

## 4.1 `MeshWalletIdentity`

### 作用
表示本地钱包的公开身份信息，用于 UI 展示、状态恢复和解锁入口识别。

### 建议结构

```swift
struct MeshWalletIdentity: Codable, Hashable {
    let address: String
    let createdAt: Date
    let unlockMethod: WalletUnlockMethod
}
```

### 字段说明
- `address`
  - 用户公开地址
  - 可显示、复制、作为充值地址
- `createdAt`
  - 钱包创建或恢复完成时间
- `unlockMethod`
  - 当前账户使用的解锁方式

### 存储位置
- Keychain：建议整体保存一份
- 也可冗余保存 address 到普通状态层，供快速显示

---

## 4.2 `WalletUnlockMethod`

### 作用
表示钱包解锁方式配置。

### 建议结构

```swift
enum WalletUnlockMethod: String, Codable, CaseIterable {
    case faceID
    case touchID
    case password
    case pin
}
```

### 说明
- 实际设备不支持 Face ID 时，不应暴露该选项
- macOS 场景下常见会退化为 Touch ID 或密码
- 若实现 PIN，则 PIN 不应单独作为明文存储对象

---

## 4.3 `OnChainSupplier`

### 作用
表示客户端内使用的链上供应商聚合模型。

### 建议结构

```swift
struct OnChainSupplier: Identifiable, Codable, Hashable {
    let supplierId: String
    let supplierIdHash: String
    let profileAddress: String
    let ownerAddress: String
    let metadataURI: String
    let expiry: Date
    let isSuspended: Bool

    var displayName: String?
    var description: String?
    var pricePerGbGoldBean: Decimal?
    var configURLs: [String]
    var tags: [String]
    var region: String?
    var website: String?

    var id: String { supplierId }
    var isActive: Bool { !isSuspended && expiry > Date() }
}
```

### 字段分层
#### 链上基础字段
- `supplierId`
- `supplierIdHash`
- `profileAddress`
- `ownerAddress`
- `metadataURI`
- `expiry`
- `isSuspended`

#### metadata 合并字段
- `displayName`
- `description`
- `pricePerGbGoldBean`
- `configURLs`
- `tags`
- `region`
- `website`

### 安全性说明
该模型属于公开业务数据，可缓存、可显示、可写测试快照。

---

## 4.4 `SupplierMetadata`

### 作用
表示供应商 metadata JSON 的标准结构。

### 建议结构

```swift
struct SupplierMetadata: Codable, Hashable {
    let name: String
    let description: String
    let region: String?
    let tags: [String]
    let pricePerGbUSDC: String
    let configURLs: [String]
    let website: String?

    enum CodingKeys: String, CodingKey {
        case name
        case description
        case region
        case tags
        case pricePerGbUSDC = "price_per_gb_usdc"
        case configURLs = "config_urls"
        case website
    }
}
```

### 字段约束
- `name`：必填，1~64 字符
- `description`：建议不超过 300 字符
- `tags`：建议不超过 8 个
- `configURLs`：至少 1 个 URL
- `pricePerGbUSDC`：字符串表示，避免 JSON 浮点误差

### 校验规则
- 必须是合法 JSON
- `configURLs` 不可为空
- 价格必须可解析为非负 Decimal
- 非法 metadata 不应导致应用崩溃

---

## 4.5 `X402Authorization`

### 作用
表示一笔配置拉取请求所需的支付授权头对象。

### 建议结构

```swift
struct X402Authorization: Codable, Hashable {
    let scheme: String
    let network: String
    let from: String
    let to: String
    let value: String
    let validAfter: String
    let validBefore: String
    let nonce: String
    let v: UInt8
    let r: String
    let s: String
}
```

### 字段说明
- `scheme`
  - 一般为 `x402`
- `network`
  - 如 `base-mainnet` / `base-sepolia`
- `from`
  - 用户地址
- `to`
  - 收款地址
- `value`
  - USDC 原子单位字符串
- `validAfter`
  - 生效时间
- `validBefore`
  - 失效时间
- `nonce`
  - 防重放值
- `v/r/s`
  - 签名结果

### 安全性说明
该对象不是私钥，但属于受限业务数据。  
可以在单次流程内存在，但不应被长期明文缓存。

---

## 4.6 `SupplierPaymentContext`

### 作用
表示当前待确认的一次支付上下文。

### 建议结构

```swift
struct SupplierPaymentContext: Identifiable, Hashable {
    let id: UUID
    let supplier: OnChainSupplier
    let amountGoldBean: Decimal
    let amountUSDCAtomic: String
    let currentBalance: Decimal
}
```

### 说明
- 该对象只应存在于 UI / Coordinator 流程中
- 不应长期持久化
- 用于支付确认 Sheet 展示

---

## 4.7 `WalletEncryptedPayload`

### 作用
表示保存在 Keychain 中的加密私钥密文对象。

### 建议结构

```swift
struct WalletEncryptedPayload: Codable, Hashable {
    let ciphertext: Data
    let nonce: Data
    let tag: Data
    let version: Int
}
```

### 说明
- 用于支持加密版本升级
- 不直接保存助记词
- 不直接保存 seed
- 真正持久化时可存为 JSON 编码后 Data

---

## 5. 持久化边界设计

## 5.1 Keychain 存储项

Keychain 中应保存以下内容：

- `wallet.identity`
- `wallet.encryptedPrivateKey`
- `wallet.salt`
- `wallet.biometricPolicy`（如需要）
- 未来版本扩展的安全材料

### 命名建议
使用明确且稳定的 service/account 组合，例如：

- service: `com.meshnetprotocol.MeshFlux.wallet`
- account: `identity`
- account: `encryptedPrivateKey`
- account: `salt`

### 原则
- 所有敏感项单独存储
- 不把多个字段拼成不透明巨对象后随意读写
- 便于后续单项迁移和故障排查

---

## 5.2 SharedPreferences 存储项

SharedPreferences 可保存以下非敏感数据：

- `openmesh.selectedSegment`
- `openmesh.lastSupplierRefreshAt`
- `openmesh.cachedSuppliers`
- `wallet.publicAddress`
- `wallet.hasWallet`
- `wallet.lastKnownBalance`
- `wallet.lastBalanceRefreshAt`

### 原则
- 可删可重建
- 不能保存私钥
- 不能保存助记词
- 不能保存密码
- 不能保存未加密签名材料

---

## 5.3 内存态存储项

仅驻留内存，不写磁盘的数据包括：

- 解锁后的私钥
- 恢复短语数组
- seed
- 支付签名流程中的 digest
- 当前支付上下文
- 当前安装中的原始配置数据

### 清理规则
以下时机必须清理：

- 应用退后台
- 用户显式锁定
- 支付完成
- 支付失败
- 安装失败
- 页面 dismiss 且流程结束

---

## 6. 钱包安全设计

## 6.1 钱包创建流程

### 流程步骤
1. 生成 12 词助记词
2. 由助记词生成 seed
3. 使用 BIP-44 路径派生 secp256k1 私钥
4. 计算以太坊地址
5. 生成随机 salt
6. 使用用户密码对私钥进行加密
7. 将密文与 salt 写入 Keychain
8. 保存 `MeshWalletIdentity`
9. 返回助记词给 UI 一次性展示
10. 用户确认后，从内存中清除助记词原文

### 安全约束
- 助记词只展示一次
- 助记词不写入任何持久层
- 私钥只以密文形式保存
- 创建成功后可直接进入 unlocked 状态，但仍需遵守后台自动锁定

---

## 6.2 钱包恢复流程

### 流程步骤
1. 用户输入 12 词恢复短语
2. 校验词数与词典合法性
3. 派生私钥与地址
4. 使用新密码加密私钥
5. 写入 Keychain
6. 覆盖或重建 `MeshWalletIdentity`
7. 清除助记词原文
8. 进入 unlocked 状态

### 安全约束
- 恢复短语输入失败时，不记录原文到日志
- 错误提示只显示“恢复短语无效”之类业务语义
- 不暴露哪个词出错

---

## 6.3 钱包解锁流程

### 解锁输入
- 密码
- 或系统生物识别
- 或平台允许的等价方式

### 解锁步骤
1. 从 Keychain 读取加密私钥与 salt
2. 使用密码派生密钥
3. 解密出私钥明文
4. 存入内存态
5. 刷新余额
6. 更新 `MeshWalletStore.state = .unlocked`

### 解锁失败处理
- 返回统一错误
- 不保留半解锁状态
- 不重复尝试自动降级为其他解锁方式
- 不输出解密中间结果

---

## 6.4 自动锁定策略

钱包必须在以下场景自动锁定：

- 应用进入后台
- 应用窗口长时间失焦（可选策略）
- 用户点击“锁定账户”
- 连续解锁失败达到阈值后
- 发生内存安全异常或严重错误后

### 锁定动作
- 清空内存中的私钥明文
- 清空当前待签名上下文
- 将状态切回 `.locked(address:)`

---

## 7. 密码学与加密设计

## 7.1 助记词规则

- 使用 BIP-39 英文 12 词
- 必须通过 checksum 校验
- 内部实现不得允许任意长度短语混用
- UI 恢复页应按空格或逐词输入方式标准化

---

## 7.2 派生路径规则

统一采用：

```text
m/44'/60'/0'/0/0
```

### 说明
- 用于 EVM 地址派生
- V2 明确不支持多钱包、多账户索引切换
- 后续如扩展多账户，需单独版本设计

---

## 7.3 地址生成规则

- 由 secp256k1 公钥生成 Ethereum address
- UI 展示统一使用校验格式或标准 hex 地址
- 业务上“地址即 ID”

### 派生结果规则
同一恢复短语必须始终派生出同一地址。

---

## 7.4 私钥加密规则

### 加密方案
- KDF：PBKDF2
- 对称加密：AES-256-GCM

### 必选要求
- 必须使用随机 salt
- 必须使用随机 nonce
- 必须校验 GCM tag
- 失败时不得返回部分明文

### 不允许
- 明文私钥落盘
- 明文 seed 落盘
- 使用固定 salt
- 使用固定 nonce

---

## 7.5 签名规则

### 适用场景
- x402 支付授权
- EIP-712 digest 签名

### 要求
- 签名必须在本地设备完成
- 私钥不出设备
- 签名前必须处于 unlocked 状态
- 签名完成后不持久化 digest 原文

---

## 8. x402 支付安全设计

## 8.1 授权对象生命周期

x402 授权对象只应在以下范围内存在：

- 支付确认后
- 配置下载请求发起前
- 单次流程中短暂存在

### 不应持久化的内容
- 完整 authorization 原文
- 签名前完整 message 原文
- 原始 digest
- 未过期授权头

---

## 8.2 时间窗口设计

授权对象必须包含：

- `validAfter`
- `validBefore`

### 规则
- 窗口应尽量短
- 过期授权必须视为无效
- 本地时间异常时应阻止继续支付

---

## 8.3 nonce 设计

nonce 必须满足：

- 每次支付唯一
- 长度固定
- 使用高质量随机源生成
- 不允许复用
- 不允许使用可预测顺序值

---

## 8.4 金额单位规则

### 内部规则
- 链上金额使用 USDC 原子单位
- 业务展示使用金豆
- 金豆 : USDC = 1 : 1

### 要求
- UI 不使用浮点做最终金额存储
- 网络传输中的金额字段以字符串表示
- 计算时使用 Decimal / BigUInt 等精确类型

---

## 9. 供应商数据安全设计

## 9.1 供应商来源可信边界

供应商数据来自两部分：

### 链上来源
- 供应商 hash / id
- profileAddress
- ownerAddress
- metadataURI
- expiry
- suspension 状态

### metadata 来源
- name
- description
- region
- tags
- website
- price
- configURLs

### 安全边界
- 链上字段优先作为可信索引基础
- metadata 视为可变展示数据，必须校验格式
- metadata 非法不能导致程序崩溃

---

## 9.2 `configURLs` 校验规则

在发起配置拉取前，必须校验：

- URL 解析成功
- scheme 为 `https`
- 不允许 `file://`
- 不允许本地回环与危险内网地址（如需要可加白或黑名单策略）
- 列表为空时不可继续支付安装

---

## 9.3 供应商缓存规则

### 可缓存
- 供应商列表
- metadata 解析结果
- 最近刷新时间

### 不可缓存
- 完整配置原文的长期副本
- 已签名 authorization
- 已过期支付上下文

### TTL 建议
- 供应商列表缓存：300 秒
- metadata 缓存：300 秒
- 错误缓存：不建议长期缓存

---

## 10. 日志与脱敏设计

## 10.1 严禁写入日志的内容

以下内容禁止进入任何普通日志：

- 私钥明文
- 助记词
- seed
- 用户密码 / PIN
- 解密后的 key material
- 完整 x402 Authorization
- 完整 nonce 原文
- 完整 digest 原文

---

## 10.2 允许记录的日志

可以记录以下业务级信息：

- 钱包是否存在
- 钱包当前状态（noWallet / locked / unlocked）
- 当前地址的缩略形式
- 供应商数量
- 某个 supplierId 的加载失败
- 支付流程阶段
- 安装成功 / 失败
- 错误分类码

### 地址脱敏建议
```text
0x1234...5678
```

---

## 10.3 错误文案策略

### 对用户
使用业务语义，例如：

- 账户已锁定
- 余额不足
- 供应商信息读取失败
- 配置获取失败，请重试
- 恢复短语无效

### 对开发日志
可记录：
- 错误类型
- 模块名
- 阶段名
- 供应商 ID 摘要

但仍不得记录敏感原文。

---

## 11. 内存安全与清理设计

## 11.1 需要主动清理的对象

以下对象在流程结束后必须清理：

- `unlockedPrivateKey`
- `mnemonicWords`
- `seed`
- `paymentContext`
- `authorization`
- `rawConfigData`

---

## 11.2 清理时机

- 创建钱包完成后
- 恢复钱包完成后
- 关闭助记词页面后
- 支付请求结束后
- 安装完成后
- 进入后台后
- 用户手动锁定后

---

## 11.3 清理原则

- 优先置空引用
- 避免在多个对象间复制敏感 Data
- 对敏感 Data 的生命周期尽可能短
- 不将敏感数据放入长生命周期单例中

---

## 12. 权限与本地安全边界

## 12.1 Keychain 使用原则

- 使用系统 Keychain 保存高敏感数据
- 如果启用生物识别，必须通过系统安全策略保护访问
- 不以自定义明文文件替代 Keychain

---

## 12.2 App Group 边界

### 原则
- 钱包私钥不应放入共享 App Group 普通文件中
- 与 VPN Extension 共享的数据仍应只限非敏感运行态
- 钱包模块与 VPN 扩展不直接共享密钥材料

---

## 12.3 文件系统使用原则

以下内容禁止写入普通文件系统：

- 私钥明文
- 助记词
- 未脱敏授权对象
- 原始签名中间结果

配置文件在安装流程中的落地属于业务需要，但必须仅限配置内容，不夹带额外敏感钱包数据。

## 13. 测试安全设计

## 13.1 Mock 设计要求

以下对象必须可 Mock：

- `BlockchainRPCClientProtocol`
- `KeychainManagerProtocol`
- `URLSessionProtocol`

### 原因
- 单元测试不应访问真实 RPC
- 单元测试不应污染真实 Keychain
- 单元测试不应发真实配置请求

## 13.2 测试中可使用的假数据

可以在测试中使用：

- 固定测试助记词
- 固定测试私钥
- 固定测试地址
- 固定测试 EIP-712 向量
- 固定测试 metadata JSON

### 但要求
- 仅限测试 target
- 不进入生产代码常量
- 不输出到生产日志

---

## 13.3 测试断言重点

应重点测试：

- 助记词生成与恢复一致性
- 同一恢复短语地址派生稳定性
- 加密后可正确解密
- 错误密码无法解密
- 后台后自动锁定
- 解锁后签名地址与 identity 地址一致
- authorization 时间窗口校验
- metadata 非法时的容错
- configURL 校验
- 敏感数据不会落入 SharedPreferences

## 14. 数据迁移与版本化设计

## 14.1 版本化必要性

钱包加密结构未来可能调整，因此敏感对象应具备版本字段。

### 建议对象
- `WalletEncryptedPayload.version`

---

## 14.2 迁移策略

### V2 首版
- 默认 `version = 1`
- 仅支持读取当前版本

### 未来版本
- 若发现旧版本结构，先迁移后写回
- 迁移失败不得破坏原有 Keychain 项
- 应先备份旧项，再写入新项

---

## 15. 失效与回滚设计

## 15.1 钱包解锁失败
- 保持锁定态
- 不写入任何新状态
- 不删除原有密文

## 15.2 恢复失败
- 不覆盖旧钱包
- 不写入半成品 identity
- 不持久化错误输入

## 15.3 支付失败
- 不持久化完整 authorization
- 不修改安装状态
- 不改变现有 provider/profile 映射

## 15.4 配置获取失败
- 不安装半成品配置
- 不写入错误映射状态
- 允许用户重新发起流程

## 17. 关键结论

V2 的数据模型与安全设计必须建立在以下核心原则上：

1. **地址公开，私钥绝不公开**  
   地址是公开身份，私钥只能本地加密保存并在短时内存中使用。

2. **助记词只展示一次，绝不落盘**  
   助记词是恢复根，不能进入普通持久化层。

3. **SharedPreferences 只管非敏感业务状态**  
   它可以保存地址、缓存和 UI 状态，但不能承担密钥保存职责。

4. **Keychain 是高敏感数据唯一持久化落点**  
   加密私钥、salt 和身份对象必须在系统安全边界内管理。

5. **支付授权是短生命周期对象**  
   x402 Authorization 只应服务于单次配置获取流程，不应长期缓存。

6. **敏感数据优先内存态、最短生命周期、尽快清理**  
   一旦流程完成、失败或退后台，就必须主动回收。

7. **安全边界必须独立于现有 VPN 核心逻辑**  
   钱包与支付不能污染现有 VPN 和 provider/profile 的稳定数据链路。

以上原则是 V2 实现过程中最容易失控、也最不能退让的底线。
