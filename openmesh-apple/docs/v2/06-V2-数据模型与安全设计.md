# 06-V2-数据模型与安全设计

## 1. 文档目的

本文档定义：

- 钱包安全边界
- signer session 设计
- 支付收据和用量凭条的数据模型
- 商用供应商热路径的安全要求

---

## 2. 钱包安全原则

1. 助记词不得明文持久化
2. 私钥不得明文持久化
3. 私钥明文只允许在 signer session 中存在于内存
4. 自动支付必须有上限
5. 钱包易用性和安全必须同时成立

---

## 3. 钱包落盘方案

推荐方案：

### 3.1 加密 blob

允许把钱包材料以加密 blob 形式存储在本地。  
这个 blob 不可直接使用，必须依赖设备绑定能力解密。

推荐实现：

1. 先生成随机 `walletDataKey`
2. 用 `walletDataKey` 加密钱包材料形成 blob
3. 再用设备绑定密钥包裹 `walletDataKey`
4. 设备绑定密钥放入 Keychain / Secure Enclave 体系

### 3.2 设备绑定保护

优先使用：

- Keychain
- Secure Enclave
- LocalAuthentication / Touch ID / 设备凭证

### 3.3 解锁流程

1. 用户首次创建或恢复钱包
2. 钱包材料被加密后落盘
3. 用户在需要自动支付前解锁一次
4. 系统通过设备绑定能力解开 `walletDataKey`
5. 再解开钱包材料
6. 解锁后只在内存中持有可签名私钥

### 3.4 快捷解锁

若用户启用本机生物识别或设备认证，可允许：

- Touch ID
- 系统认证
- 本机安全凭证

用于重新打开 signer session。  
但无论采用何种快捷解锁方式，助记词和私钥都不得明文落盘。

## 4. signer session 设计

### 4.1 目标

避免：

- 每次启动 VPN 都重新输入口令
- 每次 402 都重新输入口令

同时避免：

- 私钥长时间明文常驻

### 4.2 建议字段

- `unlockedAt`
- `unlockExpiresAt`
- `singlePaymentLimit`
- `dailyPaymentLimit`
- `dailySpent`
- `requiresManualConfirm`
- `unlockMethod`
- `deviceBindingEnabled`

### 4.3 自动锁定条件

- 超时
- App 退出
- 用户手动锁定
- 每日额度耗尽
- 设备锁屏或系统安全上下文失效

### 4.4 风险控制

若钱包已解锁但支付超限：

- 不得自动签名
- 必须要求用户重新确认

## 5. 凭条设计

## 5.1 PaymentReceipt

职责：

- 证明某个流量桶已支付
- 作为放行和恢复依据

保存位置：

- 客户端
- 供应商服务器

建议附带：

- `bucketBytes`
- `bucketSequence`
- `amountUSDC`
- `paymentReference`

## 5.2 UsageReceipt

职责：

- 证明某个时间窗口或某个桶的实际用量
- 作为后续 token 激励依据

保存位置：

- 客户端
- 供应商服务器

建议附带：

- `remainingBucketBytes`
- `bucketSequence`
- `meteringSigner`

## 5.3 恢复规则

任一方丢失最新凭条时，可向另一方恢复。  
以签名可验证的最新凭条为准。

---

## 6. 热路径安全要求

1. 热路径放行不得等待链上余额轮询
2. 支付通过后必须立即签发 PaymentReceipt
3. 预续费阈值应早于桶耗尽
4. PaymentReceipt 和 UsageReceipt 都必须可重放验证

默认建议：

- 免费桶：40MB
- 续费桶：40MB
- 预续费阈值：剩余 5MB 或 20%

---

## 7. 数据平面安全要求

1. 地址或地址派生值必须成为可统计的用户标识
2. 数据平面必须支持按地址或 session 停供
3. 节点原始统计要能汇总到地址级视图

---

## 8. 官方临时受限访问的安全边界

官方临时受限访问允许继续简化：

- 可不要求钱包
- 可不要求完整收据体系
- 可只做基础限额与日志

但不得污染商用场景的正式设计。
