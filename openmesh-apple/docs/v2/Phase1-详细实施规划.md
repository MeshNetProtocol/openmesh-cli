# Phase 1 - 详细实施规划

> 创建时间：2026-03-24
> 状态：规划中

## 1. Phase 1 目标

跑通"连接 → 统计 → 触发 402"的最小闭环，但要考虑：
1. 当前没有支持 V2Ray API 的服务器
2. 还没有流量统计服务和管理页面
3. 客户端还没有钱包功能
4. 未来需要将用户 ID 迁移到区块链钱包地址

---

## 2. 设计原则

### 2.1 用户标识的演进路径

**阶段 1（当前）**: 简单字符串 ID
```
user_id: "alice"
```

**阶段 2（过渡）**: 支持多种 ID 类型
```
user_id: "alice"  // 兼容旧格式
user_id: "0x1234...abcd"  // 区块链地址
```

**阶段 3（最终）**: 纯区块链地址
```
user_id: "0x1234...abcd"
```

### 2.2 架构设计考虑

**关键设计决策**：
- 用户标识统一使用字符串类型（支持从简单 ID 到钱包地址的平滑过渡）
- 所有 API 和数据库使用 `user_id` 字段（不使用 `username` 或 `account`）
- 认证层与业务层分离（未来可以替换认证方式）

## 4. 用户标识迁移策略

### 4.1 数据库设计

```sql
CREATE TABLE users (
    user_id TEXT PRIMARY KEY,           -- 支持 "alice" 或 "0x1234...abcd"
    user_type TEXT DEFAULT 'simple',    -- 'simple' | 'wallet'
    display_name TEXT,                  -- 显示名称
    created_at TIMESTAMP,
    updated_at TIMESTAMP
);
```

### 4.2 API 设计

**创建用户（兼容两种方式）**：

```bash
# 方式 1：简单 ID（当前）
POST /api/v1/users
{
  "user_id": "alice",
  "user_type": "simple",
  "usdc_amount": 5.0
}

# 方式 2：钱包地址（未来）
POST /api/v1/users
{
  "user_id": "0x1234567890abcdef1234567890abcdef12345678",
  "user_type": "wallet",
  "usdc_amount": 5.0
}
```

### 4.3 客户端配置

```json
{
  "user_identity": {
    "type": "simple",           // 'simple' | 'wallet'
    "value": "alice"            // 简单 ID 或钱包地址
  }
}
```

### 4.4 迁移路径

**阶段 1（当前）**：
- 使用简单字符串 ID
- 所有代码使用 `user_id` 字段

**阶段 2（过渡）**：
- 同时支持简单 ID 和钱包地址
- 添加 `user_type` 字段区分
- 客户端可选择使用钱包地址

**阶段 3（最终）**：
- 仅支持钱包地址
- 移除简单 ID 支持
- 所有用户必须使用钱包

## 5. 技术栈选择

### 5.1 服务端

- **记账服务**: Flask (已有)
- **数据库**: SQLite → PostgreSQL (生产环境)
- **流量采集**: Python + gRPC
- **管理页面后端**: Flask (扩展现有服务)

### 5.2 客户端

- **钱包**: ethers.js / web3.js
- **密钥存储**: macOS Keychain
- **UI**: Swift (原生 macOS)

### 5.3 管理页面

- **前端**: React + Vite
- **UI 库**: Ant Design / Material-UI
- **图表**: ECharts / Recharts
- **状态管理**: Zustand / Redux

## 7. 风险与应对

### 风险 1: sing-box V2Ray API 集成复杂

**应对**:
- Phase 1.1 优先验证 API 可用性
- 如果 V2Ray API 不可行，考虑其他方案（eBPF、日志解析）

### 风险 2: 钱包安全性

**应对**:
- 使用成熟的加密库
- 遵循最佳实践
- 进行安全审计

### 风险 3: 用户标识迁移复杂度

**应对**:
- 从设计阶段就考虑兼容性
- 使用抽象层隔离变化
- 提供清晰的迁移路径
