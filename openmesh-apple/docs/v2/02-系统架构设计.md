# 系统架构设计

> **文档目的**：定义基于 Hysteria2 的 V2 系统架构，说明各模块职责和数据流。

---

### 1.2 架构分层

| 层级 | 职责 | 关键技术 |
|------|------|---------|
| **Discovery Plane** | 供应商发现 | Base 链上合约 |
| **Client Plane** | 用户交互、配置管理 | Swift + sing-box |
| **Data Plane** | 流量转发、用户认证 | Hysteria2 |
| **Provider Plane** | 流量统计、配额管理、支付验证 | Go/Python + SQLite |

## 2. 客户端架构

### 2.1 模块组成

```
MeshFluxMac
├── UI Layer
│   ├── 供应商列表视图
│   ├── 钱包管理视图
│   ├── 配额监控视图
│   └── 设置视图
├── Business Logic Layer
│   ├── WalletManager（钱包管理）
│   ├── ProviderSwitcher（供应商切换）
│   ├── QuotaMonitor（配额监控）
│   └── PaymentService（支付服务）
├── Network Layer
│   ├── TUN Manager（虚拟网卡管理）
│   ├── sing-box Wrapper（sing-box 封装）
│   └── ConfigGenerator（配置生成）
└── Storage Layer
    ├── Keychain（钱包加密存储）
    └── UserDefaults（配置存储）
```

### 2.2 关键模块职责

#### TUN Manager（不变）
- 创建和管理虚拟网卡
- 路由规则配置
- 流量过滤

#### sing-box Wrapper
- 启动/停止 sing-box 进程
- 动态更新配置
- 监控连接状态

**配置示例**：
```json
{
  "inbounds": [
    {
      "type": "tun",
      "tag": "tun-in",
      "interface_name": "utun3",
      "inet4_address": "172.19.0.1/30"
    }
  ],
  "outbounds": [
    {
      "type": "hysteria2",
      "tag": "proxy",
      "server": "example.com",
      "server_port": 443,
      "password": "user_password",
      "tls": {
        "enabled": true,
        "server_name": "example.com"
      }
    }
  ]
}
```

#### WalletManager
- 创建/恢复钱包
- 钱包加密存储（macOS Keychain）
- Signer Session 管理
- x402 支付签名

#### ProviderSwitcher
- 从链上读取供应商列表
- 请求供应商配置
- 动态切换 sing-box outbound
- 不中断 TUN 层

#### QuotaMonitor
- 定期查询配额余额
- 配额不足提醒
- 触发购买流程

## 3. 服务端架构

### 3.1 Hysteria2 节点

**职责**：
- 用户认证（基于 password）
- 流量转发
- 用户级流量统计（原生）
- 暴露 Traffic Stats API

**配置示例**：
```json
{
  "listen": ":443",
  "tls": {
    "cert": "/path/to/cert.pem",
    "key": "/path/to/key.pem"
  },
  "auth": {
    "type": "password",
    "password": "your_secret"
  },
  "trafficStats": {
    "listen": ":8080",
    "secret": "stats_secret"
  }
}
```

**Traffic Stats API**：
```bash
GET http://node:8080/traffic
Authorization: Bearer stats_secret

Response:
{
  "users": {
    "user_0x1234abcd": {
      "tx": 1048576,
      "rx": 5242880
    }
  }
}
```

### 3.2 Metering Service

**职责**：
- 定期拉取各节点 Traffic Stats API
- 计算流量增量
- 按用户 ID 汇总多节点流量
- 中心化扣减配额
- 配额不足时返回阻断指令

**数据流**：
```
1. 定时任务（每 10 秒）
   ↓
2. 并发拉取所有节点 Traffic Stats API
   ↓
3. 计算流量增量（当前值 - 上次值）
   ↓
4. 按用户 ID 汇总
   ↓
5. 原子性扣减数据库中的配额
   ↓
6. 返回配额状态（充足 / 不足）
```

**核心 API**：
```bash
# 查询用户配额
GET /api/v1/users/{userId}/quota
Response:
{
  "userId": "user_0x1234abcd",
  "totalQuota": 500000000,
  "usedUpload": 50000000,
  "usedDownload": 150000000,
  "remaining": 300000000
}

# 流量上报（内部调用）
POST /api/v1/metering/report
{
  "nodeId": "node_a",
  "userId": "user_0x1234abcd",
  "txIncrement": 1000000,
  "rxIncrement": 5000000
}
```

### 3.3 Payment Service

**职责**：
- 验证 x402 支付签名
- 根据服务商价格兑换流量配额
- 记录购买历史
- 更新用户配额

**核心 API**：
```bash
POST /api/v1/users/{userId}/purchase
{
  "usdcAmount": 5.0,
  "paymentSignature": "0x...",
  "txHash": "0x..."
}

Response:
{
  "quotaId": "quota_123",
  "bandwidthPurchased": 500000000,
  "priceRate": 100,
  "newTotalQuota": 800000000,
  "purchasedAt": "2026-03-24T10:00:00Z"
}
```

### 3.4 Config Service

**职责**：
- 根据用户地址和配额状态生成 Hysteria2 配置
- 下发客户端配置
- 管理短期凭据

**核心 API**：
```bash
POST /api/v1/config/generate
{
  "userAddress": "0x1234abcd",
  "providerId": "provider_a"
}

Response:
{
  "config": {
    "type": "hysteria2",
    "server": "node-a.example.com",
    "server_port": 443,
    "password": "user_temp_password",
    "tls": {
      "enabled": true,
      "server_name": "node-a.example.com"
    }
  },
  "expiresAt": "2026-03-25T10:00:00Z"
}
```

## 4. 数据流设计

### 4.1 流量统计流

```
用户使用流量
    ↓
Hysteria2 节点统计（原生）
    ↓
暴露 Traffic Stats API
    ↓
Metering Service 定期拉取
    ↓
计算增量并汇总
    ↓
中心化扣减配额
    ↓
更新数据库
```

### 4.2 支付流

```
用户选择购买流量
    ↓
客户端检查 Signer Session
    ↓
生成 x402 支付签名
    ↓
调用 Payment Service
    ↓
验证签名和金额
    ↓
根据价格兑换配额
    ↓
更新用户配额
    ↓
返回购买结果
```

### 4.3 配置下发流

```
用户选择供应商
    ↓
客户端请求 Config Service
    ↓
Config Service 检查用户配额
    ↓
生成 Hysteria2 配置
    ↓
返回配置给客户端
    ↓
客户端更新 sing-box outbound
    ↓
重启 sing-box（TUN 层不变）
    ↓
连接建立
```

### 4.4 供应商切换流

```
用户点击切换供应商
    ↓
客户端请求新供应商配置
    ↓
Config Service 返回新配置
    ↓
客户端动态更新 sing-box outbound
    ↓
sing-box 重新连接
    ↓
TUN 层保持不变，用户无感知
```

## 5. 关键设计决策

### 5.1 为什么 TUN 层不需要改动？

```
TUN 层职责：
- 接收系统流量
- 根据路由规则转发

sing-box outbound 职责：
- 接收 TUN 转发的流量
- 使用特定协议（Hysteria2）发送到远程

两者解耦，修改 outbound 不影响 TUN
```

### 5.2 为什么使用中心化扣减？

**优点**：
- 简单直接，无需双向签名
- 无需凭条备份和恢复
- 原子性操作，避免超额使用
- 实时性要求低

**缺点**：
- 需要信任 Metering Service

**权衡**：
- 商用场景下，供应商运营 Metering Service
- 用户选择信任的供应商
- 平台提供开源代码，供应商可自行部署

### 5.3 为什么定期拉取而不是实时推送？

**定期拉取（推荐）**：
- 简单可靠
- 节点无需主动连接 Metering Service
- 容错性好（拉取失败可重试）

**实时推送**：
- 复杂度高
- 需要维护长连接
- 节点故障影响统计

**决策**：Phase 0 使用定期拉取，Phase 2 可选实时推送优化。

## 6. 扩展性设计

### 6.1 多供应商支持

```
用户可以同时：
- 在供应商 A 有 500MB 配额
- 在供应商 B 有 1GB 配额
- 在供应商 C 有 200MB 配额

每个供应商独立：
- 独立的 Metering Service
- 独立的配额管理
- 互不感知
```

### 6.2 多节点支持

```
供应商 A 部署：
- 节点 A1（美国）
- 节点 A2（欧洲）
- 节点 A3（亚洲）

Metering Service 汇总：
- 拉取 A1、A2、A3 的 Traffic Stats
- 按用户 ID 汇总
- 统一扣减配额
```

### 6.3 性能优化

**Phase 0-1**：
- 单机 Metering Service
- SQLite 数据库
- 定期拉取（10 秒）

**Phase 2+**：
- 分布式 Metering Service
- PostgreSQL + Redis
- 实时推送 + 定期对账

## 7. 安全设计

### 7.1 用户认证

- Hysteria2 使用 password 认证
- 每个用户独立密码
- 密码由 Config Service 生成
- 短期有效（24 小时）

### 7.2 流量统计安全

- Traffic Stats API 需要 secret
- 只有 Metering Service 可以访问
- 节点之间互不可见

### 7.3 支付安全

- x402 签名验证
- 防重放攻击
- 金额校验
