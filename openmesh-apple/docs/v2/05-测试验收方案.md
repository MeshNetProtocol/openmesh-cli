## 2. 功能测试

### 2.1 用户级流量统计

**测试场景 1：单用户单节点**
```
步骤：
1. 用户 A 连接节点 1
2. 下载 100MB 文件
3. 查询 Traffic Stats API

验收：
- API 返回 user_A 的流量数据
- 下载流量 ≈ 100MB（误差 < 1%）
```

**测试场景 2：单用户多节点**
```
步骤：
1. 用户 A 连接节点 1，下载 50MB
2. 用户 A 切换到节点 2，下载 50MB
3. 查询 Metering Service

验收：
- 总流量 ≈ 100MB
- 无重复计数
```

**测试场景 3：多用户并发**
```
步骤：
1. 用户 A 下载 100MB
2. 用户 B 同时下载 200MB
3. 查询各自流量

验收：
- 用户 A 流量 ≈ 100MB
- 用户 B 流量 ≈ 200MB
- 流量互不干扰
```

### 2.2 配额管理

**测试场景 4：预付费购买**
```
步骤：
1. 用户支付 5 USDC
2. 服务商价格：1 USDC = 100MB
3. 查询配额

验收：
- 配额增加 500MB
- 购买记录正确保存
```

**测试场景 5：配额扣减**
```
步骤：
1. 用户配额 500MB
2. 使用 200MB 流量
3. 查询剩余配额

验收：
- 剩余配额 = 300MB
- 扣减准确无超额
```

**测试场景 6：配额耗尽**
```
步骤：
1. 用户配额 10MB
2. 尝试下载 100MB 文件
3. 观察连接状态

验收：
- 使用 10MB 后连接阻断
- 客户端收到配额不足提示
```

**测试场景 7：配额追加**
```
步骤：
1. 用户当前配额 100MB
2. 追加购买 200MB
3. 查询总配额

验收：
- 总配额 = 300MB
- 配额正确累加
```

### 2.3 多节点场景

**测试场景 8：节点切换**
```
步骤：
1. 连接节点 A，使用 50MB
2. 切换到节点 B，使用 50MB
3. 查询总流量

验收：
- 总流量 = 100MB
- 切换过程用户无感知
- TUN 层保持连接
```

**测试场景 9：并发使用**
```
步骤：
1. 用户同时连接节点 A 和节点 B
2. 两个节点各使用 50MB
3. 查询总流量

验收：
- 总流量 = 100MB
- 两个节点流量正确汇总
```

**测试场景 10：节点故障**
```
步骤：
1. 用户连接节点 A
2. 节点 A 故障
3. 自动切换到节点 B

验收：
- 自动切换成功
- 流量统计连续
```

## 3. 性能测试

### 3.1 吞吐量测试

| 指标 | 目标 | 测试方法 |
|------|------|---------|
| 单用户吞吐量 | > 100 Mbps | iperf3 测试 |
| 10 用户并发 | > 50 Mbps/用户 | 并发 iperf3 |
| 100 用户并发 | > 10 Mbps/用户 | 压力测试 |

### 3.2 延迟测试

| 指标 | 目标 | 测试方法 |
|------|------|---------|
| 连接建立延迟 | < 500ms | 测量握手时间 |
| 流量统计延迟 | < 10s | 对比实时流量和统计 |
| API 响应时间 | < 100ms | 压测 API 接口 |

### 3.3 并发测试

| 指标 | 目标 | 测试方法 |
|------|------|---------|
| 并发连接数 | > 1000 | 并发连接测试 |
| 并发购买 | > 100 TPS | 并发购买压测 |

## 4. 安全测试

### 4.1 认证测试

**测试场景 11：密码认证**
```
步骤：
1. 使用错误密码连接
2. 使用正确密码连接

验收：
- 错误密码被拒绝
- 正确密码可连接
```

**测试场景 12：密码过期**
```
步骤：
1. 使用过期密码连接

验收：
- 连接被拒绝
- 提示密码过期
```

### 4.2 支付安全

**测试场景 13：签名验证**
```
步骤：
1. 提交无效签名的支付
2. 提交有效签名的支付

验收：
- 无效签名被拒绝
- 有效签名被接受
```

**测试场景 14：重放攻击**
```
步骤：
1. 提交相同签名两次

验收：
- 第二次提交被拒绝
- 防重放机制生效
```
