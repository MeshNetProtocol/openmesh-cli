# 实施计划

> **文档目的**：定义基于 Hysteria2 的 V2 系统分阶段实施计划。

---

## Phase 0: Hysteria2 技术验证（3-5天）

**目标**：验证 Hysteria2 用户级流量统计能力和多节点汇总的可行性。

### P0.1 环境搭建（1天）

**任务**：
- 安装 Hysteria2 服务端
- 配置多用户认证
- 启用 Traffic Stats API
- 配置 sing-box 客户端

**输出**：
- Hysteria2 服务端运行
- Traffic Stats API 可访问
- 客户端可连接

### P0.2 单节点验证（1天）

**任务**：
- 配置 2 个测试用户
- 验证用户级流量统计
- 测试 Traffic Stats API 准确性
- 对比实际流量和统计流量

**验收标准**：
- ✅ Traffic Stats API 返回用户级数据
- ✅ 流量统计准确度 > 99%
- ✅ 多用户流量互不干扰

### P0.3 多节点验证（2天）

**任务**：
- 部署 3 个 Hysteria2 节点
- 实现简单的流量汇总服务
- 验证多节点流量汇总准确性
- 测试节点切换场景

**验收标准**：
- ✅ 多节点流量正确汇总
- ✅ 无重复计数
- ✅ 节点切换流量连续

### P0.4 输出验证报告（1天）

**输出**：
- 技术可行性结论
- 性能基准数据
- 风险识别
- Phase 1 建议

---

## Phase 1: Metering Service 开发（1-2周）

**目标**：实现完整的流量统计和配额管理服务。

### P1.1 Metering Service 核心（5天）

**任务**：
- 设计数据库表结构
- 实现流量上报接口
- 实现多节点流量汇总逻辑
- 实现配额扣减逻辑（原子性）
- 实现配额查询 API

**技术栈**：
- 语言：Go/Python
- 数据库：SQLite（开发）/ PostgreSQL（生产）
- API：RESTful HTTP

**核心逻辑**：
```python
# 伪代码
def aggregate_traffic():
    for node in nodes:
        stats = fetch_traffic_stats(node)
        for user_id, traffic in stats.items():
            increment = calculate_increment(user_id, traffic)
            deduct_quota(user_id, increment)
```

### P1.2 Hysteria2 集成（3天）

**任务**：
- 实现定期拉取 Traffic Stats API（每 10 秒）
- 实现流量增量计算
- 实现配额状态同步
- 错误处理和重试机制

### P1.3 测试验证（2天）

**测试场景**：
- 单用户多节点测试
- 多用户并发测试
- 配额耗尽阻断测试
- 节点故障恢复测试

**验收标准**：
- ✅ 流量统计准确度 > 99%
- ✅ 配额扣减无超额
- ✅ API 响应时间 < 100ms

## Phase 2: Payment Service 开发（1周）

**目标**：实现预付费购买流量功能。

### P2.1 Payment Service（3天）

**任务**：
- 实现 x402 支付签名验证
- 实现 USDC → 流量配额兑换
- 实现购买记录存储
- 实现配额追加逻辑

**核心逻辑**：
```python
def purchase_bandwidth(user_address, usdc_amount, signature):
    # 1. 验证签名
    verify_x402_signature(signature, usdc_amount)

    # 2. 计算流量
    bandwidth = usdc_amount * price_rate  # 如 5 USDC * 100MB = 500MB

    # 3. 更新配额
    update_quota(user_address, bandwidth)

    # 4. 记录购买
    save_purchase_record(user_address, usdc_amount, bandwidth)
```

### P2.2 集成测试（2天）

**测试场景**：
- 端到端购买流程
- 配额追加测试
- 签名验证测试
- 并发购买测试

## Phase 3: 客户端集成（2周）

**目标**：客户端支持 Hysteria2 和流量监控。

### P3.1 配置生成（3天）

**任务**：
- 实现 Hysteria2 配置模板
- 实现动态配置生成
- 实现 sing-box 配置更新
- TUN 层保持不变

**配置示例**：
```json
{
  "outbounds": [{
    "type": "hysteria2",
    "server": "node-a.example.com",
    "server_port": 443,
    "password": "user_password",
    "tls": {
      "enabled": true,
      "server_name": "node-a.example.com"
    }
  }]
}
```

### P3.2 流量监控（4天）

**任务**：
- 实现配额查询接口调用
- 实现实时流量显示
- 实现配额不足提醒
- 实现购买流程触发

### P3.3 钱包集成（5天）

**任务**：
- 实现钱包创建/恢复
- 实现 Keychain 加密存储
- 实现 Signer Session
- 实现 x402 支付签名

## Phase 4: 供应商发现（1周）

**目标**：实现链上供应商发现和快速切换。

### P4.1 链上集成（3天）

**任务**：
- 连接 Base 测试网
- 读取 Registry 合约
- 解析供应商 Metadata
- 缓存供应商列表

### P4.2 供应商切换（2天）

**任务**：
- 实现快速切换逻辑
- 实现配置热更新
- 实现切换状态管理

**切换流程**：
```
1. 用户选择新供应商
2. 请求新供应商配置
3. 更新 sing-box outbound
4. 重启 sing-box（TUN 层不变）
5. 用户无感知切换完成
```
