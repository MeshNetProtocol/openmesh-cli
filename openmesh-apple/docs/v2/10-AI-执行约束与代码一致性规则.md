# 10-AI-执行约束与代码一致性规则

## 1. 文档目的

本文档约束 AI 和人工协作开发在 V2 中的边界。

---

## 2. 必须接受的前提

1. OpenMesh 要做的是 VPN 行业的 OpenRouter
2. 商用供应商发现与结算必须去中心化
3. 官方临时受限访问和 DIY 继续保留
4. 商用供应商热路径必须支持 x402 自动小额支付
5. 钱包是用户自持钱包，不是平台账号
6. 平台不做中心化账本

---

## 3. 绝对禁止事项

### 3.1 禁止把商用场景写回中心化账本产品

禁止：

- 设计“平台统一余额账本”作为主真相源
- 让用户先把钱充到平台再切换供应商

### 3.2 禁止把官方临时受限访问复杂化

禁止：

- 要求官方临时受限访问先接入完整商用支付体系
- 为了商用路径破坏当前简单 sing-box 节点方案

### 3.3 禁止在热路径等待链上轮询

禁止：

- 把链上 watcher 当成续费后放行的唯一依据

### 3.4 禁止弱化钱包安全

禁止：

- 助记词明文存储
- 私钥明文长期落盘
- 无上限自动支付

---

## 4. AI 输出要求

AI 在输出方案或代码时，必须说明：

1. 当前改动服务于哪一种场景
2. 是否影响官方临时受限访问
3. 是否影响 DIY / 私人节点
4. 商用热路径是否仍然是“预续费 + 自动支付 + 立即放行”
5. 钱包 signer session 是否仍符合安全边界

---

## 5. 完成定义

只有同时满足以下条件，AI 输出才算完成：

1. 没有把商用场景重新做回中心化平台
2. 没有破坏官方临时受限访问和 DIY 路径
3. 热路径不等待链上轮询
4. 钱包安全与自动支付限制清晰
5. 文档、测试、回退方案一致
