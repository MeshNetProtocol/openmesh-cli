# 12-V2-界面设计与开发指南

## 1. 文档目的

本文档指导 V2 的界面设计和开发流程，确保产出可被 AI 执行的具体任务。

## 2. 当前问题诊断

### 2.1 现有文档的不足

现有文档(01-11)更像是"产品需求文档"而不是"AI 可执行的任务清单"：

- ✅ 描述了"要做什么"(WHAT)
- ❌ 缺少"怎么做"(HOW)的具体步骤
- ❌ 缺少明确的输入/输出定义
- ❌ 缺少具体的 API 接口设计
- ❌ 缺少数据流转的详细说明
- ❌ 缺少可验证的检查点

### 2.2 需要补充的内容

要让 AI 能够执行，需要补充:

1. **具体的接口定义** - 每个模块的输入输出、函数签名
2. **数据结构定义** - 精确的 JSON schema、数据库表结构
3. **状态机定义** - 钱包解锁、支付流程、供应商切换的状态转换
4. **具体的文件清单** - 需要创建/修改哪些文件，每个文件的职责
5. **可验证的检查点** - 每一步完成后如何验证

## 3. 推荐工作流程

### 阶段 1: 界面设计 (1-2 天)

**为什么先做界面设计？**

1. **界面是产品的真相** - 很多模糊的需求，画出界面后会立刻清晰
2. **界面设计会倒逼架构决策** - 例如:
   - 供应商列表怎么展示? → 决定了需要哪些 metadata 字段
   - 钱包解锁放在哪里? → 决定了 signer session 的生命周期
   - 流量配额剩余量怎么显示? → 决定了统计数据的更新频率
3. **界面是和 AI 沟通的最佳语言** - 一张标注清楚的界面原型图，比 10 页文字描述更容易让 AI 理解

**输出物:**

```
界面原型/
├── 01-主界面.png (供应商列表、当前连接状态)
├── 02-钱包界面.png (创建/恢复/解锁)
├── 03-供应商详情.png (metadata、价格、流量配额)
├── 04-设置界面.png (自动支付限额、session 时长)
└── 05-用户流程图.png (首次使用→创建钱包→选择供应商→开始使用)
```

**工具选择:**
- Figma (推荐)
- Sketch
- 纸笔 + 拍照

### 阶段 2: 基于界面补充技术文档 (1 天)

为每个界面补充:

```
界面名称: 主界面 - 供应商列表
├── 数据模型
│   ├── Provider { id, name, status, traffic, price }
│   └── ConnectionState { connected, provider, traffic_used }
├── API 调用
│   ├── GET /api/providers → Provider[]
│   └── POST /api/connect { providerId } → ConnectionState
└── 状态变化
    ├── 初始状态: loading
    ├── 加载完成: idle
    └── 连接中: connecting
```

**输出物:**

创建 `13-V2-界面技术规格.md`，包含:
- 每个界面的数据模型
- 每个交互的 API 调用
- 每个状态变化的业务逻辑

### 阶段 3: 拆分可执行任务 (半天)

将技术规格拆分成具体任务:

```markdown
## Task 1: 实现钱包创建流程

**输入:**
- 用户点击"创建钱包"按钮

**输出:**
- 生成 12 个助记词
- 加密存储到 Keychain
- 返回钱包地址

**修改文件:**
- `WalletManager.swift` - 新增 `createWallet()` 方法
- `KeychainService.swift` - 新增 `saveEncryptedMnemonic()` 方法
- `CreateWalletView.swift` - 新增界面

**验证标准:**
- [ ] 助记词符合 BIP39 标准
- [ ] Keychain 中存储的是加密后的数据
- [ ] 可以通过助记词恢复钱包

**依赖:**
- 无

**预计工作量:**
- 2-3 小时
```

**输出物:**

创建 `14-V2-可执行任务清单.md`

### 阶段 4: AI 执行具体任务

将任务逐个交给 AI 执行，每个任务包含:

## 4. 核心界面清单

基于现有需求文档，V2 至少需要以下界面:

### 4.1 主界面 - 供应商路由

**核心元素:**
- 当前连接状态 (已连接/未连接)
- 当前供应商信息 (名称、流量使用、剩余流量)
- 供应商列表 (官方/商用/私人节点)
- 快速切换按钮

**关键交互:**
- 点击供应商 → 切换供应商
- 点击连接/断开 → 切换连接状态
- 下拉刷新 → 更新供应商列表

**需要决策的问题:**
- 供应商列表是分组显示还是平铺？
- 流量配额剩余量用进度条还是数字？
- 切换供应商需要确认吗？

### 4.2 钱包界面

**核心元素:**
- 创建钱包 / 恢复钱包
- 钱包地址
- USDC 余额 (流豆)
- 解锁 / 锁定状态
- Signer session 剩余时间

**关键交互:**
- 创建钱包 → 生成助记词 → 备份确认
- 恢复钱包 → 输入助记词 → 验证
- 解锁钱包 → 输入密码/生物识别 → 进入 signer session
- 锁定钱包 → 清除内存中的私钥

**需要决策的问题:**
- 助记词展示方式？(12 个格子 vs 列表)
- 解锁方式？(密码 + Touch ID / Face ID)
- Signer session 超时提醒？

### 4.3 供应商详情界面

**核心元素:**
- 供应商名称、描述
- 价格模型 (免费桶 + 续费价格)
- 支持的协议
- 当前流量配额状态
- 历史支付记录

**关键交互:**
- 点击"使用此供应商" → 切换
- 查看购买记录
- 查看流量使用情况

**需要决策的问题:**
- 价格怎么展示？(每 GB 价格 vs 每桶价格)
- 历史记录展示多少条？

### 4.4 设置界面

**核心元素:**
- 自动支付设置
  - 单次最大支付额度
  - 每日最大支付额度
  - Signer session 持续时长
- 网络设置 (主网/测试网)
- 私人节点导入

**关键交互:**
- 调整额度滑块
- 导入配置文件

**需要决策的问题:**
- 额度单位？(USDC vs 流豆)
- 默认值是多少？

### 4.5 首次启动流程

**流程图:**
```
启动 App
  ↓
检查是否有钱包
  ↓ 无
展示欢迎界面
  ↓
选择: [创建钱包] [恢复钱包] [稍后再说]
  ↓ 稍后再说
使用官方临时受限访问
  ↓
连接成功
```

**需要决策的问题:**
- 是否强制创建钱包？
- 官方临时受限访问的限额提示？

## 5. 数据模型示例

基于界面设计，补充精确的数据结构:

### 5.1 Provider 模型

```swift
struct Provider: Codable, Identifiable {
    let id: String                    // 链上 provider ID
    let displayName: String           // 展示名称
    let description: String           // 描述
    let configServiceURL: URL         // 配置服务地址
    let meteringServiceURL: URL       // 计量服务地址
    let supportedProtocols: [String]  // ["vmess", "vless", "trojan"]
    let pricingModel: PricingModel    // 价格模型
    let owner: String                 // 所有者地址
    let serviceSigners: [String]      // 服务签名者地址
    let status: ProviderStatus        // 当前状态
}

struct PricingModel: Codable {
    let priceRate: Decimal              // 价格比率 (1 USDC = X MB)
    let minPurchaseUSDC: Decimal        // 最小购买金额 (USDC)
    let maxPurchaseUSDC: Decimal        // 最大购买金额 (USDC)
}

enum ProviderStatus: String, Codable {
    case available    // 可用
    case connected    // 已连接
    case unavailable  // 不可用
}
```

### 5.2 Wallet 模型

```swift
struct Wallet {
    let address: String               // 钱包地址
    var balance: Decimal              // USDC 余额
    var isUnlocked: Bool              // 是否已解锁
    var signerSessionExpiresAt: Date? // Signer session 过期时间
}

struct SignerSession {
    let wallet: Wallet
    let privateKey: Data              // 仅存在于内存
    let createdAt: Date
    let expiresAt: Date
    var dailySpent: Decimal           // 今日已支付
}
```

### 5.3 TrafficQuota 模型

```swift
struct TrafficQuota: Codable {
    let quotaId: String               // 配额 ID
    let providerId: String            // 供应商 ID
    let userAddress: String           // 用户地址
    let totalBytes: Int64             // 总配额 (字节)
    var usedUpload: Int64             // 已用上传 (字节)
    var usedDownload: Int64           // 已用下载 (字节)
    var remaining: Int64              // 剩余 (字节)
    let priceRate: Decimal            // 购买时的价格比率
    let purchasedAt: Date             // 购买时间
    let updatedAt: Date               // 更新时间
}

struct PurchaseRecord: Codable {
    let recordId: String              // 记录 ID
    let usdcAmount: Decimal           // 支付金额 (USDC)
    let bandwidthPurchased: Int64     // 购买的流量 (字节)
    let priceRate: Decimal            // 价格比率
    let paymentSignature: String      // 支付签名
    let txHash: String?               // 交易哈希
    let purchasedAt: Date             // 购买时间
}
```

## 6. 状态机示例

### 6.1 钱包解锁状态机

```
[Locked]
  ↓ unlock(password)
[Unlocking]
  ↓ success
[Unlocked] → 创建 SignerSession
  ↓ timeout / lock() / dailyLimitReached
[Locked]
```

### 6.2 供应商切换状态机

```
[Idle]
  ↓ switchProvider(providerId)
[Fetching Config]
  ↓ success
[Checking Wallet]
  ↓ wallet unlocked
[Checking Quota]
  ↓ has quota
[Updating Sing-box Config]
  ↓ success
[Connected]
```

### 6.3 预付费购买状态机

```
[Monitoring Traffic]
  ↓ quota low
[Prompt User to Purchase]
  ↓ user confirms
[Auto Signing Payment]
  ↓ success
[Waiting Provider Confirmation]
  ↓ success
[Quota Purchased]
  ↓
[Monitoring Traffic]
```
