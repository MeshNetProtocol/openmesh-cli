# Phase 1 执行方案（下载即可用）- 代码基线 + 项目经理任务拆解

关联总纲：`docs/platform_bootstrap_execution_master_plan.md`  
基线快照日期：`2026-03-02`

## 0. 文档用途

本文件是 Phase 1 的执行文档，不是概念说明。目标是同时回答三件事：

1. 当前代码已经具备什么能力，缺什么能力。  
2. 在“商用供给尚未成型”情况下，如何把新用户闭环跑通。  
3. 如何把工作拆成可独立推进、可验收、可回滚的任务包。  

---

## 1. 阶段边界与成功定义

### 1.1 阶段边界（冻结）

1. Phase 1 只做新用户“下载即可用”闭环，不做商用钱包购买闭环。  
2. 平台仅提供临时受限引导能力，不承诺长期全量官方数据面。  
3. 商用提供者 UI 仅保留入口和基本占位，不在本阶段展开。  
4. 社区接力通过社区入口与导入能力完成，商业链上交易不进入本阶段。  

### 1.2 成功定义（Phase 1 Done）

1. 新用户无需钱包即可完成“连通 -> 找配置 -> 导入 -> 可连接”。  
2. 单一官方引导资源可启动第一批用户，失效时有替代路径。  
3. 老用户不被新手流程打断，仍可快速进入已安装/市场/导入流程。  

### 1.3 Out of Scope

1. 链上商用筛选、支付、结算。  
2. 钱包 onboarding 的改版。  
3. 自建节点全流程向导（Phase 2 再做）。

### 2.4 当前基线结论

1. “市场 + 导入 + 安装”能力已经比较完整。  
2. 真正缺口在“首启引导闭环”：入口、路径、兜底、来源治理、观测。  
3. Phase 1 重点应从“再做安装能力”转为“编排首启流程与运营可控性”。

## 4. 项目经理工作分解（Work Package）

### P1-WP1 首启路径与文案冻结

目标：把“下载即可用”固化为单一路径，不让用户先做复杂选择。  
输入：本文件第 2、3 章基线与差距。  
关键工作：

1. 冻结首屏双主动作：`快速连通（临时受限）`、`导入已有配置`。  
2. 明确次级动作：`去社区找配置`、`了解自建（预告）`。  
3. 冻结文案口径：`临时引导`、`非长期官方代理`、`尽快迁移社区配置`。  

交付物：

1. 首屏 IA 图（iOS/mac 共版）。  
2. 三步引导状态图（成功/失败/回退）。  
3. 文案矩阵（按钮、提示、错误态、审核口径）。  

验收：

1. PM、UE、开发、运营对同一流程无歧义。  
2. 老用户入口不被新手路径遮挡。  

依赖：无。  

### P1-WP2 引导资源与来源治理（官方临时 + 社区接力）

目标：让“1 个官方资源也能启动用户”，且能平滑切到社区接力。  
输入：官方临时资源、社区入口清单、`MeshNetProtocol.github.io` 辅助站点能力。  
关键工作：

1. 定义引导资源类型：官方临时、社区维护、好友分享。  
2. 定义来源优先级和失效切换规则（主源/镜像源/备用社区源）。  
3. 定义资源生命周期与下线策略（官方退出后社区可独立维持）。  

交付物：

1. 引导资源治理规范 v1。  
2. 社区入口清单（至少 3 个独立来源）。  
3. 故障切换与人工应急 SOP。  

验收：

1. 单源失效不导致整体首启流程中断。  
2. 官方退出后仍存在社区可达路径。  

依赖：WP1。  

### P1-WP3 首启受限连通流程（产品闭环编排）

目标：补上当前缺失的一键受限连通主路径。  
输入：WP1 IA、WP2 来源治理。  
关键工作：

1. 设计“受限连通”状态机：准备中、可用、受限到期、不可达。  
2. 设计失败分级：网络不可达、资源失效、导入失败。  
3. 定义与现有 Market/导入页面的跳转编排，不破坏旧功能。  

交付物：

1. 首启状态机图 + 页面流转图。  
2. 错误码与用户提示映射表。  
3. 兼容策略（iOS Tab 结构、mac 菜单结构）。  

验收：

1. 新用户可以不理解 VPN 术语仍走通三步。  
2. 老用户可直接进入原有 Market/已安装/导入链路。  

依赖：WP1、WP2。  

### P1-WP4 导入闭环增强（从“可导入”到“易导入”）

目标：复用现有强导入能力，降低新手失败率。  
输入：现有导入实现（URL/文本/base64 + 安装向导）。  
关键工作：

1. 提供“社区内容一键粘贴导入”最短路径。  
2. 统一 iOS/mac 的导入失败提示分级。  
3. 增加来源标签记录策略（官方临时/社区/分享）。  

交付物：

1. 导入 UX 统一规范。  
2. 失败提示标准文本与恢复动作。  
3. 来源标签字段定义（供后续统计和治理）。  

验收：

1. 导入失败时用户能看到下一步可执行动作。  
2. 导入后默认可一键切换到新 provider。  

依赖：WP3。  

### P1-WP5 可观测性与 KPI 落地

目标：从“感觉可用”变成“数据可验证可迭代”。  
输入：WP3 流程状态机、WP4 导入事件。  
关键工作：

1. 定义漏斗指标：下载 -> 首启 -> 受限连通 -> 社区访问 -> 导入成功 -> 首次连接成功。  
2. 定义关键时长指标：下载到可用中位时长。  
3. 定义异常指标：资源失效率、导入失败率、回退触发率。  

交付物：

1. 指标字典 v1。  
2. 版本对比口径与发布后复盘模板。  
3. 运营日报字段模板。  

验收：

1. 每个阶段节点都有可统计事件。  
2. 能比较版本迭代前后效果。  

依赖：WP3、WP4。  

### P1-WP6 发布与风险回滚

目标：确保 Phase 1 上线可灰度、可回滚、可应急。  
输入：WP1~WP5 产物。  
关键工作：

1. 制定灰度策略（小流量 -> 全量）。  
2. 制定回滚开关（入口回退、源切换、文案降级）。  
3. 制定应急联动（社区入口变更、镜像切换、公告模板）。  

交付物：

1. 上线检查清单。  
2. 回滚 Runbook。  
3. 应急沟通模板（App 内提示 + 社区公告）。  

验收：

1. 任何单点故障都可在短时间内回退。  
2. 运营和开发有统一应急脚本。  

依赖：WP1~WP5。

## 7. KPI 与阶段门槛

### 7.1 过程指标

1. 首启进入“快速连通”入口比例。  
2. 社区入口点击率。  
3. 导入尝试率与导入成功率。  

### 7.2 结果指标

1. 下载到首次可用成功率。  
2. 下载到首次可用中位时长。  
3. 首次可用后 24h 仍可连接比例。  

### 7.3 Phase 1 退出门槛

1. 上述结果指标连续稳定两个发布周期。  
2. 官方临时资源降级后，社区接力路径仍可用。  
3. 客服/社群人工救援占比显著下降。

## 8. 风险与回滚

| 风险 | 触发信号 | 应对策略 |
|---|---|---|
| 官方单源失效 | 首启失败率突增 | 切换到备用社区源，入口文案降级 |
| 审核口径误读 | 审核反馈聚焦“长期代理” | 启用审核版文案包，强调临时引导定位 |
| 新手流失 | 首屏停留高、导入率低 | 收敛动作数量，强化一键入口 |
| 老用户体验下降 | 老用户 Market/已安装访问路径变长 | 保留老路径快捷入口，不强制走引导 |

## 9. 与 Phase 2 的衔接条件

满足以下条件后进入 `docs/platform_phase_2_private_node_growth.md`：

1. Phase 1 闭环稳定，首启可用不依赖高强度人工干预。  
2. 社区入口可持续维护，并形成基础贡献者池。  
3. 可以开始把“有能力用户”引导到自建与分享路径。
