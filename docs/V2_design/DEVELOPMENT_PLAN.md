# V1 开发计划文档

基于当前已确认的 V1 目标架构，定义开发阶段、阶段目标、核心任务、负责人角色建议、预计工期、风险点、阻塞项、交付物和完成标准，用于指导实际执行。

> 说明：
> - 本文档描述的是 V1 目标实现的开发计划，不是当前代码现状说明。
> - `validation/phase4` 中的文件型 POC 仅作为技术验证输入，不直接作为正式实现。
> - 计划默认以单实例后端、2-3 台 Xray 服务器、Base Mainnet 为目标部署形态。
> - 工期为建议值，默认在关键依赖满足、评审及时、测试环境可用的前提下估算。

---

## 1. V1 目标范围

### 目标能力

V1 需要完成以下闭环：

1. 用户选择套餐并完成授权
2. 服务端创建订阅并执行首次扣费
3. 服务端按周期自动续费
4. 服务端同步订阅状态到 Xray 准入控制
5. 客户端可查询订阅状态、完成连接和基础验证
6. 系统完成基础测试、部署和上线准备

### 不包含内容

以下内容不属于 V1 范围：

- 链上套餐管理
- 链上订阅状态管理
- 多实例部署
- Redis / WebSocket / 通知服务
- 10+ Xray 服务器集群能力
- V2 的高可用和扩容能力

---

## 2. 执行原则

- 先锁定职责边界，再进入实现，避免边做边改方向
- 先完成主业务闭环，再做跨模块联调和上线准备
- 每个阶段结束后必须做一次阶段验收，未达标不进入下一阶段
- 每个阶段都要同步更新相关文档，避免代码和文档脱节
- 关键模块优先做集成测试样例，避免最后才发现接口不兼容

## 8. Phase 1：智能合约与链上能力

### 参考验证实现

- 参考目录：`docs/V2_design/validation/phase4/`
- 核心参考文档：`docs/V2_design/validation/phase4/README.md`
- 核心参考代码：`docs/V2_design/validation/phase4/contracts/`
- 关键参考结论：`VPNCreditVaultV4` 已验证的最小链上职责为绑定、permit 授权、charge 扣费、chargeId 去重

### 阶段目标

完成 `VPNCreditVault` 合约的最小链上职责，使其可以支持身份绑定、授权、扣费和去重，为服务端订阅流程提供稳定的链上基础。

### 建议负责人角色

- 主责：合约工程师
- 协作：后端工程师、技术负责人

### 预计工期

- 1-2 周

### 前置条件

- V1 链上职责边界已确认
- 不再引入链上套餐管理和链上订阅状态管理

### 核心任务

#### 5.1 合约设计
- 明确 `identity -> payer` 绑定模型
- 明确 permit 授权入口和参数约束
- 明确 `chargeId` 去重规则
- 明确续费扣费相关事件和错误处理边界

#### 5.2 合约实现
- 实现 identity 与 payer 的绑定能力
- 实现 Permit 授权与扣费调用路径
- 实现续费所需的链上扣费能力
- 实现 `chargeId` 防重复扣费逻辑
- 补齐必要事件定义

#### 5.3 合约测试
- 编写 Foundry 单元测试
- 覆盖首次授权和首次扣费流程
- 覆盖重复扣费失败场景
- 覆盖参数异常和权限异常场景
- 覆盖续费相关核心路径

#### 5.4 部署准备
- 编写部署脚本
- 明确环境变量与配置项
- 整理部署后需要同步给前后端的参数
- 预留主网部署检查项

### 交付物

- `VPNCreditVault` 合约源码
- Foundry 测试
- 部署脚本
- 合约接口说明
- 合约部署参数清单

### 风险点

- 合约职责再次膨胀，重新引入链上套餐或链上状态
- `chargeId` 规则设计不严谨，导致重复扣费风险
- 事件定义不足，给服务端状态同步带来困难
- 测试覆盖不全，主网前暴露设计缺陷

### 主要阻塞项

- 合约职责边界未冻结
- 服务端对合约接口的调用需求未明确
- 测试网部署参数或钱包准备不完整

### 实施约束

- 必须以 `phase4` 已验证的 `VPNCreditVaultV4` 设计边界为基础推进
- 若要偏离 `phase4` 已验证职责，必须先单独评审并更新总览文档
- 任何新增链上职责都必须明确证明是 V1 必需，而不是旧方案回流

### 完成标准

- 合约职责与 V1 文档一致
- Foundry 测试通过
- 支持服务端发起首次扣费和续费扣费
- 不包含链上套餐管理和链上订阅状态管理
- 部署脚本可在测试环境执行

- 设计套餐、订阅、授权记录、扣费记录、事件镜像等数据表

- 设计自动续费相关字段

#### 6.2 套餐与订阅能力

- 实现 Permit 签名验证
- 实现首次授权和首次扣费流程
- 实现续费扣费流程
- 实现链上事件监听与必要的状态对账
- 处理扣费成功、失败、重复请求等场景

- API 测试
- 与合约的集成测试
- 关键状态流转测试

- 订阅管理服务代码

### 主要阻塞项

- 合约事件或接口仍在变化
- 数据库 schema 尚未评审

## 13. 跨阶段依赖与关键阻塞点

### 关键依赖

- Phase 2 依赖 Phase 1 的合约接口稳定
- Phase 3 依赖 Phase 2 的用户状态和接口稳定
- Phase 4 依赖 Phase 2 和 Phase 3 的联调接口稳定
- Phase 5 依赖前 4 个阶段主链路打通

### 高优先级阻塞点

- 合约职责变化
- 服务端状态机变化
- Xray 环境不稳定
- 客户端接口频繁变更
- 测试环境与主网部署准备不足

### 建议管理动作

- 每个阶段开始前做一次范围确认
- 每周固定检查一次阻塞项
- 关键接口冻结后再进入下游联调
- 阶段结束时同步更新总览文档和相关模块文档

## 15. 推荐执行顺序

建议严格按以下顺序推进：

1. Phase 1：先锁定最小链上职责
2. Phase 2：再完成服务端主流程
3. Phase 3：接入 Xray 准入和流量统计
4. Phase 4：进行客户端联调
5. Phase 5：最后统一测试、部署和发布准备

> 原则：先完成主业务闭环，再做联调和上线准备，避免并行推进导致职责边界反复变动。
