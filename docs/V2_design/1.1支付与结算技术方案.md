## 一、方案背景与目标

### 1.1 商业模式说明

OpenMesh V2 的商业模式从"去中心化直接结算"调整为"平台托管结算模式"。本方案提供完整的支付与结算技术架构,实现以下核心功能:

- 用户通过 Coinbase Commerce 或其他钱包支付订阅费
- USDC 支付到供应商的智能合约托管账户
- 供应商提现时平台自动扣除 10% 手续费(可调整)
- 供应商上线时缴纳 100 USDC 上线费(可调整)

### 1.2 技术目标

- **零 Gas 费用户体验**: 通过 Coinbase Commerce 实现用户支付零 Gas 费
- **安全的资金托管**: 通过智能合约实现供应商资金隔离和安全托管
- **自动化结算**: 供应商提现时自动扣除平台手续费
- **多钱包支持**: 支持 Coinbase、MetaMask、WalletConnect 等多种钱包
- **完整的支付闭环**: 从支付入口到链上结算的完整技术方案

### 1.3 核心功能

**支付功能**:
- Coinbase Commerce 支付链接生成
- 多钱包支持(Coinbase、MetaMask、WalletConnect)
- QR 码扫码支付
- 支付状态实时同步

**结算功能**:
- 供应商独立托管合约
- 自动手续费扣除
- 随时提现,无限制
- 链上事件监听和同步

**管理功能**:
- 供应商注册和上线
- 手续费比例配置
- 供应商暂停/恢复
- 多签钱包管理

## 二、技术选型

### 2.1 Base 网络选择

**选择理由**:

1. **成本优势**
   - Gas 费最低: ~$0.50/笔交易
   - 比以太坊主网便宜 95%(主网 ~$50/笔)
   - 相比其他 L2 降低 60% 成本

2. **技术支持**
   - 原生 USDC 支持(Circle 官方发行)
   - x402 协议已部署 Base Sepolia 测试网
   - EVM 完全兼容,开发工具成熟

3. **生态优势**
   - Coinbase 官方支持,生态快速增长
   - 交易确认速度: 1-3 秒
   - 开发者友好,文档完善

**备选方案**: Polygon(如未来需要多链支持)

### 2.2 USDC 稳定币

**选择理由**:

1. **原生支持**
   - Circle 在 Base 上原生发行 USDC
   - 无需跨链桥接,降低风险
   - 与 Coinbase 生态深度集成

2. **流动性优势**
   - 市场流动性最好的稳定币之一
   - Coinbase 用户可直接使用账户余额支付
   - 广泛的钱包和交易所支持

3. **合规性**
   - Circle 受监管,合规性强
   - 定期审计和储备金证明
   - 降低平台合规风险

### 2.3 Coinbase Commerce 集成

**核心优势**:

1. **零 Gas 费**
   - Coinbase 赞助用户支付的 Gas 费
   - 用户只需支付订阅金额
   - 显著降低用户门槛

2. **用户体验**
   - Coinbase 用户可直接用账户余额支付
   - 支持 Face ID/指纹验证
   - 一键支付,无需复杂操作

3. **集成方案**
   - 官方 SDK 和 API 支持
   - Webhook 实时通知
   - 支持订阅模式(自动续订)

4. **成本**
   - 平台手续费: 1%
   - 相比节省的开发成本和用户 Gas 费,完全可接受

## 三、支付方案设计

### 3.1 Coinbase Commerce 集成

**支付链接生成**:

```
POST /api/v1/subscriptions/create-payment

Request:
{
  "userId": "user_xxx",
  "providerId": "provider_xxx",
  "tierId": "tier_standard"
}

Response:
{
  "paymentId": "payment_xxx",
  "hostedUrl": "https://commerce.coinbase.com/charges/xxx",
  "deepLink": "coinbase://pay?charge=xxx",
  "qrCodeData": "ethereum:0x...?value=6000000&token=USDC"
}
```

**Coinbase Commerce SDK 集成**:
- 后端使用 Coinbase Commerce API 创建支付请求
- 指定收款地址为供应商的智能合约地址
- 设置 Webhook 接收支付通知
- 支持订阅模式配置

### 3.2 多钱包支持

**支持的钱包**:

1. **Coinbase 账户**(推荐)
   - 现有用户直接用账户余额支付
   - 零 Gas 费
   - Face ID/指纹验证

2. **Coinbase Wallet**
   - 移动钱包 App
   - 深链接唤起: `coinbase://pay?charge=xxx`

3. **MetaMask**
   - 浏览器扩展或移动 App
   - 深链接唤起: `metamask://send?...`

4. **WalletConnect**
   - 支持任何 WalletConnect 兼容钱包
   - 扫码连接

5. **QR 码支付**
   - 任何钱包扫码支付
   - EIP-681 标准支付链接

**支付链接格式(EIP-681)**:

```
ethereum:0x{escrowAddress}@8453/transfer?
  address=0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913&
  uint256=6000000
```

参数说明:
- `0x{escrowAddress}`: 供应商托管合约地址
- `@8453`: Base 链 ID
- `address`: USDC 合约地址(Base)
- `uint256`: 金额(6 USDC = 6000000)

### 3.4 自动续订方案

**方案 A: Coinbase Commerce 订阅模式(推荐)**

```
1. 首次订阅时,用户授权自动续订
   ↓
2. Coinbase Commerce 存储授权
   ↓
3. 到期时,Coinbase 自动扣款
   ↓
4. 通过 Webhook 通知后端
   ↓
5. 后端更新订阅状态
```

**优势**:
- 真正的自动续订
- 无需用户再次操作
- Coinbase 处理支付失败重试
- 零 Gas 费

**方案 B: EIP-3009 授权(备选)**

```
1. 用户首次订阅时,签署 EIP-3009 授权
   ↓
2. 授权智能合约在到期时自动扣款
   ↓
3. 客户端在到期时调用智能合约执行扣款
   ↓
4. 无需用户再次签名
```

**注意**: 方案 B 需要用户承担 Gas 费,用户体验不如方案 A

## 四、智能合约架构

### 4.1 整体架构

采用 **Factory + Minimal Proxy (EIP-1167)** 模式:

```
ProviderEscrowFactory (工厂合约 - 单例)
├── 创建供应商托管合约(Minimal Proxy)
├── 收取供应商上线费(100 USDC,可调整)
├── 管理平台手续费钱包地址
├── 管理平台管理员权限
└── 触发 ProviderRegistered 事件

ProviderEscrowImplementation (实现合约 - 单例)
├── 接收用户订阅支付(USDC)
├── 跟踪总存款和总提现金额
├── 执行提现并扣除平台手续费(10%,可调整)
├── 支持管理员暂停/恢复功能
└── 触发 Deposited、Withdrawn 事件

ProviderEscrowProxy (代理合约 - 每个供应商一个)
└── 最小代理,委托调用到 Implementation
```

**架构优势**:

- Gas 效率: 部署成本降低 ~10 倍
- 隔离性: 每个供应商独立合约实例
- 可维护性: 升级 Implementation 即可影响所有代理
- 成熟模式: Gnosis Safe、Uniswap 等项目验证

### 4.2 Factory 合约设计

**职责**:
- 供应商注册和合约创建
- 收取上线费
- 管理全局配置(手续费比例、上线费金额)

**关键函数**:

```solidity
// 供应商注册(收取上线费)
function registerProvider(
    address provider,
    bytes32 providerId,
    ISignatureTransfer.PermitTransferFrom calldata permit,
    bytes calldata signature
) external returns (address escrowAddress);

// 管理员更新配置
function updatePlatformFeeRate(uint256 newRate) external onlyAdmin;
function updateOnboardingFee(uint256 newFee) external onlyAdmin;
function updatePlatformFeeWallet(address newWallet) external onlyAdmin;

// 管理员暂停/恢复供应商
function pauseProvider(address escrowAddress) external onlyAdmin;
function unpauseProvider(address escrowAddress) external onlyAdmin;
```

**关键事件**:

```solidity
event ProviderRegistered(
    bytes32 indexed providerId,
    address indexed provider,
    address escrowAddress,
    uint256 onboardingFee
);

event PlatformFeeRateUpdated(uint256 oldRate, uint256 newRate);
event OnboardingFeeUpdated(uint256 oldFee, uint256 newFee);
event ProviderPaused(address indexed escrowAddress);
event ProviderUnpaused(address indexed escrowAddress);
```

### 4.3 Escrow 合约设计

**职责**:
- 接收订阅支付
- 处理供应商提现
- 自动扣除平台手续费

**关键函数**:

```solidity
// 初始化(由 Factory 调用)
function initialize(
    address _provider,
    address _platformFeeWallet,
    uint256 _platformFeeRate,
    address _factory
) external;

// 接收订阅支付(由 x402 合约或 Coinbase Commerce 调用)
function deposit(
    address user,
    uint256 amount,
    bytes32 subscriptionId
) external;

// 供应商提现
function withdraw(uint256 amount) external onlyProvider whenNotPaused;

// 查询可用余额
function availableBalance() external view returns (uint256);
```

**关键事件**:

```solidity
event Deposited(
    address indexed provider,
    address indexed user,
    uint256 amount,
    bytes32 indexed subscriptionId,
    uint256 timestamp
);

event Withdrawn(
    address indexed provider,
    uint256 providerAmount,
    uint256 platformFee,
    uint256 timestamp
);
```

### 4.4 合约安全设计

**安全模式: Checks-Effects-Interactions**

```solidity
function withdraw(uint256 amount) external onlyProvider nonReentrant whenNotPaused {
    // CHECKS - 验证
    require(amount > 0, "Invalid amount");
    uint256 available = totalDeposited - totalWithdrawn;
    require(amount <= available, "Insufficient balance");

    // EFFECTS - 状态更新(在外部调用之前)
    uint256 platformFee = (amount * platformFeeRate) / 10000;  // 基点计算
    uint256 providerAmount = amount - platformFee;
    totalWithdrawn += amount;

    // INTERACTIONS - 外部调用(最后执行)
    IERC20(USDC).transfer(platformFeeWallet, platformFee);
    IERC20(USDC).transfer(provider, providerAmount);

    emit Withdrawn(provider, providerAmount, platformFee);
}
```

**安全机制**:

1. **ReentrancyGuard** (OpenZeppelin)
   - 防止重入攻击
   - 所有涉及转账的函数使用 `nonReentrant` 修饰符

2. **Pausable 模式** (OpenZeppelin)
   - 管理员可暂停单个供应商合约
   - 紧急情况下停止提现功能
   - 不影响其他供应商

3. **访问控制**
   - `onlyProvider`: 只有供应商可提现
   - `onlyAdmin`: 只有管理员可调整配置
   - `onlyFactory`: 只有工厂可初始化

### 4.5 链上事件监听

**事件监听架构**:

```
Base 区块链
    ↓ 事件流
Event Listener Service (Go)
    ├── 监听 ProviderRegistered
    ├── 监听 Deposited
    ├── 监听 Withdrawn
    ├── 监听 ProviderPaused/Unpaused
    ↓ 解析验证
PostgreSQL 数据库
    ├── providers 表
    ├── subscriptions 表
    ├── transactions 表
    ↓ 状态更新
Subscription Management System
```

**Go 事件监听实现**:

```go
// 使用 go-ethereum 订阅事件
query := ethereum.FilterQuery{
    Addresses: []common.Address{factoryAddress},
    Topics: [][]common.Hash{
        {depositedEventSignature},
    },
}

logs := make(chan types.Log)
sub, err := client.SubscribeFilterLogs(context.Background(), query, logs)

// 处理事件
for {
    select {
    case log := <-logs:
        handleDepositedEvent(log)
    case err := <-sub.Err():
        // 重连逻辑
    }
}
```

**容错机制**:
- 存储最后处理的区块号
- 启动时从上次位置恢复
- 指数退避重试(RPC 失败)
- 事件数据验证(防止脏数据)

**乐观确认**:
- 收到 Webhook 通知立即激活订阅
- 异步等待链上确认(1-3 秒)
- 处理罕见失败情况(余额不足、签名无效)

## 五、供应商结算

### 5.1 供应商上线流程

**注册流程**:

```
1. 供应商准备
   - 拥有 EVM 钱包地址
   - 钱包中有 ≥100 USDC(上线费)

2. 调用 Factory.registerProvider()
   - 提供 providerId(唯一标识)
   - 提供 provider 地址(提现接收地址)
   - 提交 Permit2 签名(支付上线费)

3. Factory 执行
   - 验证 Permit2 签名
   - 收取 100 USDC 上线费
   - 通过 CREATE2 部署 Minimal Proxy
   - 初始化代理合约
   - 触发 ProviderRegistered 事件

4. 链下服务处理
   - 监听 ProviderRegistered 事件
   - 更新数据库(providerId → escrowAddress 映射)
   - 供应商可以开始配置套餐
```

**Gas 成本估算**:

| 操作 | Gas 成本 | 美元成本(Base) |
|------|---------|---------------|
| Factory 部署 | ~800,000 | $2-5(一次性) |
| 供应商注册 | ~150,000 | $0.50-1.00 |
| 用户订阅支付 | ~100,000 | $0.30-0.50 |
| 供应商提现 | ~120,000 | $0.40-0.60 |

### 5.2 供应商提现流程

**提现流程**:

```
1. 供应商查询可用余额
   - 调用 escrow.availableBalance()
   - 查看可提现金额

2. 供应商发起提现
   - 调用 escrow.withdraw(amount)
   - 指定提现金额

3. 智能合约执行
   - 验证余额充足
   - 计算平台手续费(10%)
   - 转账到平台手续费钱包
   - 转账到供应商钱包
   - 触发 Withdrawn 事件

4. 链下服务处理
   - 监听 Withdrawn 事件
   - 更新供应商余额
   - 记录提现历史
   - 记录平台手续费收入
```

**提现策略**:
- 无限制,随时提现
- 无最低金额限制
- 无频率限制
- 供应商承担 Gas 费

### 5.3 手续费管理

**平台手续费**:
- 默认比例: 10%
- 可通过管理员函数调整
- 基点计算(10000 = 100%)
- 自动从提现金额中扣除

**供应商上线费**:
- 默认金额: 100 USDC
- 可通过管理员函数调整
- 一次性收取
- 用于覆盖合约部署成本

**多签钱包管理**:
- 平台手续费钱包使用 Gnosis Safe
- 至少 2/3 多签
- 管理员操作需要多人确认
- 定期审计和对账

## 六、安全性设计

### 6.1 智能合约安全

**安全审计策略**:

根据产品决策,采用 **内部审计 + 测试网验证** 方案:

1. **内部安全审查**
   - 代码审查(多人)
   - 静态分析工具(Slither、Mythril)
   - 手动漏洞检查(OWASP Top 10)

2. **测试覆盖**
   - 单元测试覆盖率 >90%
   - 集成测试(与 x402 集成)
   - 模糊测试(Foundry Fuzz)
   - 不变量测试(关键属性验证)

3. **测试网验证**
   - Base Sepolia 部署 2+ 周
   - 模拟真实场景测试
   - 监控异常行为
   - 限制初期供应商数量(≤5)

### 6.2 运营安全

**多签钱包**:
- 平台手续费钱包使用 Gnosis Safe
- 至少 2/3 多签
- 管理员操作需要多人确认

**不可升级设计**:
- 合约部署后不可升级
- 发现严重漏洞需重新部署
- 降低复杂性和攻击面

**紧急响应**:
- 管理员可暂停单个供应商合约
- 暂停权限归项目管理员账号
- 明确的事件响应流程
- 24/7 监控和告警

**监控告警**:
- 异常大额提现告警
- Gas 价格突增告警
- 事件同步延迟告警
- 合约余额异常告警

### 6.3 支付安全

**Coinbase Commerce 安全**:
- Webhook 签名验证
- HTTPS 加密传输
- 支付金额验证
- 防重放攻击

**链上验证**:
- 双重验证机制(Webhook + 链上事件)
- 交易哈希验证
- 金额和地址验证
- 防止脏数据

| Coinbase Commerce 支付 | $0 | **零 Gas 费** |
| MetaMask 支付 | ~100,000 | $0.30-0.50 |
| 自动续订(Coinbase) | $0 | **零 Gas 费** |
| 套餐升级 | ~100,000 | $0.30-0.50 |

### 7.3 供应商成本

| 项目 | 成本 | 说明 |
|------|------|------|
| 上线费 | 100 USDC | 一次性,可调整 |
| 提现 Gas | $0.40-0.60/次 | 由供应商承担 |
| 平台手续费 | 10% | 自动扣除,可调整 |

### 7.4 收入模型

**平台收入来源**:
1. 供应商上线费: 100 USDC/个
2. 订阅收入分成: 10%
3. Coinbase Commerce 手续费: 由用户支付,平台不承担

**示例计算**:
- 100 个供应商上线: 100 × 100 = 10,000 USDC
- 月订阅收入 $10,000: 10% = 1,000 USDC
- **月度收入**: ~1,000 USDC(稳定后)

## 九、风险评估与缓解

### 9.1 技术风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| 智能合约漏洞 | 高 | 中 | 全面测试 + 内部审查 + 测试网验证 |
| Coinbase Commerce 不可用 | 中 | 低 | 保留 MetaMask 和 QR 码备选 |
| Gas 价格突增 | 中 | 低 | Base 费用稳定,监控告警 |
| RPC 服务故障 | 中 | 中 | 多 RPC 提供商 + 自动切换 |
| 事件同步延迟 | 低 | 中 | 存储区块号 + 恢复逻辑 |

### 9.2 业务风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| 供应商欺诈 | 中 | 低 | 声誉系统 + 用户评价 |
| 用户投诉 | 低 | 中 | 取消订阅机制(不考虑退款) |
| 监管变化 | 高 | 低 | 法律咨询 + 合规监控 |
| 采用率低 | 中 | 中 | 官方供应商先行 + 市场推广 |
| 依赖 Coinbase | 中 | 低 | 多钱包支持作为备选 |

### 9.3 运营风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| 密钥管理 | 高 | 低 | 多签钱包 + 硬件钱包 |
| 紧急响应 | 中 | 低 | Pausable 模式 + 应急手册 |
| 人员流失 | 中 | 中 | 文档完善 + 知识传承 |
