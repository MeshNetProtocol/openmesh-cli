# Phase 1 验证阶段 - Bug修复方案设计

基于前端联调阶段暴露出的两个核心问题，在此输出相关的修复与逻辑修正方案。

## Bug 1: Free 套餐不应该通过上链发交易来“订阅”
**问题分析**：
当前前端在订阅下拉列表中提供了 `Free 套餐 (0 USDC)` 供用户选择并点击提交。这是一个逻辑违背——Free 应是系统内的“兜底”设定。只要 VPN 节点在链上（或服务端）查不到该身份有任何有效的付费记录，就应该无条件将其降级/默认为 Free 用户，并仅仅在本地或通过中心化 Tracker 统计其每日 100MB 流量。0 USDC 的上链“订阅”不仅没有任何安全意义，反而会占用链上存储和额外的 Permit 签名负担。

**修改方案**：
1. **前端交互移除 (`index.html / app.js`)**：
   - 取消“初次订阅”列表中 `Plan 1 (Free)` 的选项，用户首开合约级订阅只能是从 `Basic`、`Premium` 或 `测试套餐` 起步。
   - 当用户查询自己的订阅数据为空时，前端界面应当默认显示卡片提示：“当前状态：未付费 (采用默认 Free 模式，每日 100MB)”。
2. **降级与解约 (`app.js`)**：
   - 现有的 `[下月降级]` 如果用户选择降级为 Free，在我们的业务概念里，其本质上是不再续费任何高级套餐。由于 V2 体系里我们有 `cancelPendingChange` 和通过 `cancelSubscription` 关闭续费功能的操作。最优雅的降级至 Free 方式应该是：调用 **取消自动续费**。服务过期后自动回落至 Free 状态。
   - 若要保留“指明下个月降级成 `planId=1` （Free 占位符）”的功能也可以，但更推荐将其在界面上直译为“取消后续扣款”。
3. **服务端识别 (`traffic-tracker.js`)**：
   - 确保当用户流量上报但查询链上无活跃 Sub 时，使用 `Plan 1` 的硬编码限额：100MB，超额即挂断。

## Bug 2: 丢失了短周期自动扣款的“测试套餐”
**问题分析**：
在真实环境测试智能合约的自动续费逻辑 (`RenewalService`) 时，测试者不可能干坐着等 30 天/365 天让 `executeRenewal()` 被激活。之前的体系保留了一个“0.1 USDC / 30分钟”的测试条目，用以在同等环境下验证大批量的 CDP 节点发起的自动续费是否异常扣款。在对套餐规格正规化改造时，它被错误覆盖了。

**修改方案**：
引入一个新的 `Plan 4 (测试桩)` 专门用于开发期间的自动续费时间轴验证。
1. **合约层面 (若涉及重新发测试网)**：
   在 `VPNSubscriptionV2.sol` 的构造函数或 Relayer 操作中，确保注入测试档位：
   - ID: `4`
   - Price: `0.1 USDC`
   - Period: `1800` (30 分钟)
2. **后端 Prepare API (`index.js`) 补丁**：
   在 `/api/subscription/prepare` 中计算 `maxAmount` 时支持 `Plan 4` 的极低金额：
   ```javascript
   if (planId === 4) amountNum = 0.1; // 100,000 个六位精度 USDC 单位
   ```
3. **前端入口补丁 (`index.html`)**：
   重新补上黄色的⚠️提示栏：
   ```html
   <option value="4" style="background: #fff3cd; color: #856404;">⚠️ 测试套餐 - 0.1 USDC / 30 分钟 (仅测试网)</option>
   ```

## Bug 3: 前后端分离访问导致的 501 Unsupported Method (POST)
**问题分析**：
开发者在使用类似于 `python -m http.server 8080` 或 IDE 插件等内置的简易静态服务器加载前端 `index.html` 时，如果 `app.js` 将 `API_BASE` 配置为相对路径 `/api`，浏览器会向这个简易静态服务器发起以 `/api/subscription/prepare` 为路径的 POST 请求。由于静态文件服务器通常仅支持 GET 请求去获取文件内容，因此会抛出 `501 Unsupported Method ('POST')` 异常，而这笔请求根本没有抵达真正的订阅 Node.js 服务。

**修改方案**：
Node 后台服务已通过环境配置固定跑在 3000 端口，前端和后端实际上是经典的跨域分离部署状态。因此在不通过 Nginx 或网关反向代理合并端口的前提下：
- **前端 (`app.js`)**：必须将 `CONFIG.API_BASE` 硬编码指向 Node 服务器的实际跑测端口：`http://localhost:3000/api`。后端已完全配套支持 CORS 跨域请求（`Access-Control-Allow-Origin: *`）。
