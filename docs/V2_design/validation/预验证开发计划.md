## 1. 预验证目标

在正式开发前,通过构建 3 个最小功能原型来验证核心技术方案的可行性:

1. **流量统计原型**: 验证 Hysteria2 流量收集的功能性、完整性和性能
2. **订阅支付原型**: 验证套餐订阅和自动续订的完整流程
3. **集成验证原型**: 验证流量管控与订阅系统的集成

### 1.1 核心验证点

- Hysteria2 流量统计的准确性 (目标 > 99%)
- 流量数据上报的实时性 (目标 < 1 分钟延迟)
- MetaMask 支付集成的可行性
- 订阅到期和续订逻辑的正确性
- 流量超额断线机制的有效性
- 多节点切换的流量统计一致性

### 1.2 技术约束

- 所有服务在本地运行
- 数据存储使用临时文件 (JSON)
- 数据层封装接口,便于后续替换数据库
- 服务端代码使用 Go
- 客户端使用 sing-box
- 一切从简,快速验证

## 3. 原型 1: 流量统计验证

### 3.1 目标

验证 Hysteria2 流量统计的准确性、实时性和多节点切换场景。

### 3.2 功能范围

**核心功能**:
- 2 个 Hysteria2 节点 (node1, node2)
- 流量收集服务 (接收节点上报)
- 简单的流量统计界面
- sing-box 客户端配置 (支持节点切换)

**验证场景**:
- 单节点流量统计
- 节点切换后流量累加
- 流量统计准确性 (与实际传输对比)
- 流量上报延迟测试

### 3.3 数据模型

```go
// data/storage.go - 数据层接口
type TrafficStorage interface {
    // 记录流量
    RecordTraffic(userID string, nodeID string, bytes int64) error
    
    // 查询用户总流量
    GetUserTraffic(userID string) (int64, error)
    
    // 查询用户在指定节点的流量
    GetUserTrafficByNode(userID string, nodeID string) (int64, error)
    
    // 获取所有用户流量
    GetAllTraffic() (map[string]int64, error)
}

// data/file_storage.go - 文件存储实现
type FileStorage struct {
    dataFile string
}

// 数据文件格式 (JSON)
{
  "users": {
    "user1": {
      "total": 1048576,
      "by_node": {
        "node1": 524288,
        "node2": 524288
      },
      "last_updated": "2026-03-31T10:00:00Z"
    }
  }
}
```

### 3.4 任务卡

#### 任务 1.1: 搭建 Hysteria2 节点
- **功能**: 配置 2 个 Hysteria2 节点,启用流量统计 API
- **输入**: Hysteria2 配置模板
- **输出**: 2 个可运行的 Hysteria2 节点
- **验收**: 节点可正常连接,流量统计 API 可访问

#### 任务 1.2: 实现流量收集服务
- **功能**: Go 服务接收节点流量上报,聚合统计
- **输入**: 节点上报的流量数据
- **输出**: 聚合后的用户流量统计
- **验收**: 可接收上报,数据正确聚合并持久化

#### 任务 1.3: 实现数据存储层
- **功能**: 封装数据存储接口,实现文件存储
- **输入**: 流量数据
- **输出**: JSON 文件持久化
- **验收**: 数据可读写,接口清晰,易于替换

#### 任务 1.4: 实现流量统计界面
- **功能**: 简单 Web 界面展示流量统计
- **输入**: 用户 ID
- **输出**: 总流量、各节点流量、实时更新
- **验收**: 界面可访问,数据展示正确

#### 任务 1.5: 配置 sing-box 客户端
- **功能**: 配置 sing-box 支持 2 个节点切换
- **输入**: 节点地址和密钥
- **输出**: sing-box 配置文件
- **验收**: 可连接节点,可手动切换

#### 任务 1.6: 流量准确性测试
- **功能**: 传输已知大小文件,验证统计准确性
- **输入**: 测试文件 (1MB, 10MB, 100MB)
- **输出**: 统计误差报告
- **验收**: 误差 < 1%

#### 任务 1.7: 节点切换测试
- **功能**: 在 2 个节点间切换,验证流量累加
- **输入**: 节点切换操作
- **输出**: 流量正确累加
- **验收**: 切换后总流量 = node1 + node2

#### 任务 1.8: 实时性测试
- **功能**: 测试流量上报到界面显示的延迟
- **输入**: 流量传输
- **输出**: 延迟时间
- **验收**: 延迟 < 1 分钟

## 5. 原型 3: 集成验证

### 5.1 目标

将流量统计和订阅系统集成,验证流量管控和余量展示。

### 5.2 功能范围

**核心功能**:
- 集成流量统计和订阅管理
- 流量额度检查
- 超额断线机制
- 80% 用量提醒
- 统一的管理界面

**验证场景**:
- 订阅激活后可使用 VPN
- 流量达到 80% 触发提醒
- 流量超额后断开连接
- 订阅过期后禁止连接
- 续订成功后恢复服务

- **输入**: 流量上报

#### 任务 3.5: 实现超额断线

#### 任务 3.6: 实现统一界面

- **验收**: 界面展示完整,数据实时更新

    GetExpiringSubscriptions(beforeTime time.Time) ([]*Subscription, error)
}

// TierStorage 套餐存储接口
type TierStorage interface {
    GetAllTiers() ([]*Tier, error)
    GetTier(tierID string) (*Tier, error)
}

func NewStorage(storageType string) Storage {
    switch storageType {
    case "file":
        return NewFileStorage("./data")
    case "postgres":
        return NewPostgresStorage(connStr)
    case "sqlite":
        return NewSQLiteStorage(dbPath)
    default:
        return NewFileStorage("./data")
    }
}
