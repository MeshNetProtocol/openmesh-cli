# 文档审阅清单

用于审阅 `docs/V2_design/` 下的规划、设计、验证、模块说明等文档，减少方向性错误、状态冲突和术语混乱。

---

## 1. 文档定位

- [ ] 先确认这份文档的类型：开发计划、技术设计、验证记录、模块说明、会议结论，还是归档文档
- [ ] 文档开头已明确说明“描述的是目标方案、当前现状，还是验证结果”
- [ ] 没有把“计划”写成“现状”，也没有把“验证结论”写成“正式实现”
- [ ] 文档标题、摘要、章节结构与文档定位一致

---

## 2. 架构与职责一致性

- [ ] 模块职责与当前确认方案一致
- [ ] 链上、服务端、客户端职责边界清晰
- [ ] 没有把服务端职责误写到合约侧
- [ ] 没有把客户端职责误写到服务端侧
- [ ] 没有重复定义同一职责由多个模块承担
- [ ] 架构图、模块列表、正文描述三者一致

---

## 3. 验证文档与目标方案边界

- [ ] 明确区分“技术验证实现”和“正式目标架构”
- [ ] `validation/phase*` 文档被表述为验证过程或验证结论，而不是正式交付实现
- [ ] 如果验证实现与目标架构不同，文档中已明确说明差异
- [ ] 没有把 POC、临时脚本、文件型状态管理误写成最终形态
- [ ] 验证环境和目标环境区分清楚（例如测试网与主网）

---

## 4. 状态与进度表述

- [ ] 模块状态、里程碑状态、任务状态之间不冲突
- [ ] “规划中 / 待开发 / 开发中 / 已完成 / 待适配”等状态语义统一
- [ ] 没有出现“正文说待开发、表格说已完成”这类冲突
- [ ] 里程碑依赖关系合理
- [ ] 阶段完成度和备注相互匹配

---

## 5. 计划任务可执行性

- [ ] 每个 Phase 都有清晰可执行的任务，而不是抽象口号
- [ ] 任务顺序符合依赖关系
- [ ] 没有保留已经明确废弃或不再采用的任务
- [ ] 任务能映射到具体模块或交付物
- [ ] 任务颗粒度适中，不要过大也不要过细

---

## 6. 技术栈表述

- [ ] 明确写清是“当前实现使用”还是“目标方案使用”
- [ ] 没有把验证期临时技术栈误写成长期方案
- [ ] 没有把可选技术写成既定事实
- [ ] 同一模块的技术栈在全文中保持一致

---

## 7. 术语统一

- [ ] 同一个概念只使用一种主称呼
- [ ] 合约名、模块名、服务名、目录名与代码/文档现状一致
- [ ] 没有混用旧名、新名、临时名
- [ ] 中英文术语前后一致
- [ ] 首次出现的缩写有足够上下文

---

## 8. 废弃方案清理

- [ ] 没有残留已废弃组件、合约、接口、流程名称
- [ ] 没有残留已经被否定的技术路线
- [ ] 如果保留历史信息，已明确标记为“历史方案”或“归档内容”
- [ ] 旧方案不会被误读为现行方案

---

## 9. V1 / V2 或阶段演进关系

- [ ] 高版本能力建立在低版本能力之上，依赖关系合理
- [ ] 没有默认假设前一阶段已完成，但文档别处又显示未完成
- [ ] 对比表写的是“目标能力对比”还是“当前落地对比”，语义清楚
- [ ] 未来规划部分不会让人误解为当前实施状态

---

## 10. 图表与正文一致性

- [ ] 表格、流程图、架构图与正文描述一致
- [ ] 图中组件没有在正文缺失说明
- [ ] 正文中的关键模块在图中也能找到
- [ ] 图例和状态符号含义明确

---

## 11. 发布前最后复核

提交前，至少逐条问自己：

- [ ] 这句话描述的是目标、现状，还是验证结果？
- [ ] 这项能力到底属于链上、服务端，还是客户端？
- [ ] 这个组件/合约/接口今天还存在吗？
- [ ] 这项任务真的要做，还是旧方案残留？
- [ ] 第一次接手项目的人会不会被这份文档误导？

---

## 推荐审阅流程

### 快速审阅（5 分钟）

适用于小改动：

- [ ] 查有没有旧术语残留
- [ ] 查状态是否冲突
- [ ] 查是否把验证实现写成正式方案

### 标准审阅（15-30 分钟）

适用于计划文档、模块设计文档更新：

- [ ] 过一遍本清单第 1-10 节
- [ ] 对照相关代码目录、验证文档、历史设计文档核对一次
- [ ] 检查图表、表格、正文是否一致

### 重要变更审阅

适用于改架构、改模块职责、改阶段路线图：

- [ ] 明确列出“变更前 / 变更后”
- [ ] 明确哪些旧内容需要删除
- [ ] 明确哪些验证文档会因此失效或需要补充说明
- [ ] 更新相关总览文档、模块文档、验证文档之间的交叉引用

---

## 适用范围

建议用于以下文档：

- `docs/V2_design/PROJECT_OVERVIEW.md`
- `docs/V2_design/modules/*`
- `docs/V2_design/validation/*`
- 新增的设计方案、模块说明、阶段总结、迁移计划文档

---

## 维护原则

- 这份清单本身也应保持简洁，只保留高频、易错、会误导团队的检查项
- 若后续连续出现同类文档问题，应把该问题沉淀为新的审阅项
- 若某些检查项长期无实际价值，应及时删除，避免清单失效
