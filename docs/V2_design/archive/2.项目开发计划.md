## 一、项目概况

### 1.1 项目目标

OpenMesh V2 是流量行业的 OpenRouter,提供统一的 VPN 流量供应商接入平台。本期核心目标是交付官方流量管理系统和套餐管理系统,支持供应商自行部署并接入平台。

### 1.2 技术栈现状

**后端技术栈**:
- Go 1.25.0 (核心业务逻辑)
- sing-box 1.13.1 (VPN 流量转发)
- Hysteria2 (底层传输协议)
- x402 协议 (去中心化支付结算)
- Gin (HTTP 框架)

**前端技术栈**:
- Swift (iOS/macOS 客户端)
- Kotlin (Android 客户端)
- Windows 原生客户端

**基础设施**:
- Cloudflare Workers + D1 (market-api,供应商发现服务)
- 以太坊/Solana (x402 支付结算)

### 1.3 现有代码库结构

```
openmesh-cli/
├── go-cli-lib/          # Go 核心库,包含 VPN 逻辑
├── market-api/          # Cloudflare Workers API,供应商发现
├── x402/                # x402 支付协议实现
├── sing-box/            # sing-box 集成
├── hysteria/            # Hysteria2 协议
├── openmesh-apple/      # iOS/macOS 客户端
├── openmesh-android/    # Android 客户端
├── openmesh-win/        # Windows 客户端
└── docs/                # 文档
```

## 四、关键依赖与风险

### 4.1 技术依赖

| 依赖项 | 当前状态 | 风险等级 | 应对措施 |
|--------|---------|---------|---------|
| x402 协议 | 已集成基础功能 | 中 | 需验证订阅场景适配性,可能需要扩展 |
| sing-box | v1.13.1 | 低 | 稳定版本,需确认流量控制扩展能力 |
| Hysteria2 | 已集成 | 低 | 成熟协议,风险可控 |
| Cloudflare Workers | 已部署 market-api | 低 | 现有基础设施可复用 |

### 4.2 关键风险

**风险 1: 设备数管理并发一致性**
- **描述**: 多设备几乎同时连接时,设备数判断可能出现竞态
- **影响**: 可能允许超出上限的设备连接
- **应对**: 使用分布式锁或数据库事务保证一致性
- **责任人**: 架构师

**风险 2: x402 协议与订阅场景适配**
- **描述**: x402 协议主要面向单次支付,订阅场景可能需要扩展
- **影响**: 可能需要修改协议或实现自定义扩展
- **应对**: 尽早验证协议适配性,必要时与 x402 社区沟通
- **责任人**: 架构师

**风险 3: 自动续订成功率**
- **描述**: 钱包余额不足、Session 过期等导致续订失败
- **影响**: 用户体验下降,服务中断
- **应对**: 提前 7 天余额提醒,宽限期内多次重试
- **责任人**: 工程师

**风险 4: 带宽限速实现一致性**
- **描述**: 不同供应商的带宽限速实现可能与套餐标称不符
- **影响**: 用户体验不一致,平台信誉受损
- **应对**: 制定带宽限速验收测试规范,供应商上架前必须通过
- **责任人**: 项目负责人

**风险 5: 链上支付确认延迟**
- **描述**: 区块链交易存在确认等待时间
- **影响**: 支付成功后服务激活延迟
- **应对**: 收到支付签名后先激活订阅,链上确认作为事后审计
- **责任人**: 架构师
