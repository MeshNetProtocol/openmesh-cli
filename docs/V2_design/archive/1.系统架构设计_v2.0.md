## 1. 引言

### 1.1 文档目的

本文档是 OpenMesh V2 的系统架构设计文档,基于已冻结的需求文档 v2.0 和已确认的架构决策,提供完整的技术架构蓝图,用于指导后续的详细设计和开发实施。

本文档的主要用途:
- 为工程师提供模块设计、接口定义和技术选型依据
- 为测试团队提供系统边界、集成点和测试策略输入
- 为项目负责人提供架构风险评估和技术决策追溯
- 为后续 API 设计、数据库设计、合约设计提供架构约束

### 1.2 架构目标

OpenMesh V2 的架构设计遵循以下核心目标:

1. **简单优先**:避免过度设计,优先选择简单可行的方案
2. **安全第一**:身份私钥保护、智能合约安全、API 安全
3. **用户体验**:零 Gas 费、毫秒级支付确认、流畅的跨设备体验
4. **可维护性**:模块化设计、清晰的职责边界、完善的监控
5. **向后兼容**:V1 私有节点功能完整保留

**三层架构说明**:

**第一层 - 流量层**:
- 客户端通过 sing-box 连接 Hysteria2 节点
- Hysteria2 节点提供 VPN 流量转发服务
- Hysteria2 节点使用 Traffic Stats API 统计用户流量

**第二层 - 统计层**:
- 客户端从流量统计服务查询流量使用情况
- Hysteria2 节点定期上报流量到流量统计服务
- 流量统计服务检测超额并通知客户端

**第三层 - 结算层**:
- 客户端展示订阅信息
- 用户通过第三方钱包支付
- Coinbase Commerce 处理支付并通知订阅服务
- 订阅服务与流量统计服务交互,确认用户是否可用
- 智能合约托管资金并处理提现

### 2.2 系统分层架构

**客户端层**
- 职责: 用户交互、身份管理、支付触发、VPN 连接
- 技术: 原生应用 (iOS/Android/macOS/Windows)
- 关键特性: 本地安全存储、跨平台一致性、V1/V2 功能隔离

**API 网关层**
- 职责: 请求路由、身份认证、请求签名验证
- 技术: HTTP/HTTPS (RESTful API)
- 关键特性: 无状态设计、水平扩展、防重放攻击

**业务逻辑层**
- 职责: 核心业务逻辑处理、状态管理、事件处理
- 技术: 微服务架构 (可选单体起步)
- 关键特性: 模块化、清晰的职责边界、易于测试

**数据持久化层**
- 职责: 数据存储、查询优化、事务管理
- 技术: PostgreSQL
- 关键特性: ACID 保证、关系型数据模型、备份恢复

**区块链交互层**
- 职责: 链上事件监听、交易提交、合约调用
- 技术: Base RPC + 智能合约
- 关键特性: 多 RPC 容错、事件同步、Gas 优化

**子模块**:


### 3.2 后端服务模块

#### 3.2.1 用户身份服务

**职责**:
- 用户身份验证
- 请求签名验证
- 用户信息管理

**核心接口**:
```
GET /api/v1/subscriptions
Headers:
  X-Signature: 0x... (用户对请求内容的签名)
  X-Timestamp: 1234567890
  X-Nonce: random_string

- 从签名中恢复用户地址
- 验证签名有效性
- 返回该用户的订阅列表
```

**技术实现**:
- EIP-712 签名验证 (Typed Structured Data)
- 从签名恢复 EVM 地址 (ecrecover)
- 防重放攻击 (domain + nonce + timestamp)
- 无需客户端传递地址,从签名中恢复

#### 3.2.2 供应商管理服务

**职责**:
- 供应商注册
- 供应商信息管理
- 套餐管理
- 供应商榜单排序

**核心接口**:
```
POST /api/v1/providers/register
- 供应商注册
- 创建托管合约

GET /api/v1/providers
- 获取供应商列表
- 支持排序和筛选

POST /api/v1/providers/{id}/tiers
- 创建套餐
- 配置价格和流量

PUT /api/v1/providers/{id}/tiers/{tierId}
- 更新套餐
- 上架/下架
```

**榜单排序机制**:
```
排序算法:
1. 基于供应商缴纳的平台手续费总额
2. 手续费越高,排名越靠前
3. 形成 PoW (Proof of Work) 机制
4. 鼓励供应商提供优质服务

计算公式:
score = total_platform_fees_paid
```

**完整支付流程**:
```
1. 客户端创建支付请求
   POST /api/v1/payments/create
   - 指定 redirect_url (支付完成后返回的 URL)
   - 指定 cancel_url (用户取消支付返回的 URL)
   ↓
2. 后端生成 Coinbase Commerce 支付链接
   - 创建 Charge
   - 设置 redirect_url 和 cancel_url
   - 返回 hosted_url 给客户端
   ↓
3. 客户端打开 Coinbase Commerce 支付页面
   - 在浏览器或 App 内打开 hosted_url
   - 用户在 Coinbase 页面完成支付
   ↓
4. 支付完成后,Coinbase Commerce 执行两个操作:
   
   操作 A (用户侧):
   - Coinbase Commerce 重定向用户回到 redirect_url
   - URL 参数包含支付结果 (charge_id, status)
   - 客户端从 URL 参数知道支付成功
   
   操作 B (服务端):
   - Coinbase Commerce 发送 Webhook 到后端
   - 后端验证 Webhook 签名
   - 后端解析支付事件 (charge:confirmed)
   - 后端更新订阅状态为 Active
   - 后端激活 VPN 服务
   ↓
5. 客户端处理重定向
   - 解析 URL 参数,获取 charge_id
   - 查询一次订阅状态确认激活
   GET /api/v1/subscriptions?chargeId={charge_id}
   ↓
6. 显示订阅成功
```

**关键点**:
- **无需轮询**: Coinbase Commerce 通过重定向告诉客户端支付结果
- **双重确认**: 重定向(用户侧) + Webhook(服务端) 确保可靠性
- **毫秒级响应**: Coinbase Commerce 支付确认是毫秒级的
- **客户端只需查询一次**: 收到重定向后查询订阅状态即可

**核心接口**:
```
POST /api/v1/payments/create
Request:
  Headers:
    X-Signature: 0x... (EIP-712 签名)
  Body:
    {
      "providerId": "provider_xxx",
      "tierId": "tier_standard",
      "redirectUrl": "openmesh://payment/success",
      "cancelUrl": "openmesh://payment/cancel"
    }
Response:
  {
    "chargeId": "charge_xxx",
    "hostedUrl": "https://commerce.coinbase.com/charges/xxx",
    "expiresAt": "2026-03-30T11:00:00Z"
  }

POST /api/v1/webhooks/coinbase
- 接收 Coinbase Commerce Webhook
- 验证签名
- 处理 charge:confirmed 事件
- 激活订阅

GET /api/v1/subscriptions?chargeId={charge_id}
```

```
    ↓

    ↓
```

### 4.7 链上事件同步流程

```
Base 网络产生事件:
    - ProviderRegistered
    - Deposited
    - Withdrawn
    - ProviderPaused/Unpaused
    ↓
链上事件监听服务 (Go):
    WebSocket 订阅事件
    ↓
    接收到新事件
    ↓
    解析事件数据
    - 事件类型
    - 事件参数
    - 区块号
    - 交易哈希
    ↓
    验证事件
    - 验证合约地址
    - 验证事件签名
    - 验证数据完整性
    ↓
    开始数据库事务
    ↓
    更新业务状态
    - 供应商注册 → 创建供应商记录
    - 支付成功 → 激活订阅
    - 提现成功 → 记录提现历史
    ↓
    更新同步状态
    - 记录最后处理的区块号
    - 记录事件处理时间
    ↓
    提交事务
    ↓
    触发后续业务逻辑
    - 通知客户端
    - 通知供应商系统
    ↓
    继续监听下一个事件

异常处理:
    RPC 连接失败
    ↓
    指数退避重试
    - 1 秒, 2 秒, 4 秒, 8 秒...
    - 最大 60 秒
    ↓
    切换到备用 RPC
    ↓
    从上次区块号恢复
    ↓
    继续监听

    事件处理失败
    ↓
    回滚数据库事务
    ↓
    记录错误日志
    ↓
    重试处理 (最多 3 次)
    ↓
    失败后告警
```

**容错机制**:
- 存储最后处理的区块号
- 启动时从上次位置恢复
- 多 RPC 提供商切换
- 事件数据验证
- 数据库事务保证一致性

### 7.3 API 安全

#### 7.3.1 身份认证

**EIP-712 签名验证 (Typed Structured Data)**:
```go
// EIP-712 Domain
type Domain struct {
    Name              string
    Version           string
    ChainId           *big.Int
    VerifyingContract common.Address
}

// 请求消息结构
type PaymentRequest struct {
    ProviderId string
    TierId     string
    Nonce      string
    Timestamp  int64
}

// 验证 EIP-712 签名
func VerifyEIP712Signature(signature string, request PaymentRequest) (common.Address, error) {
    // 1. 构造 EIP-712 Domain
    domain := Domain{
        Name:    "OpenMesh V2",
        Version: "1",
        ChainId: big.NewInt(8453), // Base
    }

    // 2. 构造 TypedData
    typedData := apitypes.TypedData{
        Types: apitypes.Types{
            "EIP712Domain": []apitypes.Type{
                {Name: "name", Type: "string"},
                {Name: "version", Type: "string"},
                {Name: "chainId", Type: "uint256"},
            },
            "PaymentRequest": []apitypes.Type{
                {Name: "providerId", Type: "string"},
                {Name: "tierId", Type: "string"},
                {Name: "nonce", Type: "string"},
                {Name: "timestamp", Type: "uint256"},
            },
        },
        PrimaryType: "PaymentRequest",
        Domain:      domain,
        Message:     request,
    }

    // 3. 计算 EIP-712 哈希
    hash, _ := typedData.HashStruct(typedData.PrimaryType, typedData.Message)

    // 4. 从签名恢复地址
    pubKey, _ := crypto.SigToPub(hash, signature)
    address := crypto.PubkeyToAddress(*pubKey)

    return address, nil
}
```

**优势**:
- 用户在钱包中看到结构化数据,而非哈希
- 内置 domain 隔离,防止跨应用重放
- 更安全,更专业

#### 7.3.2 防重放攻击

**Nonce + Timestamp**:
```go
type SignedRequest struct {
    Address   string `json:"address"`
    Message   string `json:"message"`
    Nonce     string `json:"nonce"`
    Timestamp int64  `json:"timestamp"`
    Signature string `json:"signature"`
}

func ValidateRequest(req SignedRequest) error {
    // 1. 检查时间戳 (5 分钟内有效)
    if time.Now().Unix() - req.Timestamp > 300 {
        return errors.New("request expired")
    }
    
    // 2. 检查 nonce (Redis 存储已使用的 nonce)
    if redis.Exists(req.Nonce) {
        return errors.New("nonce already used")
    }
    
    // 3. 验证签名
    if !VerifySignature(req.Address, req.Message, req.Signature) {
        return errors.New("invalid signature")
    }
    
    // 4. 存储 nonce (5 分钟过期)
    redis.Set(req.Nonce, true, 300)
    
    return nil
}
```

#### 7.3.3 Webhook 安全

**Coinbase Commerce Webhook 验证**:
```go
func VerifyWebhook(signature, payload string) bool {
    // 1. 获取 Webhook Secret
    secret := os.Getenv("COINBASE_WEBHOOK_SECRET")
    
    // 2. 计算 HMAC
    mac := hmac.New(sha256.New, []byte(secret))
    mac.Write([]byte(payload))
    expectedSignature := hex.EncodeToString(mac.Sum(nil))
    
    // 3. 比较签名
    return hmac.Equal([]byte(signature), []byte(expectedSignature))
}
```

- 优先级: P1

**密钥管理**:
- 风险: 管理员私钥泄露
- 缓解: 多签钱包 + 硬件钱包
