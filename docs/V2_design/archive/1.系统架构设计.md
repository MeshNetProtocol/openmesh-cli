## 一、架构概览

### 1.1 系统定位

OpenMesh V2 是流量行业的 OpenRouter，提供统一的 VPN 流量供应商接入平台。

**核心价值**：
- 平台托管结算：智能合约自动管理资金，扣除 10% 手续费
- 统一接入协议：基于 x402 的标准化订阅支付
- Fair Use 管理：月度流量额度，超额降速不断线
- 供应商自主：可使用官方系统或自建系统

### 1.2 整体架构

系统采用四层架构设计：

```
┌─────────────────────────────────────────────────────────┐
│                    客户端层                              │
│  iOS/Android/Windows/macOS 客户端                       │
│  职责：订阅购买、流量使用、状态展示                       │
└─────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────┐
│                  后端服务层                              │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐   │
│  │套餐管理系统   │ │事件监听服务   │ │x402支付集成  │   │
│  └──────────────┘ └──────────────┘ └──────────────┘   │
│  数据层：PostgreSQL + Redis                            │
└─────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────┐
│              智能合约层（Base 网络）                      │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐   │
│  │Factory 合约   │ │Escrow 实现    │ │Proxy 合约     │   │
│  └──────────────┘ └──────────────┘ └──────────────┘   │
└─────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────┐
│                  流量管理层                              │
│  Hysteria2 + 准入控制 + Fair Use 流量统计                │
└─────────────────────────────────────────────────────────┘
```

### 1.3 技术栈选型

| 层级 | 技术选型 | 选择理由 |
|------|---------|---------|
| 智能合约 | Solidity + Base | Gas 费低、原生 USDC、x402 已部署 |
| 支付集成 | Coinbase Commerce + 多钱包 | 零 Gas 费、Coinbase 账户直接支付、2-3 周快速上线 |
| 后端服务 | Go 1.25 + Gin | 高性能、现有技术栈 |
| 数据存储 | PostgreSQL + Redis | 成熟稳定、支持复杂查询 |
| 流量管理 | sing-box + Hysteria2 | 现有集成、性能优秀 |
| 区块链交互 | go-ethereum | 官方 SDK、功能完整 |

### 1.4 支付方案：Coinbase Commerce + 多钱包支持

**核心策略**：聚焦 USDC + Base 链，采用 Coinbase Commerce 作为主要支付方式，同时支持多种钱包。

**支付方式**：
- **主要方式**：Coinbase Commerce（支持 Coinbase 账户 + Coinbase Wallet）
- **备选方式**：MetaMask 深链接（浏览器用户）
- **通用方式**：EIP-681 标准支付链接 + QR 码（任何钱包）

**关键优势**：
- ✅ **零 Gas 费**：用户支付无需承担 Gas 费
- ✅ **Coinbase 账户集成**：现有 Coinbase 用户直接用账户余额支付
- ✅ **快速上线**：2-3 周开发时间（vs 内置钱包 12-16 周）
- ✅ **原生自动续订**：Coinbase Commerce 支持订阅模式
- ✅ **多钱包兼容**：支持 MetaMask、WalletConnect 等

**用户身份系统**：
- 用户 ID = email/手机号（独立账号系统）
- 用户可绑定多个钱包地址
- 通过签名验证钱包所有权

**详细方案**：参见 [1_2.架构决策-Base+USDC专属方案.md](1_2.架构决策-Base+USDC专属方案.md)

### 1.5 核心设计原则

1. **职责分离**：各层独立，通过接口交互
2. **事件驱动**：链上事件驱动链下状态同步
3. **乐观确认**：支付签名即激活，链上确认异步
4. **可观测性**：完整的日志、监控、告警

## 三、智能合约层架构

### 3.1 模块划分

**ProviderEscrowFactory（工厂合约）**
- 职责：供应商注册、合约创建、全局配置管理
- 状态：platformFeeRate, onboardingFee, admin, providers
- 事件：ProviderRegistered, ConfigUpdated

**ProviderEscrowImplementation（实现合约）**
- 职责：资金托管、提现处理、手续费扣除
- 状态：provider, totalDeposited, totalWithdrawn, paused
- 事件：Deposited, Withdrawn

**ProviderEscrowProxy（代理合约）**
- 职责：委托调用到 Implementation
- 模式：EIP-1167 Minimal Proxy（每个供应商一个）

### 3.2 接口定义

**Factory.registerProvider**
- 输入：provider 地址、providerId、Permit2 签名
- 输出：escrowAddress
- 前置条件：permit 金额 >= onboardingFee、providerId 未注册
- 后置条件：创建 Proxy、收取上线费、触发事件

**Escrow.withdraw**
- 输入：提现金额
- 输出：无
- 前置条件：调用者 == provider、金额 <= 可用余额、未暂停
- 后置条件：扣除手续费、转账、更新状态、触发事件

**Escrow.deposit**（被动接收）
- 通过 x402 Permit2 直接转入 USDC
- 触发 Deposited 事件供后端监听

### 3.3 与 x402 协议集成

**集成方式**：
- 用户支付时，witness.to 指向供应商 Escrow 合约地址
- 调用 x402ExactPermit2Proxy.settle()
- USDC 直接转入 Escrow 合约
- Escrow 触发 Deposited 事件

**数据流**：
```
客户端 → 构建 Permit2 签名（witness.to = escrowAddress）
      → 调用 x402ExactPermit2Proxy.settle()
      → USDC 转入 Escrow 合约
      → 触发 Deposited 事件
      → 后端监听事件并激活订阅
```

## 四、后端服务层架构

### 4.1 模块划分

**套餐管理系统**
- 职责：套餐 CRUD、订阅管理、状态查询
- 接口：REST API
- 数据：PostgreSQL（订阅、套餐、用户）

**事件监听服务**
- 职责：监听链上事件、同步到数据库
- 事件：ProviderRegistered、Deposited、Withdrawn
- 容错：存储最后处理区块号、指数退避重试

**x402 支付集成**
- 职责：验证支付签名、查询合约地址
- 协议：x402 订阅扩展协议
- 集成：与套餐管理系统交互

### 4.2 模块间接口

**套餐管理 → 数据库**
- 写入：创建订阅、更新状态、记录流量
- 读取：查询订阅、查询套餐、查询用户

**事件监听 → 套餐管理**
- 调用：激活订阅、记录提现、更新供应商状态
- 数据：事件参数（user、amount、subscriptionId）

**x402 集成 → 套餐管理**
- 查询：获取供应商合约地址
- 验证：验证支付签名有效性

### 4.3 数据一致性保证

**乐观确认机制**：
- 收到支付签名立即激活订阅
- 异步等待链上确认（1-3 秒）
- 失败时回滚订阅状态

**事件同步机制**：
- 存储最后处理的区块号
- 启动时从上次位置恢复
- 事件数据验证后再写入数据库

## 五、流量管理层架构

### 5.1 模块划分

**订阅准入控制**
- 职责：验证用户订阅状态、允许/拒绝连接
- 数据源：Redis（订阅状态缓存）
- 逻辑：检查订阅是否有效、是否在宽限期

**Fair Use 流量统计**
- 职责：实时计量流量、更新 Redis 计数器
- 数据：Redis（流量计数、设备列表）
- 频率：每 10 秒批量上报

**降速策略执行**
- 职责：流量超额时执行降速
- 触发：流量达到套餐额度
- 策略：由供应商配置降速幅度

### 5.2 数据结构设计

**Redis 数据结构**：
```
subscription:{sub_id}:status → "active" | "quota_exceeded" | "expired"
subscription:{sub_id}:traffic:total → Integer (bytes)
subscription:{sub_id}:quota → Integer (bytes)
subscription:{sub_id}:active_devices → Set<device_id>
```

### 5.3 流量管理流程

**连接准入**：
```
1. 设备请求连接
2. 查询 Redis 订阅状态
3. 状态 == "active" → 允许连接
4. 状态 == "quota_exceeded" 或 "expired" → 拒绝连接
5. 添加设备到活跃列表
```

**流量统计**：
```
1. 设备传输数据
2. 每 10 秒批量上报流量
3. Redis 原子递增计数器
4. 检查是否超出限额
5. 超额 → 更新状态为 "quota_exceeded"
6. 达到 80% → 发送预警通知
```

**降速执行**：
```
1. 流量超额触发
2. 查询供应商降速策略
3. 应用带宽限制
4. 通知客户端降速状态
```

## 六、关键业务流程设计

### 6.1 供应商上线流程

```
1. 供应商发起注册
   - 输入：provider 地址、providerId
   - 准备：100 USDC 上线费

2. 客户端构建 Permit2 签名
   - 授权 Factory 合约收取 100 USDC

3. 调用 Factory.registerProvider()
   - Factory 验证签名
   - 收取 100 USDC 到平台钱包
   - 通过 CREATE2 部署 Minimal Proxy
   - 初始化 Proxy（绑定 provider 地址）
   - 触发 ProviderRegistered 事件

4. 后端监听事件
   - 捕获 ProviderRegistered 事件
   - 创建供应商记录（PostgreSQL）
   - 存储 providerId → escrowAddress 映射
   - 更新供应商状态为 "active"

5. 供应商配置套餐
   - 通过套餐管理系统创建套餐
   - 配置价格、流量额度、节点覆盖
```

### 6.2 用户订阅购买流程

```
1. 用户选择套餐
   - 浏览供应商列表
   - 选择套餐等级

2. 客户端查询合约地址
   - 调用后端 API：GET /providers/{providerId}/escrow
   - 返回：escrowAddress

3. 客户端构建支付
   - 构建 Permit2 签名
   - witness.to = escrowAddress
   - amount = 套餐价格

4. 调用 x402ExactPermit2Proxy.settle()
   - 验证 Permit2 签名
   - USDC 转入 escrowAddress
   - 触发 Deposited 事件

5. 后端监听事件
   - 捕获 Deposited 事件
   - 创建订阅记录（PostgreSQL）
   - 初始化流量计数器（Redis）
   - 设置订阅状态为 "active"
   - 返回订阅成功

6. 用户连接 VPN
   - 流量管理层验证订阅状态
   - 允许连接并开始计量
```

### 6.3 自动续订流程

```
1. 定时任务检查到期订阅
   - 查询到期前 3 天的订阅
   - 筛选开启自动续订的订阅

2. 客户端发起续订
   - 使用 Signer Session 构建签名
   - witness.to = escrowAddress
   - amount = 套餐价格

3. 调用 x402ExactPermit2Proxy.settle()
   - 验证签名并转账
   - 触发 Deposited 事件

4. 后端处理续订
   - 捕获 Deposited 事件
   - 更新订阅有效期（+30 天）
   - 重置流量计数器（Redis）
   - 恢复订阅状态为 "active"

5. 续订失败处理
   - 进入 24 小时宽限期
   - 每 6 小时重试一次
   - 宽限期结束仍失败 → 状态改为 "expired"
```

### 6.4 供应商提现流程

```
1. 供应商发起提现
   - 调用 Escrow.withdraw(amount)
   - 签名验证（msg.sender == provider）

2. 合约执行提现
   - 检查可用余额（totalDeposited - totalWithdrawn）
   - 计算手续费（amount * platformFeeRate / 10000）
   - 转账 (amount - fee) 到 provider
   - 转账 fee 到 platformFeeWallet
   - 更新 totalWithdrawn
   - 触发 Withdrawn 事件

3. 后端记录提现
   - 捕获 Withdrawn 事件
   - 记录提现历史（PostgreSQL）
   - 更新供应商余额统计
```

## 七、数据模型设计

### 7.1 PostgreSQL Schema

**providers 表**
- id, provider_id, provider_address, escrow_address
- onboarding_fee, status, created_at

**subscriptions 表**
- id, user_id, provider_id, tier_id
- status, expires_at, auto_renew
- created_at, updated_at

**tiers 表**
- id, provider_id, name, price_usdc
- monthly_quota_gb, bandwidth_mbps

**transactions 表**
- id, subscription_id, type, amount
- tx_hash, block_number, timestamp

**withdrawals 表**
- id, provider_id, amount, platform_fee
- tx_hash, timestamp

### 7.2 Redis 数据结构

**订阅状态**：
- Key: `subscription:{sub_id}:status`
- Value: "active" | "quota_exceeded" | "expired"

**流量计数**：
- Key: `subscription:{sub_id}:traffic:total`
- Value: Integer (bytes)

**流量限额**：
- Key: `subscription:{sub_id}:quota`
- Value: Integer (bytes)

**活跃设备**：
- Key: `subscription:{sub_id}:active_devices`
- Value: Set<device_id>

### 7.3 智能合约状态

**Factory 状态**：
- platformFeeRate: uint256（基点，1000 = 10%）
- onboardingFee: uint256（100 USDC）
- platformFeeWallet: address
- admin: address
- providers: mapping(bytes32 => address)

**Escrow 状态**：
- provider: address
- totalDeposited: uint256
- totalWithdrawn: uint256
- paused: bool

```
│  ┌──────────────┐  ┌──────────────┐               │

                       ↓

```

### 9.2 网络架构

**外部访问**：
- 客户端 → Base RPC（Alchemy/Infura）
- 客户端 → 后端 API（HTTPS）
- 客户端 → 流量节点（Hysteria2/UDP）

**内部通信**：
- 后端 API ↔ PostgreSQL（内网）
- 后端 API ↔ Redis（内网）
- 事件监听 → Base RPC（外网）
- 流量节点 ↔ Redis（内网）

### 9.3 监控与运维

**监控指标**：
- 智能合约：Gas 消耗、交易成功率、事件延迟
- 后端服务：API 响应时间、订阅成功率、事件同步延迟
- 流量管理：在线设备数、流量吞吐量、连接成功率

**告警规则**：
- 事件同步延迟 > 5 分钟
- API 错误率 > 1%
- Gas 价格突增 > 50%
- Redis/PostgreSQL 不可用

**日志收集**：
- 结构化日志（JSON 格式）
- 集中式存储（ELK/Loki）
- 关键事件追踪（订阅、支付、提现）

## 十、非功能性需求

### 10.1 性能指标

- 订阅购买响应时间：< 3 秒（含链上确认）
