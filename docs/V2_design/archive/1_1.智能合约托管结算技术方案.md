## 一、方案背景

OpenMesh V2 的商业模式从"去中心化直接结算"调整为"平台托管结算模式"。本方案提供完整的智能合约架构设计，实现以下核心功能：

- 平台为每个供应商创建独立的智能合约
- 用户通过 x402 协议将订阅费支付到平台指定的智能合约
- 供应商提现时平台自动扣除 10% 手续费（可调整）
- 供应商上线时缴纳 100 USDC 上线费（可调整）

## 二、区块链网络选择

### 推荐方案：**Base**

**选择理由：**

1. **成本优势**
   - Gas 费最低：~$0.50/笔交易
   - 比以太坊主网便宜 95%（主网 ~$50/笔）
   - [相比其他 L2 降低 60% 成本](https://markaicode.com/base-layer-2-developer-comparison/)

2. **技术支持**
   - [原生 USDC 支持](https://usdc.org/guides/usdc-on-different-blockchains)（Circle 官方发行）
   - x402 协议已部署 Base Sepolia 测试网
   - EVM 完全兼容，开发工具成熟

3. **生态优势**
   - Coinbase 官方支持，生态快速增长
   - 交易确认速度：1-3 秒
   - 开发者友好，文档完善

**备选方案：** Polygon（如未来需要多链支持）

## 三、智能合约架构设计

### 3.1 整体架构

采用 **Factory + Minimal Proxy (EIP-1167)** 模式：

```
ProviderEscrowFactory (工厂合约 - 单例)
├── 创建供应商托管合约（Minimal Proxy）
├── 收取供应商上线费（100 USDC，可调整）
├── 管理平台手续费钱包地址
├── 管理平台管理员权限
└── 触发 ProviderRegistered 事件

ProviderEscrowImplementation (实现合约 - 单例)
├── 接收用户订阅支付（USDC）
├── 跟踪总存款和总提现金额
├── 执行提现并扣除平台手续费（10%，可调整）
├── 支持管理员暂停/恢复功能
└── 触发 Deposited、Withdrawn 事件

ProviderEscrowProxy (代理合约 - 每个供应商一个)
└── 最小代理，委托调用到 Implementation
```

**架构优势：**

- [Gas 效率](https://rareskills.io/post/eip-1167-minimal-proxy-standard-with-initialization-clone-pattern)：部署成本降低 ~10 倍
- 隔离性：每个供应商独立合约实例
- 可维护性：升级 Implementation 即可影响所有代理
- 成熟模式：Gnosis Safe、Uniswap 等项目验证

### 3.2 核心合约详细设计

#### ProviderEscrowFactory

**职责：**
- 供应商注册和合约创建
- 收取上线费
- 管理全局配置（手续费比例、上线费金额）

**关键函数：**

```solidity
// 供应商注册（收取上线费）
function registerProvider(
    address provider,
    bytes32 providerId,
    ISignatureTransfer.PermitTransferFrom calldata permit,
    bytes calldata signature
) external returns (address escrowAddress);

// 管理员更新配置
function updatePlatformFeeRate(uint256 newRate) external onlyAdmin;
function updateOnboardingFee(uint256 newFee) external onlyAdmin;
function updatePlatformFeeWallet(address newWallet) external onlyAdmin;

// 管理员暂停/恢复供应商
function pauseProvider(address escrowAddress) external onlyAdmin;
function unpauseProvider(address escrowAddress) external onlyAdmin;
```

**关键事件：**

```solidity
event ProviderRegistered(
    bytes32 indexed providerId,
    address indexed provider,
    address escrowAddress,
    uint256 onboardingFee
);

event PlatformFeeRateUpdated(uint256 oldRate, uint256 newRate);
event OnboardingFeeUpdated(uint256 oldFee, uint256 newFee);
event ProviderPaused(address indexed escrowAddress);
event ProviderUnpaused(address indexed escrowAddress);
```

#### ProviderEscrowImplementation

**职责：**
- 接收订阅支付
- 处理供应商提现
- 自动扣除平台手续费

**关键函数：**

```solidity
// 初始化（由 Factory 调用）
function initialize(
    address _provider,
    address _platformFeeWallet,
    uint256 _platformFeeRate,
    address _factory
) external;

// 接收订阅支付（由 x402 合约调用）
function deposit(
    address user,
    uint256 amount,
    bytes32 subscriptionId
) external;

// 供应商提现
function withdraw(uint256 amount) external onlyProvider whenNotPaused;

// 查询可用余额
function availableBalance() external view returns (uint256);
```

**安全模式：Checks-Effects-Interactions**

```solidity
function withdraw(uint256 amount) external onlyProvider nonReentrant whenNotPaused {
    // CHECKS - 验证
    require(amount > 0, "Invalid amount");
    uint256 available = totalDeposited - totalWithdrawn;
    require(amount <= available, "Insufficient balance");

    // EFFECTS - 状态更新（在外部调用之前）
    uint256 platformFee = (amount * platformFeeRate) / 10000;  // 基点计算
    uint256 providerAmount = amount - platformFee;
    totalWithdrawn += amount;

    // INTERACTIONS - 外部调用（最后执行）
    IERC20(USDC).transfer(platformFeeWallet, platformFee);
    IERC20(USDC).transfer(provider, providerAmount);

    emit Withdrawn(provider, providerAmount, platformFee);
}
```

**关键事件：**

```solidity
event Deposited(
    address indexed provider,
    address indexed user,
    uint256 amount,
    bytes32 indexed subscriptionId,
    uint256 timestamp
);

event Withdrawn(
    address indexed provider,
    uint256 providerAmount,
    uint256 platformFee,
    uint256 timestamp
);
```

### 3.3 与 x402 协议集成

**支付流程：**

```
1. 用户选择订阅套餐
   ↓
2. 客户端生成 Permit2 签名
   witness.to = 供应商托管合约地址
   ↓
3. 调用 x402ExactPermit2Proxy.settle()
   ↓
4. USDC 直接转入供应商托管合约
   ↓
5. 托管合约触发 Deposited 事件
   ↓
6. 链下服务监听事件并激活订阅
```

**关键优势：**
- 复用现有 x402 基础设施
- 无需重新实现支付逻辑
- 用户体验一致（Permit2 签名）
- 支持 gasless 支付

**集成代码示例（Go）：**

```go
// 构建 witness 指向供应商托管合约
witness := &evmclient.Witness{
    To:         providerEscrowAddress,  // 供应商托管合约地址
    ValidAfter: uint64(time.Now().Unix()),
}

// 调用 x402 支付
result, err := lib.MakeX402Payment(
    x402ProxyURL,
    privateKeyHex,
    witness,
)
```

## 四、供应商上线流程

### 4.1 注册流程

```
1. 供应商准备
   - 拥有 EVM 钱包地址
   - 钱包中有 ≥100 USDC（上线费）

2. 调用 Factory.registerProvider()
   - 提供 providerId（唯一标识）
   - 提供 provider 地址（提现接收地址）
   - 提交 Permit2 签名（支付上线费）

3. Factory 执行
   - 验证 Permit2 签名
   - 收取 100 USDC 上线费
   - 通过 CREATE2 部署 Minimal Proxy
   - 初始化代理合约
   - 触发 ProviderRegistered 事件

4. 链下服务处理
   - 监听 ProviderRegistered 事件
   - 更新数据库（providerId → escrowAddress 映射）
   - 供应商可以开始配置套餐
```

## 五、链上链下同步方案

### 5.1 事件监听架构

```
Base 区块链
    ↓ 事件流
Event Listener Service (Go)
    ├── 监听 ProviderRegistered
    ├── 监听 Deposited
    ├── 监听 Withdrawn
    ├── 监听 ProviderPaused/Unpaused
    ↓ 解析验证
PostgreSQL 数据库
    ├── providers 表
    ├── subscriptions 表
    ├── transactions 表
    ↓ 状态更新
Subscription Management System
```

### 5.2 关键事件

**ProviderRegistered**
```solidity
event ProviderRegistered(
    bytes32 indexed providerId,
    address indexed provider,
    address escrowAddress,
    uint256 onboardingFee
);
```
**处理逻辑：**
- 在数据库中创建供应商记录
- 存储 providerId → escrowAddress 映射
- 更新供应商状态为 "active"

**Deposited**
```solidity
event Deposited(
    address indexed provider,
    address indexed user,
    uint256 amount,
    bytes32 indexed subscriptionId,
    uint256 timestamp
);
```
**处理逻辑：**
- 激活用户订阅
- 记录交易历史
- 更新订阅有效期
- 重置流量使用量

**Withdrawn**
```solidity
event Withdrawn(
    address indexed provider,
    uint256 providerAmount,
    uint256 platformFee,
    uint256 timestamp
);
```
**处理逻辑：**
- 记录提现历史
- 更新供应商余额
- 记录平台手续费收入

### 5.3 实现要点

**Go 事件监听服务：**

```go
// 使用 go-ethereum 订阅事件
query := ethereum.FilterQuery{
    Addresses: []common.Address{factoryAddress},
    Topics: [][]common.Hash{
        {depositedEventSignature},
    },
}

logs := make(chan types.Log)
sub, err := client.SubscribeFilterLogs(context.Background(), query, logs)

// 处理事件
for {
    select {
    case log := <-logs:
        handleDepositedEvent(log)
    case err := <-sub.Err():
        // 重连逻辑
    }
}
```

**容错机制：**
- 存储最后处理的区块号
- 启动时从上次位置恢复
- 指数退避重试（RPC 失败）
- 事件数据验证（防止脏数据）

**乐观确认：**
- 收到支付签名立即激活订阅
- 异步等待链上确认（1-3 秒）
- 处理罕见失败情况（余额不足、签名无效）

## 六、安全性设计

### 6.1 智能合约安全

**安全模式：**

1. **ReentrancyGuard**（OpenZeppelin）
   - 防止重入攻击
   - 所有涉及转账的函数使用 `nonReentrant` 修饰符

2. **Checks-Effects-Interactions 模式**
   - 先验证条件
   - 再更新状态
   - 最后执行外部调用

3. **Pausable 模式**（OpenZeppelin）
   - 管理员可暂停单个供应商合约
   - 紧急情况下停止提现功能
   - 不影响其他供应商

4. **访问控制**
   - `onlyProvider`：只有供应商可提现
   - `onlyAdmin`：只有管理员可调整配置
   - `onlyFactory`：只有工厂可初始化

**安全审计策略：**

根据产品决策，采用 **内部审计 + 测试网验证** 方案：

1. **内部安全审查**
   - 代码审查（多人）
   - 静态分析工具（Slither、Mythril）
   - 手动漏洞检查（OWASP Top 10）

2. **测试覆盖**
   - 单元测试覆盖率 >90%
   - 集成测试（与 x402 集成）
   - 模糊测试（Foundry Fuzz）
   - 不变量测试（关键属性验证）

3. **测试网验证**
   - Base Sepolia 部署 2+ 周
   - 模拟真实场景测试
   - 监控异常行为
   - 限制初期供应商数量（≤5）

### 6.2 运营安全

**多签钱包：**
- 平台手续费钱包使用 Gnosis Safe
- 至少 2/3 多签
- 管理员操作需要多人确认

**不可升级设计：**
- 合约部署后不可升级
- 发现严重漏洞需重新部署
- 降低复杂性和攻击面

**紧急响应：**
- 管理员可暂停单个供应商合约
- 暂停权限归项目管理员账号
- 明确的事件响应流程
- 24/7 监控和告警

**监控告警：**
- 异常大额提现告警
- Gas 价格突增告警
- 事件同步延迟告警
- 合约余额异常告警

## 十、已确认的产品决策

基于与项目负责人的讨论，以下决策已确认：

### 10.1 区块链网络
✅ **Base**（确认）
- 先单链部署
- 后续根据需求评估多链

### 10.2 安全审计
✅ **内部审计 + 测试网验证**
- 不进行外部审计（节省成本和时间）
- 依赖全面测试和长期测试网验证

### 10.3 提现策略
✅ **无限制，随时提现**
- 供应商可随时提现任意金额
- 无最低金额限制
- 无频率限制

### 10.4 退款政策
✅ **不支持退款**
- 区块链交易不可逆
- 简化合约逻辑
- 用户可取消订阅（当期服务继续）

### 10.5 KYC 要求
✅ **不需要 KYC**
- 供应商上线无需身份验证
- 降低准入门槛

### 10.6 争议处理
✅ **暂不考虑争议仲裁**
- 用户不满意可取消订阅
- 依赖供应商声誉机制

- 平台手续费：10%（默认，管理员可调整）

- 写入智能合约，通过管理员函数修改

- 暂停/恢复供应商权限归项目管理员账号
- 使用多签钱包管理
- 明确的权限范围和流程
