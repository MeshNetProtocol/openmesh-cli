# OpenMesh V2 技术方案设计

## 背景

OpenMesh V2 是流量行业的 OpenRouter，提供统一的 VPN 流量供应商接入平台。本文档设计三个核心技术方案，并确认剩余的开放问题。

**设计目标**:
1. x402 协议与订阅场景的适配方案
2. 设备数管理的并发一致性方案
3. 流量管理系统的部署架构
4. 确认开放问题 Q2/Q6/Q7

---

## 一、x402 协议与订阅场景的适配方案

### 1.1 现状分析

**x402 协议当前能力**:
- 支持单次支付验证（V1 和 V2 版本）
- 基于 EVM/SVM 的签名验证机制
- 支持 PaymentRequirements 和 PaymentPayload 结构
- 具备扩展字段 `Extra` 和 `Extensions`

**订阅场景的特殊需求**:
- 周期性支付（30天自动续订）
- 订阅状态管理（有效/宽限期/过期/已取消）
- 订阅等级信息传递
- 续订意图表达（首次购买 vs 续订）

### 1.2 技术方案

#### 方案：基于 x402 V2 扩展字段实现订阅协议

**核心设计思路**: 不修改 x402 核心协议，通过 `Extensions` 字段传递订阅特定信息。

**1.2.1 订阅支付请求扩展**

在 `PaymentRequired` 响应中，通过 `Extensions` 字段传递订阅信息：

```json
{
  "x402Version": 2,
  "accepts": [{
    "scheme": "evm-exact",
    "network": "eip155:1",
    "asset": "eip155:1/erc20:0xA0b86991c6218b36c1d19D4a2e9Eb0cE3606eB48",
    "amount": "6000000",
    "payTo": "0x...",
    "maxTimeoutSeconds": 300
  }],
  "extensions": {
    "openmesh.subscription": {
      "version": "1.0",
      "subscriptionId": "sub_xxx",
      "tierId": "tier_standard",
      "tierName": "标准版",
      "billingCycle": 30,
      "bandwidthMbps": 200,
      "monthlyQuotaGB": 300,
      "isRenewal": false
    }
  }
}
```

**1.2.2 订阅支付响应扩展**

客户端在 `PaymentPayload` 中通过 `Extensions` 字段传递订阅意图：

```json
{
  "x402Version": 2,
  "payload": {
    "signature": "0x...",
    "message": {...}
  },
  "accepted": {...},
  "extensions": {
    "openmesh.subscription": {
      "action": "subscribe",
      "userId": "user_xxx",
      "deviceId": "device_xxx",
      "autoRenew": true,
      "signerSessionId": "session_xxx"
    }
  }
}
```

**1.2.3 续订场景处理**

续订时，`isRenewal` 标记为 `true`，并携带原订阅 ID：

```json
{
  "extensions": {
    "openmesh.subscription": {
      "action": "renew",
      "subscriptionId": "sub_xxx",
      "userId": "user_xxx",
      "autoRenew": true
    }
  }
}
```

**1.2.4 订阅验证流程**

```
客户端                    套餐管理系统                x402 验证
  |                           |                         |
  |--请求订阅---------------->|                         |
  |                           |--返回 402 + 订阅扩展--->|
  |<-PaymentRequired----------|                         |
  |                           |                         |
  |--构建支付 + 订阅扩展----->|                         |
  |                           |--验证签名-------------->|
  |                           |<-验证通过---------------|
  |                           |--创建订阅记录           |
  |                           |--激活流量权限           |
  |<-订阅成功-----------------|                         |
```

### 1.3 协议规范定义（Q2 答案）

**Q2: x402 订阅协议接口规范由谁定义、在哪里发布、如何版本管理？**

**方案**:

1. **定义方**: OpenMesh 架构团队负责定义订阅扩展规范
2. **发布位置**:
   - 主规范文档: `docs/protocols/x402-subscription-extension-v1.md`
   - OpenAPI 规范: `docs/api/subscription-api-v1.yaml`
   - 代码实现: `x402/go/extensions/openmesh/subscription.go`
3. **版本管理**:
   - 扩展版本独立于 x402 核心版本
   - 使用语义化版本 (v1.0, v1.1, v2.0)
   - 在 `extensions.openmesh.subscription.version` 字段中声明
   - 向后兼容原则：新版本必须能处理旧版本客户端
4. **发布流程**:
   - 规范文档提交到 GitHub 仓库
   - 通过 PR review 流程审核
   - 合并后自动发布到文档站点
   - 供应商通过 GitHub Releases 订阅更新通知

### 1.4 实现要点

**服务端实现**:
```go
// x402/go/extensions/openmesh/subscription.go
package openmesh

type SubscriptionExtension struct {
    Version        string `json:"version"`
    SubscriptionID string `json:"subscriptionId,omitempty"`
    TierID         string `json:"tierId"`
    TierName       string `json:"tierName"`
    BillingCycle   int    `json:"billingCycle"`
    BandwidthMbps  int    `json:"bandwidthMbps"`
    MonthlyQuotaGB int    `json:"monthlyQuotaGB"`

func InjectSubscriptionExtension(pr *types.PaymentRequired, sub *SubscriptionExtension) {
    if pr.Extensions == nil {
        pr.Extensions = make(map[string]interface{})
    }
    pr.Extensions["openmesh.subscription"] = sub
}
```

## 二、流量限制与账号管理方案

### 2.1 需求调整说明

**原需求**: 设备数限制（如 3 台设备同时在线）
**新需求**: 流量总量限制（不限设备数,只限制总流量）

**设计理念**:
- 更贴合大模型使用方式
- 用户可以在任意多台设备上使用
- 当总流量超出套餐限制时,关闭该账号下所有设备的访问

### 2.2 技术方案

#### 方案：基于流量计量 + 实时阈值检查

**2.2.1 架构设计**

```
┌─────────────┐
│  Device 1   │───┐
└─────────────┘   │
                  │    ┌──────────────────┐
┌─────────────┐   ├───>│  流量管理系统    │
│  Device 2   │───┤    │  (Hysteria2)     │
└─────────────┘   │    │  - 流量计量      │
                  │    │  - 阈值检查      │
┌─────────────┐   │    └──────────────────┘
│  Device N   │───┘            │
└─────────────┘                ▼
                        ┌─────────────┐
                        │   Redis     │
                        │  - 流量计数  │

                        └─────────────┘
```

**2.2.2 数据结构设计**

**Redis 数据结构**:

```
# 流量计数器（原子操作,单位: bytes）
subscription:{sub_id}:traffic:upload -> Integer
subscription:{sub_id}:traffic:download -> Integer
subscription:{sub_id}:traffic:total -> Integer

# 账号状态
subscription:{sub_id}:status -> String {
    "active" | "quota_exceeded" | "expired"
}

# 流量限额（从套餐配置读取,缓存到 Redis）
subscription:{sub_id}:quota -> Integer (bytes)

# 设备连接列表（用于批量断开）
subscription:{sub_id}:active_devices -> Set {
    "device_id_1",
    "device_id_2",
    ...
}
```

**2.2.3 流量计量与检查流程**

```go
// 流量上报（每个数据包传输后调用）
func ReportTraffic(subID, deviceID string, uploadBytes, downloadBytes int64) error {
    // 1. 原子递增流量计数
    uploadKey := fmt.Sprintf("subscription:%s:traffic:upload", subID)
    downloadKey := fmt.Sprintf("subscription:%s:traffic:download", subID)
    totalKey := fmt.Sprintf("subscription:%s:traffic:total", subID)

    redis.IncrBy(uploadKey, uploadBytes)
    redis.IncrBy(downloadKey, downloadBytes)
    newTotal := redis.IncrBy(totalKey, uploadBytes+downloadBytes)

    // 2. 检查是否超出限额
    quotaKey := fmt.Sprintf("subscription:%s:quota", subID)
    quota := redis.Get(quotaKey)

    if newTotal > quota {
        // 超出限额,触发账号封禁
        return HandleQuotaExceeded(subID)
    }

    // 3. 接近限额时预警（80%）
    if float64(newTotal) > float64(quota)*0.8 {
        go SendQuotaWarning(subID, newTotal, quota)
    }

    return nil
}

// 超限处理
func HandleQuotaExceeded(subID string) error {
    // 1. 更新账号状态
    statusKey := fmt.Sprintf("subscription:%s:status", subID)
    redis.Set(statusKey, "quota_exceeded", 0)

    // 2. 获取所有活跃设备
    devicesKey := fmt.Sprintf("subscription:%s:active_devices", subID)
    devices := redis.SMembers(devicesKey)

    // 3. 断开所有设备连接
    for _, deviceID := range devices {
        hysteria2.CloseConnection(deviceID)
    }

    // 4. 清空设备列表
    redis.Del(devicesKey)

    // 5. 记录到数据库
    db.LogQuotaExceeded(subID, time.Now())

    return nil
}
```

**2.2.4 设备连接准入控制**

```go
func HandleDeviceConnect(subID, deviceID string) error {
    // 1. 检查订阅状态
    statusKey := fmt.Sprintf("subscription:%s:status", subID)
    status := redis.Get(statusKey)

    if status == "quota_exceeded" {
        return errors.New("流量额度已用尽,请升级套餐或等待下个计费周期")
    }

    if status == "expired" {
        return errors.New("订阅已过期,请续订")
    }

    // 2. 检查订阅是否存在
    sub := db.GetSubscription(subID)
    if sub == nil {
        return errors.New("订阅不存在")
    }

    // 3. 添加设备到活跃列表
    devicesKey := fmt.Sprintf("subscription:%s:active_devices", subID)
    redis.SAdd(devicesKey, deviceID)

    return nil
}

// 设备断开连接
func HandleDeviceDisconnect(subID, deviceID string) {
    devicesKey := fmt.Sprintf("subscription:%s:active_devices", subID)
    redis.SRem(devicesKey, deviceID)
}
```

**2.2.5 流量统计与重置**

```go
// 获取流量使用情况
func GetTrafficUsage(subID string) (*TrafficUsage, error) {
    uploadKey := fmt.Sprintf("subscription:%s:traffic:upload", subID)
    downloadKey := fmt.Sprintf("subscription:%s:traffic:download", subID)
    totalKey := fmt.Sprintf("subscription:%s:traffic:total", subID)
    quotaKey := fmt.Sprintf("subscription:%s:quota", subID)

    upload := redis.Get(uploadKey)
    download := redis.Get(downloadKey)
    total := redis.Get(totalKey)
    quota := redis.Get(quotaKey)

    return &TrafficUsage{
        Upload:     upload,
        Download:   download,
        Total:      total,
        Quota:      quota,
        Percentage: float64(total) / float64(quota) * 100,
    }, nil
}

// 月度流量重置（定时任务,每月1号执行）
func ResetMonthlyTraffic() {
    subs := db.GetActiveSubscriptions()
    for _, sub := range subs {
        // 1. 归档上月流量数据到 PostgreSQL
        usage, _ := GetTrafficUsage(sub.ID)
        db.ArchiveTrafficUsage(sub.ID, usage, time.Now().AddDate(0, -1, 0))

        // 2. 重置 Redis 计数器
        uploadKey := fmt.Sprintf("subscription:%s:traffic:upload", sub.ID)
        downloadKey := fmt.Sprintf("subscription:%s:traffic:download", sub.ID)
        totalKey := fmt.Sprintf("subscription:%s:traffic:total", sub.ID)

        redis.Set(uploadKey, 0, 0)
        redis.Set(downloadKey, 0, 0)
        redis.Set(totalKey, 0, 0)

        // 3. 恢复账号状态（如果因流量超限被封禁）
        statusKey := fmt.Sprintf("subscription:%s:status", sub.ID)
        if redis.Get(statusKey) == "quota_exceeded" {
            redis.Set(statusKey, "active", 0)
        }
    }
}
```

### 2.3 性能优化

**批量流量上报**:
- 客户端每 10 秒批量上报一次流量,而非每个数据包都上报
- 减少 Redis 写入频率,提升性能

**流量检查频率**:
- 每次上报时检查是否超限
- 接近限额（80%）时增加检查频率

**缓存策略**:
- 套餐限额缓存到 Redis,避免频繁查询数据库
- 流量数据每 5 分钟同步到 PostgreSQL

### 2.4 容错与降级

**Redis 不可用时的降级策略**:
1. 切换到数据库计量（性能下降但保证功能）
2. 临时允许流量超限,事后补偿
3. 记录异常日志,人工介入

**数据一致性保证**:
- Redis 数据定期同步到 PostgreSQL
- PostgreSQL 作为权威数据源
- 系统重启时从 PostgreSQL 恢复 Redis 状态

### Q2: x402 订阅协议接口规范由谁定义、在哪里发布、如何版本管理？

**答案**: 见第一章节 1.3

### Q6: 供应商链上注册资料审核机制是否需要，还是任何人都可以注册？

**建议方案**: 采用"无需审核 + 社区治理"模式

**理由**:
1. **去中心化原则**: OpenMesh 定位为开放平台，不应成为中心化审核方
2. **降低准入门槛**: 鼓励更多供应商接入，促进生态繁荣
3. **市场自我调节**: 用户评价和口碑会自然淘汰劣质供应商

**具体方案**:
- 任何人都可以在链上注册成为供应商（支付少量 gas 费）
- 注册时需质押一定数量的代币（如 100 USDC）作为保证金
- 用户可以对供应商进行评分和举报
- 社区 DAO 可以投票冻结恶意供应商的保证金
- 客户端默认按评分排序供应商列表

**风险控制**:
- 客户端显示供应商注册时间、服务用户数、评分等信誉指标
- 新供应商标记"新供应商"徽章，提醒用户谨慎选择
- 平台提供供应商黑名单机制（由 DAO 治理）

### Q7: 自动续订失败通知是否需要 App 推送以外的渠道（如邮件），涉及用户信息收集

**确认方案**: 用户主动查询订阅状态，App 仅提示未订阅

**架构理念**:
1. **OpenMesh App 是集成端**: 不做业务处理，只负责展示和路由
2. **业务由供应商处理**: 订阅管理、支付、续订等由供应商系统负责
3. **用户与供应商直接交互**: OpenMesh 不介入用户和供应商之间的业务关系

**具体方案**:

**App 端行为**:
- 当用户尝试连接 VPN 但连接失败时，App 检查订阅状态
- 如果订阅不存在或已过期，显示提示："尚未订阅该供应商，请先购买订阅"
- 不主动推送续订提醒或失败通知
- 用户可以在 App 内查看所有供应商的订阅状态（通过查询供应商 API）

**供应商管理系统行为**:
- 提供订阅状态查询 API（供 App 调用）
- 展示用户的订阅详情：到期时间、流量使用、订阅状态
- 用户主动访问供应商管理界面时，可以看到续订失败原因
- 供应商可以选择是否实现自己的通知机制（如邮件、Telegram）

**用户体验流程**:
1. 用户尝试连接 VPN

4. 用户点击"查看详情"，跳转到供应商管理界面
5. 用户在供应商界面充值并手动续订
