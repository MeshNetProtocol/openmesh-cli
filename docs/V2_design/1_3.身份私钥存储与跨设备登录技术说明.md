## 1. 问题定义与场景澄清

### 1.1 核心问题

**钱包私钥在 OpenMesh V2 中的作用是什么,以及如何存储和跨设备迁移。**

### 1.2 场景澄清

**重要**: 本项目中的钱包私钥**不是**用于资产存储和支付,而是用于**身份标识**。

```
身份钱包 (本文档讨论的对象):
- 钱包地址 = 用户ID (类似用户名)
- 钱包私钥 = 身份凭证 (类似密码/API Key)
- 用途: 标识用户身份,签名订阅请求
- 特点: 不存储资产,不用于支付

支付钱包 (不在本文档范围):
- 用户可以用任何其他钱包支付订阅费用
- 与身份钱包分离
```

**类比**:
```
传统方式:
- 用户名: user@email.com
- 密码: ********
- 跨设备: 输入邮箱+密码

本项目方式:
- 用户ID: 0x1234...abcd (身份钱包地址)
- 凭证: 0xabcd...1234 (身份钱包私钥)
- 跨设备: 输入私钥 (类似 API Key)
```

### 1.3 为什么选择区块链地址作为用户ID

- **无需中心化存储**: 不需要维护用户名/邮箱数据库
- **隐私保护**: 不收集手机号、邮箱等个人信息
- **去中心化**: 用户完全控制自己的身份
- **跨平台一致**: 同一身份可在任何设备使用

## 2. 身份私钥的应用场景

### 2.1 本项目中的核心场景

1. **用户身份标识**
   - 钱包地址作为用户的唯一标识符
   - 系统通过地址识别用户,而不是邮箱或手机号

2. **订阅请求签名**
   - 用户订阅服务时,使用身份私钥签名请求
   - 证明"我是 0xAAAA,我要订阅服务"
   - 支付由其他钱包完成

3. **跨设备登录**
   - 用户在新设备上输入身份私钥
   - 系统派生出地址,完成登录
   - 类似在新设备上输入 API Key

### 2.2 不适用的场景

身份私钥**不用于**:
- 存储资产(身份钱包不持有资金)
- 支付订阅费用(由其他钱包支付)
- 加密用户数据(数据加密应使用专门的加密密钥)

## 3. 技术实现方案

### 3.1 存储方案: 类 API Key 模式

**核心原则**: 身份私钥类似 API Key,存储在本地安全存储中,不需要额外的设备密钥对加密。

**实现**:
```
1. 身份私钥明文存储在本地安全存储中
2. 受操作系统级别的访问控制保护
3. 应用解锁后可直接读取(不需要每次生物识别)
4. 跨设备迁移: 直接复制私钥字符串
```

### 3.2 平台实现方案

#### 3.2.1 iOS 平台

**使用技术**: Keychain Services

**实现**:
```swift
// 存储身份私钥
let query: [String: Any] = [
    kSecClass as String: kSecClassGenericPassword,
    kSecAttrAccount as String: "identity_private_key",
    kSecValueData as String: privateKeyData,
    kSecAttrAccessible as String: kSecAttrAccessibleWhenUnlockedThisDeviceOnly
]
SecItemAdd(query as CFDictionary, nil)

// 读取身份私钥
let query: [String: Any] = [
    kSecClass as String: kSecClassGenericPassword,
    kSecAttrAccount as String: "identity_private_key",
    kSecReturnData as String: true
]
var result: AnyObject?
SecItemCopyMatching(query as CFDictionary, &result)
```

**安全特性**:
- `kSecAttrAccessibleWhenUnlockedThisDeviceOnly`: 设备解锁后可访问,不会同步到 iCloud
- 其他应用无法访问
- 设备锁定时无法读取

#### 3.2.2 Android 平台

**使用技术**: EncryptedSharedPreferences

**实现**:
```kotlin
// 创建加密的 SharedPreferences
val masterKey = MasterKey.Builder(context)
    .setKeyScheme(MasterKey.KeyScheme.AES256_GCM)
    .build()

val sharedPreferences = EncryptedSharedPreferences.create(
    context,
    "identity_prefs",
    masterKey,
    EncryptedSharedPreferences.PrefKeyEncryptionScheme.AES256_SIV,
    EncryptedSharedPreferences.PrefValueEncryptionScheme.AES256_GCM
)

// 存储身份私钥
sharedPreferences.edit()
    .putString("identity_private_key", privateKey)
    .apply()

// 读取身份私钥
val privateKey = sharedPreferences.getString("identity_private_key", null)
```

**安全特性**:
- 自动使用 Android Keystore 加密数据
- 其他应用无法访问
- 设备锁定时无法读取

#### 3.2.3 macOS 平台

**使用技术**: Keychain Services (与 iOS 相同)

**实现**: 与 iOS 相同

**安全特性**:
- 与 iOS 相同的安全级别
- 不会通过 iCloud Keychain 同步(因为使用 `ThisDeviceOnly`)

#### 3.2.4 Windows 平台

**使用技术**: Windows Credential Manager 或 DPAPI

**方案 1: Credential Manager**
```csharp
using Windows.Security.Credentials;

// 存储身份私钥
var vault = new PasswordVault();
vault.Add(new PasswordCredential(
    "OpenMeshIdentity",
    "identity_private_key",
    privateKey
));

// 读取身份私钥
var credential = vault.Retrieve("OpenMeshIdentity", "identity_private_key");
var privateKey = credential.Password;
```

**方案 2: DPAPI (Data Protection API)**
```csharp
using System.Security.Cryptography;

// 存储身份私钥
byte[] privateKeyBytes = Encoding.UTF8.GetBytes(privateKey);
byte[] encryptedBytes = ProtectedData.Protect(
    privateKeyBytes,
    null,
    DataProtectionScope.CurrentUser
);
File.WriteAllBytes(keyFilePath, encryptedBytes);

// 读取身份私钥
byte[] encryptedBytes = File.ReadAllBytes(keyFilePath);
byte[] privateKeyBytes = ProtectedData.Unprotect(
    encryptedBytes,
    null,
    DataProtectionScope.CurrentUser
);
string privateKey = Encoding.UTF8.GetString(privateKeyBytes);
```

**安全特性**:
- 绑定到当前用户账户
- 其他用户无法访问
- 用户注销时无法读取

## 4. 跨设备登录流程

### 4.1 导出身份私钥

**界面设计**:
```
账号设置
┌─────────────────────────────────────┐
│ 用户ID (身份地址)                    │
│ 0x1234...abcd                       │
│ [复制地址]                          │
├─────────────────────────────────────┤
│ API Key (身份凭证)                   │
│ [显示 API Key] [复制] [生成二维码]   │
│                                     │
│ ⚠️ 重要提示:                        │
│ 这是您的身份凭证,类似其他服务的      │
│ API Key。任何获得此凭证的人都可以:   │
│ • 以您的身份订阅服务                │
│ • 查看您的订阅记录                  │
│ • 消耗您已购买的流量额度            │
│                                     │
│ 建议:                               │
│ • 不要分享给任何人                  │
│ • 妥善保管备份                      │
│ • 仅在安全环境下导出                │
└─────────────────────────────────────┘
```

**导出流程**:
```
1. 用户点击"显示 API Key"
2. (可选) 要求生物识别或密码验证
3. 显示 API Key 明文: 0xabcd...1234
4. 提供"复制"和"生成二维码"按钮
5. 显示红色警告提示
```

### 4.2 导入身份私钥

**界面设计**:
```
登录/导入账号
┌─────────────────────────────────────┐
│ 选择登录方式:                        │
│                                     │
│ ○ 创建新账号                        │
│   系统将自动生成新的身份钱包         │
│                                     │
│ ● 导入已有账号                      │
│   输入您的 API Key                   │
│                                     │
│ ┌─────────────────────────────────┐ │
│ │ 0x                              │ │
│ │                                 │ │
│ └─────────────────────────────────┘ │
│                                     │
│ [扫描二维码] [粘贴]                  │
│                                     │
│ [导入]                              │
└─────────────────────────────────────┘
```

**导入流程**:
```
1. 用户选择"导入已有账号"
2. 用户粘贴 API Key 字符串或扫描二维码
3. 系统验证 API Key 格式 (0x 开头, 64 个十六进制字符)
4. 系统派生出钱包地址
5. 显示"登录为 0x1234...abcd"
6. 将 API Key 存储到本地安全存储
7. 完成登录
```

### 4.3 二维码格式

**编码格式**:
```json
{
  "type": "openmesh_identity",
  "version": "1.0",
  "private_key": "0xabcd...1234"
}
```

**生成**:
```
1. 将 JSON 字符串编码为二维码
2. 建议使用纠错级别 M (15% 容错)
3. 显示在界面上供扫描
```

**扫描**:
```
1. 扫描二维码
2. 解析 JSON
3. 验证 type 和 version
4. 提取 private_key
5. 自动填充到导入界面
```

## 5. 安全性分析

### 5.1 风险模型

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 设备丢失 | 他人可能获取 API Key | 设备解锁保护 + 本地安全存储 |
| API Key 泄露 | 他人可以冒充用户身份,消耗流量 | 取消订阅功能 + 用户教育 |
| 恶意应用 | 尝试读取 API Key | 操作系统访问控制 |

### 5.2 与传统钱包的风险对比

```
传统钱包 (存储资产):
- 私钥泄露 = 资金损失 (不可逆)
- 风险等级: 极高
- 保护措施: 硬件钱包、多重签名、冷存储

本项目 (身份标识):
- API Key 泄露 = 流量被消耗 (可控损失)
- 风险等级: 中等
- 保护措施: 本地安全存储 + 取消订阅
```

### 5.3 核心保护措施

#### 5.3.1 本地安全存储

**实现**:
- 使用操作系统提供的安全存储机制
- 其他应用无法访问
- 设备锁定时无法读取

**平台**:
- iOS: Keychain
- Android: EncryptedSharedPreferences
- macOS: Keychain
- Windows: Credential Manager / DPAPI

#### 5.3.2 取消订阅功能 (API Key 泄露时的补救措施)

**类比 Twitter Blue 订阅模式**:
```
Twitter Blue 订阅:
- 用户通过 MetaMask 钱包为 Twitter 账号支付会员费
- 用户可以随时取消订阅
- 取消后,当前周期结束时服务停止

本项目订阅模式:
- 用户通过任何钱包为身份 ID (0xAAAA) 支付订阅费
- 用户可以随时取消订阅
- 取消后,当前周期结束时服务停止
```

**如果 API Key 泄露**:
1. 用户发现异常(流量消耗异常、订阅记录异常)
2. 用户立即取消所有订阅
3. 当前周期结束后,服务自动停止
4. 攻击者无法继续消耗流量
5. 用户生成新 API Key,重新订阅

**关键点**:
- 支付钱包与身份 API Key 分离
- 任何钱包都可以为某个身份 ID 支付订阅费
- 用户可以通过取消订阅来止损
- 不需要复杂的 API Key 轮换机制

#### 5.3.3 用户教育

1. **创建账号时**
   ```
   "您的 API Key 已生成。

   重要提示:
   • API Key 是您身份的唯一凭证
   • 请立即备份并妥善保管
   • 设备丢失后需要 API Key 才能恢复账号
   • 不要分享给任何人"
   ```

2. **导出 API Key 时**
   ```
   "⚠️ 安全警告

   您即将导出 API Key。任何获得 API Key 的人都可以:
   • 以您的身份订阅服务
   • 消耗您已购买的流量额度
   • 查看您的订阅记录

   如果发现 API Key 泄露:
   • 立即取消所有订阅
   • 生成新账号并重新订阅

   [我已了解风险,继续导出]"
   ```

3. **账号设置界面**
   ```
   "💡 安全建议

   • 定期检查订阅记录,发现异常及时取消订阅
   • 不要在公共设备上登录
   • 如果怀疑 API Key 泄露,请立即取消订阅并生成新账号"
   ```

## 7. 需求文档修正建议

基于本技术分析,建议修正以下需求:

### 7.1 FR-002 修正

**原需求**:
> 系统应在用户选择创建账号时自动生成新的 EVM 钱包,同时生成或获取当前设备的设备密钥对;钱包私钥必须由设备密钥对保护后再存储到设备本地安全存储中。

**修正为**:
> 系统应在用户选择创建账号时自动生成新的 EVM 钱包作为身份标识;钱包私钥存储到设备本地安全存储中(iOS Keychain / Android EncryptedSharedPreferences / macOS Keychain / Windows Credential Manager),受操作系统级别的访问控制保护。

**修正理由**:
- 身份私钥不涉及资产,不需要设备密钥对加密
- 本地安全存储已提供足够的保护
- 简化实现,提升用户体验

### 7.2 FR-004 删除

**原需求**:
> 系统应在 iOS 使用 Keychain / Secure Enclave 能力、在 Android 使用 Keystore / AndroidKeyStore 能力、在 Windows 使用 CNG / Windows Hello 相关能力生成并持有设备密钥对。

**删除理由**:
- 身份私钥不需要设备密钥对加密
- 此需求不适用于当前场景

### 7.3 NFR-SEC-001 修正

**原需求**:
> 钱包私钥必须先经设备相关密钥保护后再存储于设备本地安全存储中,不得以明文形式上传服务器或静默落盘。

**修正为**:
> 身份钱包私钥存储于设备本地安全存储中,受操作系统级别的访问控制保护,不得上传服务器。

**修正理由**:
- 明确"身份钱包"与"支付钱包"的区别
- 去掉"设备相关密钥保护"的要求
- 保留"不上传服务器"的核心安全要求

### 7.4 NFR-SEC-002 修正

**原需求**:
> 移动端应支持 Face ID、Touch ID 或同等级生物识别机制,Windows 端应支持 Windows Hello 或同等级本机访问控制能力。

**修正为**:
> 应用启动时可选支持生物识别或密码验证(Face ID / Touch ID / Windows Hello),用于应用级访问控制。身份私钥读取依赖操作系统的设备解锁状态,不需要每次读取时都进行生物识别验证。

**修正理由**:
- 生物识别用于应用级保护,不是私钥读取的必要条件
- 避免频繁验证影响用户体验
- 类似 API Key 的使用模式

## 10. 总结

### 10.1 核心方案

- **身份私钥 = API Key**: 用于身份标识,不涉及资产
- **存储方式**: 本地安全存储,受操作系统保护
- **跨设备**: 直接复制私钥字符串,类似 API Key
- **安全措施**: 本地安全存储 + 用户教育 + 异常检测

### 10.2 与原方案的主要差异

| 维度 | 原方案 (设备密钥对加密) | 新方案 (本地安全存储) |
|------|----------------------|---------------------|
| 复杂度 | 高 | 低 |
| 跨设备 | 需要显式导出/导入 | 直接复制 API Key |
| 生物识别 | 每次读取都需要 | 可选,应用级保护 |
| 硬件依赖 | 依赖 Secure Enclave 等 | 不依赖 |
| 适用场景 | 资产钱包 | 身份标识 |
