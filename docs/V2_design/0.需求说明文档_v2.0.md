## 1. 引言

### 1.1 文档目的

本文档用于将 OpenMesh V2 当前 PRD 重构为符合软件工程规范的软件需求规格说明书，作为后续架构设计、任务拆解、研发实现、测试验收和需求追溯的统一基线。

本文档的主要用途如下：

- 为架构师提供功能边界、外部接口、设计约束和非功能目标
- 为工程师提供可实现、可测试、可追踪的需求定义
- 为测试团队提供验收标准、测试范围和场景依据
- 为项目负责人提供范围控制、优先级管理和变更评估依据

本文档是需求基线文档，不替代技术架构设计文档、开发计划文档和实施细节说明。涉及实现方案时，以“约束”和“接口要求”为主，具体技术实现仍由架构师和工程师继续细化。

### 1.2 项目范围

OpenMesh V2 是一个面向 VPN 流量订阅场景的平台化系统，其目标是让终端用户像使用 OpenRouter 统一接入 AI 模型一样，统一发现、购买和使用不同流量供应商提供的 VPN 服务。

本期范围内包含：

- 可选的身份账号系统，仅在订阅公开供应商服务时启用
- 用户对公开供应商套餐的浏览、购买、续订、取消、升级和降级
- 月度流量额度管理、状态管理、服务暂停与恢复
- 基于 x402 协议和 Base 网络的订阅支付闭环
- 平台为供应商创建专属托管结算智能合约
- 供应商上线费收取、收入提现与平台手续费扣除
- 官方开源流量管理系统和套餐管理系统
- 自建系统供应商基于协议和接口规范接入平台
- V1 私有节点功能完整保留且与账号体系解耦

本期范围外不包含：

- 多链生产部署
- 法币支付能力
- 退款流程
- 强制 KYC
- 可升级智能合约
- 平台替供应商提供长期托管运维服务

### 1.3 定义、缩写和术语

| 术语 | 定义 |
|------|------|
| OpenMesh V2 | 本期目标产品，提供公开供应商订阅、支付、流量控制与统一接入能力 |
| V1 私有节点功能 | 已上线版本中的私有节点创建、导入、分享和访问能力，本期必须保留 |
| 用户 / 订阅者 | 购买并使用 VPN 服务的终端用户 |
| 流量供应商 | 提供 VPN 节点、套餐和流量服务的服务提供方 |
| OpenMesh 平台 | 提供供应商发现、支付协议、结算托管、参考实现和接入规范的平台方 |
| x402 协议 | 本项目用于订阅支付和验证的标准化支付协议 |
| Hysteria2 | 本项目统一的 VPN 数据通道协议 |
| 身份钱包 | 用于生成用户 ID 和签名订阅请求的钱包，不存储资产，不用于支付 |
| 身份私钥 / API Key | 身份钱包对应的私钥，作为用户身份凭证，可用于跨设备导入 |
| 支付钱包 | 用户用于支付订阅费用的外部钱包，与身份钱包分离 |

| USDC | 本期订阅支付和供应商结算使用的稳定币 |
| Fair Use / 月度流量额度 | 套餐在一个计费周期内允许消耗的总流量硬上限 |
| 供应商托管合约 | 平台为每个供应商创建的专属收款与提现智能合约 |
| Minimal Proxy | 基于 EIP-1167 的最小代理模式，用于降低供应商合约部署成本 |
| 宽限期（Grace Period） | 订阅到期后允许用户继续使用服务的 24 小时时间窗口 |
| 订阅状态 | 表示订阅当前是否可用的状态，包括 Active、Grace、Expired、Canceled |
| 本地安全存储 | 操作系统提供的安全存储能力，用于保存身份私钥，不上传服务器 |

## 2. 总体描述

### 2.1 产品视角

OpenMesh V2 在产品层面是对 OpenMesh V1 的扩展，而不是替代。系统需要在保留私有节点“开箱即用”体验的前提下，新增公开供应商订阅、托管结算和统一接入能力。

系统上下文如下：

```text
终端用户
  ├── OpenMesh 客户端（移动端 / 桌面端）
  │     ├── 私有节点功能（V1）
  │     ├── 账号中心（V2）
  │     ├── 订阅管理与流量展示
  │     └── 支付与续订触发
  │
  ├── OpenMesh 平台服务
  │     ├── 供应商发现
  │     ├── 套餐管理系统（官方开源）
  │     ├── 流量管理系统（官方开源）
  │     ├── 订阅状态管理
  │     └── 链上事件监听与同步
  │
  ├── 外部协议与基础设施
  │     ├── x402 支付协议
  │     ├── Hysteria2 VPN 通道
  │     ├── Base 网络 / RPC
  │     ├── 智能合约（供应商托管合约）
  │     └── PostgreSQL
  │
  └── 流量供应商
        ├── 官方系统部署方
        └── 自建系统接入方
```

#### 2.1.1 系统接口

- 与 x402 协议集成，用于首次支付、续订支付和支付验证
- 与供应商托管结算智能合约集成，用于订阅收款、余额查询和提现
- 与 Hysteria2 集成，用于 VPN 流量接入、断线控制和服务准入
- 与事件监听服务集成，用于链上事件同步到业务数据库

#### 2.1.2 用户接口

- 移动端客户端：支持身份账号创建、订阅购买、用量查看、自动续订控制
- 桌面端客户端：支持 Windows 和 macOS，支持 API Key 导入、导出、二维码展示、订阅查看、支付和连接状态提示
- 供应商管理界面：支持套餐配置、收入查看和提现操作

#### 2.1.3 硬件接口

- iOS Keychain
- Android Keystore（通过 EncryptedSharedPreferences 使用）
- macOS Keychain
- Windows Credential Manager / DPAPI
- Face ID / Touch ID / Android 生物识别能力 / Windows Hello（可选应用级访问控制）

#### 2.1.4 软件接口

- Base RPC 服务
- PostgreSQL 数据库，用于持久化供应商、订阅、账单、流量统计和链上事件同步结果
- 本期不要求 Redis 或其他额外缓存中间件；按月续订和 24 小时宽限的业务复杂度下，简单的数据库加应用层方案足以满足需求
- OpenZeppelin 合约库
- EncryptedSharedPreferences
- Windows Credential Manager / DPAPI

#### 2.1.5 通信接口

- HTTP / HTTPS：客户端与平台 API 通信
- WebSocket：订阅状态、支付状态和用量更新
- UDP：Hysteria2 数据通道

### 2.2 产品功能概述

OpenMesh V2 的核心功能由以下几部分组成：

1. 可选账号系统：为公开订阅场景提供身份钱包账号、API Key 导入导出和跨设备迁移能力。
2. 供应商发现与套餐订阅：让用户浏览供应商、比较套餐并完成购买。
3. 订阅生命周期管理：维护状态、宽限期、续订、取消、升级和降级。
4. 月度流量控制：对硬性流量额度进行统计、提醒、断线和恢复控制。
5. 托管结算：为每个供应商创建专属合约并管理收入提现。
6. 官方参考实现：提供开源流量管理系统和套餐管理系统。
7. 自建系统接入：允许供应商在遵守协议时保留自有后端。
8. 向后兼容：保持 V1 私有节点能力完整可用。

### 2.3 用户特征

| 用户类型 | 特征 | 关键关注点 |
|------|------|------|
| 终端用户（订阅者） | 关注购买流程简洁、连接稳定、用量透明、续订不中断 | 钱包安全、支付成功率、连接体验、清晰提示 |
| 流量供应商 | 关注接入成本、套餐管理灵活性、收入透明度和提现效率 | 上线门槛、协议兼容、财务清晰、部署自由度 |
| 平台管理员 | 关注系统稳定性、结算闭环、供应商准入和风险控制 | 合规风险、事件同步、异常监控、收入管理 |

### 2.4 约束条件

| 编号 | 约束 | 说明 |
|------|------|------|
| C-001 | Base 单链约束 | 本期仅支持 Base 网络 |
| C-002 | EVM 兼容约束 | 身份标识采用 EVM 地址，支付能力应兼容 EVM 生态 |
| C-003 | 协议约束 | VPN 统一接入协议为 Hysteria2 |
| C-004 | 合约约束 | 智能合约部署后不可升级 |
| C-005 | 退款约束 | 本期不支持退款 |
| C-006 | 费用约束 | 供应商上线费 100 USDC，提现手续费 10% |
| C-007 | 部署约束 | 官方系统以开源代码方式交付，供应商自行部署 |
| C-008 | 兼容约束 | V2 不得破坏 V1 私有节点能力 |
| C-009 | 安全约束 | 用户私钥不得上传服务器 |
| C-010 | 身份与支付分离约束 | 身份钱包不持有资产，不直接承担订阅支付功能 |

### 2.5 假设和依赖

| 编号 | 假设 / 依赖 | 说明 |
|------|------|------|
| A-001 | 用户具备可用的 EVM 钱包能力 | 公开订阅场景依赖钱包签名和支付 |
| A-002 | Base 网络稳定可用 | 依赖链上支付确认、事件监听和合约调用 |
| A-003 | x402 协议可持续使用 | 依赖支付协议可用性与版本稳定性 |
| A-004 | Hysteria2 协议满足流量管理需求 | 依赖协议实现断线、准入和节点接入规范 |
| A-005 | 供应商遵守接入规范 | 供应商需正确实现协议接口和流量转发规范 |
| A-006 | 第三方 RPC 服务稳定 | 依赖多 RPC 提供商保障读写可用性 |
| A-007 | 终端设备具备安全存储能力 | iOS、Android、macOS、Windows 设备均需具备本地安全存储能力 |

## 3. 具体需求

### 3.1 功能需求

#### 3.1.1 账号与身份能力

| 需求 ID | 需求名称 | 描述 | 优先级 | 验收标准 ID |
|------|------|------|------|------|
| FR-001 | 可选账号与 V1 解耦 | 系统应允许用户在不创建账号的情况下继续使用全部 V1 私有节点功能；仅在订阅公开供应商服务时要求创建账号。 | P0 | AC-001 |
| FR-002 | 创建账号时自动生成身份钱包 | 系统应在用户选择创建账号时自动生成新的 EVM 身份钱包；钱包地址展示为用户 ID，身份私钥作为 API Key 存储到设备本地安全存储中。 | P0 | AC-002 |
| FR-003 | 身份私钥导入 | 系统仅支持通过身份私钥导入已有账号，不支持助记词备份、助记词恢复或助记词导入。 | P0 | AC-003 |
| FR-004 | 本地安全存储 | 系统应在 iOS 使用 Keychain、在 Android 使用 EncryptedSharedPreferences、在 macOS 使用 Keychain、在 Windows 使用 Credential Manager 或 DPAPI 存储身份私钥，受操作系统级访问控制保护。 | P0 | AC-004 |
| FR-005 | API Key 显式导出与跨设备导入 | 系统应支持用户在已登录设备上显式展示身份私钥，将其展示为 `0x` 开头的十六进制字符串，并提供复制按钮、二维码展示按钮和高风险红色提示元素；其他设备仅支持通过该 API Key 字符串或二维码完成导入。 | P1 | AC-005 |
| FR-006 | 账号中心独立界面 | 系统应将账号、订阅、余额等能力置于独立界面，不干扰私有节点默认入口。 | P0 | AC-006 |

#### 3.1.2 供应商发现与订阅购买

| 需求 ID | 需求名称 | 描述 | 优先级 | 验收标准 ID |
|------|------|------|------|------|
| FR-007 | 供应商发现 | 系统应向用户展示可订阅供应商及其套餐、价格、流量额度和节点覆盖范围。 | P0 | AC-007 |
| FR-008 | 首次订阅购买 | 用户应能在不超过 3 个主要操作步骤内完成套餐选择、身份确认和订阅支付。 | P0 | AC-008 |
| FR-009 | 重复购买限制 | 同一用户对同一供应商在同一有效周期内不得重复购买同一等级套餐。 | P0 | AC-009 |
| FR-010 | 多供应商并行订阅 | 用户应可同时向多个不同供应商订阅，且各订阅相互独立。 | P0 | AC-010 |
| FR-011 | 套餐定义与展示 | 供应商应可定义多等级套餐，并为每个等级配置价格、月度流量额度、节点覆盖范围和套餐名称。 | P0 | AC-011 |
| FR-012 | 套餐上下架 | 供应商应能上架或下架套餐；套餐下架不得影响已订阅用户当前周期服务。 | P0 | AC-012 |

#### 3.1.3 流量管理与订阅生命周期

| 需求 ID | 需求名称 | 描述 | 优先级 | 验收标准 ID |
|------|------|------|------|------|
| FR-013 | 月度流量统计 | 系统应按订阅周期统计用户总流量使用量，并向客户端提供额度、已用量、剩余量和重置时间。 | P0 | AC-013 |
| FR-014 | 80% 用量提醒 | 当用户流量达到套餐额度 80% 时，系统应触发明确提醒。 | P0 | AC-014 |
| FR-015 | 超额断线 | 当用户流量超过套餐额度时，系统应立即断开所有设备连接并拒绝新的连接请求。 | P0 | AC-015 |
| FR-016 | 订阅状态管理 | 系统应维护 Active、Grace、Expired、Canceled 四种订阅状态，并依据状态决定是否允许 VPN 使用。 | P0 | AC-016 |
| FR-017 | 自动续订 | 系统应在到期前 3 天开始尝试自动续订。自动续订通过以下方式实现：主方案为 Coinbase Commerce 订阅授权，由 Coinbase 自动处理扣款；备选方案为 EIP-3009 智能合约授权，由合约自动执行扣款。 | P0 | AC-017 |
| FR-018 | 续订失败与宽限期 | 当自动续订失败时，系统应立即通知用户并进入 24 小时宽限期。 | P0 | AC-018 |
| FR-019 | 套餐升级 | 用户升级到更高等级套餐时，系统应按剩余天数收取差价并立即生效。 | P1 | AC-019 |
| FR-020 | 套餐降级 | 用户降级到更低等级套餐时，系统应在下一个续订周期生效。 | P2 | AC-020 |
| FR-021 | 订阅管理 | 用户应可查看活跃订阅、自动续订状态、历史支付记录，并执行续订、取消和升级操作。 | P1 | AC-021 |
| FR-022 | 服务暂停与恢复 | 系统应在订阅过期或流量超额时暂停服务，在续订成功后自动恢复服务。 | P0 | AC-022 |

#### 3.1.5 官方系统与接入模式

| 需求 ID | 需求名称 | 描述 | 优先级 | 验收标准 ID |
|------|------|------|------|------|

### 附录 B：用户场景

#### 场景 1：新用户首次订阅

1. 用户打开客户端并创建新身份账号或导入已有 API Key。
2. 用户浏览供应商列表和套餐等级，选择目标套餐。
3. 用户确认费用并使用支付钱包完成支付。
4. 系统提示订阅成功、显示到期时间与流量额度。
5. VPN 服务立即可用。

#### 场景 2：流量达到 80% 后收到提醒

1. 用户在多台设备使用 VPN。
2. 本月流量达到套餐额度 80%。
3. 客户端推送提醒通知。
4. 若继续使用至超额，连接立即断开并提示恢复方式。

#### 场景 3：自动续订成功

1. 订阅到期前 3 天开始续订尝试。
2. 支付钱包余额充足且自动续订授权有效。
3. 系统完成自动续订。
4. 用户收到续订成功通知，服务无中断。

#### 场景 4：自动续订失败后在宽限期补救

1. 自动续订因支付钱包余额不足失败。
2. 用户收到失败原因和宽限截止时间通知。
3. 用户补足余额并手动续订。
4. 新周期从原到期时间起算，服务不中断。

#### 场景 5：宽限期结束后服务暂停并恢复

1. 宽限期结束仍未续订。
2. 系统断开 VPN 并提示订阅已过期。
3. 用户续订成功后，服务在 1 分钟内恢复。

#### 场景 6：切换供应商

1. 用户当前订阅供应商 A。
2. 用户发现供应商 B 并购买新套餐。
3. 新订阅生效，用户可选择是否保留旧订阅。
4. 用户侧无需感知供应商后端是否使用官方系统。

#### 场景 7：供应商使用官方系统上架套餐

1. 新供应商支付 100 USDC 上线费。
2. 平台创建专属托管合约。
3. 供应商部署官方系统并配置套餐。
4. 用户端可发现该供应商并进行订阅。

#### 场景 8：供应商使用自有系统接入

1. 供应商支付上线费并获得合约。
2. 供应商在自建系统中实现 x402 接口和流量规范。
3. 平台完成注册后对外展示该供应商。
4. 用户可正常订阅和使用其服务。

#### 场景 9：供应商提现收入

1. 供应商查看合约余额和收入明细。
2. 供应商发起提现。
3. 系统展示提现金额、平台手续费和实收金额。
4. 提现完成后保留记录。

#### 场景 10：从已登录设备导出 API Key 并导入另一台设备

1. 用户在已登录设备中进入账号管理界面，点击“导出 API Key”。
2. 系统可选要求用户完成本机安全验证。
3. 系统从本地安全存储中读取身份私钥，并展示为 `0x` 开头的十六进制字符串。
4. 界面同时提供“复制”按钮、“展示二维码”按钮，以及红色高风险提示。
5. 用户在另一台设备上通过粘贴私钥字符串或扫描二维码完成导入。
6. 新设备完成地址校验后登录成功。

#### 场景 11：无账号使用私有节点

1. 用户打开 OpenMesh。
2. 用户导入朋友分享的私有节点配置。
3. 用户无需创建账号即可直接使用。

#### 场景 12：从私有节点升级到公开订阅

1. 用户长期使用 V1 私有节点能力。
2. 用户浏览公开供应商并点击订阅。
3. 系统此时引导创建身份账号。
4. 用户完成支付后，可同时使用私有节点和公开订阅。

### 附录 E：风险与缓解

| 风险 | 说明 | 缓解措施 |
|------|------|------|
| 智能合约安全风险 | 合约一旦上线将承载真实资金 | 采用内部审计、测试覆盖和 Base Sepolia 验证 |
| 合约不可升级风险 | 严重漏洞无法原地修复 | 限制初期供应商数量并建立紧急响应与重新部署方案 |
| 链上 Gas 波动 | 网络拥堵可能影响成本和体验 | 选择 Base 并监控 Gas 波动 |
| 供应商服务欺诈风险 | 收款后服务质量可能不达标 | 建立信誉机制和取消订阅能力 |
| 托管合规风险 | 资金托管模式可能触发合规要求 | 由法务持续评估并约束上线范围 |
| 自建系统质量参差 | 第三方实现可能偏离协议 | 提供 SDK、测试工具和一致性校验 |
| 用户不理解流量硬上限 | 可能认为订阅等于无限流量 | 在订购前显式展示额度与超额断线规则 |
| 余额不足导致续订失败 | 用户可能未维持余额 | 提前提醒余额不足并提供手动续订入口 |
| 自动续订授权失效 | Coinbase Commerce 授权或智能合约授权可能过期或失效 | 失败提示中明确要求重新授权，并提供清晰的授权流程指引 |
| 链上确认延迟 | 交易确认时间会影响即时体验 | 采用乐观激活与事后链上审计结合策略 |
| RPC 故障 | 第三方 RPC 不稳定会影响支付和同步 | 使用多 RPC 提供商和自动切换机制 |
| 私钥丢失风险 | 用户未提前导出私钥且设备损坏或丢失时，账号将无法恢复 | 在导出私钥入口中明确恢复风险，并要求用户自行承担保管责任 |
| 私钥泄露风险 | 用户主动展示、复制或分享私钥后，第三方可消耗该账号下的流量 | 在导出界面显示红色高风险提示，并要求显式确认后才展示私钥或二维码 |
| V2 影响 V1 风险 | 新功能可能破坏已上线私有节点能力 | 彻底解耦账号中心和私有节点模块，并做回归测试 |
