# 基于区块链地址的订阅式 VPN 服务 - 技术方案与开发计划

## 一、系统总体需求

### 1.1 业务需求

构建一个基于区块链地址认证的订阅式 VPN 服务系统,用户通过区块链支付订阅费用后,可以使用集成了 Sing-box 的客户端访问 Xray 服务器进行 VPN 转发。系统需要支持:

- **用户身份管理**: 使用区块链地址替代传统 email 作为用户唯一标识
- **订阅付费**: 用户支付订阅费后自动获得访问权限,支持自动续费
- **动态用户管理**: 实时启用/禁用用户,无需重启服务
- **流量统计**: 实时记录每个用户在每个服务器的流量使用情况
- **多服务器支持**: 单个管理服务可管理多个 Xray 服务器
- **客户端接入**: 提供易用的客户端,支持多平台

### 1.2 技术需求

- **高可用性**: 服务器故障不影响其他服务器,支持故障转移
- **实时性**: 用户状态变更实时生效,流量统计实时更新
- **安全性**: 通信加密,防止滥用,保护用户隐私
- **可扩展性**: 支持水平扩展,可添加更多服务器和功能模块
- **可维护性**: 模块化设计,便于独立开发、测试和部署

## 四、模块 1: Xray 流量统计模块 (已验证)

### 4.1 模块需求描述

#### 4.1.1 功能需求

1. **用户级别流量统计**: 实时记录每个用户的上行和下行流量
2. **多服务器支持**: 支持同时统计多个 Xray 服务器的流量
3. **流量汇总**: 自动汇总用户在所有服务器的总流量
4. **流量持久化**: 定期保存流量数据到文件,防止 Xray 重启后数据丢失
5. **重启检测**: 自动检测 Xray 重启,保留历史流量数据
6. **实时推送**: 通过 WebSocket 向客户端实时推送流量更新
7. **动态用户管理**: 支持动态添加/删除用户,无需重启 Xray

#### 4.1.2 非功能需求

1. **性能**: 流量统计不影响 VPN 转发性能,查询延迟 < 100ms
2. **可靠性**: 数据持久化间隔 5 秒,确保数据不丢失
3. **准确性**: 流量统计误差 < 1%
4. **可扩展性**: 支持 10+ Xray 服务器,1000+ 并发用户

### 4.2 技术方案描述

#### 4.2.1 核心技术选型

- **Xray Stats API**: 使用 Xray 内置的 StatsService 进行流量统计
- **xtlsapi 库**: 使用 [hiddify/xtlsapi](https://github.com/hiddify/xtlsapi) Python 库调用 Xray gRPC API
- **Go HTTP 服务**: 使用 Go 开发订阅管理服务,负责流量统计和持久化
- **WebSocket**: 使用 Gorilla WebSocket 库实现实时推送

#### 4.2.2 关键技术方案

**1. 流量持久化机制**

问题: Xray 的流量统计存储在内存中,重启后数据丢失

解决方案: 定期查询并持久化流量数据

实现原理:
```
1. 定时任务每 5 秒查询所有 Xray 服务器的流量
2. 计算增量: 当前流量 - 上次记录的流量
3. 检测重启: 如果当前流量 < 上次流量,说明 Xray 重启了
4. 累加到历史总流量并保存到 users.json
```

**2. 多服务器流量汇总**

场景: 用户可以连接到不同地区的多个 Xray 服务器

实现方案:
- 为每个用户维护一个 `servers` 字段,记录在每个服务器的流量
- 定期查询所有服务器,计算每个服务器的增量
- 汇总所有服务器的流量到 `total_uplink` 和 `total_downlink`

**3. WebSocket 实时推送**

问题: 传统轮询方式效率低,延迟高,服务器压力大

解决方案: 使用 WebSocket 实时推送流量更新

实现方案:
- HTTP 服务维护所有活跃的 WebSocket 连接
- 流量统计模块每次更新后触发推送事件
- WebSocket 服务器广播到所有连接的客户端

**4. 动态用户管理**

问题: Xray 的 `xray api adu` 命令行工具不工作

解决方案: 使用 xtlsapi 库直接调用 Xray 的 gRPC API

实现方案:
- 添加用户: `XrayClient.add_client()` → gRPC `HandlerService.AlterInbound`
- 删除用户: `xray api rmu` 命令 → gRPC `HandlerService.AlterInbound`
- 查询用户: `xray api inbounduser` 命令 → gRPC `HandlerService.GetInboundUsers`

  ```go
  if currentUplink < lastXrayUplink {

      // 正常情况,累加增量
      delta := currentUplink - lastXrayUplink
      totalUplink += delta
      lastXrayUplink = currentUplink
  }
  ```

## 五、模块 2: 区块链支付模块 (待验证)

### 5.1 模块需求描述

#### 5.1.1 功能需求

1. **支付接收**: 接收用户的区块链支付交易
2. **交易验证**: 验证支付交易的真实性和有效性
3. **订阅激活**: 支付成功后自动激活用户订阅
4. **自动续费**: 订阅到期前自动从用户钱包扣款
5. **支付回调**: 提供支付回调接口供区块链网络调用
6. **退款处理**: 支持订阅取消后的退款流程

#### 5.1.2 非功能需求

1. **安全性**: 防止支付欺诈,验证交易签名
2. **可靠性**: 支付处理失败自动重试
3. **幂等性**: 重复的支付回调不会重复激活订阅
4. **实时性**: 支付确认后 30 秒内激活订阅

### 5.2 技术方案描述

#### 5.2.1 核心技术选型

- **区块链网络**: 支持 Ethereum、Polygon 等 EVM 兼容链
- **智能合约**: 使用 Solidity 开发订阅支付合约
- **Web3 库**: 使用 ethers.js 或 web3.js 与区块链交互
- **支付网关**: 可选集成第三方支付网关(如 Coinbase Commerce)

#### 5.2.2 关键技术方案

**1. 智能合约设计**

功能:
- 接收用户支付
- 记录订阅信息(用户地址、订阅时长、金额)
- 触发支付事件,通知后端服务
- 支持自动续费授权

**2. 支付验证流程**

```
1. 用户发起支付交易 → 智能合约
2. 智能合约触发 PaymentReceived 事件
3. 后端监听事件 → 获取交易哈希
4. 验证交易: 检查金额、发送者、接收者
5. 验证通过 → 激活订阅
```

**3. 自动续费机制**

方案 A: 用户授权智能合约自动扣款
- 用户一次性授权合约从钱包扣款
- 到期时合约自动执行扣款
- 扣款成功自动延长订阅

方案 B: 定时检查余额并请求支付
- 到期前发送续费通知
- 用户手动确认支付
- 支付成功后延长订阅

### 6.1 模块需求描述

#### 6.1.2 非功能需求

2. **一致性**: 数据库订阅状态与 Xray 服务器状态保持一致

4. **可扩展性**: 支持水平扩展,多实例部署

### 6.2 技术方案描述

#### 6.2.1 核心技术选型

- **Web 框架**: Go + Gin 框架
- **数据库**: PostgreSQL (支持事务和复杂查询)
- **缓存**: Redis (缓存用户状态,减少数据库查询)
- **定时任务**: Go cron 库
- **消息队列**: 可选 RabbitMQ/Kafka (异步处理订阅变更)

#### 6.2.2 关键技术方案

**1. 订阅状态机**

状态定义:
- `pending`: 等待支付确认
- `active`: 订阅有效,用户可访问
- `expiring`: 即将到期(24 小时内)
- `expired`: 订阅已过期,用户无法访问
- `suspended`: 订阅暂停(如违规)
- `cancelled`: 订阅已取消

状态转换:
```
pending → (支付成功) → active
active → (接近到期) → expiring
expiring → (续费成功) → active
expiring → (续费失败) → expired
active → (手动暂停) → suspended
suspended → (恢复) → active
```

**2. 订阅与 Xray 状态同步**

问题: 数据库订阅状态与 Xray 服务器用户列表可能不一致

解决方案: 双向同步机制
- 订阅变更时立即同步到 Xray
- 定期(每 6 小时)从 Xray 查询用户列表并对比
- 发现不一致时自动修复

**3. 定时任务调度**

任务类型:
- 续费检查任务(每小时): 查询即将到期的订阅,触发续费
- 过期清理任务(每天): 查询已过期订阅,从 Xray 删除用户
- 状态同步任务(每 6 小时): 同步数据库与 Xray 状态
- 流量统计任务(每 5 秒): 查询并更新用户流量(已在模块 1 实现)

**4. 多服务器用户管理**

策略选择:
- **广播模式**: 用户订阅后添加到所有 Xray 服务器
- **分配模式**: 根据地理位置或负载分配服务器
- **混合模式**: 高级用户访问所有服务器,普通用户访问指定服务器

实现方案:
```go
// 添加用户到所有服务器
func (s *SubscriptionService) ActivateUser(user *User) error {
    for _, server := range s.xrayServers {
        if err := server.AddUser(user); err != nil {
            // 记录失败,继续处理其他服务器
            log.Error("Failed to add user to server", server.Name, err)
        }
    }
    return nil
}
```

  ```go
      expiring := s.db.FindExpiringSubscriptions(24 * time.Hour)
      
      for _, sub := range expiring {
          // 触发续费流程
          if err := s.paymentService.AutoRenew(sub); err != nil {
              // 续费失败,标记为过期
  ```

```
3. 用户在钱包中确认连接
4. 获取用户区块链地址

```

支付流程:
```
1. 用户选择订阅套餐
2. 调用智能合约的 subscribe() 方法
3. 用户在钱包中确认交易
4. 等待交易确认
5. 轮询后端 API 查询订阅状态
6. 订阅激活后自动连接 VPN
```

- **关键代码**:
  ```typescript
  class SingboxManager {
    private process: ChildProcess | null = null;
    
    async start(config: SingboxConfig): Promise<void> {
      const configPath = await this.writeConfig(config);
      this.process = spawn('./bin/sing-box', ['run', '-c', configPath]);
      
      this.process.stdout?.on('data', (data) => {
        this.handleLog(data.toString());
      });
      
      this.process.on('exit', (code) => {
        this.handleExit(code);
      });
    }
    
    async stop(): Promise<void> {
      if (this.process) {
        this.process.kill();
        this.process = null;
      }
    }
  }
  ```

#### 任务 2: 钱包集成

- **关键代码**:
  ```typescript
  class WalletService {
    private provider: ethers.BrowserProvider | null = null;
    
    async connect(): Promise<string> {
      if (window.ethereum) {
        this.provider = new ethers.BrowserProvider(window.ethereum);
        const accounts = await this.provider.send('eth_requestAccounts', []);
        return accounts[0];
      }
      throw new Error('No wallet found');
    }
    
    async subscribe(plan: string, amount: string): Promise<string> {
      const signer = await this.provider!.getSigner();
      const contract = new ethers.Contract(CONTRACT_ADDRESS, ABI, signer);
      const tx = await contract.subscribe(plan, { value: amount });
      await tx.wait();
      return tx.hash;
    }
  }
  ```

  ```typescript
    async fetchStatus(address: string): Promise<void> {
      const response = await fetch(`${API_BASE}/api/users/${address}/status`);
      const data = await response.json();
      this.subscription = data.subscription;
    }

      const response = await fetch(`${API_BASE}/api/subscriptions`, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ user_address: address, payment_tx_hash: txHash })
      });
      const data = await response.json();
      this.subscription = data;
    }
  }
  ```

  ```typescript
  class TrafficService {
    private ws: WebSocket | null = null;
    
    connect(address: string): void {
      this.ws = new WebSocket(`${WS_BASE}/ws`);
      
      this.ws.onmessage = (event) => {
        const data = JSON.parse(event.data);
        if (data.type === 'traffic_update') {
          const userTraffic = data.users.find(u => u.email === address);
          if (userTraffic) {
            this.updateTraffic(userTraffic);
          }
        }
      };
    }
    
    disconnect(): void {
      if (this.ws) {
        this.ws.close();
        this.ws = null;
      }
    }
  }
  ```

## 八、附录

### 8.1 安全考虑

#### 8.1.1 认证安全
- UUID 唯一性: 确保每个区块链地址对应唯一的 UUID
- UUID 保密: UUID 不应公开,只在客户端配置中使用
- 区块链验证: 验证支付交易的真实性和有效性

#### 8.1.2 通信安全
- VLESS 加密: Sing-box 和 Xray 之间使用 VLESS 协议加密通信
- gRPC TLS: HTTP 服务和 Xray 之间的 gRPC 通信使用 TLS 加密
- API 认证: HTTP 服务的管理接口需要认证

#### 8.1.3 防滥用
- 流量限制: 对每个用户设置流量配额
- 连接限制: 限制单个用户的并发连接数
- 异常检测: 监控异常流量模式,自动暂停可疑用户

### 8.2 监控与运维

#### 8.2.1 监控指标
- 用户指标: 活跃用户数、新增用户数、流失用户数
- 订阅指标: 订阅转化率、续费率、平均订阅时长
- 服务指标: Xray 服务器负载、连接数、流量统计
- 支付指标: 支付成功率、自动续费成功率

#### 8.2.2 日志记录
- 操作日志: 记录所有用户添加/删除操作
- 支付日志: 记录所有支付回调和处理结果
- 错误日志: 记录所有错误和异常情况
- 审计日志: 记录管理员操作

#### 8.2.3 故障恢复
- 状态同步: 定期同步数据库和 Xray 的用户状态
- 自动修复: 检测到不一致时自动修复
- 手动干预: 提供管理接口用于手动修复异常

### 8.3 部署建议

#### 8.3.1 服务器配置
- HTTP 服务: 2 核 4GB 内存,可处理 10000+ 用户
- Xray 服务器: 4 核 8GB 内存,可支持 1000+ 并发连接
- 数据库: PostgreSQL 或 MySQL,主从复制

#### 8.3.2 网络配置
- Xray 端口: 10086/10087 (VLESS)、10085/10088 (gRPC API)
- HTTP 服务端口: 8080 (API)、443 (HTTPS)
- 防火墙: 只开放必要端口,限制 gRPC API 访问

#### 8.3.3 高可用
- HTTP 服务: 多实例部署,使用负载均衡
- Xray 服务器: 多服务器部署,客户端支持故障转移
- 数据库: 主从复制,定期备份
